🔥 Architecture Design Questions

Format: Practice these until you can answer confidently in 60-90 seconds. Every answer follows the Situation → Approach → Trade-offs structure.

Q1: "Design a Kafka-based system for an e-commerce platform processing 100K orders/hour with real-time inventory updates."

"I'd create three topics: order-events (12 partitions — 100K orders/hour ≈ 28/sec, room for 12 parallel consumers), inventory-updates (compacted, key=warehouse-sku, 20 partitions), and payment-results (6 partitions). All with RF=3, min.insync.replicas=2, acks=all.

Partition key for orders is order_id for lifecycle event ordering. Inventory uses compaction so consumers always see latest stock level. Schema Registry with Avro and BACKWARD compatibility.

Three consumer groups: order processing (EOS with transactions), inventory service (reads compacted topic), and analytics (Flink). Trade-off: EOS transaction complexity, but data integrity justifies it."

Q2: "You have a clickstream of 50K events/second. Design the Kafka topic configuration."

"50K events/sec × 500-byte avg = 25 MB/s. Each partition ~10 MB/s, so 3 minimum — but I'd provision 40 partitions for consumer parallelism and growth.

Config: acks: 1 (clickstream tolerates some loss), retention.ms: 604800000 (7 days), compression.type: lz4 (~60% reduction), batch.size: 65536 and linger.ms: 10 for producer batching.

Key = user_id for per-user ordering. If ordering isn't needed, use null keys with sticky partitioning. Trade-off: 40 partitions increases rebalancing time but throughput benefits outweigh this."

Q3: "Design a partition key strategy for a multi-tenant SaaS platform."

"The naive approach — tenant_id — creates hot partitions. If Walmart has 10M events/day and a small business has 100, Walmart's partition is 100,000x hotter.

Option A: Composite key — tenant_id + entity_id. Distributes large tenants across partitions while maintaining per-entity ordering.

Option B: Dedicated topics per tier — Premium tenants get their own topic (SLA isolation). Standard tenants share a multi-tenant topic.

Option C: Custom partitioner — Hash on entity_id for large tenants, tenant_id for small ones.

I'd choose Option A — simple, distributes load, maintains per-entity ordering."

Q4: "How would you migrate from a monolithic Kafka topic to a microservices event architecture?"

"Strangler fig pattern: (1) Shadow topic — Create domain-specific topics, deploy a 'splitter' consumer that routes events by type. (2) Dual write — New services produce to both monolith and domain topic. (3) Consumer migration — Migrate consumers one-by-one, each independently reversible. (4) Decommission — Stop dual write once all consumers migrated.

Key: Register schemas in advance with RecordNameStrategy. Each phase should be independently deployable and rollbackable."

Q5: "Design a system where Kafka consumers need strict global order."

"Option A: Single partition — Guarantees global order but limits throughput to one consumer. Viable only for <1K events/sec.

Option B: Sequential processing layer — Multiple partitions for transport, then a downstream service buffers and reorders by sequence number. Scalable but complex.

Option C: Challenge the requirement — Most 'global ordering' requirements are actually 'per-entity ordering.' Does order-123's events really need ordering relative to order-456? Usually no. Use entity ID as partition key.

I'd always challenge the requirement first. In 15 years I've seen exactly two legitimate global ordering needs."

Q6: "Design Kafka-based event sourcing for an order management system."

"Event store: order-events topic, compacted + 90-day retention, key = order_id. Events: OrderCreated, ItemAdded, PaymentReceived, Shipped.

Materialized views: Consumer groups rebuild state — one CG → PostgreSQL, another → Elasticsearch, another → Redis.

Snapshots: For orders with 50+ events, publish snapshots to order-snapshots compacted topic. On startup, read snapshot first, then replay.

Schema evolution: Schema Registry with FULL_TRANSITIVE compatibility — every version must be cross-readable since events live forever."

Q7: "How would you handle Kafka across multiple data centers?"

"Active-Passive: One primary, one DR. MirrorMaker 2 for async replication. RPO = seconds. Challenge: offset translation.

Active-Active: Both accept writes. MM2 replicates bidirectionally with topic prefixing. Challenge: deduplication, conflict resolution.

Stretch cluster: Single cluster across DCs with rack-aware replication. Zero RPO but requires <10ms inter-DC latency.

For most cases: Active-Passive for simplicity. Test failover monthly."

🔍 Debugging Scenarios

These are the "tell me about a time..." questions. Show your debugging instincts.

S1: "Consumer lag growing on 3 of 12 partitions. Others are fine."

📛 Diagnosis: Hot partitions — disproportionate data on 3 partitions (hot keys) or one consumer is slower (GC, resource-constrained host).
✅ Fix: If hot keys → custom partitioner or add random suffix to hot keys. If consumer issue → check the specific host's CPU, memory, GC logs.

S2: "After deployment, consumers rebalancing every 60 seconds in a loop."

📛 Diagnosis: Rebalancing storm — consumers evicted faster than they can rejoin. max.poll.interval.ms too low, or session.timeout.ms too low.
✅ Fix: Increase max.poll.interval.ms to 300000 (5 min). Reduce max.poll.records. Use group.instance.id for static membership during rolling deploys.

S3: "We lost messages in production. acks=1, replication.factor=3."

📛 Diagnosis: With acks=1, only leader acknowledges. Leader dies before followers replicate → messages lost. Classic unacknowledged replication gap.
✅ Fix: Change to acks=all + min.insync.replicas=2. Trade-off: ~3-5ms additional latency per produce.

S4: "Consumer group shows 'EMPTY' state but application pods are running."

📛 Diagnosis: Consumers aren't connecting — wrong bootstrap.servers, wrong group.id, or SASL/SSL misconfiguration.
✅ Fix: Verify connection config. Common cause: environment variable overrides — production pods reading staging Kafka config.

S5: "Debezium CDC connector: 'WAL segment already removed' error."

📛 Diagnosis: PostgreSQL removed WAL segments Debezium hasn't consumed. Connector fell behind and replication slot's WAL was recycled.
✅ Fix: Increase max_wal_size and wal_keep_size. For recovery: recreate connector with snapshot.mode=initial. Monitor replication slot lag.

S6: "Producer getting 'NOT_ENOUGH_REPLICAS' intermittently."

📛 Diagnosis: acks=all with min.insync.replicas=2 but ISR has shrunk below minimum. A broker is down, disk full, or network partition.
✅ Fix: Identify unhealthy broker and restore. If disk full, add disk or lower retention. Temporary (dangerous): lower min.insync.replicas=1 but this sacrifices durability.

S7: "Offset commits failing with 'CommitFailedException'."

📛 Diagnosis: Consumer was evicted (processing took longer than max.poll.interval.ms), partitions reassigned. Old consumer tries to commit for partitions it no longer owns → duplicates!
✅ Fix: Increase max.poll.interval.ms or reduce max.poll.records. Implement ConsumerRebalanceListener to detect partition revocation.

S8: "Kafka cluster disk usage growing faster than expected despite retention.ms."

📛 Diagnosis: Log compaction backlog, abandoned consumer groups (__consumer_offsets growth), transaction markers, or retention.bytes not set.
✅ Fix: Check per-topic disk with kafka-log-dirs.sh. Delete abandoned consumer groups. Increase log.cleaner.threads. Set both retention.ms AND retention.bytes.

🎯 Exactly-Once & Ordering

Q1: "Can you have exactly-once without Kafka transactions?"

"Yes — if your sink is idempotent. Writing to a database with an upsert using the Kafka record's offset or business key as dedup key. The consumer may process twice (at-least-once), but the idempotent sink ensures the effect is applied only once. Many teams achieve 'effectively exactly-once' this way without transaction overhead."

Q2: "Messages ordered by customer_id, 12 partitions, some customers 100x more traffic. How to handle hot partition?"

"Approach 1: Sub-key partitioning — Use customer_id + order_id as key. Spreads load but loses per-customer ordering.

Approach 2: Separate topic for hot customers — Route VIP customers (top 1%) to a dedicated topic.

Approach 3: Local ordering buffer — Keep customer_id as key, scale the consumer vertically for the hot partition.

The right answer depends on whether per-customer ordering is a hard business requirement."

Q3: "Explain 'exactly-once' within Kafka vs end-to-end exactly-once."

"Within Kafka: Idempotent producers + transactions guarantee consume-transform-produce pipelines process each record exactly once via producer IDs and sequence numbers.

End-to-end: Sink must participate. (1) 2PC: Flink pre-commits during checkpoint, full commit on completion. (2) Idempotent sink: Upsert with dedup key. Works with any upsert-capable store.

Hard truth: true E2E exactly-once requires BOTH source AND sink to cooperate. Non-transactional REST API? Best you get is at-least-once with client-side dedup."

Q4: "Kafka Streams app writes to DB. How to prevent duplicates on crash?"

"The changelog + offset pattern: Store Kafka consumer offset alongside business data in the same DB transaction.

BEGIN TRANSACTION; INSERT INTO orders (order_id, ...) VALUES (...); UPDATE consumer_offsets SET offset = 42 WHERE topic = 'order-events' AND partition = 3; COMMIT;

On restart, read stored offset from DB and seek consumer there. DB becomes source of truth for consumption progress."

Q5: "max.in.flight.requests.per.connection=5 with idempotence — how does Kafka maintain ordering?"

"With idempotence, the broker tracks sequence numbers per producer per partition. Even with 5 in-flight requests, if request 3 arrives before request 2, the broker buffers request 3 and waits. Once request 2 arrives and is written, request 3 follows in order. Without idempotence, max.in.flight > 1 can cause ordering issues on retries. With idempotence, max.in.flight=5 is safe AND gives better throughput."

Q6: "A consumer reads from a compacted topic. Can it see duplicates for the same key?"

"Yes! Compaction is asynchronous. Between runs, the topic contains multiple records per key. Also, the active segment (currently written to) is never compacted. Consumers should always treat compacted topic records as upserts — apply the latest value per key and handle duplicates gracefully."

⚡ Flink Challenges

F1: "Flink watermark is stuck. Events consumed but no windows firing."

"Common causes: (1) Idle source partition — one Kafka partition has no data, watermark is min across all partitions. Fix: withIdleness(Duration.ofMinutes(5)). (2) Wrong timestamp extraction — extracting seconds instead of milliseconds. (3) Single key dominating with monotonous watermarks.

Debugging: Check Flink web UI → watermarks tab. Per-subtask watermarks — the lowest is your bottleneck."

F2: "200GB state, checkpoints taking 10 minutes, causing backpressure."

"Five optimizations: (1) Incremental checkpoints (RocksDB only) — only delta since last checkpoint. (2) Increase checkpoint interval — 15-20 min with minPauseBetweenCheckpoints = 5 min. (3) Local recovery — state.backend.local-recovery=true. (4) Tune RocksDB — increase block cache. (5) State TTL — review if all 200GB is necessary."

F3: "Two Kafka topics joined by user_id. One has 50 events/sec, the other 5000/sec. Handling skew?"

"(1) Pre-aggregate the high-volume stream into 1-minute summaries per user before the join. (2) Broadcast the small stream — 50/sec × small events fits in memory, avoids shuffle entirely. (3) Salting — add random salt (0-9) to hot keys, join 10 copies, then de-duplicate.

I'd start with broadcast since 50/sec easily fits in memory."

F4: "Implement a real-time leaderboard (top 10 products by sales) in Flink."

order-events → KeyBy(product_id) → Sliding Window (1hr, 1min slide) → Sum(amount) → WindowAll → Top-N → Sink (Redis)

The WindowAll is a single-parallelism bottleneck — acceptable because input is already aggregated. Sliding window (1hr/1min) means 60 overlapping windows per key — monitor state size."

F5: "Enrich order events with user profile data from a REST API without blocking?"

"Use Flink's AsyncIO operator — non-blocking REST calls, maintains throughput. ordered_wait preserves order (buffers); unordered_wait for higher throughput.

Better for hot path: cache user profiles in Flink state (refreshed periodically) or use a broadcast stream of profile changes. REST calls per event don't scale at 50K events/sec."

F6: "Handle schema evolution in a running Flink job without downtime?"

"(1) Take a savepoint. (2) Update job code with new schema handling. (3) Update Schema Registry with compatible version. (4) Redeploy from savepoint.

For state schema evolution: Flink supports migration with Avro/POJO serializers. For breaking changes: new job with fresh state from historical offset.

Critical rule: never change the uid() of a stateful operator between versions — Flink maps savepoint state to operators by UID."

⚡ Rapid Fire — 20 Quick Q&A

Click each question to reveal the answer. Aim for <30 seconds each.

1. What is a Kafka consumer group?

A set of consumers that cooperate to consume from a topic. Each partition is read by exactly one consumer in the group.

2. What happens when a Kafka broker dies?

Partitions where it was leader get new leaders from ISR. Producers/consumers reconnect. No data loss if RF>1 and ISR maintained.

3. What is ISR?

In-Sync Replicas — replicas (including leader) that are fully caught up. acks=all waits for all ISR members.

4. What is auto.offset.reset?

What consumers do with no committed offset: earliest (from beginning), latest (only new), none (throw error).

5. What is log compaction?

Kafka retains only the latest record per key, removing older duplicates. Used for stateful topics (inventory, config).

6. What is a tombstone in Kafka?

A record with a valid key but null value. After compaction, tells Kafka to eventually delete the key entirely.

7. What is min.insync.replicas?

Minimum replicas that must acknowledge when acks=all. If ISR < min.insync, writes fail with NOT_ENOUGH_REPLICAS.

8. How does Kafka ensure ordering?

Ordering guaranteed within a partition. Use a consistent message key to route related messages to the same partition.

9. What is a Flink watermark?

A timestamp assertion: "No events with time ≤ W will arrive." Used to trigger windows and track event-time progress.

10. What is backpressure in Flink?

When a downstream operator can't keep up, it signals upstream to slow down. Flink handles this via credit-based flow control.

11. What is operator chaining in Flink?

Combining compatible operators (map + filter) into a single task to avoid serialization overhead. Broken at shuffle boundaries (keyBy).

12. What is a Flink savepoint?

A manual, portable snapshot of job state. Used for upgrades, rescaling, migration. Persists until manually deleted.

13. What is the Outbox Pattern?

Write business data + event to the database in one transaction. CDC captures the outbox table and publishes to Kafka. Avoids dual-write inconsistency.

14. What is Debezium?

Open-source CDC platform. Reads database WAL/binlog and produces change events to Kafka. Supports PostgreSQL, MySQL, MongoDB.

15. What is read_committed isolation?

Consumer setting that filters out records from aborted Kafka transactions. Required for end-to-end exactly-once.

16. What is a Flink checkpoint barrier?

A marker injected into the data stream. When an operator receives barriers from all inputs, it snapshots state. Based on Chandy-Lamport algorithm.

17. What is session.timeout.ms?

How long a consumer can go without a heartbeat before being evicted. Default: 45s (Kafka 3.x).

18. What is RocksDB in Flink?

An embedded key-value store used as a state backend. Stores state on local disk, supports incremental checkpoints, handles terabytes.

19. What is Schema Registry compatibility?

BACKWARD: new schema reads old data. FORWARD: old reads new. FULL: both directions. FULL_TRANSITIVE: all versions cross-compatible.

20. What is MirrorMaker 2?

Kafka's cross-cluster replication tool. Replicates topics, offsets, and ACLs between clusters. Used for DR and geo-replication.

🧭 Technology Choice Scenarios

TC1: "Batch data in Spark, now need real-time with sub-second latency. Flink or Spark Streaming?"

"If sub-second is a hard requirement, Flink. Spark's micro-batch adds 100ms+ inherent latency. However, if 1-2 seconds is acceptable, Spark Streaming reduces operational complexity — one framework for both.

My recommendation: Start with Spark Streaming. If latency proves insufficient, introduce Flink for the specific sub-second pipelines."

TC2: "Message queue for task distribution across 50 worker pods. Kafka, SQS, or RabbitMQ?"

"SQS (or RabbitMQ if on-prem). This is a task queue pattern, not event streaming. Kafka is optimized for ordered logs and replay. For task distribution, you want: competing consumers, message-level acknowledgment, automatic DLQ.

SQS: visibility timeouts, DLQ, auto-scaling with queue depth, serverless. Kafka would work but adds unnecessary complexity. RabbitMQ if you need on-prem, priority queues, or complex routing."

TC3: "Event-driven microservices — events through Kafka or direct REST?"

"Kafka for async events, REST for sync queries.

Kafka when: no immediate response needed, multiple consumers, temporal decoupling, event replay. REST when: immediate response required, request-reply pattern.

Hybrid: Order → REST → Payment (sync, need result). Order → Kafka → Notification (async). Order → Kafka → Analytics (async). Anti-pattern: Kafka as synchronous request-reply."

TC4: "Replicate DB changes to data lake in real-time. Kafka + Debezium or direct pipeline?"

"Kafka + Debezium. Three advantages: (1) Decoupling — DB doesn't know about the lake. Kafka buffers if lake goes down. (2) Fan-out — multiple consumers: lake, search index, cache invalidation. (3) Minimal DB impact — WAL reading, zero query load.

Direct pipeline only for: simple one-off migration, very low volume, or when operational simplicity outweighs architectural benefits."

TC5: "100K events/sec, 500GB state. Managed Flink or self-managed?"

"At this scale, self-managed on Kubernetes (Flink K8s Operator). Managed services have state size caps, limited RocksDB tuning, and scaling isn't instant.

Middle ground: Start managed for faster time-to-market. When you hit limitations (and you will at 500GB state), migrate via savepoints. Key question: can your team handle K8s + Flink operations?"

TC6: "Financial transactions with exactly-once and audit compliance. What stack?"

"Kafka (EOS) + Flink + PostgreSQL with comprehensive audit:

Kafka: acks=all, min.insync.replicas=2, idempotence, transactions. Long retention (1 year) — it IS your audit log.

Flink: Exactly-once checkpointing with RocksDB. 2PC sinks to PostgreSQL.

PostgreSQL: Idempotent upserts with transaction ID as dedup key. PITR enabled.

Monitoring: Periodically reconcile Kafka offsets with DB record counts. This is NOT over-engineering for financial data."

🏆 Final Takeaways

The Interview Gauntlet survival guide — nail these principles and you'll crush any streaming interview.

1. Structure Answers

Situation → Approach → Trade-offs. Interviewers reward structured thinking.

2. Discuss Trade-offs

There's no perfect solution — show you understand the costs.

3. Draw Diagrams

Even verbally: "imagine three boxes..." Visuals signal architectural thinking.

4. Challenge Requirements

"Do you really need global ordering, or per-entity?" shows senior thinking.

5. Know the Numbers

Partition throughput (~10MB/s), checkpoint overhead (~5%), EOS latency (~5-10%).

6. Lead with Simple

"Start with X, if we hit limitation Y, evolve to Z." Shows pragmatism.

7. Name Real Tools

"I'd use Debezium for CDC" beats "I'd use a change capture tool." Specificity = experience.

8. Admit Boundaries

"I haven't used Pulsar in production, but my understanding is..." beats making things up.

9. Connect to Business

"This prevents duplicate charges" beats "this achieves exactly-once."

10. Practice Rapid Fire

If you can answer all 20 in <30 seconds each, you're interview-ready. 🔥
