🏗️ Flink Architecture — JobManager, TaskManagers & Parallelism

Apache Flink is a true stream processing engine — it processes events one at a time (not micro-batches). This gives it lower latency and more natural semantics for event-driven applications.

┌─────────────────────────────────────────────────────────────┐ │ Flink Cluster │ │ ┌───────────────────────┐ │ │ │ JobManager (JM) │ ← The "brain" │ │ │ • Schedules tasks │ │ │ │ • Manages checkpoints│ │ │ │ • Coordinates recovery│ │ │ │ • REST API (:8081) │ │ │ └──────────┬────────────┘ │ │ │ distributes work │ │ ┌────────┼─────────┬────────────┐ │ │ ▼ ▼ ▼ ▼ │ │ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │ TM-1 │ │ TM-2 │ │ TM-3 │ │ TM-4 │ ← TaskManagers │ │ │[slot]│ │[slot]│ │[slot]│ │[slot]│ ("the muscles") │ │ │[slot]│ │[slot]│ │[slot]│ │[slot]│ ← Task slots │ │ └──────┘ └──────┘ └──────┘ └──────┘ (unit of resource) │ └─────────────────────────────────────────────────────────────┘

Parallelism Levels

# Operator-level parallelism stream.key_by(lambda x: x.user_id) \ .window(TumblingEventTimeWindows.of(Time.minutes(5))) \ .aggregate(CountAgg()) \ .set_parallelism(8) # This operator runs 8 parallel instances # Job-level parallelism env.set_parallelism(4) # Default for all operators

Task Slots vs Parallelism: A TaskManager has a fixed number of slots. Each slot gets a fraction of the TM's memory. Operators from the same job can share a slot (operator chaining), reducing serialization overhead. Total cluster parallelism = num_task_managers × slots_per_task_manager.

💡 Interview Insight

"Flink has a JobManager (coordinator) and TaskManagers (workers). The key difference from Spark Streaming: Flink processes events one at a time with continuous operators, not micro-batches. This means lower latency (milliseconds vs seconds), true event-time processing, and more natural windowing semantics."

⏰ Event Time vs Processing Time

Time semantics are the most critical concept in stream processing. Getting this wrong means your results are wrong.

Event happens Enters Kafka Flink processes at source (broker time) (wall clock) │ │ │ Timeline: ───────────┼─────────────────────┼──────────────────┼────── │ │ │ EVENT TIME INGESTION TIME PROCESSING TIME Example — Mobile order placed at 2:00 PM, phone was offline: Event Time: 2:00 PM ← when the user clicked "Buy" Ingestion Time: 2:05 PM ← when phone reconnected, event hit Kafka Processing Time: 2:05.3 PM ← when Flink processed the event

❌ Processing Time Approach

  • 2:00-2:01 window: 100 orders
  • 2:01-2:02 window: 200 orders ← includes delayed mobile orders from 2:00!
  • ⚠️ Result: WRONG — attributes revenue to the wrong minute

✅ Event Time Approach

  • 2:00-2:01 window: 150 orders ← includes late mobile orders, correctly placed
  • 2:01-2:02 window: 150 orders
  • ✅ Result: CORRECT — revenue attributed to when orders happened
Time SemanticUse WhenShopStream Example
Event TimeResults must reflect when events happenedRevenue reports, fraud time windows
Processing TimeLow latency matters more than accuracyReal-time monitoring dashboards
Ingestion TimeNo event timestamp availableLegacy systems without timestamps
# Bounded out-of-orderness (most common) # "Events may arrive up to 5 seconds late" strategy = WatermarkStrategy \ .for_bounded_out_of_orderness(Duration.of_seconds(5)) \ .with_timestamp_assigner( lambda event, _: event.timestamp_ms ) order_stream = env.from_source(kafka_source, strategy, "Orders")
💡 Interview Insight

"Event time whenever correctness matters — revenue calculations, fraud detection windows, SLA measurements. Processing time when you need lowest latency and approximate results are acceptable. The cost of event time is complexity (watermarks, late data handling). In practice, 90% of production streaming jobs should use event time."

🌊 Watermarks & Late Data Handling

Watermarks are Flink's mechanism to track progress in event time. A watermark W(t) declares: "No events with timestamp ≤ t will arrive after this point."

Event stream (time on x-axis, arrival order top-to-bottom): Events: e(2:01) e(2:03) e(2:02) e(2:04) e(2:03) e(2:06) Watermarks: W(1:56) W(1:58) W(1:58) W(1:59) W(1:59) W(2:01) ────────────────────────────────────────────────────→ With boundedOutOfOrderness = 5 seconds: Watermark = max_event_time_seen - 5 seconds After seeing e(2:06): watermark = 2:06 - 5s = 2:01 → Window [2:00-2:01) can now FIRE → Any event with timestamp < 2:01 arriving after this is "late"

📏 forBoundedOutOfOrderness

Best for most real-world streams with some disorder. watermark = maxTimestamp - maxOutOfOrderness. Higher bound → more late events accepted but higher latency.

📐 forMonotonousTimestamps

Best for perfectly ordered sources (rare). watermark = lastEventTimestamp. Zero tolerance for out-of-order = potential data loss.

🔧 Custom WatermarkStrategy

Best for sources with known delay patterns. You implement the logic — e.g., different delays per data center.

Handling Late Data

result = order_stream \ .key_by(lambda e: e.category) \ .window(TumblingEventTimeWindows.of(Time.minutes(1))) \ .allowed_lateness(Time.minutes(5)) # Accept late data up to 5 min .side_output_late_data(late_output_tag) # Route very late data to side output .aggregate(RevenueAggregator()) # Late data flow: # Within window → normal processing # After watermark but within allowedLateness → window RE-FIRES # After allowedLateness → routed to side output (too late)

ShopStream Watermark Design

🛒 Order Events

boundedOutOfOrderness = 10s. Mobile app orders may have slight delays, but 10s covers most cases.

🖱️ Clickstream

boundedOutOfOrderness = 30s. Browser events from slow connections can be significantly delayed.

📦 Inventory Updates

forMonotonousTimestamps(). Database CDC events arrive in WAL order — perfectly monotonic.

💡 Interview Insight

"By default, late data is dropped. With `allowedLateness`, the window state is kept longer and the window re-fires with updated results. For data after even the allowed lateness, I use side outputs for batch reconciliation. The key trade-off: longer watermark delays increase accuracy but also increase state size and result latency."

🪟 Window Types — Tumbling, Sliding, Session & Global

Windows group elements for bounded computations over an unbounded stream. Choosing the right window type is a critical design decision.

TUMBLING WINDOWS (fixed size, no overlap) ┌────────┐┌────────┐┌────────┐┌────────┐ │ 0-5min ││ 5-10m ││10-15m ││15-20m │ Every event in exactly ONE window └────────┘└────────┘└────────┘└────────┘ SLIDING WINDOWS (fixed size, with overlap) ┌──────────────┐ │ 0-10 min │ └──┬───────────┘ Size=10min, Slide=5min ┌──────────────┐ → each event in 2 windows │ 5-15 min │ └──┬───────────┘ SESSION WINDOWS (dynamic, gap-based) ┌───────────────┐ ┌────────┐ ┌──────────────────────┐ │ User A session│ │ (gap │ │ User A session │ │ click,click │ │ >30min)│ │ click,add-cart,buy │ └───────────────┘ └────────┘ └──────────────────────┘ GLOBAL WINDOWS (you control trigger) ┌──────────────────────────────────────────────────────┐ │ Everything in one window, custom trigger decides │ └──────────────────────────────────────────────────────┘

📊 Tumbling

Revenue per minute for dashboard. TumblingEventTimeWindows.of(Time.minutes(1))

📈 Sliding

Avg order value, last hour, updated every 5 min. SlidingEventTimeWindows.of(Time.hours(1), Time.minutes(5))

👤 Session

Shopping session duration per user. EventTimeSessionWindows.withGap(Time.minutes(30))

🌍 Global + Trigger

Process every 100 orders per warehouse. GlobalWindows.create() + CountTrigger.of(100)

⚠️ Memory Impact

Tumbling (1 min): State = 1 window per key Sliding (1hr/5min): State = 12 overlapping windows per key ← 12x more! Session (30min gap): State = variable, depends on user activity Global: State grows unbounded without proper trigger/evictor ← dangerous!
💡 Interview Insight

"Session windows with a 30-minute gap. Unlike tumbling or sliding windows which have fixed boundaries, session windows are dynamic — they open when a user's first event arrives and close when no event arrives for 30 minutes. One consideration: session windows can merge — if a late event bridges the gap between two sessions, Flink automatically merges them."

🧠 State Management — The Heart of Stateful Streaming

State is what makes Flink powerful. Unlike stateless map/filter, stateful operators remember information across events.

🔑 Keyed State (most common)

  • ValueState<T> → single value per key
  • ListState<T> → list of values per key
  • MapState<K,V> → map of key-values per key
  • ReducingState → aggregated value per key
  • Partitioned by key — each key has independent state

⚙️ Operator State (rare)

  • Not partitioned by key
  • Each parallel instance has its own state
  • Used in sources/sinks
  • Example: Kafka source stores partition offsets
  • Instance 0: {P0: offset 45023}

State Backends

🏎️ HashMapStateBackend

  • ✅ In JVM heap memory — very fast (ns access)
  • ⚠️ Limited by heap size, GC pressure
  • ❌ Checkpoints: full snapshot only
  • Best: state < 10 GB, low latency critical

🪨 EmbeddedRocksDBStateBackend

  • ✅ On local disk (SSD) — no GC pressure
  • ✅ Supports incremental checkpoints!
  • ⚠️ Slower (μs access, serde overhead)
  • Best: state > 10 GB, production workloads

Fraud Detection with Keyed State

class FraudDetector(KeyedProcessFunction): def open(self, runtime_context): self.order_count = runtime_context.get_state( ValueStateDescriptor("order-count", Types.INT())) self.last_reset_time = runtime_context.get_state( ValueStateDescriptor("last-reset", Types.LONG())) def process_element(self, event, ctx): current_count = self.order_count.value() or 0 current_count += 1 self.order_count.update(current_count) # Register timer to reset count after 5 minutes ctx.timer_service().register_event_time_timer( event.timestamp + 5 * 60 * 1000) # Fraud rule: more than 10 orders in 5 minutes if current_count > 10: yield FraudAlert(user_id=ctx.get_current_key(), count=current_count, severity="HIGH") def on_timer(self, timestamp, ctx): self.order_count.clear() # Reset counter

State TTL — Preventing Unbounded Growth

ttl_config = StateTtlConfig \ .new_builder(Time.hours(24)) \ .set_update_type(StateTtlConfig.UpdateType.OnCreateAndWrite) \ .set_state_visibility(StateTtlConfig.StateVisibility.NeverReturnExpired) \ .build() state_descriptor = ValueStateDescriptor("user-profile", Types.STRING()) state_descriptor.enable_time_to_live(ttl_config) # State auto-cleaned after 24 hours of no updates
💡 Interview Insight

"Three strategies: (1) State TTL — configure automatic expiration. (2) RocksDB state backend — stores state on disk, supports terabytes without GC pressure. (3) Application-level cleanup — use timers to explicitly clear state after a business-defined period. Without these, state grows indefinitely and eventually crashes the TaskManager."

🛡️ Checkpointing & Fault Tolerance

Checkpointing is how Flink achieves exactly-once processing even when failures occur. It's based on the Chandy-Lamport distributed snapshot algorithm.

HOW CHECKPOINTING WORKS: 1. JobManager injects checkpoint BARRIERS into all source streams 2. Barriers flow through the dataflow graph with the data 3. When an operator receives barriers from ALL inputs → snapshot state 4. State snapshots stored to durable storage (S3, HDFS) Source 1: ──[e1]──[e2]──║BARRIER║──[e3]──[e4]──→ │ Source 2: ──[e5]──[e6]──║BARRIER║──[e7]──→ │ ┌───────┴───────┐ │ Operator │ │ waits for ALL │ │ barriers │ │ → snapshot │ └───────┬───────┘ │ ──║BARRIER║──→ downstream

Barrier Alignment = Exactly-Once

Operator with 2 inputs: Input 1: ──[e1]──║B║──[e2]──[e3]──→ Input 2: ──[e4]──[e5]──[e6]──║B║──→ Step 1: Barrier arrives on Input 1 → BLOCK Input 1, buffer e2, e3 Step 2: Continue processing Input 2 (e4, e5, e6) Step 3: Barrier arrives on Input 2 → snapshot state → release barrier Step 4: Unblock Input 1, process buffered e2, e3 Result: Snapshot reflects exactly events BEFORE barrier on ALL inputs → exactly-once semantics!

Checkpoints vs Savepoints

FeatureCheckpointSavepoint
Triggered byFlink (automatic, periodic)User (manual)
PurposeFault recoveryPlanned operations
Use caseCrash recoveryJob upgrade, rescale, migration
IncrementalYes (RocksDB)No (always full)
CleanupAuto-cleaned on job cancelPersisted until manual delete

Recovery from Failure

Normal operation: CP1 ─────── CP2 ─────── CP3 ──── CRASH! ↑ Recovery: Restore from CP3 • Kafka offsets → reset to CP3 position • State → restored from CP3 snapshot • Reprocess events from CP3 offset → exactly-once: uncommitted transaction aborted, downstream read_committed consumers never see duplicates
💡 Interview Insight

"Three mechanisms: (1) Checkpointing snapshots state and Kafka offsets atomically using Chandy-Lamport with barrier alignment. (2) Kafka consumer offsets rewind on recovery. (3) Kafka producer uses transactions — output committed only when checkpoint completes via 2PC. If the job crashes, the incomplete transaction is aborted. End result: each input event affects the output exactly once."

📝 Flink SQL, Table API & CDC

Flink SQL brings SQL to stream processing. Dynamic tables are the key abstraction — a SQL table that continuously updates as new events arrive.

Traditional SQL

  • SELECT category, SUM(amount) FROM orders GROUP BY category;
  • Runs once, returns static result

✨ Flink SQL

  • Same SQL — but continuously updates as new orders arrive!
  • Result is a "dynamic table" (changelog stream)
-- Real-time revenue per category per minute SELECT category, TUMBLE_START(event_time, INTERVAL '1' MINUTE) AS window_start, COUNT(*) AS order_count, SUM(amount) AS total_revenue, AVG(amount) AS avg_order_value FROM order_events WHERE event_type = 'CREATED' GROUP BY category, TUMBLE(event_time, INTERVAL '1' MINUTE); -- Temporal join: price valid AT THE TIME of the order SELECT o.order_id, p.price, o.quantity, o.quantity * p.price AS total FROM orders AS o JOIN product_prices FOR SYSTEM_TIME AS OF o.event_time AS p ON o.product_id = p.product_id;

Flink CDC — Stream Database Changes Directly

-- Read directly from PostgreSQL via CDC (no Kafka needed!) CREATE TABLE inventory ( sku STRING, warehouse_id STRING, quantity INT, updated_at TIMESTAMP(3), PRIMARY KEY (sku, warehouse_id) NOT ENFORCED ) WITH ( 'connector' = 'postgres-cdc', 'hostname' = 'postgres', 'database-name' = 'shopstream', 'table-name' = 'inventory' ); -- Real-time low-stock alerts SELECT sku, warehouse_id, quantity FROM inventory WHERE quantity < 10; -- Continuously emits when stock drops below 10
💡 Interview Insight

"Flink SQL for standard analytics — aggregations, joins, windowed computations. DataStream API for complex event processing — custom state, async I/O, ML model inference. In practice, start with SQL and drop to DataStream only when SQL can't express the logic. Flink also allows mixing both in the same pipeline."

This comparison comes up in virtually every streaming interview.

🚀 Choose Flink when

  • ✅ Sub-second latency required
  • ✅ Complex stateful processing (fraud, CEP)
  • ✅ Session windows needed
  • ✅ Fine-grained event-time control
  • ✅ Large state (100s of GB) with RocksDB

⚡ Choose Spark Streaming when

  • ✅ Team already uses Spark for batch
  • ✅ Seconds-level latency is acceptable
  • ✅ Need batch + streaming unification
  • ✅ Heavy ML integration (MLlib)
  • ✅ Broader ecosystem (Delta Lake, Iceberg)
💡 Interview Insight

"I'd choose Flink for fraud detection. Three reasons: (1) Latency — millisecond response time; Spark adds 100ms+ delay. (2) State management — first-class keyed state with RocksDB, TTL, and timers. (3) Session windows — native in Flink; Spark doesn't have them. I'd choose Spark Streaming if the team already had Spark expertise and latency requirements were relaxed to seconds."

🎯 Key Takeaways

1. True Streaming

Flink processes per-event with millisecond latency, not micro-batches.

2. Architecture

JobManager coordinates; TaskManagers execute. Parallelism = slots × TMs.

3. Event Time

Almost always correct for business logic. Processing time only for monitoring.

4. Watermarks

Track event-time progress. boundedOutOfOrderness handles real-world disorder.

5. Windows

Tumbling for fixed reports, sliding for rolling metrics, session for user behavior.

6. Keyed State

Enables complex per-entity processing. Use RocksDB for production-scale state.

7. State TTL

Prevents unbounded growth — always configure cleanup for production jobs.

8. Checkpointing

Chandy-Lamport + Kafka transactions = end-to-end exactly-once.

9. Flink SQL

Fast path for standard analytics. DataStream API for complex stateful logic.

10. Flink vs Spark

Flink wins for low-latency, stateful, event-driven. Spark wins for batch-heavy teams.
