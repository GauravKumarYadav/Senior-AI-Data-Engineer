🎯 Exactly-Once Semantics (EOS)

The holy grail of stream processing: every record is processed exactly once — no duplicates, no data loss. Kafka achieves this through three layered mechanisms.

Layer 1 — Idempotent Producer

Without idempotence: Producer sends r5 → Broker writes r5 → ACK lost Producer retries → Broker writes r5 AGAIN → duplicate! With idempotence (enable.idempotence=true): Each producer gets a Producer ID (PID) Each record gets a sequence number per partition Broker deduplicates: "PID=7, seq=5 already written → skip"

Layer 2 — Transactions

Transactions let you atomically write to multiple topics and commit consumer offsets together — essential for "consume-transform-produce" patterns.

┌─────────────────────────────────────────────────┐ │ TRANSACTION BOUNDARY │ │ │ │ 1. Read from "order-events" (offset 42) │ │ 2. Process: enrich order with user profile │ │ 3. Write to "enriched-orders" topic │ │ 4. Write to "notification-events" topic │ │ 5. Commit consumer offset (42 → 43) │ │ │ │ ALL succeed together, or ALL roll back │ └─────────────────────────────────────────────────┘

Layer 3 — read_committed Consumers

Setting isolation.level=read_committed makes consumers only see records from committed transactions:

Timeline of partition: [r1:committed] [r2:txn-A] [r3:txn-A] [r4:committed] [r5:txn-A] read_uncommitted: sees r1, r2, r3, r4, r5 (all records) read_committed: sees r1, then waits... txn-A commits → sees r2, r3, r4, r5 txn-A aborts → sees r1, r4 (r2, r3, r5 filtered out)
ComponentOverhead
Idempotent producer~1-3% latency increase
Transactions~5-10% throughput decrease
read_committedSlight delay (waits for txn)
💡 Interview Insight

"Through three layers: (1) Idempotent producers assign sequence numbers per partition so the broker deduplicates retries. (2) Transactions extend this across partitions and topics — you atomically produce to multiple topics and commit offsets in one operation. (3) read_committed consumers only see records from committed transactions. Together, these enable end-to-end exactly-once in consume-transform-produce pipelines."

📜 Schema Registry — Data Contracts

When 15 microservices produce and consume events, you need data contracts. Without schema enforcement, a producer can change a field name and silently break every downstream consumer.

┌──────────┐ ┌─────────────────┐ ┌──────────┐ │ Producer │ ──→ │ Schema Registry │ ←── │ Consumer │ │ │ │ │ │ │ │ 1. Register │ Stores: │ │ 3. Fetch schema │ schema │ • Schema ID │ │ by ID │ 2. Send msg │ • Versions │ │ 4. Deserialize │ with ID=1 │ • Compat rules │ │ with schema └──────────┘ └─────────────────┘ └──────────┘ Wire format: ┌──────┬───────────┬──────────────────────────┐ │ 0x00 │ Schema ID │ Avro/Protobuf payload │ │ 1B │ 4 bytes │ variable │ └──────┴───────────┴──────────────────────────┘

The schema is not embedded in every message — only a 4-byte ID. This saves enormous bandwidth. The consumer fetches and caches the schema from the registry.

"Three reasons: (1) Contract enforcement — Schema Registry validates every message against a registered schema. (2) Size efficiency — only a 4-byte ID per message, cutting sizes by 70-80% vs JSON. In microservices with 15 teams, this prevents 'schema surprise' production incidents."

🔄 Schema Evolution & Compatibility

Schemas change — that's inevitable. The question is: how do you change them without breaking running consumers?

BACKWARD (default): New schema can READ old data → Deploy new CONSUMERS first, then new PRODUCERS FORWARD: Old schema can READ new data → Deploy new PRODUCERS first, then new CONSUMERS FULL: Both BACKWARD + FORWARD → Deploy in any order — safest, but most restrictive NONE: No compatibility checking → Chaos mode. Don't do this in production!

What Changes Are Safe?

ChangeBACKWARDFORWARDFULL
Add field with default✅✅✅
Add field without default❌✅❌
Remove field with default✅❌❌
Remove field without default❌❌❌
Rename field❌❌❌

Safe Evolution Example

// Version 1 (current) {"order_id": "string"}, {"amount": "double"}, {"customer_id": "string"} // Version 2 (add optional field) {"order_id": "string"}, {"amount": "double"}, {"customer_id": "string"}, {"shipping_address": ["null","string"], "default": null}

💡 Interview Insight

"Never make a breaking change directly. Use a multi-step approach: (1) Add the new field with a default value (backward compatible). (2) Deploy all consumers to handle both formats. (3) Deploy producers to write the new field. (4) After a full retention period, optionally remove the deprecated field. For truly incompatible changes, create a new topic (e.g., order-events-v2) and run both in parallel during migration."

🔌 Kafka Connect — Bridging the Data World

Kafka Connect streams data between Kafka and external systems without custom producer/consumer code. It's the integration backbone of ShopStream.

Kafka Connect Cluster ┌────────────────────────┐ │ Connect Workers │ External Systems │ │ Kafka Topics │ Source Connectors │ PostgreSQL ────►│ Debezium PG │──► inventory-cdc MySQL ────►│ Debezium MySQL │──► orders-cdc │ │ │ Sink Connectors │ Elasticsearch◄───│ ES Sink │◄── order-events S3 (archive) ◄───│ S3 Sink │◄── clickstream └────────────────────────┘

Debezium CDC (Change Data Capture)

Debezium reads the PostgreSQL WAL to capture every INSERT, UPDATE, DELETE as a Kafka event. No polling, no application changes, no triggers.

PostgreSQL WAL → Debezium → Kafka Topic Before: inventory table: {sku: "A100", qty: 50} UPDATE: SET qty = 45 WHERE sku = 'A100' Debezium emits: { "before": {"sku": "A100", "qty": 50}, "after": {"sku": "A100", "qty": 45}, "op": "u", "source": {"table": "inventory"} }

Dead Letter Queue (DLQ)

Records that can't be processed (deserialization errors, schema mismatches) go to the DLQ topic instead of crashing the connector:

{ "errors.tolerance": "all", "errors.deadletterqueue.topic.name": "dlq-inventory-sink", "errors.deadletterqueue.topic.replication.factor": 3, "errors.deadletterqueue.context.headers.enable": true }
💡 Interview Insight

"Kafka Connect for standard integrations — databases (via Debezium CDC), cloud storage, search engines. It handles offset management, parallelism, fault tolerance, and schema translation out of the box. I'd write a custom consumer only when I need complex business logic or the target system has no existing connector. Connect also supports SMTs for lightweight transforms and DLQs for poison pill handling."

"I monitor consumer lag via Kafka Exporter + Prometheus + Grafana. Alert at two thresholds: warning if lag >1000 records, critical if >10000. ☁️ Managed Kafka — Cloud Offerings

Running your own Kafka cluster is operationally complex. Managed services handle the infrastructure so you focus on building.

🟣 Confluent Cloud

Full Kafka ecosystem: Schema Registry, ksqlDB, managed connectors. Multi-cloud (AWS, GCP, Azure). Best for rich ecosystem needs.

🟠 AWS MSK

Kafka-native on AWS. Glue Schema Registry. MSK Connect. Cost-effective for AWS-native teams. Serverless mode available.

🔵 Azure Event Hubs

Kafka protocol compatible (not actual Kafka). Simple event streaming for Azure shops. Per-throughput-unit pricing.

💡 Interview Insight

"Self-managed makes sense when you need full control or strict data residency. Managed services are better for most teams — they handle broker provisioning, patching, replication, and monitoring. I'd evaluate: (1) team Kafka expertise, (2) operational budget, (3) feature needs (Schema Registry, Connect, ksqlDB), and (4) cloud lock-in tolerance."

🧩 Advanced Producer & Consumer Patterns

Pattern 1 — Outbox Pattern

Solves the dual-write problem: service writes to DB AND produces to Kafka. If Kafka fails after DB commit → inconsistency.

┌─────────────────────────────┐ │ PostgreSQL │ │ ┌─────────────┬───────────┐ │ ┌─────────┐ │ │ orders │ outbox │ │────►│ Debezium│───► Kafka │ │ (business) │ (events) │ │ CDC │ │ │ └─────────────┴───────────┘ │ └─────────┘ │ BEGIN; │ │ INSERT INTO orders(...); │ │ INSERT INTO outbox(...); │ │ COMMIT; │ └─────────────────────────────┘

Pattern 2 — Dead Letter Topic + Retry

Main Topic ──► Consumer ──► Process │ (failure) ▼ Retry-1 Topic ──► (delay 1 min) ──► Consumer │ (failure) ▼ Retry-2 Topic ──► (delay 5 min) │ (failure) ▼ DLQ Topic (manual review)

Pattern 3 — Event Sourcing

Command: "Place Order #123" │ ▼ ┌──────────────────────────────────────────────────────────┐ │ order-events (compacted, key=order_id) │ │ [order-123:Created] [order-123:Paid] [order-123:Shipped] │ └──────────────────────────────────────────────────────────┘ │ ▼ Materialize (consumer rebuilds state) ┌──────────────────────┐ │ Order #123 │ │ Status: Shipped │ │ Amount: $99.50 │ │ History: full audit │ └──────────────────────┘
💡 Interview Insight

"The Outbox Pattern: instead of writing to the database and Kafka separately (dual-write problem), I write the business data and an outbox event in a single database transaction. Then Debezium CDC captures the outbox table changes and publishes them to Kafka. This guarantees that if the DB transaction commits, the event will eventually reach Kafka. It's eventual consistency with guaranteed delivery."

🎯 Key Takeaways

🎯 Exactly-Once

Three layers: idempotent producer + transactions + read_committed consumers.

📜 Schema Registry

Essential governance — use Avro for data pipelines, enforce BACKWARD compatibility.

🔄 Schema Evolution

Always add fields with defaults, never rename or remove directly. Plan multi-step migrations.

🔌 Kafka Connect

Standard integrations out of the box. Debezium CDC is the gold standard for database change capture.

📊 Consumer Lag

The #1 production metric. Monitor, alert, and have a scaling playbook ready.

☁️ Managed Kafka

Confluent for ecosystem, MSK for AWS-native, Event Hubs for Azure simplicity.

🏗️ Outbox Pattern

Solves the dual-write problem between databases and Kafka via CDC.

💰 EOS Overhead

Transactions cost ~5-10% throughput. Use where exactly-once matters (payments), not everywhere.
