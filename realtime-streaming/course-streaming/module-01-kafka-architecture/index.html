📋 Topics — The Logical Backbone

A topic in Kafka is a named, logical channel to which producers write records and from which consumers read. Think of it as an infinitely growing, append-only, immutable log.

ShopStream example: You'd create separate topics for order-events, clickstream, inventory-updates, and payment-results. Each topic isolates a domain concern. A single Kafka cluster can host thousands of topics.

┌──────────────────────────────────────────────────────┐ │ Kafka Cluster │ │ │ │ Topic: "order-events" │ │ ┌────┬────┬────┬────┬────┬────┬────┬────┐ │ │ │ r0 │ r1 │ r2 │ r3 │ r4 │ r5 │ r6 │ r7 │→ ... │ │ └────┴────┴────┴────┴────┴────┴────┴────┘ │ │ │ │ Topic: "clickstream" │ │ ┌────┬────┬────┬────┬────┬────┐ │ │ │ r0 │ r1 │ r2 │ r3 │ r4 │ r5 │→ ... │ │ └────┴────┴────┴────┴────┴────┘ │ └──────────────────────────────────────────────────────┘

Key Properties of Topics

PropertyDescription
Append-onlyRecords are always added to the end
ImmutableOnce written, records cannot change
DurablePersisted to disk, replicated
Retention-basedOld records removed by time or size
Ordered per partitionNot globally ordered across topic

Topics are logical — they don't map 1:1 to physical files. Under the hood, each topic is split into partitions, and each partition maps to a set of log segment files on disk.

💡 Interview Insight

"A topic is a named, append-only, immutable log that acts as a logical channel for a category of events. Producers publish records to topics, and consumers subscribe to them. Topics are split into partitions for parallelism and are persisted to disk with configurable retention. The immutability guarantee is what enables Kafka's replay capability — any consumer can re-read the entire history from any offset."

⚡ Partitions — The Unit of Parallelism

Partitions are where Kafka's scalability magic happens. Each topic is divided into one or more partitions, and each partition is an independent, ordered, append-only log. Ordering is guaranteed within a partition, but NOT across partitions.

Topic: "order-events" (3 partitions) Partition 0: [r0] [r1] [r2] [r3] [r4] → ... Partition 1: [r0] [r1] [r2] [r3] → ... Partition 2: [r0] [r1] [r2] [r3] [r4] [r5] → ... Key "order-123" → hash("order-123") % 3 = Partition 1 Key "order-456" → hash("order-456") % 3 = Partition 0

🔀 Parallelism

12 partitions = up to 12 consumers reading in parallel. Partition count sets the ceiling for consumer parallelism.

📐 Ordering

By using order_id as the key, all events for order-123 (created, paid, shipped) land in the same partition — in order.

🚀 Throughput

Each partition handles ~10 MB/s writes. 12 partitions = ~120 MB/s total throughput for a topic.

Choosing Partition Count

FactorGuidance
Consumer parallelismPartitions ≥ max consumers
Throughput targetMore partitions = more throughput
Ordering requirementsEvents needing order share a key
Rebalancing timeMore partitions = slower rebalance

Critical rule: You can increase partitions later, but you can never decrease them. And increasing partitions breaks key-based ordering guarantees for existing keys.

💡 Interview Insight

"I consider three factors: (1) target throughput divided by per-partition throughput gives minimum partitions, (2) maximum consumer parallelism needed — you can't have more active consumers than partitions, and (3) ordering requirements — all events that must be ordered together share a partition key. I also factor in that partitions can only be increased (never decreased), and increasing them breaks existing key-based routing."

🖥️ Brokers & Replication

A broker is a single Kafka server. A Kafka cluster is a group of brokers working together. Replication copies each partition across multiple brokers for fault tolerance.

Kafka Cluster (3 Brokers) — RF=3 Broker 1 Broker 2 Broker 3 ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ P0 (LEADER) │ │ P0 (follower)│ │ P0 (follower)│ │ P1 (follower)│ │ P1 (LEADER) │ │ P1 (follower)│ │ P2 (follower)│ │ P2 (follower)│ │ P2 (LEADER) │ └──────────────┘ └──────────────┘ └──────────────┘ Write to P0 → Broker 1 (leader) → replicates to Broker 2, 3

In-Sync Replicas (ISR)

The ISR set contains the leader plus all followers that are "caught up" (within replica.lag.time.max.ms, default 30s). A follower that falls behind is removed from ISR. acks=all means "all ISR members acknowledged," not "all replicas."

The acks Setting

SettingBehaviorDurabilityLatency
acks=0Fire and forgetLowest — may lose data~0.5ms
acks=1Wait for leader onlyMedium — loses if leader dies~2ms
acks=allWait for all ISR replicasHighest — survives broker loss~5-10ms
💡 Interview Insight

"Through replication. Each partition is replicated across multiple brokers — typically RF=3. One replica is the leader (handles reads/writes), and followers replicate passively. The ISR tracks which followers are caught up. With acks=all, a produce request only succeeds when all ISR members acknowledge. Combined with min.insync.replicas=2, this guarantees data survives any single broker failure."

📤 Producers — Publishing Messages

Producers are client applications that publish records to Kafka topics. A record consists of a key (optional), a value (the payload), a timestamp, and optional headers.

Producer Record Structure: ┌───────────────────────────────────────────┐ │ Key: "order-123" (byte[]) │ │ Value: {"event":"created","amount":99.50}│ │ Timestamp: 1712345678000 │ │ Headers: [("source","checkout-svc")] │ │ Topic: "order-events" │ └───────────────────────────────────────────┘

Partition Selection Logic

if record.partition is explicitly set: → use that partition elif record.key is not None: → partition = hash(key) % num_partitions # deterministic! else: → sticky partitioning (batch to one partition, then rotate)

Critical Producer Configs

ConfigValueWhy
acksallDurability for orders
enable.idempotencetruePrevent duplicates on retry
linger.ms5Wait 5ms to batch more records
compression.typelz4Good compression/speed ratio

Idempotent producers solve a subtle problem: if a producer sends a record, the broker writes it, but the ACK is lost, the producer retries and creates a duplicate. With enable.idempotence=true, the producer attaches a sequence number — the broker deduplicates retries.

from confluent_kafka import Producer config = { 'bootstrap.servers': 'kafka-1:9092', 'acks': 'all', 'enable.idempotence': True, 'compression.type': 'lz4', } producer = Producer(config) producer.produce( topic='order-events', key='order-123', value='{"event":"created"}', callback=on_delivery, ) producer.flush()

💡 Interview Insight

"If a key is provided, Kafka hashes the key (murmur2) modulo the number of partitions — this guarantees all records with the same key go to the same partition, preserving per-key ordering. If no key is provided, Kafka uses 'sticky partitioning' — it fills a batch to one partition, then rotates for better batching efficiency."

📥 Consumers & Consumer Groups

Consumers pull records from Kafka (not push!). This pull model lets each consumer control its own read rate. Consumers are organized into consumer groups for parallel processing.

Topic: "order-events" (4 partitions) Consumer Group: "order-processing-service" ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ Partition 0│ │ Partition 1│ │ Partition 2│ │ Partition 3│ └─────┬──────┘ └─────┬──────┘ └─────┬──────┘ └─────┬──────┘ ▼ ▼ ▼ ▼ ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐ │Consumer A │ │Consumer B │ │Consumer C │ │Consumer D │ └───────────┘ └───────────┘ └───────────┘ └───────────┘ Rule: Each partition → exactly 1 consumer in a group Each consumer → can read from 1+ partitions

The golden rule: 4 partitions + 4 consumers = 1 each (ideal). 4 partitions + 6 consumers = 4 active + 2 idle (wasted!).

Multiple consumer groups read the same topic independently — each group maintains its own offsets. This enables the pub/sub pattern: order-processing, analytics, and fraud-detection all consume the same stream.

Offset Management

Partition 0: [r0][r1][r2][r3][r4][r5][r6][r7][r8][r9] ↑ ↑ committed offset latest offset Consumer lag = latest - committed = 9 - 5 = 4 records behind
StrategyHowRisk
Auto-commitCommits every 5s automaticallyMay lose records on crash
Manual commitCall commitSync() after processingMore control, must handle
💡 Interview Insight

"The extra consumers sit idle. Kafka's rule is: each partition is assigned to exactly one consumer within a group. So with 4 partitions and 6 consumers, only 4 are active. The idle ones serve as hot standbys — if an active consumer dies, an idle one picks up its partition during rebalancing."

🔄 Consumer Rebalancing

Rebalancing redistributes partitions among consumers when membership changes — a consumer joins, leaves, crashes, or partition count changes.

😱 Eager Rebalance (legacy)

  • ❌ Revoke ALL partitions from ALL consumers
  • ❌ Full stop-the-world: 10-30 seconds
  • ❌ 500K orders/day → ~350 orders missed per rebalance

✅ Cooperative Sticky (default 3.x)

  • ✅ Only revoke partitions that need to MOVE
  • ✅ Most consumers never stop processing
  • ✅ Partial pause: ~2-5 seconds

Static Group Membership

Each consumer instance gets a persistent group.instance.id. When a consumer restarts (same ID), Kafka recognizes it and skips rebalancing — essential for rolling deployments.

# Tuning tips for production config = { 'group.instance.id': 'order-processor-pod-3', # Static! 'partition.assignment.strategy': 'cooperative-sticky', 'session.timeout.ms': 45000, # Default 10s is aggressive 'heartbeat.interval.ms': 15000, # 1/3 of session timeout 'max.poll.interval.ms': 300000, # 5 min for slow processing }
💡 Interview Insight

"Three strategies: (1) Use cooperative sticky assignment so only affected partitions are revoked. (2) Use static group membership with group.instance.id for rolling deployments — the broker recognizes a restarted consumer and skips rebalancing. (3) Tune session.timeout.ms and max.poll.interval.ms to avoid false evictions from GC pauses."

🏛️ ZooKeeper (Legacy) vs KRaft

Kafka historically depended on ZooKeeper for cluster metadata. Starting with Kafka 3.3, KRaft (Kafka Raft) replaces ZooKeeper entirely. ZooKeeper is fully removed in Kafka 4.0.

🐘 ZooKeeper (Legacy)

  • ❌ Separate system to manage (3-5 nodes)
  • ❌ ~200K partition practical limit
  • ❌ Controller failover ~30 seconds
  • ❌ High operational overhead
  • ❌ Separate ACL system

⚡ KRaft Mode

  • ✅ Single system — metadata built-in
  • ✅ Millions of partitions supported
  • ✅ Controller failover ~5 seconds
  • ✅ Low operational overhead
  • ✅ Unified Kafka security
KRaft MODE (no ZooKeeper): ┌─────────────────────────────────┐ │ Kafka Brokers │ │ Broker 1 Broker 2 Broker 3 │ │ (voter) (voter) (voter) │ │ │ │ Controller quorum built-in │ │ Metadata in __cluster_metadata │ └─────────────────────────────────┘
💡 Interview Insight

"ZooKeeper was an external dependency that created operational complexity, limited partition scalability to ~200K, and caused slow controller failovers (~30s). KRaft embeds metadata management directly into Kafka using Raft consensus. This eliminates an external system, enables millions of partitions, reduces failover to ~5 seconds, and unifies security."

💾 Log Segments & Retention

Each partition is stored as a sequence of log segment files on disk. Understanding this physical layout is key to understanding retention and compaction.

Partition 0 directory: /kafka-data/order-events-0/ 00000000000000000000.log ← Segment 1 (offsets 0-9999) 00000000000000000000.index ← Offset index 00000000000000010000.log ← Segment 2 (offsets 10000-19999) 00000000000000020000.log ← Active segment (being written)

Retention Strategies

StrategyConfigBehavior
Time-basedretention.ms=604800000Delete segments older than 7 days
Size-basedretention.bytes=10737418240Delete oldest when total >10GB
Compactioncleanup.policy=compactKeep latest value per key forever

Log Compaction

BEFORE compaction: [k1:v1] [k2:v1] [k1:v2] [k3:v1] [k2:v2] [k1:v3] AFTER compaction: [k3:v1] [k2:v2] [k1:v3] ← latest value per key retained Tombstone: [k2:null] → eventually removes k2 entirely

📦 order-events

Time-based (90 days). Full event history for replay, old orders purged.

📊 inventory-levels

Log compaction. Key = warehouse-sku. Always want latest count, not history.

🖱️ clickstream

Time-based (7 days). High volume, short-lived analytical value.

💡 Interview Insight

"Time-based retention suits event streams where you care about the sequence (order lifecycle, clickstream). Log compaction suits state/snapshot topics where you only care about the latest value per key — like inventory levels or user profiles. Compacted topics act like a distributed key-value store on top of Kafka."

🎯 Key Takeaways

📋 Topics

Logical, immutable, append-only logs — the fundamental Kafka abstraction.

⚡ Partitions

Unit of parallelism AND ordering. Choose partition keys wisely. Can increase, never decrease.

🛡️ Replication

RF=3, min.insync.replicas=2, acks=all — your durability guarantee.

📤 Producers

Key-based hashing for deterministic partition routing. Idempotence prevents duplicates.

📥 Consumer Groups

Enable parallel processing. Partition count caps parallelism. Offset = your contract.

🔄 Rebalancing

Use cooperative sticky + static membership. Tune timeouts to avoid false evictions.

⚡ KRaft

Replaces ZooKeeper: simpler ops, faster failover, millions of partitions.

💾 Retention

Log compaction = state store. Time-based = event streams. Match to business needs.
