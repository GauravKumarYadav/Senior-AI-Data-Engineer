🔬 EXPLAIN ANALYZE — Reading Query Plans

EXPLAIN shows you what the database plans to do. EXPLAIN ANALYZE actually runs the query and shows what it really did, with timing and row counts.

-- PostgreSQL syntax EXPLAIN ANALYZE SELECT c.name, COUNT(*) AS order_count, SUM(o.amount) AS total_spent FROM customers c JOIN orders o ON c.customer_id = o.customer_id WHERE o.order_date >= '2024-01-01' GROUP BY c.name ORDER BY total_spent DESC LIMIT 10;

Decoding the Key Elements

Node Type (cost=startup..total rows=estimated width=bytes) (actual time=start..end rows=actual loops=N)
ElementMeaning
cost=startup..totalEstimated cost in arbitrary units (lower = better)
rows=Estimated rows (EXPLAIN) or actual (ANALYZE)
actual time=Real wall-clock time in milliseconds

Common Scan Types

📖 Seq Scan

Full table scan — reads every row. Fine for small tables or queries returning >10-15% of rows.

🔍 Index Scan

B-tree lookup + table fetch. The workhorse for selective queries.

⚡ Index Only Scan

Reads only from the index — no table access! Requires a covering index.

🗺️ Bitmap Scan

Index → bitmap → batch table fetch. Great for medium-selectivity queries.

Red flags: Seq Scan on a large table with a selective WHERE, actual rows ≫ estimated rows (bad statistics), nested loops with high loops= count.

💡 Interview Insight

"I start with EXPLAIN ANALYZE to get the actual execution plan. I look for: (1) Seq Scans on large tables with selective filters — need indexes. (2) Big discrepancies between estimated and actual row counts — stale statistics, fix with ANALYZE. (3) The most expensive node — follow cost numbers and actual time to find the bottleneck."

🌳 Index Types — B-tree, Hash, GIN, GiST

B-tree — The Default Workhorse

B-tree indexes support equality, range queries, sorting, and prefix matching. O(log n) lookup — for a 10M-row table, that's ~23 comparisons.

CREATE INDEX idx_orders_date ON orders (order_date); -- These ALL use the B-tree index: WHERE order_date = '2024-01-15' -- Equality WHERE order_date > '2024-01-01' -- Range WHERE order_date BETWEEN '2024-01' AND '2024-06' -- Range ORDER BY order_date -- Sorting WHERE order_date IS NULL -- NULL lookup

GIN — Generalized Inverted Index

Designed for composite values — arrays, JSONB, and full-text search. Maps each element/key to the rows containing it.

-- Full-text search CREATE INDEX idx_products_search ON products USING GIN (to_tsvector('english', description)); -- JSONB containment CREATE INDEX idx_orders_metadata ON orders USING GIN (metadata); -- Array operations CREATE INDEX idx_products_tags ON products USING GIN (tags);

GiST — Generalized Search Tree

Supports spatial data, range types, and nearest-neighbor (KNN) searches.

-- PostGIS spatial queries CREATE INDEX idx_stores_location ON stores USING GIST (location); SELECT name, ST_Distance(location, ST_MakePoint(-73.98, 40.75)) AS distance FROM stores ORDER BY location <-> ST_MakePoint(-73.98, 40.75) -- KNN search LIMIT 5;

💡 Interview Insight

"For a JSONB column storing product attributes, I'd use a GIN index. GIN maps each key-value pair to the rows containing it, enabling efficient containment queries with @>. For single-key lookups, a B-tree expression index like CREATE INDEX ON products ((attributes->>'color')) is smaller and faster."

📐 Composite Indexes — Column Order Matters

The Leftmost Prefix Rule

A composite index on (a, b, c) can satisfy queries on (a), (a, b), or (a, b, c) — but never (b) or (c) alone. Think of a phone book sorted by (last, first, city).

CREATE INDEX idx_orders_composite ON orders (customer_id, order_date, status); -- ✅ Uses full index: WHERE customer_id = 100 AND order_date = '2024-01-15' AND status = 'shipped' -- ✅ Uses first two columns: WHERE customer_id = 100 AND order_date > '2024-01-01' -- ✅ Uses first column only: WHERE customer_id = 100 -- ❌ Cannot use this index: WHERE order_date = '2024-01-15' WHERE status = 'shipped'

Column Order Strategy — ERS

Rule: Equality columns first → Range/inequality next → Sort columns last.

-- Query: WHERE status = 'active' AND created_at > '2024-01-01' ORDER BY amount DESC -- Optimal index: CREATE INDEX idx_optimal ON orders (status, created_at, amount); -- ^equality ^range ^sort

Why Column Order Matters — 5000× Difference

❌ Index A: (order_date, customer_id)

  • Scans ALL orders for the date range
  • Then filters for the customer
  • Reads ~30,000 rows → filters to 6

✅ Index B: (customer_id, order_date)

  • Jumps straight to the customer
  • Then scans their date range
  • Reads ~6 rows directly

The Range Column Trap

An index can efficiently handle at most one range condition. After the first range predicate, subsequent columns aren't used for seeking.

CREATE INDEX idx_orders ON orders (amount, order_date); -- ✅ Uses both columns (amount is equality): WHERE amount = 100 AND order_date > '2024-01-01' -- ⚠️ Uses amount's range only (order_date becomes a filter): WHERE amount > 50 AND order_date > '2024-01-01'
💡 Interview Insight

"I follow the ERS principle: Equality first (maximally narrows search), Range next (B-tree handles one range scan), Sort last (avoids a separate sort step). I also consider query frequency — if 80% of queries filter by customer_id, it should be leftmost even if other orderings suit rare queries better."

📦 Covering Indexes & Index-Only Scans

The Extra Trip Problem

A regular index lookup is a two-step process: search the index → find row pointers, then fetch actual rows from the table (heap). The heap fetch is often the expensive part — scattered random I/O.

Covering Indexes to the Rescue

A covering index includes all columns the query needs, eliminating the heap fetch entirely.

-- Regular index: still needs heap fetch for order_date and amount CREATE INDEX idx_orders_customer ON orders (customer_id); -- Covering index: includes everything the query needs! CREATE INDEX idx_orders_covering ON orders (customer_id) INCLUDE (order_date, amount);

INCLUDE vs Composite

Feature(a, b, c) composite(a) INCLUDE (b, c)
Filter on b or c✅ Yes❌ No
Sort on b or c✅ Yes❌ No
Index-only scan✅ Yes✅ Yes
Index sizeLarger (b,c in tree)Smaller (b,c in leaf only)
Maintenance costHigherLower

Rule of thumb: Use INCLUDE for columns you SELECT but never filter/sort on. Use composite keys for columns you filter or sort on.

Visibility Map & VACUUM

-- Check if your query gets an Index Only Scan EXPLAIN ANALYZE SELECT order_date, amount FROM orders WHERE customer_id = 123; -- Look for: "Index Only Scan using idx_orders_covering" -- "Heap Fetches: 0" → pure index-only scan! -- "Heap Fetches: 500" → visibility map needs VACUUM
💡 Interview Insight

"A covering index includes all columns a query needs — filter and output — so the database answers entirely from the index. I use the INCLUDE clause to add non-searchable columns to the leaf pages, keeping the tree small. The tradeoff is increased index size and write overhead, so I reserve them for high-frequency, performance-critical queries."

🤝 Join Algorithms — NL, Hash, Merge

Nested Loop Join

FOR each row in outer_table: ← "Driving" table FOR each row in inner_table: ← "Probe" table IF join_condition matches: output combined row

Best when: Outer table is small, inner table has an index on the join column. 10 rows × index lookup = ~200 operations.

Hash Join

Step 1: Build hash table from SMALLER table (build side) Step 2: Probe hash table for each row of LARGER table (probe side) = O(n + m) total — fast when smaller table fits in work_mem

Best when: No useful indexes, both tables moderately sized, equi-join. Worst: build table spills to disk.

Merge Join (Sort-Merge)

Step 1: Sort both tables on the join key (or use index order) Step 2: Walk both sorted lists simultaneously, matching rows If already sorted: just O(n + m) — the fastest possible!

Best when: Both inputs are already sorted from an index or preceding ORDER BY.

Comparison Matrix

AlgorithmTimeMemoryBest Scenario
Nested LoopO(n × m) worstO(1)Small outer + indexed inner
Hash JoinO(n + m)O(smaller)Large × large, no index, equi
Merge JoinO(n log n + m log m)O(sort bufs)Pre-sorted inputs
💡 Interview Insight

"Nested Loop is best for small-large joins with an index. Hash Join is optimal for large equi-joins without indexes — O(n+m). Merge Join excels when both inputs are already sorted. Non-equi joins (>, <, LIKE) can only use Nested Loop. The optimizer considers table sizes, indexes, join predicate type, and work_mem."

📊 Statistics, ANALYZE & Selectivity

Why the Optimizer Needs Statistics

The query optimizer is a cost-based planner. It estimates the cost of different strategies and picks the cheapest. It needs: row counts, distinct values, data distribution, and selectivity.

What ANALYZE Collects

StatisticWhat It StoresUsed For
n_distinctNumber of distinct valuesGROUP BY size estimation
null_fracFraction of NULLsNULL handling
most_common_valsFrequent values + frequenciesSkewed data estimation
histogram_boundsEvenly-spaced boundariesRange query estimation
correlationPhysical vs logical orderIndex scan vs bitmap choice

When Statistics Go Wrong

-- Symptom: estimated rows = 100, actual rows = 500,000 -- Cause: stale statistics after bulk load/delete -- Fix: ANALYZE orders; -- For unusual distributions, increase statistics target: ALTER TABLE orders ALTER COLUMN status SET STATISTICS 1000; -- default is 100 ANALYZE orders;

Extended Statistics (PostgreSQL 10+)

Standard stats are per-column. Correlated columns (city + zip) cause the optimizer to assume independence, leading to wildly wrong estimates.

-- Without extended stats, optimizer assumes independence: -- P(city='NYC' AND zip='10001') = 0.05 × 0.0001 = 0.000005 -- Reality: 0.003 (600× higher!) CREATE STATISTICS stat_city_zip (dependencies, ndistinct) ON city, zip_code FROM addresses; ANALYZE addresses;
💡 Interview Insight

"A query slow after a bulk load? The table statistics are stale. The optimizer uses old row counts and distributions, making bad plan choices — like Seq Scan instead of Index Scan. Fix: run ANALYZE. In production, configure autovacuum/autoanalyze more aggressively with per-table settings for autovacuum_analyze_threshold and scale_factor."

🔄 Query Rewriting & Materialized Views

Predicate Pushdown

The optimizer pushes WHERE filters as early as possible — ideally into the scan layer so fewer rows flow through joins and aggregations.

-- The optimizer rewrites this: SELECT * FROM ( SELECT c.name, o.amount FROM customers c JOIN orders o ON c.customer_id = o.customer_id ) sub WHERE order_date > '2024-01-01';
🔽 Pushes the filter down:
 
Filter applied during scan,
not after the join completes.
 
Fewer rows flow through
the entire pipeline → faster.

Join Reordering

The optimizer rearranges join order to minimize intermediate result sizes — starting with the most selective filter produces smaller intermediates.

Materialized Views

A materialized view is a cached query result stored as a physical table.

CREATE MATERIALIZED VIEW mv_daily_revenue AS SELECT DATE_TRUNC('day', order_date) AS day, category, COUNT(*) AS order_count, SUM(amount) AS revenue FROM orders o JOIN products p ON o.product_id = p.product_id GROUP BY DATE_TRUNC('day', order_date), category; -- Concurrent refresh (no lock, needs unique index) CREATE UNIQUE INDEX idx_mv_unique ON mv_daily_revenue (day, category); REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_revenue;

When to Use Materialized Views

✅ Good Use Cases

  • Dashboard aggregations over millions of rows
  • Complex joins that rarely change
  • Pre-computed rollups for BI tools

❌ Bad Use Cases

  • Data that must be real-time
  • High-write-rate tables
  • Simple queries that are already fast
💡 Interview Insight

"For a dashboard querying billions of rows, I'd create materialized views for common aggregation patterns — daily/weekly rollups — refreshed on a schedule with CONCURRENTLY to avoid blocking reads. This turns a 30-second aggregation into a 10ms lookup. For real-time needs, I'd layer a live query over recent data on top."

🧩 Table Partitioning — Range, List, Hash

Why Partition?

Partitioning splits a logical table into physical sub-tables. The database prunes partitions that don't match query filters — scanning 10M rows instead of 500M.

Range Partitioning (Time-Series)

CREATE TABLE orders ( order_id BIGINT, customer_id INT, order_date DATE, amount DECIMAL(10,2) ) PARTITION BY RANGE (order_date); CREATE TABLE orders_2024_01 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2024-02-01'); CREATE TABLE orders_2024_02 PARTITION OF orders FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

List & Hash Partitioning

📋 List Partitioning

Best for categorical data — regions, statuses. PARTITION BY LIST (region) with FOR VALUES IN ('east', 'west').

#️⃣ Hash Partitioning

Guarantees even distribution. PARTITION BY HASH (customer_id) with MODULUS 4, REMAINDER 0.

Partitioning Comparison

TypeBest ForPruning OnEven Distribution
RangeTime-series, datesRange predicatesDepends on data
ListRegions, categoriesIN / = predicatesDepends on data
HashEven distribution needed= predicates✅ Guaranteed

Pitfalls

⚠️ Partition key must be in every unique/PK constraint. Cross-partition queries (no filter on partition key) scan ALL partitions. Too many partitions (>1000) slow planning.

💡 Interview Insight

"I'd partition when a table exceeds tens of millions of rows and queries filter on a specific column. Range by month for time-series — enables partition pruning. Dropping old partitions is instantaneous vs DELETE on millions of rows. Key: partition key must be in all unique constraints, keep count reasonable."

🎯 Key Takeaways

🔬 EXPLAIN ANALYZE

Your #1 diagnostic tool. Read bottom-up, compare estimated vs actual rows, look for Seq Scans on large tables with selective filters.

🌳 Index Types

B-tree handles 90% of cases. GIN for JSONB/arrays/full-text. GiST for spatial. Hash almost never. Know what each supports.

📐 Composite Index Order

ERS rule: Equality → Range → Sort. Leftmost prefix rule: (A, B, C) helps A, A+B, A+B+C — never B or C alone.

📦 Covering Indexes

INCLUDE eliminates heap fetches for high-frequency queries. Query answered entirely from the index.

🤝 Join Algorithms

NL (small × indexed-large), Hash (large × large, no index), Merge (pre-sorted). Non-equi joins → Nested Loop only.

📊 Statistics & ANALYZE

Stale stats = bad plans. Run ANALYZE after bulk changes. Extended stats for correlated columns.

🔄 Materialized Views

Pre-computed results for dashboards. REFRESH CONCURRENTLY avoids blocking. Trade freshness for speed.

🧩 Partitioning

Range for time-series, List for categories, Hash for even distribution. Pruning only helps when filtering on partition key.
