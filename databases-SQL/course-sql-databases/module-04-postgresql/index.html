🗃️ JSONB — The Best of Both Worlds

Product attributes vary wildly — a laptop has RAM and screen size, a shirt has fabric and fit. Creating 200 nullable columns is ugly. An EAV table is a query nightmare. JSONB gives you schema flexibility inside a relational database.

CREATE TABLE products ( product_id SERIAL PRIMARY KEY, name TEXT NOT NULL, category TEXT NOT NULL, price DECIMAL(10,2) NOT NULL, attributes JSONB DEFAULT '{}' ); INSERT INTO products (name, category, price, attributes) VALUES ('MacBook Pro 16"', 'Electronics', 2499.99, '{"ram_gb": 32, "storage_tb": 1, "color": "Space Gray"}'), ('Wool Sweater', 'Clothing', 89.99, '{"size": "L", "color": "Navy", "material": "Merino Wool"}');

JSON vs JSONB — Always Use JSONB

FeatureJSONJSONB
StorageRaw text (preserves whitespace)Binary decomposed
Indexing (GIN)❌ Not supported✅ Full GIN support
Comparison ops❌ No equality✅ =, @>, <@, ?, ?&
Read/query speedSlower (reparse each time)Faster (pre-parsed)

JSONB Operators

-- -> returns JSONB (with quotes) SELECT attributes->'color' FROM products; -- "Space Gray" -- ->> returns TEXT (no quotes) SELECT attributes->>'color' FROM products; -- Space Gray -- @> "contains" — GIN indexable! SELECT * FROM products WHERE attributes @> '{"color": "Navy"}'; -- ? "key exists" SELECT * FROM products WHERE attributes ? 'motor';

Indexing JSONB

-- GIN index: ALL keys and values (most flexible) CREATE INDEX idx_attrs ON products USING GIN (attributes); -- B-tree expression index: specific path (smaller, faster) CREATE INDEX idx_color ON products ((attributes->>'color')); -- Numeric range on JSONB field: CREATE INDEX idx_ram ON products (((attributes->>'ram_gb')::int));
💡 Interview Insight

"I'd use JSONB for truly variable attributes where different rows have fundamentally different fields — like product specs across categories. I keep high-cardinality, JOIN-key columns as regular columns and put the 'long tail' in JSONB. Always index with GIN for containment, and expression indexes for specific paths. The anti-pattern is dumping everything into JSONB — you lose type safety, FK constraints, and optimizer insight."

🧩 Declarative Partitioning & Partition Pruning

Partitioning shines when a table has hundreds of millions of rows and queries filter on a specific column — usually a date.

-- Step 1: Partitioned parent CREATE TABLE orders ( order_id BIGINT NOT NULL, customer_id INT NOT NULL, order_date DATE NOT NULL, amount DECIMAL(10,2) ) PARTITION BY RANGE (order_date); -- Step 2: Create quarterly partitions CREATE TABLE orders_2024_q1 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2024-04-01'); CREATE TABLE orders_2024_q2 PARTITION OF orders FOR VALUES FROM ('2024-04-01') TO ('2024-07-01'); -- Step 3: DEFAULT partition catches strays CREATE TABLE orders_default PARTITION OF orders DEFAULT; -- Step 4: Indexes on parent → auto-created on children CREATE INDEX idx_orders_date ON orders (order_date);

Partition Pruning — Proof It Works

EXPLAIN (ANALYZE, COSTS OFF) SELECT SUM(amount) FROM orders WHERE order_date BETWEEN '2024-04-01' AND '2024-06-30'; -- Output: -- Aggregate -- → Append -- Subplans Removed: 3 ← 3 partitions pruned! -- → Seq Scan on orders_2024_q2 ← only Q2 scanned

Automated Partition Management

CREATE OR REPLACE FUNCTION create_monthly_partition(target_date DATE) RETURNS void AS $$ DECLARE partition_name TEXT; start_date DATE; end_date DATE; BEGIN start_date := DATE_TRUNC('month', target_date); end_date := start_date + INTERVAL '1 month'; partition_name := 'orders_' || TO_CHAR(start_date, 'YYYY_MM'); EXECUTE format( 'CREATE TABLE IF NOT EXISTS %I PARTITION OF orders FOR VALUES FROM (%L) TO (%L)', partition_name, start_date, end_date ); END; $$ LANGUAGE plpgsql;
💡 Interview Insight

"Partitioning's killer feature for data retention: detach and drop a partition — instantaneous regardless of row count. Compare to DELETE on millions of rows which generates dead tuples, bloats the table, and takes hours. I schedule cron jobs to create new partitions ahead of time and detach expired ones."

🧹 VACUUM & AUTOVACUUM — Taming Dead Tuples

Why VACUUM Exists: MVCC

PostgreSQL uses MVCC. UPDATEs don't modify rows in-place — they mark old versions as "dead" and create new ones. Dead tuples pile up.

-- What an UPDATE actually does in the heap: -- Page 42: -- │ order_id=1, status='pending' [xmax=100] DEAD │ ← old, invisible -- │ order_id=1, status='shipped' [xmin=100] LIVE │ ← new, visible

What VACUUM Does

♻️ Reclaim Space

Marks dead tuples as reusable so new inserts can use that space.

👁️ Update Visibility Map

Tells the planner which pages are "all-visible" — enables Index Only Scans.

❄️ Freeze Old Tuples

Prevents transaction ID wraparound (PostgreSQL's XID is 32-bit ≈ 4 billion).

VACUUM vs VACUUM FULL

FeatureVACUUMVACUUM FULL
Lock levelDoes NOT block reads/writesExclusive lock (blocks!)
Space reclaimedWithin existing pagesReturns space to OS
Table rewrittenNoYes (full rewrite)
When to useRegularly / autovacuumExtreme bloat only

Monitoring Bloat

SELECT schemaname, relname, n_live_tup, n_dead_tup, ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct, last_vacuum, last_autovacuum FROM pg_stat_user_tables ORDER BY n_dead_tup DESC LIMIT 10;

AUTOVACUUM Tuning

-- Default: vacuum when dead tuples > 50 + 0.2 × table_size -- 10M row table: 50 + 0.2 × 10M = 2,000,050 dead tuples → 20% bloat! -- Tune for hot tables: ALTER TABLE orders SET ( autovacuum_vacuum_threshold = 1000, autovacuum_vacuum_scale_factor = 0.01, -- 1% instead of 20% autovacuum_analyze_threshold = 500, autovacuum_analyze_scale_factor = 0.005 );
💡 Interview Insight

"Bloat happens when dead tuples accumulate faster than VACUUM reclaims them. I diagnose by checking n_dead_tup in pg_stat_user_tables. For mild bloat, tune autovacuum more aggressively (lower scale_factor). For severe bloat, pg_repack rebuilds the table online without an exclusive lock — unlike VACUUM FULL. Prevention is key: tune autovacuum per-table for high-write workloads."

🔌 Connection Pooling with PgBouncer

The Problem: PostgreSQL's Process Model

PostgreSQL spawns a dedicated OS process per connection (~5–10 MB RAM each). 500 connections = 2.5–5 GB just for connection overhead.

❌ Without Pooling

  • 500 app threads → 500 backend processes
  • ~5 GB RAM overhead
  • OS context-switching nightmare

✅ With PgBouncer

  • 500 app threads → PgBouncer → 20 backends
  • ~200 MB RAM overhead
  • Time-shared connections

Pool Modes

ModeMultiplexingSession StateUse Case
SessionLow✅ FullLegacy apps
Transaction ⭐High❌ Per-txn onlyMost web apps
StatementMaximum❌ NoneSimple read queries

Key Configuration

; pgbouncer.ini [pgbouncer] pool_mode = transaction max_client_conn = 1000 ; max clients to PgBouncer default_pool_size = 20 ; backends per db/user pair min_pool_size = 5 ; keep 5 backends warm reserve_pool_size = 5 ; extra backends for burst query_wait_timeout = 120 ; max wait for a backend

Monitoring

-- Connect to PgBouncer admin console SHOW POOLS; -- cl_waiting > 0 → clients waiting! Increase pool_size -- sv_active maxed → pool saturated -- sv_idle high → pool oversized, reduce to save resources
💡 Interview Insight

"10K concurrent connections? Direct connections would OOM. PgBouncer in transaction mode with ~50-100 backends. Web requests hold a connection for milliseconds, so 50 backends serve 10K clients via time-sharing. Monitor cl_waiting — if clients queue, increase pool_size cautiously. For read-heavy workloads, add read replicas behind a separate pool."

📈 pg_stat_statements — Finding Slow Queries

Enable It

-- postgresql.conf: -- shared_preload_libraries = 'pg_stat_statements' CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

The Query Performance Dashboard

-- Top 10 queries by total execution time SELECT LEFT(query, 80) AS query_preview, calls, ROUND(total_exec_time::numeric, 2) AS total_time_ms, ROUND(mean_exec_time::numeric, 2) AS avg_time_ms, rows, ROUND((100.0 * shared_blks_hit / NULLIF(shared_blks_hit + shared_blks_read, 0))::numeric, 2) AS cache_hit_pct FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10;

Combining with pg_stat_user_tables

-- Find tables with sequential scan problems SELECT relname, seq_scan, idx_scan, ROUND(100.0 * idx_scan / NULLIF(seq_scan + idx_scan, 0), 1) AS idx_pct, n_live_tup FROM pg_stat_user_tables WHERE n_live_tup > 10000 ORDER BY seq_scan DESC LIMIT 10; -- High seq_scan + low idx_pct = needs indexes!
💡 Interview Insight

"Step 1: Enable pg_stat_statements, sort by total_exec_time. Step 2: EXPLAIN ANALYZE each top offender. Step 3: Check pg_stat_user_tables for high seq_scan counts. Step 4: Check cache_hit_pct — if low, working set exceeds shared_buffers. I focus on total time, not just average, because a 1ms query called 10M times is worse than a 10s query called once."

🔒 Locking — Advisory, Row-Level & Deadlocks

Row-Level Locking with SELECT FOR UPDATE

-- Two checkout processes grab the same inventory item BEGIN; SELECT quantity FROM inventory WHERE product_id = 42 FOR UPDATE; -- Locks row → other processes WAIT here UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 42; COMMIT; -- FOR UPDATE variants: FOR UPDATE -- Exclusive lock, wait if locked FOR UPDATE NOWAIT -- Error immediately if locked FOR UPDATE SKIP LOCKED -- Skip locked rows (job queues!) FOR SHARE -- Shared lock (allow other readers)

SKIP LOCKED — Building a Job Queue

-- Worker grabs next available job BEGIN; SELECT job_id, payload FROM job_queue WHERE status = 'pending' ORDER BY created_at LIMIT 1 FOR UPDATE SKIP LOCKED; -- If another worker locked a job, this skips it! COMMIT;

Advisory Locks

Advisory locks are application-level coordination primitives.

-- Prevent duplicate cron job execution SELECT pg_try_advisory_lock(hashtext('nightly_report')); -- Returns false if another instance already running

Deadlock Prevention

📋 Consistent Order

Always lock rows in the same order (e.g., ascending PK) across all transactions.

⏱️ Keep Txns Short

Acquire locks late, release early. Less time holding = less chance of deadlock.

⚡ Fail Fast

Use NOWAIT or SET lock_timeout = '5s' instead of waiting indefinitely.

💡 Interview Insight

"To prevent double-charging: SELECT ... FOR UPDATE on the inventory row for an exclusive lock before decrementing. For distributed systems, combine with an advisory lock on order ID hash to prevent duplicate submission. Set lock_timeout to fail fast. Key principle: pessimistic locking for inventory, idempotency keys for payments."

📥 COPY — Bulk Loading at Wire Speed

INSERT vs COPY Performance

🐌 INSERT (row by row)

10M rows ≈ 45 minutes. Per-statement parsing, planning, and executor overhead for every row.

🚀 COPY (binary pipe)

10M rows ≈ 45 seconds. Compact binary protocol with minimal per-row overhead. 60× faster.

COPY Syntax

-- Load from CSV COPY orders (order_id, customer_id, order_date, amount, status) FROM '/data/exports/orders_2024.csv' WITH (FORMAT csv, HEADER true, DELIMITER ','); -- Export to CSV COPY (SELECT * FROM orders WHERE order_date >= '2024-01-01') TO '/tmp/recent_orders.csv' WITH (FORMAT csv, HEADER true);

Optimization Checklist for Bulk Loads

-- 1. Drop indexes before load DROP INDEX idx_orders_date; -- 2. Disable triggers ALTER TABLE orders DISABLE TRIGGER ALL; -- 3. Increase maintenance_work_mem SET maintenance_work_mem = '1GB'; -- 4. Load the data COPY orders FROM '/data/orders_bulk.csv' WITH (FORMAT csv, HEADER true); -- 5. Rebuild indexes (faster than incremental) CREATE INDEX idx_orders_date ON orders (order_date); -- 6. Re-enable triggers + update stats ALTER TABLE orders ENABLE TRIGGER ALL; ANALYZE orders;

COPY vs \copy

COPY (server-side)

  • File must be on the PostgreSQL server
  • Runs as the postgres OS user
  • Fastest option

\copy (client-side)

  • File is on the client machine
  • Streams through psql connection
  • Slightly slower, more convenient
💡 Interview Insight

"100M rows? COPY, not INSERT. Before: drop indexes, disable triggers, bump maintenance_work_mem. Split file and load partitions in parallel with multiple COPY sessions. After: rebuild indexes, re-enable triggers, ANALYZE. This achieves 500K+ rows/second. For one-time migrations, temporarily set wal_level=minimal and fsync=off (with crash-reload understanding)."

🎯 Key Takeaways

🗃️ JSONB

Document flexibility in PostgreSQL. Use -> for JSON, ->> for text, @> for containment. GIN index for flexible queries, expression index for specific paths.

🧩 Partitioning

Declarative partitioning enables partition pruning. Range by date for time-series. Detach+drop for instant data purge. Always create a DEFAULT partition.

🧹 VACUUM

Reclaims dead tuples from MVCC. Autovacuum defaults are too conservative for hot tables — tune scale_factor to 0.01. Use pg_repack over VACUUM FULL.

🔌 PgBouncer

Transaction mode multiplexes hundreds of connections onto a small pool. Session-level state (SET, prepared stmts) doesn't persist across transactions.

📈 pg_stat_statements

Sort by total_exec_time for highest impact targets. Combine with pg_stat_user_tables for seq scan detection.

🔒 Locking

FOR UPDATE serializes concurrent access. SKIP LOCKED for job queues. Advisory locks for app-level coordination. Consistent lock ordering prevents deadlocks.

📥 COPY

10-100× faster than INSERT. Drop indexes before, rebuild after. \copy for client-side, COPY for server-side. Always ANALYZE after bulk loads.
