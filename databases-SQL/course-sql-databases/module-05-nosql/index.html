🔑 DynamoDB — Key-Value at Planet Scale

DynamoDB is a fully managed key-value + document database from AWS. Think of it as a distributed hash map where every item is identified by a primary key and stored across partitions based on that key.

Partition Key + Sort Key

The partition key (PK) determines data distribution. The sort key (SK) enables range queries within a partition.

Table: orders PK: customer_id SK: order_date#order_id ┌─────────────┬────────────────────┬─────────┬──────────┐ │ customer_id │ order_date#oid │ amount │ status │ │ (PK) │ (SK) │ │ │ ├─────────────┼────────────────────┼─────────┼──────────┤ │ CUST#001 │ 2024-01-15#ORD-101 │ 59.99 │ shipped │ │ CUST#001 │ 2024-02-20#ORD-205 │ 124.50 │ pending │ │ CUST#001 │ 2024-03-08#ORD-312 │ 89.00 │ shipped │ │ CUST#002 │ 2024-01-03#ORD-055 │ 250.00 │ delivered│ │ CUST#002 │ 2024-04-11#ORD-401 │ 34.99 │ pending │ └─────────────┴────────────────────┴─────────┴──────────┘

Query Capabilities

# Get all orders for a customer (efficient — queries ONE partition) table.query( KeyConditionExpression=Key('customer_id').eq('CUST#001') ) # Get orders for a customer in a date range (sort key range) table.query( KeyConditionExpression=( Key('customer_id').eq('CUST#001') & Key('order_date_oid').between('2024-01-01', '2024-03-31') ) ) # ❌ This is a SCAN (reads entire table — expensive!): # "Find all orders with status = 'pending'" without an index

Global Secondary Indexes (GSIs)

A GSI is a full copy of the table with a different key schema, maintained automatically by DynamoDB.

Base Table: PK=customer_id, SK=order_date#order_id GSI-1 (by status): PK=status, SK=order_date GSI-2 (by product): PK=product_id, SK=order_date "All pending orders" → Query GSI-1 where PK='pending' ✅ "Orders for product Y" → Query GSI-2 ✅
⚠️ GSI Gotchas

Each GSI doubles your write cost (DynamoDB replicates writes). GSIs are eventually consistent. Max 20 per table. A hot partition on a GSI can throttle writes to the base table.

Single-Table Design Pattern

Instead of multiple tables, DynamoDB experts put everything in one table using prefixed keys:

┌────────────┬───────────────────┬───────────────────────────┐ │ PK │ SK │ Attributes │ ├────────────┼───────────────────┼───────────────────────────┤ │ CUST#001 │ PROFILE │ name, email, created_at │ │ CUST#001 │ ORD#2024-01-15 │ amount, status, items │ │ PROD#A100 │ METADATA │ name, price, category │ │ PROD#A100 │ REVIEW#2024-01-20 │ rating, comment, user_id │ └────────────┴───────────────────┴───────────────────────────┘ Customer profile: PK='CUST#001', SK='PROFILE' Customer orders: PK='CUST#001', SK begins_with('ORD#') Product reviews: PK='PROD#A100', SK begins_with('REVIEW#')
💡 Interview Gold

"DynamoDB design is access-pattern-first — the opposite of SQL where you normalize first and query later. I start by listing every access pattern, then work backward to design keys. The partition key should distribute data evenly, and the sort key enables the range queries I need. The cardinal sin is designing DynamoDB like a relational database — you'll end up with scans everywhere."

📄 MongoDB — Documents & Aggregation

MongoDB stores data as BSON documents (binary JSON). Each document is schema-flexible — different documents in the same collection can have different fields.

// orders collection { _id: ObjectId("65a1b2c3d4e5f6a7b8c9d0e1"), customer: { id: "CUST-001", name: "Alice Chen", email: "alice@example.com" }, items: [ { product_id: "PROD-A100", name: "Wireless Mouse", qty: 2, price: 29.99 }, { product_id: "PROD-B200", name: "USB-C Hub", qty: 1, price: 49.99 } ], total: 109.97, status: "shipped", order_date: ISODate("2024-03-15T14:30:00Z") }

Key difference from SQL: Related data (customer info, line items, shipping) is embedded in a single document — no JOINs needed.

When to Embed vs Reference

✅ Embed (denormalize)

  • 1:1 or 1:few relationships
  • Data always read together
  • Rarely changes after creation
  • Under 16 MB document limit
  • Ex: Order ← embed → line items

🔗 Reference (normalize)

  • 1:many or many:many relationships
  • Data read independently
  • Changes frequently
  • Risk of exceeding 16 MB limit
  • Ex: Product ← reference → reviews

Aggregation Pipeline

A sequence of stages that transform documents — conceptually like Unix pipes.

// Top 5 customers by spend in Q1 2024 db.orders.aggregate([ { $match: { order_date: { $gte: ISODate("2024-01-01"), $lt: ISODate("2024-04-01") } }}, { $group: { _id: "$customer.id", total_spend: { $sum: "$total" }, order_count: { $sum: 1 } }}, { $sort: { total_spend: -1 } }, { $limit: 5 } ]);
📝 "Top 5 customers by total spend"
 
🔍 Stage 1: Filter to Q1 2024 only
 
 
 
📊 Stage 2: Group by customer, sum totals
 
 
 
📈 Stage 3: Sort by spend descending
✂️ Stage 4: Take top 5

$unwind — Exploding Arrays

// Which products generate the most revenue? db.orders.aggregate([ { $unwind: "$items" }, // One doc per item (like UNNEST) { $group: { _id: "$items.product_id", product_name: { $first: "$items.name" }, total_revenue: { $sum: { $multiply: ["$items.qty", "$items.price"] } } }}, { $sort: { total_revenue: -1 } }, { $limit: 10 } ]);

MongoDB Indexes

// Single-field index db.orders.createIndex({ "customer.id": 1 }); // Compound index (same leftmost-prefix rules as SQL!) db.orders.createIndex({ status: 1, order_date: -1 }); // Multikey index (auto-indexes array elements) db.orders.createIndex({ "items.product_id": 1 }); // TTL index (auto-delete after expiration — perfect for sessions!) db.sessions.createIndex({ created_at: 1 }, { expireAfterSeconds: 3600 });
💡 Interview Gold

"I'd choose MongoDB when data is naturally document-shaped and varies across records — product catalogs, user-generated content, IoT events. Embedded documents eliminate JOINs for read-heavy patterns. But I'd stick with PostgreSQL when I need ACID transactions across entities, complex joins, or strong schema enforcement. With PostgreSQL's JSONB, many document use cases can be handled without leaving the relational world."

⚡ Redis — The In-Memory Swiss Army Knife

Redis is an in-memory data structure store. The "value" in each key-value pair can be one of several rich data structures.

📝 STRING

Simple key→value. Caching, counters, distributed locks. SET, GET, INCR.

📋 HASH

Field→value maps (like a row). HSET, HGET, HGETALL. More memory-efficient than serialized JSON.

📜 LIST

Ordered sequence. Queues, activity feeds. LPUSH, RPOP, BRPOP (blocking pop).

🎯 SET

Unique members. Intersections, unions, membership tests. SADD, SINTER, SUNION.

🏆 SORTED SET

Members ordered by score. Leaderboards, rankings, rate limiting. ZADD, ZREVRANGE, ZINCRBY.

📡 STREAM

Append-only event log with consumer groups. Like lightweight Kafka. XADD, XREADGROUP.

Strings — Caching & Counters

-- Cache a serialized object with TTL SET product:42:details '{"name":"Wireless Mouse","price":29.99}' EX 3600 GET product:42:details -- Atomic counter (page views, inventory) INCR product:42:view_count -- 1, 2, 3, ... INCRBY product:42:stock -1 -- decrement stock atomically -- Distributed lock (simplified) SET lock:checkout:order-789 "worker-3" NX EX 30 -- NX = only set if not exists, EX 30 = expire in 30 seconds

Sorted Sets — Leaderboards & Rate Limiting

-- Leaderboard: members ordered by score ZADD leaderboard 9500 "alice" 8200 "bob" 7800 "charlie" ZREVRANGE leaderboard 0 2 WITHSCORES -- top 3 ZINCRBY leaderboard 500 "charlie" -- charlie now 8300, overtakes bob! -- Rate limiting: score = timestamp, member = request ID ZADD ratelimit:user:1001 1711987200 "req-abc" ZCOUNT ratelimit:user:1001 (NOW-60) +inf -- requests in last 60s ZREMRANGEBYSCORE ratelimit:user:1001 0 (NOW-60) -- cleanup old
💡 Interview Gold

"For a real-time leaderboard, Redis Sorted Sets are perfect. On each sale: ZINCRBY leaderboard:daily <revenue> <seller_id>. Top 10: ZREVRANGE leaderboard:daily 0 9 WITHSCORES. O(log N) updates, O(log N + M) ranges — sub-millisecond with millions of sellers."

🧩 Redis Patterns — Caching, Pub/Sub & Streams

Cache-Aside (Lazy Loading)

Request for product:42 ┌──────────┐ 1. GET product:42 ┌──────────┐ │ │ ◄──────────────────────│ │ │ App │ Cache MISS (nil) │ Redis │ │ Server │ │ Cache │ │ │ 2. SELECT * FROM ┌──────────┐ │ │ ─────────────────────→ │PostgreSQL│ │ │ ← result ───────────│ │ │ │ 3. SET product:42 └──────────┘ │ │ ─────────────────────→ Redis (EX 3600) └──────────┘ Next request → Cache HIT → skip database entirely!
def get_product(product_id: int) -> dict: cache_key = f"product:{product_id}" # 1. Try cache first cached = redis.get(cache_key) if cached: return json.loads(cached) # 2. Cache miss — query database product = db.execute("SELECT * FROM products WHERE id = %s", product_id) # 3. Populate cache with TTL redis.set(cache_key, json.dumps(product), ex=3600) return product

Eviction Policies

🔥 Common Policies

  • allkeys-lru — evict least recently used (general caching)
  • allkeys-lfu — evict least frequently used
  • volatile-lru — LRU only among keys with TTL

📋 Others

  • noeviction — reject writes (critical data)
  • volatile-ttl — evict closest to expiration
  • allkeys-random — random eviction

Pub/Sub vs Streams

📢 Pub/Sub

  • Fire-and-forget broadcasting
  • ⚠️ Messages LOST if no subscriber listening
  • No persistence or replay
  • No consumer groups

📡 Streams

  • Append-only log (like lightweight Kafka)
  • ✅ Messages persist until deleted
  • Consumer groups + load balancing
  • Message acknowledgement (XACK)
💡 Interview Gold

"Cache invalidation strategies: (1) TTL-based — simple, for slightly-stale-OK data. (2) Write-through — delete cache key on every DB write. (3) Event-driven — CDC emits events, consumer refreshes cache. I prevent cache stampedes with mutex locks or probabilistic early expiration."

🕸️ Neo4j — Graphs & Cypher

Relational databases handle graphs poorly — recursive self-joins with unknown depth, exponential query complexity. Graph databases store relationships as first-class citizens.

👥 Social Networks

"Friends of friends who bought X"

🎯 Recommendations

"Customers who bought A also bought B"

🔍 Fraud Detection

"Find circular money flows"

🌐 Knowledge Graphs

"What entities are related to X?"

The Property Graph Model

Nodes (entities) have labels and properties: (:Customer {id: "C001", name: "Alice", tier: "gold"}) (:Product {id: "P100", name: "Wireless Mouse", price: 29.99}) Relationships have types, direction, and properties: (:Customer)-[:PURCHASED {date: "2024-03-15", qty: 2}]->(:Product) (:Customer)-[:REVIEWED {rating: 5}]->(:Product) Visual: (Alice)──PURCHASED──→(Wireless Mouse)──BELONGS_TO──→(Electronics) │ ↑ └──REVIEWED {★★★★★}──────┘

Cypher Query Language

// Find all products Alice purchased MATCH (c:Customer {name: "Alice"}) -[:PURCHASED]->(p:Product) RETURN p.name, p.price; // "Customers who bought X also bought..." MATCH (target:Product {name: "Wireless Mouse"}) <-[:PURCHASED]-(cust) -[:PURCHASED]->(rec:Product) WHERE rec <> target RETURN rec.name, COUNT(cust) AS co_purchases ORDER BY co_purchases DESC LIMIT 5;
📦 Find Alice's purchases
🔗 Traverse PURCHASED edges
 
📋 Return product name & price
 
🎯 Collaborative filtering!
← who bought the mouse?
→ what else did they buy?
🚫 Exclude the mouse itself
📊 Rank by co-purchase count
 
✂️ Top 5 recommendations

Cypher vs SQL — Variable Depth

🐘 SQL (friends-of-friends)

  • Requires recursive CTEs or N self-joins
  • Gets ugly at 3+ hops, impossible at 6+
  • Each hop = full-table join

🕸️ Cypher

  • -[:KNOWS*2]->(fof) — 2 hops
  • Change *2 to *6 — same simplicity!
  • O(1) per hop via pointer traversal
💡 Interview Gold

"I'd use a graph database when the core value is in the relationships, not the entities — and queries need variable-depth traversals. Classic examples: social networks, recommendations, fraud detection. The key indicator is when SQL needs recursive CTEs or multiple self-joins. Graph databases make multi-hop traversals O(1) per hop. But for CRUD, aggregations, and reporting — SQL is still king."

🧭 SQL vs NoSQL — The Decision Framework

It's not SQL or NoSQL — it's SQL and NoSQL. Most production systems use polyglot persistence.

Decision Flowchart

🔍 Complex queries, JOINs, ad-hoc analysis?

→ PostgreSQL. No question.

🚀 Simple key lookups at 100K+ req/s?

→ DynamoDB / Cassandra. Design keys carefully.

📄 Flexible documents, varying schemas?

→ MongoDB (or PostgreSQL JSONB if you also need ACID).

⚡ Sub-ms latency for hot data?

→ Redis as a cache in front of your primary DB.

🕸️ Relationship traversals?

→ Neo4j / Neptune. Friends-of-friends, recommendations, fraud.

🔎 Full-text search with relevance?

→ Elasticsearch (or PostgreSQL full-text for simpler needs).

Common Anti-Patterns

❌ "We chose MongoDB because it's web-scale"

Then needed JOINs, transactions, complex reports. Should have used PostgreSQL.

❌ "Everything in Redis for speed"

Redis is RAM-bound. 500GB in Redis costs a fortune. Use it as a CACHE, not primary storage.

❌ "Normalized DynamoDB"

12 tables with "joins" in app code = slow & expensive. Use single-table design or switch to SQL.

❌ "Time-series in a regular SQL table"

10B rows, queries timing out. Use TimescaleDB, InfluxDB, or partitioned tables.

💡 Interview Gold

"I default to PostgreSQL — it handles 90% of use cases including JSONB for documents. I add specialized stores only when PostgreSQL becomes a bottleneck for specific access patterns. Three questions guide my choice: (1) Access patterns — ad-hoc or predefined? (2) Consistency — strict ACID or eventual OK? (3) Scale profile — read-heavy, write-heavy, or both?"

🎯 Key Takeaways

🔑 DynamoDB

Access-pattern-first design. Partition key for distribution, sort key for range queries. GSIs for additional patterns. Single-table design packs related entities with prefixed keys.

📄 MongoDB

Embed related data to avoid JOINs. Embed for 1:few read-together; reference for 1:many or frequently-updated. Aggregation pipeline maps to SQL concepts on documents.

⚡ Redis

A data structure server, not just cache. Strings for caching, Hashes for objects, Lists for queues, Sets for membership, Sorted Sets for leaderboards. Choose the right structure.

🕸️ Neo4j

Excels at relationship-heavy queries. Cypher makes variable-depth traversals trivial (*2..6) compared to recursive CTEs in SQL.

🏗️ The Golden Rule

Default to PostgreSQL for new projects — it handles relational data, JSONB documents, full-text search, and even time-series. Add specialized NoSQL stores only when you hit a specific bottleneck. Polyglot persistence is the norm in production: PostgreSQL as source of truth, Redis for caching, and maybe DynamoDB or Elasticsearch for specific high-scale access patterns.
