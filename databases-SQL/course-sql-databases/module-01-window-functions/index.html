🪟 What Are Window Functions?

Imagine you work at an e-commerce company. Your manager asks: "Show me each order alongside the customer's total lifetime spend." Without window functions, you'd write something ugly — a subquery or a self-join:

-- The painful way (correlated subquery) SELECT o.order_id, o.customer_id, o.amount, (SELECT SUM(o2.amount) FROM orders o2 WHERE o2.customer_id = o.customer_id) AS lifetime_spend FROM orders o;

This fires a subquery for every single row. On a million-row table, that's a million subqueries. Window functions solve this elegantly:

-- The window function way SELECT order_id, customer_id, amount, SUM(amount) OVER (PARTITION BY customer_id) AS lifetime_spend FROM orders;

The Mental Model: A Window Into Your Data

Think of a window function as a sliding glass panel that you hold up against your result set. Each row can "see" other rows — its neighbors, its group, or the entire table — and compute something based on what it sees. Crucially, the row count never changes. Unlike GROUP BY, which collapses rows, window functions add columns to existing rows.

┌─────────────────────────────────────────────────────┐ │ Regular Query Result Set (all rows preserved) │ │ ┌─────────┐ │ │ │ Window │ ← Each row peeks through its window │ │ │ Frame │ to see neighboring rows and compute │ │ └─────────┘ aggregates WITHOUT collapsing them │ └─────────────────────────────────────────────────────┘

Anatomy of a Window Function Call

function_name(args) OVER ( [PARTITION BY column1, column2, ...] -- Defines groups (like GROUP BY but no collapse) [ORDER BY column3 ASC/DESC, ...] -- Defines ordering within each partition [frame_clause] -- Defines which rows in the partition to consider )
ComponentPurposeRequired?
PARTITION BYSplits rows into groupsOptional
ORDER BYOrders rows within partitionDepends

💡 Interview Insight

"What's the difference between GROUP BY and window functions?" — "GROUP BY collapses rows into groups and returns one row per group. Window functions compute across a set of rows related to the current row but preserve every row in the output. This means I can show per-row detail alongside aggregate calculations — like showing each order's amount next to the customer's average order value — without self-joining or subquerying."

🏅 ROW_NUMBER, RANK, and DENSE_RANK

These three ranking functions are the bread and butter of SQL interviews. They all assign numbers to rows within a partition, but handle ties differently.

SELECT product_id, category, revenue, ROW_NUMBER() OVER (PARTITION BY category ORDER BY revenue DESC) AS row_num, RANK() OVER (PARTITION BY category ORDER BY revenue DESC) AS rank_val, DENSE_RANK() OVER (PARTITION BY category ORDER BY revenue DESC) AS dense_rank_val FROM product_sales;

The Analogy: A Race Finish

🏁 ROW_NUMBER()

The bib number at the finish line. Even if two runners cross at the same time, one gets #1 and the other #2. No ties, ever. Non-deterministic for ties.

🥇 RANK()

The Olympic medal system. Two tie for gold? Both get rank 1, next gets rank 3. Gaps appear after ties.

🏆 DENSE_RANK()

The podium step system. Two tie for 1st? They share step 1, next gets step 2. No gaps, ever.

Values: 100 100 90 80 80 70 │ │ │ │ │ │ ROW_NUM: 1 2 3 4 5 6 ← Always sequential, no gaps, no ties RANK: 1 1 3 4 4 6 ← Ties share rank, gaps after ties DENSE_RANK:1 1 2 3 3 4 ← Ties share rank, NO gaps

Classic Interview Pattern: Top-N Per Group

"Find the top 2 highest-revenue products per category."

WITH ranked AS ( SELECT product_id, category, revenue, ROW_NUMBER() OVER (PARTITION BY category ORDER BY revenue DESC) AS rn FROM product_sales ) SELECT product_id, category, revenue FROM ranked WHERE rn <= 2;

Why ROW_NUMBER here and not RANK? If two products tie for #2, RANK would return both (giving you 3 results). ROW_NUMBER guarantees exactly 2 rows per category. The choice depends on the business requirement — ask your interviewer!

💡 Interview Insight

"When would you choose DENSE_RANK over RANK?" — "I'd use DENSE_RANK when I care about the number of distinct ranking levels rather than positions. For example, if asked 'find products with the 3rd highest revenue,' DENSE_RANK ensures I find that 3rd tier even when ties push RANK's values past 3."

↔️ LEAD, LAG, and Gap Analysis

LAG() and LEAD() let each row reach out and grab values from adjacent rows in the ordered partition. Think of standing in a line and glancing at the person ahead or behind you.

LAG(column, offset, default) OVER (PARTITION BY ... ORDER BY ...) LEAD(column, offset, default) OVER (PARTITION BY ... ORDER BY ...) -- offset: how many rows back (LAG) or forward (LEAD). Default: 1 -- default: value when no row exists (first/last row). Default: NULL

Month-over-Month Revenue Change

SELECT month, revenue, LAG(revenue, 1) OVER (ORDER BY month) AS prev_month_revenue, revenue - LAG(revenue, 1) OVER (ORDER BY month) AS mom_change, ROUND( 100.0 * (revenue - LAG(revenue, 1) OVER (ORDER BY month)) / LAG(revenue, 1) OVER (ORDER BY month), 2 ) AS mom_pct_change FROM monthly_revenue;

YoY Comparison

SELECT month, revenue, LAG(revenue, 12) OVER (ORDER BY month) AS same_month_last_year, ROUND( 100.0 * (revenue - LAG(revenue, 12) OVER (ORDER BY month)) / NULLIF(LAG(revenue, 12) OVER (ORDER BY month), 0), 2 ) AS yoy_pct_change FROM monthly_revenue;

Pro tip: Always wrap the denominator in NULLIF(..., 0) to avoid division-by-zero errors. Interviewers love to see that defensive coding habit.

Gap Analysis: Finding Missing Sequences

"Find gaps in order IDs (missing order numbers)."

-- ❌ WRONG: Can't use window functions in WHERE! SELECT order_id, LEAD(order_id) OVER (ORDER BY order_id) - order_id AS gap_size FROM orders WHERE LEAD(order_id) OVER (ORDER BY order_id) - order_id > 1;
-- ✅ CORRECT: Wrap in CTE, then filter WITH order_gaps AS ( SELECT order_id, LEAD(order_id) OVER (ORDER BY order_id) AS next_order_id FROM orders ) SELECT order_id, next_order_id, (next_order_id - order_id) AS gap_size FROM order_gaps WHERE next_order_id - order_id > 1;
💡 Interview Insight

"Can you use a window function in a WHERE clause?" — "No. Window functions are evaluated after WHERE, HAVING, and GROUP BY — they're in the same phase as SELECT. To filter on a window function result, I wrap the query in a CTE or subquery. The SQL logical processing order is: FROM → WHERE → GROUP BY → HAVING → SELECT (window functions here) → ORDER BY → LIMIT."

🎯 FIRST_VALUE, LAST_VALUE, NTH_VALUE

The Trap: Default Frame Specification

These functions return a specific row's value from the window frame. But here's where most people get burned — the default frame.

When ORDER BY is specified, the default frame is ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. This means LAST_VALUE() doesn't give you the last value in the partition — it gives you the current row. This is the single most common window function bug.

-- ❌ BUGGY: LAST_VALUE sees only up to current row SELECT order_id, customer_id, order_date, amount, FIRST_VALUE(amount) OVER (PARTITION BY customer_id ORDER BY order_date) AS first_order_amt, LAST_VALUE(amount) OVER (PARTITION BY customer_id ORDER BY order_date) AS last_order_amt -- BUG! FROM orders;
-- ✅ FIXED: Extend the frame to the entire partition SELECT order_id, customer_id, order_date, amount, FIRST_VALUE(amount) OVER ( PARTITION BY customer_id ORDER BY order_date ) AS first_order_amt, LAST_VALUE(amount) OVER ( PARTITION BY customer_id ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING ) AS last_order_amt FROM orders;
FunctionReturnsFrame Matters?Common Gotcha
FIRST_VALUE()Value from first row in frameLess so*Usually works with default
LAST_VALUE()Value from last row in frameYESMust extend frame!
NTH_VALUE(n)Value from nth row in frameYESReturns NULL if < n rows

*FIRST_VALUE usually works because the default frame starts at UNBOUNDED PRECEDING, which includes the first row.

Named Windows — DRY Principle

SELECT DISTINCT customer_id, FIRST_VALUE(product_name) OVER w AS first_product_bought, LAST_VALUE(product_name) OVER w AS latest_product_bought, FIRST_VALUE(order_date) OVER w AS first_purchase_date, LAST_VALUE(order_date) OVER w AS latest_purchase_date FROM orders JOIN products USING (product_id) WINDOW w AS ( PARTITION BY customer_id ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING );

Pro tip: Use the WINDOW clause to define a named window and reuse it. Avoids repeating the same OVER(...) five times — cleaner and less error-prone.

💡 Interview Insight

"What's the default frame when ORDER BY is present?" — "RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. This means aggregate window functions compute a running total by default, and LAST_VALUE() will just return the current row's value — not the actual last row. To fix this, I explicitly specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING."

🖼️ Frame Specification: ROWS vs RANGE vs GROUPS

The frame clause defines exactly which rows the window function considers. This separates good SQL engineers from great ones.

ROWS | RANGE | GROUPS BETWEEN { UNBOUNDED PRECEDING | n PRECEDING | CURRENT ROW } AND { UNBOUNDED FOLLOWING | n FOLLOWING | CURRENT ROW }

ROWS — Physical Row Count

-- 3-day moving average (exactly 3 rows) SELECT order_date, daily_revenue, AVG(daily_revenue) OVER ( ORDER BY order_date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW ) AS moving_avg_3day FROM daily_sales;
Row 1: AVG of [Row 1] = Row 1 value Row 2: AVG of [Row 1, Row 2] = average of 2 Row 3: AVG of [Row 1, Row 2, Row 3] = average of 3 ← full window Row 4: AVG of [Row 2, Row 3, Row 4] = average of 3 ← slides forward

RANGE — Logical Value Range

-- Include all rows with revenue within ±100 of current value SELECT order_id, revenue, COUNT(*) OVER ( ORDER BY revenue RANGE BETWEEN 100 PRECEDING AND 100 FOLLOWING ) AS nearby_count FROM orders;

Side-by-Side Comparison

Data (ORDER BY value): 10, 20, 20, 20, 30, 40 Current row: 20 (second occurrence), frame = "1 PRECEDING AND 1 FOLLOWING" ROWS: [20, 20, 20] ← 1 physical row before + current + 1 after RANGE: [20, 20, 20] ← all values within 19..21 GROUPS: [10, 20, 20, 20, 30] ← 1 peer group before + current group + 1 after
Frame TypeBest ForHandles Ties
ROWSFixed-size sliding windowsIgnores ties
RANGEValue-based ranges, date intervalsGroups peers
GROUPSPeer-group-based windowsGroups peers
💡 Interview Insight

"What's the difference between ROWS and RANGE?" — "ROWS counts physical rows — 2 PRECEDING means exactly 2 rows before. RANGE operates on logical values — 2 PRECEDING means all rows whose ORDER BY value is within 2 of the current row's. The key difference shows up with duplicates: RANGE treats all peers as a single unit, while ROWS treats each as individual. For moving averages with date gaps, use RANGE; for fixed-size sliding windows, use ROWS."

📊 Running Aggregates: SUM, AVG, COUNT OVER

Running Totals

The most common analytics interview question: "Calculate a running total of revenue by date."

SELECT order_date, daily_revenue, SUM(daily_revenue) OVER (ORDER BY order_date) AS running_total, SUM(daily_revenue) OVER ( ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_total_explicit -- Same thing, just explicit FROM daily_sales;

Moving Averages

-- 7-day moving average SELECT order_date, daily_revenue, ROUND(AVG(daily_revenue) OVER ( ORDER BY order_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ), 2) AS moving_avg_7day, MIN(daily_revenue) OVER ( ORDER BY order_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS rolling_min_7day FROM daily_sales;

Cumulative Percentage

"What percentage of total revenue has been earned by each date?"

SELECT order_date, daily_revenue, SUM(daily_revenue) OVER (ORDER BY order_date) AS running_total, ROUND( 100.0 * SUM(daily_revenue) OVER (ORDER BY order_date) / SUM(daily_revenue) OVER (), -- Empty OVER = grand total 2 ) AS cumulative_pct FROM daily_sales;

The trick: SUM(...) OVER () with an empty OVER clause gives the grand total across the entire result set — perfect for computing percentages without a self-join.

💡 Interview Insight

"How do you compute a running total that resets each month?" — "I'd partition by the month and order by date within each partition. The running total automatically resets at each partition boundary: SUM(amount) OVER (PARTITION BY DATE_TRUNC('month', order_date) ORDER BY order_date)"

📈 PERCENT_RANK, CUME_DIST, NTILE

PERCENT_RANK — Relative Standing

PERCENT_RANK() returns the relative rank as a percentage: (rank - 1) / (total_rows - 1). First row = 0, last row = 1.

SELECT customer_id, total_spend, PERCENT_RANK() OVER (ORDER BY total_spend) AS pct_rank FROM customer_lifetime_value;

CUME_DIST — Cumulative Distribution

CUME_DIST() returns the fraction of rows ≤ the current row: count(values ≤ current) / total_rows. Always > 0, last row = 1.

Key Difference

📊 PERCENT_RANK

  • Formula: (rank-1)/(n-1)
  • Min value: 0
  • Meaning: "% of rows ranked lower"

📈 CUME_DIST

  • Formula: count(≤ current)/n
  • Min value: 1/n (never 0)
  • Meaning: "% of rows ≤ this value"

NTILE — Divide Into Buckets

NTILE(n) divides the ordered partition into n roughly equal buckets. Perfect for percentiles and segments.

-- Segment customers for marketing SELECT customer_id, total_spend, CASE NTILE(4) OVER (ORDER BY total_spend DESC) WHEN 1 THEN 'VIP (Top 25%)' WHEN 2 THEN 'High Value' WHEN 3 THEN 'Medium Value' WHEN 4 THEN 'Low Value' END AS customer_segment FROM customer_lifetime_value;

Caveat: NTILE distributes rows as evenly as possible, but 10 rows in 4 buckets gives sizes 3, 3, 2, 2. For precise percentiles, prefer PERCENT_RANK or CUME_DIST.

💡 Interview Insight

"How would you find the median using window functions?" — "I'd use PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY value) if available. Otherwise, I can use ROW_NUMBER: rank all values, then pick the middle one(s) with WHERE rn IN (FLOOR((total+1)/2.0), CEIL((total+1)/2.0)) and average them — handles both odd and even row counts."

🔗 Practical: Sessionization with Window Functions

You have a user_events table with timestamped page views. A "session" is a sequence of events where no gap exceeds 30 minutes. Assign a session ID to each event. This is a classic interview question at Google, Meta, and Amazon.

Step-by-Step Approach

-- Step 1: Calculate gap from previous event WITH event_gaps AS ( SELECT user_id, event_time, page_url, LAG(event_time) OVER ( PARTITION BY user_id ORDER BY event_time ) AS prev_event_time, EXTRACT(EPOCH FROM ( event_time - LAG(event_time) OVER ( PARTITION BY user_id ORDER BY event_time ) )) / 60 AS gap_minutes FROM user_events ), -- Step 2: Flag session boundaries (gap > 30 min or first event) session_boundaries AS ( SELECT *, CASE WHEN gap_minutes > 30 OR gap_minutes IS NULL THEN 1 ELSE 0 END AS is_new_session FROM event_gaps ), -- Step 3: Running SUM of boundary flags = session ID sessions AS ( SELECT *, SUM(is_new_session) OVER ( PARTITION BY user_id ORDER BY event_time ) AS session_id FROM session_boundaries ) SELECT user_id, event_time, page_url, session_id FROM sessions ORDER BY user_id, event_time;

How It Works — Visually

User A's events: Time: 10:00 10:05 10:20 11:15 11:18 11:25 Gap (min): NULL 5 15 55 3 7 New session? 1 0 0 1 0 0 SUM flags: 1 1 1 2 2 2 └── Session 1 ──┘ └── Session 2 ──┘

The trick is brilliant: converting boolean flags (0/1) into a running sum. Each new session increments the counter, and all events within share the same sum value.

💡 Interview Insight

"Walk me through how you'd sessionize clickstream data." — "Three-step window function approach: 1) LAG to compute time gaps per user, 2) Flag rows where gap exceeds 30 min or no previous event, 3) Running SUM of flags creates a monotonically increasing session counter. This is O(n log n) for the sort and O(n) for the scan — much more efficient than self-joins."

🎯 Key Takeaways

🏅 Ranking Functions

ROW_NUMBER = unique sequential (no ties). RANK = ties with gaps. DENSE_RANK = ties without gaps. Choose based on whether you want exactly N results or inclusive-of-ties.

↔️ LAG/LEAD

Access adjacent rows for period-over-period comparisons, gap analysis, and change detection. Always handle NULLs for first/last rows.

⚠️ LAST_VALUE Trap

The default frame RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW means LAST_VALUE returns the current row. Always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.

🖼️ Frame Types

ROWS = physical row count. RANGE = logical values. GROUPS = peer groups. Use ROWS for sliding windows, RANGE for value ranges.

🚫 No WHERE Filtering

Window functions execute in the SELECT phase — after WHERE, GROUP BY, HAVING. Wrap in a CTE to filter on window function results.

🔗 Sessionization Pattern

LAG → flag boundaries → running SUM. Applies to sessions, islands, streaks, and any consecutive-row grouping problem.
