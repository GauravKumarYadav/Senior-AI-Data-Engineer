📝 Common Table Expressions: Your Query's Outline

A Common Table Expression (CTE) is a named temporary result set that exists only for the duration of a single query. Think of it as giving a name to a subquery and placing it at the top of your SQL — like defining variables before using them.

WITH monthly_revenue AS ( SELECT DATE_TRUNC('month', order_date) AS month, SUM(amount) AS revenue FROM orders GROUP BY DATE_TRUNC('month', order_date) ) SELECT month, revenue, revenue - LAG(revenue) OVER (ORDER BY month) AS mom_change FROM monthly_revenue;

Why CTEs Beat Nested Subqueries

Without CTEs, complex queries become "Russian nesting dolls" — subqueries inside subqueries. Compare:

❌ Nested Subqueries

  • Reads inside-out (unnatural)
  • Can't reuse intermediate results
  • Indentation hell
  • Hard to debug step-by-step

✅ Chained CTEs

  • Reads top-to-bottom like a story
  • Each CTE references prior ones
  • Clean, flat indentation
  • Comment out steps to debug

Chaining Multiple CTEs

WITH step1 AS (...), step2 AS (SELECT ... FROM step1 ...), -- Can reference step1 step3 AS (SELECT ... FROM step1, step2) -- Can reference step1 AND step2 SELECT * FROM step3;

💡 Interview Insight

"Are CTEs materialized or inlined?" — "It depends on the engine. PostgreSQL < 12 always materialized CTEs — creating an optimization barrier. Since v12, non-recursive CTEs are inlined by default, but you can force materialization with AS MATERIALIZED. In MySQL 8.0+, CTEs referenced multiple times are materialized. Key takeaway: don't assume CTE = materialization, and test with EXPLAIN."

🔄 Recursive CTEs: Trees, Graphs, Sequences

A recursive CTE references itself. It has two parts connected by UNION ALL:

🌱 Base Case (Anchor)

The starting point — doesn't reference the CTE. Seeds the first rows.

🔁 Recursive Step

References the CTE itself to build upon previous results. Must have a termination condition.

WITH RECURSIVE cte_name AS ( -- Base case: seed rows SELECT ... UNION ALL -- Recursive step: join CTE to itself SELECT ... FROM cte_name JOIN some_table ON ... WHERE termination_condition ) SELECT * FROM cte_name;

Example: Organizational Hierarchy

"Given an employees table with a manager_id column, show the full reporting chain."

WITH RECURSIVE org_chart AS ( -- Base case: top-level executives (no manager) SELECT employee_id, name, title, manager_id, 1 AS level, name::TEXT AS reporting_chain FROM employees WHERE manager_id IS NULL UNION ALL -- Recursive: find direct reports of known employees SELECT e.employee_id, e.name, e.title, e.manager_id, oc.level + 1, oc.reporting_chain || ' → ' || e.name FROM employees e JOIN org_chart oc ON e.manager_id = oc.employee_id ) SELECT employee_id, name, title, level, reporting_chain FROM org_chart ORDER BY reporting_chain;
-- Output: employee_id | name | title | level | reporting_chain 1 | Alice | CEO | 1 | Alice 2 | Bob | VP Eng | 2 | Alice → Bob 4 | Diana | Sr Eng | 3 | Alice → Bob → Diana 3 | Charlie | VP Sales | 2 | Alice → Charlie

Example: Date Series Generation

-- Generate dates for the last 30 days (useful for gap-filling) WITH RECURSIVE date_series AS ( SELECT CURRENT_DATE - INTERVAL '30 days' AS dt UNION ALL SELECT dt + INTERVAL '1 day' FROM date_series WHERE dt < CURRENT_DATE ) SELECT dt::DATE FROM date_series;
💡 Interview Insight

"When would you use a recursive CTE?" — "For hierarchical or graph-like data: org charts, category trees, bill-of-materials, file paths, friend-of-friend queries. Also great for generating sequences like date series for gap-filling. The pattern is always: anchor seeds rows, recursive step extends by one level, WHERE clause prevents infinite recursion. For deep hierarchies (1000+ levels), I'd consider a materialized path or nested set model."

↕️ Lateral Joins: Correlated Subqueries Done Right

You want to find the 3 most recent orders for each customer. A LATERAL join (PostgreSQL, MySQL 8+) or CROSS APPLY (SQL Server) lets a subquery reference columns from preceding tables in the FROM clause.

-- PostgreSQL: LATERAL SELECT c.customer_id, c.name, recent.order_id, recent.amount, recent.order_date FROM customers c CROSS JOIN LATERAL ( SELECT o.order_id, o.amount, o.order_date FROM orders o WHERE o.customer_id = c.customer_id ORDER BY o.order_date DESC LIMIT 3 ) AS recent;
-- SQL Server: CROSS APPLY (same semantics) SELECT c.customer_id, c.name, recent.order_id, recent.amount, recent.order_date FROM customers c CROSS APPLY ( SELECT TOP 3 o.order_id, o.amount, o.order_date FROM orders o WHERE o.customer_id = c.customer_id ORDER BY o.order_date DESC ) AS recent;

CROSS JOIN LATERAL vs LEFT JOIN LATERAL

CROSS JOIN LATERAL = CROSS APPLY → Excludes customers with 0 matches LEFT JOIN LATERAL = OUTER APPLY → Includes all customers, NULLs if no matches
-- Include customers even if they have no orders SELECT c.customer_id, c.name, recent.order_id, recent.amount FROM customers c LEFT JOIN LATERAL ( SELECT o.order_id, o.amount FROM orders o WHERE o.customer_id = c.customer_id ORDER BY o.order_date DESC LIMIT 3 ) AS recent ON TRUE; -- ON TRUE required for LEFT JOIN LATERAL in PG
ApproachLimitation
Window function + filterCan't LIMIT per group easily
Correlated subqueryReturns only 1 value
Self-join + ROW_NUMBER CTEMore verbose, full sort required
LATERAL subqueryClean, efficient, supports LIMIT
💡 Interview Insight

"What is a lateral join, and when would you use one?" — "A LATERAL join allows a subquery in FROM to reference columns from preceding tables — essentially a correlated subquery that can return multiple rows. I'd use it for top-N-per-group with LIMIT, table-valued functions per row, or unnesting arrays. In PostgreSQL it's JOIN LATERAL, in SQL Server it's CROSS APPLY. Often more efficient than ROW_NUMBER CTE because the optimizer can use index seek + limit per group."

🧮 GROUPING SETS, CUBE & ROLLUP

Sometimes you need multiple levels of aggregation in a single query — totals by product, by category, by region, and a grand total.

GROUPING SETS — Pick Your Levels

-- All levels in one pass SELECT category, region, SUM(revenue) AS total_revenue FROM sales GROUP BY GROUPING SETS ( (category), -- Subtotal per category (region), -- Subtotal per region (category, region), -- Detail per category+region () -- Grand total );

NULLs indicate "all values" at that level. Use GROUPING() to distinguish aggregated NULLs from real NULLs:

SELECT CASE WHEN GROUPING(category) = 1 THEN 'ALL' ELSE category END AS category, CASE WHEN GROUPING(region) = 1 THEN 'ALL' ELSE region END AS region, SUM(revenue) AS total_revenue FROM sales GROUP BY GROUPING SETS ((category), (region), (category, region), ());

ROLLUP — Hierarchical Subtotals

-- ROLLUP(a, b, c) = GROUPING SETS ((a,b,c), (a,b), (a), ()) SELECT EXTRACT(YEAR FROM order_date) AS yr, EXTRACT(QUARTER FROM order_date) AS qtr, EXTRACT(MONTH FROM order_date) AS mo, SUM(amount) AS revenue FROM orders GROUP BY ROLLUP ( EXTRACT(YEAR FROM order_date), EXTRACT(QUARTER FROM order_date), EXTRACT(MONTH FROM order_date) );

CUBE — All Possible Combinations

-- CUBE(a, b) = GROUPING SETS ((a,b), (a), (b), ()) SELECT category, region, SUM(revenue) AS total_revenue FROM sales GROUP BY CUBE (category, region);

Quick Reference

SyntaxGenerates# Sets (n cols)
GROUPING SETSExactly what you specifyExplicit
ROLLUP (a, b, c)(a,b,c), (a,b), (a), ()n + 1
CUBE (a, b, c)All subsets of {a, b, c}2^n
💡 Interview Insight

"When would you use ROLLUP vs CUBE?" — "ROLLUP is for hierarchical dimensions — geographic (city → state → country) or time (day → month → year) where there's a natural parent-child relationship. CUBE is for cross-tabulation where every combination matters — like revenue by both category AND region independently. Both are more efficient than multiple GROUP BY + UNION ALL queries because the database scans data once."

⚡ UNION ALL vs UNION: The Performance Trap

✅ UNION ALL

  • Simple concatenation — no overhead
  • Keeps all rows including duplicates
  • Streaming — fast on any size
  • ~2 sec on 20M rows

⚠️ UNION

  • Hash/sort to deduplicate — expensive
  • Removes duplicate rows
  • Must process entire result set
  • ~15+ sec on 20M rows
-- Under the hood: UNION ALL: Table A rows ──┐ ├──→ Result (fast, streaming) Table B rows ──┘ UNION: Table A rows ──┐ ├──→ [Hash/Sort to deduplicate] ──→ Result (slower) Table B rows ──┘

Rules of Thumb

1️⃣ Default to UNION ALL

Unless you specifically need deduplication.

2️⃣ Check Disjointness

If sources are partitioned by date range, UNION's dedup is wasted work.

3️⃣ Types Must Match

Column count and types must match. Names come from the first branch.

Common Pattern: Combining Event Tables

-- Events are unique — definitely use UNION ALL SELECT event_id, user_id, 'click' AS event_type, created_at FROM click_events UNION ALL SELECT event_id, user_id, 'purchase', created_at FROM purchase_events UNION ALL SELECT event_id, user_id, 'refund', created_at FROM refund_events ORDER BY created_at;
💡 Interview Insight

"When should you use UNION vs UNION ALL?" — "UNION ALL should be the default — it's a simple concatenation with no overhead. UNION adds a deduplication step (hash or sort) that can be very expensive on large datasets. I only use UNION when I specifically need to eliminate duplicates, and even then I ask: could the sources already be disjoint? If combining partitioned tables or event streams with unique IDs, UNION's dedup is pure waste."

🔍 Correlated vs Non-Correlated Subqueries

📋 Non-Correlated

  • Runs once, independently
  • Produces a fixed result
  • No reference to outer query
  • Generally fast

🔗 Correlated

  • Runs once per outer row
  • References outer query columns
  • Dependent on outer query
  • Can be slow (but optimizers help)
-- Non-correlated: runs once, returns a single value SELECT customer_id, name, total_spend FROM customers WHERE total_spend > ( SELECT AVG(total_spend) FROM customers );
-- Correlated: references c.customer_id from outer query SELECT c.customer_id, c.name, (SELECT COUNT(*) FROM orders o WHERE o.customer_id = c.customer_id) AS order_count -- Runs per customer FROM customers c;
-- Visual comparison: Non-Correlated: Correlated: ┌──────────────────┐ ┌──────────────────┐ │ Outer Query │ │ Outer Query │ │ ┌──────────┐ │ │ Row 1 → ┌────┐ │ │ │ Subquery │───┼─ Runs ONCE │ │ SQ │─┼─ Run │ └──────────┘ │ │ Row 2 → │ │─┼─ Run │ │ │ Row 3 → │ │─┼─ Run │ │ │ └────┘ │ └──────────────────┘ └──────────────────┘

Correlated Subquery with EXISTS

The most efficient use of correlated subqueries — EXISTS short-circuits as soon as it finds one match:

-- "Find customers with at least one order over $500" SELECT c.customer_id, c.name FROM customers c WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id AND o.amount > 500 );
💡 Interview Insight

"What's the difference, and which is faster?" — "Non-correlated runs once and produces a fixed result. Correlated references the outer query, so conceptually it runs once per outer row. Performance-wise, non-correlated is generally faster, but modern optimizers can decorrelate many correlated subqueries into joins. The exception: EXISTS with short-circuit evaluation is very efficient for semi-join patterns."

⚔️ EXISTS vs IN vs JOIN: The Showdown

"Find customers who have placed orders" — three ways to do it:

-- Method 1: IN SELECT customer_id, name FROM customers WHERE customer_id IN (SELECT customer_id FROM orders); -- Method 2: EXISTS SELECT c.customer_id, c.name FROM customers c WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id); -- Method 3: JOIN SELECT DISTINCT c.customer_id, c.name FROM customers c JOIN orders o ON c.customer_id = o.customer_id;

🚨 The NULL Trap with NOT IN

-- ❌ DANGER: Returns NO ROWS if subquery contains any NULL! SELECT * FROM customers WHERE customer_id NOT IN (SELECT manager_id FROM employees); -- x NOT IN (1, 2, NULL) → x!=1 AND x!=2 AND x!=NULL → always UNKNOWN -- ✅ SAFE: Always use NOT EXISTS instead SELECT * FROM customers c WHERE NOT EXISTS ( SELECT 1 FROM employees e WHERE e.manager_id = c.customer_id );

Performance Guide

Small subquery, large outer table: IN ≈ EXISTS ≈ JOIN (optimizer converts to similar plans) Large subquery, small outer table: EXISTS > IN > JOIN (EXISTS short-circuits) Both tables large: JOIN with hash ≈ EXISTS with hash (optimizer chooses hash semi-join) General advice: EXISTS → semi-joins (especially NOT) JOIN → need columns from both tables IN → small, known-no-NULLs lists
💡 Interview Insight

"When do you choose EXISTS over IN or JOIN?" — "I default to EXISTS for semi-join patterns and especially NOT EXISTS over NOT IN — because NOT IN with NULLs in the subquery returns no rows due to three-valued logic, which is a subtle but devastating bug. I use JOIN when I need columns from both tables. In modern PostgreSQL, the optimizer often converts all three to the same plan, but EXISTS with NOT is the only one that's always NULL-safe."

🎯 Key Takeaways

📝 CTEs = Readability

Transform nested subqueries into sequential, named steps. Chain with commas; each references the ones above it.

🔄 Recursive CTEs

Anchor → UNION ALL → recursive step with termination. Use for hierarchies, graph traversal, and sequence generation.

↕️ LATERAL Joins

Correlated subquery in FROM that returns multiple rows. Perfect for top-N-per-group with LIMIT. Use LEFT JOIN LATERAL ... ON TRUE to keep unmatched rows.

🧮 GROUPING SETS

Multi-level aggregation in one pass. ROLLUP = hierarchical (n+1 levels). CUBE = all combinations (2^n levels). Way faster than UNION ALL.

⚡ UNION ALL First

Default to UNION ALL. UNION's deduplication requires hash/sort of entire result — skip it unless you genuinely need dedup.

🚨 NOT IN = NULL Trap

If the subquery returns even one NULL, NOT IN gives empty results. Always prefer NOT EXISTS for anti-semi-joins. Top-tier interview gotcha.
