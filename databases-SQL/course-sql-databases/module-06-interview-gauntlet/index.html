🔥 Architecture & Design Questions

These are the schema design and system architecture questions that separate senior engineers from the pack. Every answer shows why, not just what.

Q1: "Design a schema for an e-commerce order system at millions of orders/day"

CREATE TABLE orders ( order_id BIGINT GENERATED ALWAYS AS IDENTITY, customer_id BIGINT NOT NULL REFERENCES customers(customer_id), order_date TIMESTAMPTZ NOT NULL DEFAULT NOW(), status TEXT NOT NULL DEFAULT 'pending', total_amount DECIMAL(12,2) NOT NULL, PRIMARY KEY (order_id, order_date) ) PARTITION BY RANGE (order_date); CREATE TABLE order_items ( item_id BIGINT GENERATED ALWAYS AS IDENTITY, order_id BIGINT NOT NULL, order_date TIMESTAMPTZ NOT NULL, product_id BIGINT NOT NULL REFERENCES products(product_id), quantity INT NOT NULL CHECK (quantity > 0), unit_price DECIMAL(10,2) NOT NULL, -- price at time of purchase! FOREIGN KEY (order_id, order_date) REFERENCES orders );
🎯 Key Design Decisions

Partitioned by order_date for time-range performance + data retention. unit_price on order_items captures price at purchase (products change price!). JSONB attributes on products for variable attributes. GENERATED ALWAYS AS IDENTITY over SERIAL — modern PostgreSQL best practice.

Q2: "500M-row orders table, dashboard aggregation takes 45 seconds. Fix it."

1️⃣ Materialized View

Pre-compute daily/monthly rollups. Refresh nightly. Dashboard queries hit the mat view (ms, not seconds).

2️⃣ Partition Pruning

Partition orders by month — scans ~10M rows per month instead of 500M.

3️⃣ Covering Index

CREATE INDEX ... ON orders (order_date) INCLUDE (total_amount) — index-only scan, no heap fetches.

4️⃣ Check Fundamentals

pg_stat_statements for frequency + cache hit rate. VACUUM ANALYZE if stats are stale.

Q3: "Implement a flash sale — 100 units, thousands of concurrent buyers"

-- Atomic UPDATE with RETURNING (preferred) UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 42 AND quantity > 0 RETURNING quantity; -- If no rows returned → out of stock -- Atomic, no race condition! -- Single statement = implicit lock

📉 Decrement stock by 1

✅ AND stock is still positive
📋 Return new quantity

Q4: "Design an audit trail for the orders table"

CREATE TABLE orders_audit ( audit_id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, order_id BIGINT NOT NULL, operation TEXT NOT NULL, -- INSERT, UPDATE, DELETE changed_at TIMESTAMPTZ DEFAULT NOW(), old_values JSONB, new_values JSONB ); CREATE OR REPLACE FUNCTION audit_orders() RETURNS TRIGGER AS $$ BEGIN INSERT INTO orders_audit (order_id, operation, old_values, new_values) VALUES ( COALESCE(NEW.order_id, OLD.order_id), TG_OP, CASE WHEN TG_OP != 'INSERT' THEN to_jsonb(OLD) END, CASE WHEN TG_OP != 'DELETE' THEN to_jsonb(NEW) END ); RETURN COALESCE(NEW, OLD); END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_orders_audit AFTER INSERT OR UPDATE OR DELETE ON orders FOR EACH ROW EXECUTE FUNCTION audit_orders();
💡 Why JSONB?

Using JSONB for old_values/new_values means the audit table never needs to change when the orders schema evolves. Schema-proof audit logging.

🏗️ More Architecture — Multi-Tenant & CAP

Q5: "How would you design a database for multi-tenant SaaS?"

🔗 Shared Table + tenant_id

Low isolation, low complexity. Best for many small tenants. Use Row-Level Security to prevent cross-tenant leaks.

📦 Schema Per Tenant

Medium isolation, medium complexity. Good for compliance requirements with moderate tenant counts.

🏠 Database Per Tenant

High isolation, high complexity. Enterprise customers with strict data separation needs.

-- Most common: shared tables with Row-Level Security ALTER TABLE orders ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_isolation ON orders USING (tenant_id = current_setting('app.tenant_id')::int); -- Every query automatically filtered — no accidental data leaks SET app.tenant_id = '42'; SELECT * FROM orders; -- Only sees tenant 42's orders

Q6: "Explain the CAP theorem"

CAP: A distributed system can guarantee at most 2 of 3: C — Consistency: Every read gets the most recent write A — Availability: Every request gets a response P — Partition tolerance: System works despite network failures CP (Consistency + Partition Tolerance): PostgreSQL, MongoDB → Rejects writes during partitions to stay consistent AP (Availability + Partition Tolerance): DynamoDB, Cassandra → Accepts writes during partitions, resolves later In practice: partitions WILL happen, so you choose CP vs AP. Use CP for source-of-truth (payments). Use AP for high-availability reads (product catalog).

Q7: "Systematically diagnose a slow 6-table join"

1️⃣ EXPLAIN ANALYZE

Get the actual execution plan with real timing.

2️⃣ Find the Bottleneck

Which node takes longest? Estimated vs actual rows match?

3️⃣ Check Seq Scans

Large tables doing sequential scans? Add indexes for selective filters.

4️⃣ Check Spills

Hash joins spilling to disk? Increase work_mem for this session.

⚠️ work_mem Gotcha

work_mem is per-operation-per-connection. 256MB × 100 connections × 3 operations = 75 GB. Set it per-session with SET work_mem = '256MB', not globally!

✍️ Query Writing Challenges

Challenge 1: Top 3 products by revenue per category

WITH ranked AS ( SELECT p.category, p.name, SUM(oi.quantity * oi.unit_price) AS revenue, RANK() OVER ( PARTITION BY p.category ORDER BY SUM(oi.quantity * oi.unit_price) DESC ) AS rn FROM order_items oi JOIN products p ON oi.product_id = p.product_id GROUP BY p.category, p.name ) SELECT category, name, revenue FROM ranked WHERE rn <= 3 ORDER BY category, revenue DESC;
💡 RANK vs ROW_NUMBER

RANK: ties both appear (3 rows could become 4). ROW_NUMBER: strictly N rows per group, arbitrary tiebreaker. State your choice and explain why.

Challenge 2: Customers with 3 consecutive months of increasing orders

WITH monthly AS ( SELECT customer_id, DATE_TRUNC('month', order_date) AS month, COUNT(*) AS cnt FROM orders GROUP BY 1, 2 ), flagged AS ( SELECT *, CASE WHEN cnt > LAG(cnt, 1) OVER w AND LAG(cnt, 1) OVER w > LAG(cnt, 2) OVER w AND LAG(cnt, 2) OVER w > LAG(cnt, 3) OVER w THEN 1 ELSE 0 END AS three_month_increase FROM monthly WINDOW w AS (PARTITION BY customer_id ORDER BY month) ) SELECT DISTINCT customer_id FROM flagged WHERE three_month_increase = 1;

Challenge 3: Sessionize clickstream with 30-min timeout

WITH gaps AS ( SELECT user_id, event_time, page_url, EXTRACT(EPOCH FROM ( event_time - LAG(event_time) OVER ( PARTITION BY user_id ORDER BY event_time) )) / 60.0 AS gap_minutes FROM clickstream ), boundaries AS ( SELECT *, CASE WHEN gap_minutes > 30 OR gap_minutes IS NULL THEN 1 ELSE 0 END AS new_session FROM gaps ) SELECT user_id, SUM(new_session) OVER ( PARTITION BY user_id ORDER BY event_time ) AS session_id, event_time, page_url FROM boundaries;
🎯 The Pattern

LAG → detect gaps → flag boundaries → running SUM of flags = session IDs. This sessionization pattern appears in ~60% of SQL interviews.

✍️ More Query Challenges

Challenge 4: 7-day rolling average including zero-order days

WITH date_series AS ( SELECT generate_series( (SELECT MIN(order_date) FROM orders), (SELECT MAX(order_date) FROM orders), '1 day'::interval )::date AS day ), daily_revenue AS ( SELECT d.day, COALESCE(SUM(o.total_amount), 0) AS revenue FROM date_series d LEFT JOIN orders o ON o.order_date::date = d.day GROUP BY d.day ) SELECT day, revenue, ROUND(AVG(revenue) OVER ( ORDER BY day ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ), 2) AS rolling_7day_avg FROM daily_revenue;
💡 Key Insight

You must generate a complete date series and LEFT JOIN to it. Otherwise days with zero orders are missing and the rolling average window shifts incorrectly.

Challenge 5: Market basket analysis — products bought together

SELECT p1.name AS product_a, p2.name AS product_b, COUNT(DISTINCT oi1.order_id) AS co_purchase_count FROM order_items oi1 JOIN order_items oi2 ON oi1.order_id = oi2.order_id AND oi1.product_id < oi2.product_id -- avoid dupes & self-pairs JOIN products p1 ON oi1.product_id = p1.product_id JOIN products p2 ON oi2.product_id = p2.product_id GROUP BY p1.name, p2.name HAVING COUNT(DISTINCT oi1.order_id) >= 10 ORDER BY co_purchase_count DESC LIMIT 20;
🎯 The < Trick

oi1.product_id < oi2.product_id ensures each pair appears once (A,B but not B,A) and excludes self-pairs. Classic interview pattern!

Challenge 6: Median order value per category

-- PostgreSQL: use PERCENTILE_CONT SELECT p.category, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY o.total_amount) AS median, AVG(o.total_amount) AS mean FROM orders o JOIN order_items oi ON o.order_id = oi.order_id JOIN products p ON oi.product_id = p.product_id GROUP BY p.category;

Challenge 7: Customers who repurchased within 7 days of first order

WITH first_orders AS ( SELECT customer_id, MIN(order_date) AS first_order_date FROM orders GROUP BY customer_id ) SELECT DISTINCT o.customer_id FROM orders o JOIN first_orders f ON o.customer_id = f.customer_id WHERE o.order_date > f.first_order_date AND o.order_date <= f.first_order_date + INTERVAL '7 days'; -- Identifies high early engagement — key retention metric

🔧 Performance Debugging Scenarios

Scenario 1: Seq Scan despite having an index on email

-- The query: SELECT * FROM customers WHERE LOWER(email) = 'alice@example.com'; -- Problem: index is on `email`, but LOWER() prevents index usage -- Fix: expression index CREATE INDEX idx_customers_email_lower ON customers (LOWER(email));

Scenario 2: Index scan but still slow (500K heap fetches)

-- Index finds 500K rows, but each requires random I/O heap fetch -- Fix 1: Covering index (Index Only Scan, zero heap fetches) CREATE INDEX ... ON orders (customer_id) INCLUDE (order_date, amount, status); -- Fix 2: Narrow the SELECT list (don't SELECT *) -- Fix 3: CLUSTER to physically reorder table (one-time, exclusive lock) CLUSTER orders USING idx_orders_customer;

Scenario 3: DISTINCT on 100M rows crawling

-- Problem: DISTINCT on non-indexed column → massive sort SELECT DISTINCT customer_id FROM orders WHERE status = 'active'; -- Fix: use EXISTS (checks membership, stops at first match) SELECT customer_id FROM customers c WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id AND o.status = 'active' );

Scenario 4: UPDATE millions of rows takes 30 minutes

-- Problem: single transaction updating millions → WAL bloat + long locks -- Fix: batch in chunks of 10,000 DO $$ DECLARE rows_updated INT; BEGIN LOOP UPDATE orders SET status = 'archived' WHERE order_id IN ( SELECT order_id FROM orders WHERE order_date < '2023-01-01' AND status != 'archived' LIMIT 10000 ); GET DIAGNOSTICS rows_updated = ROW_COUNT; EXIT WHEN rows_updated = 0; COMMIT; END LOOP; END $$;
🎯 Production Tips

COUNT(*) slow on 100M rows? Use SELECT reltuples::bigint FROM pg_class for an approximate instant count. Adding indexes blocking writes? Use CREATE INDEX CONCURRENTLY (can't run in a transaction, but won't block DML).

⚠️ Tricky Gotchas

These edge cases separate senior engineers from intermediates. The fix is always the same: understand what the database actually does, not what you assume.

💀 NULL in NOT IN

WHERE x NOT IN (1, 2, NULL) → returns ZERO rows! Any NULL makes the whole expression UNKNOWN. Fix: use NOT EXISTS.

➗ Integer Division

SELECT 5/2 → returns 2, not 2.5! Always cast to DECIMAL before dividing for percentages: 5::numeric / 2.

📅 BETWEEN is Inclusive

With timestamps, BETWEEN '2024-01-01' AND '2024-01-31' misses Jan 31 14:30! Use half-open: >= '2024-01-01' AND < '2024-02-01'.

🔤 Implicit Type Cast

WHERE phone_number = 5551234567 on a TEXT column → PostgreSQL casts the column, killing the index. Use correct type: '5551234567'.

🎲 LIMIT Without ORDER BY

Which 10 rows? Different every time! Non-deterministic. Always pair LIMIT with ORDER BY.

📊 COUNT(*) vs COUNT(col)

COUNT(*) counts ALL rows. COUNT(column) counts non-NULL values only. 1000 orders, 200 with discount_code → COUNT(*)=1000, COUNT(discount_code)=200.

Window Function Default Frame Bug

-- This LOOKS right but has a subtle bug: SELECT order_date, amount, SUM(amount) OVER ( ORDER BY order_date ) AS running_total FROM orders; -- Default: RANGE BETWEEN UNBOUNDED -- PRECEDING AND CURRENT ROW -- Duplicate dates = peers = "jump" -- Fix: use ROWS explicitly SUM(amount) OVER ( ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW )
😱 Looks correct...
 
 
 
 
 
⚠️ Default frame uses RANGE
🐛 Same-date rows grouped as peers
📈 Running total "jumps" unexpectedly
 
✅ ROWS processes one row at a time
🎯 Correct incremental running total
 
🎯 More Gotchas

UPDATE with non-unique JOIN → PostgreSQL picks an arbitrary matching row, no error. READ COMMITTED isolation → long-running reports can see inconsistent data (rows inserted mid-scan). Fix: BEGIN ISOLATION LEVEL REPEATABLE READ.

Q2: "Redis caching strategy for 1M-product catalog"

🔑 Key Structure

product:{id}:detail → full JSON (TTL 1hr). product:{id}:price → just price (TTL 5min). category:{id}:list → sorted set by popularity.

🔥 Cache Warming

On deploy, pre-cache top 1000 products (80/20 rule). 10% hot products = ~200MB — comfortable for Redis.

🗑️ Invalidation

Product update → DELETE key (lazy reload). Price change → DELETE + publish to channel. Eviction: allkeys-lfu.

Q3: "Design a rate limiter using Redis"

# Sliding window using sorted sets def is_rate_limited(user_id: str, max_requests=100, window_secs=60) -> bool: key = f"ratelimit:{user_id}" now = time.time() pipe = r.pipeline() pipe.zremrangebyscore(key, 0, now - window_secs) # Remove old pipe.zadd(key, {f"{now}:{uuid4()}": now}) # Add current pipe.zcard(key) # Count in window pipe.expire(key, window_secs) # Auto-cleanup _, _, request_count, _ = pipe.execute() return request_count > max_requests
💡 Why Sorted Set?

Score = timestamp enables efficient removal of expired entries with ZREMRANGEBYSCORE. The window slides precisely — no fixed bucket boundaries that cause edge-case bursts.

⚡ Rapid Fire — 20 Quick Q&A

1. EXPLAIN ANALYZE vs EXPLAIN?

ANALYZE actually runs the query, showing real timing and row counts.

2. Default PG index type?

B-tree.

3. Window function in WHERE?

No — wrap in a CTE or subquery.

4. DELETE vs TRUNCATE?

DELETE: row-by-row, logged, triggers fire. TRUNCATE: instant drop, minimal logging, no triggers.

5. VACUUM FULL vs VACUUM?

FULL rewrites the table, returns disk space to OS. Requires exclusive lock.

6. What's a covering index?

Index with all columns a query needs → Index Only Scan, no heap access.

7. Leftmost prefix rule?

Index (A,B,C) helps A, A+B, A+B+C — never B or C alone.

8. COALESCE(x, 0)?

Returns x if non-NULL, otherwise 0.

9. Concurrent writes same row?

Row-level locking. Second writer waits until first commits/rolls back.

10. UNION vs UNION ALL?

UNION deduplicates (sorts). UNION ALL keeps all rows (faster).

11. What's a deadlock?

Two transactions each hold a lock the other needs. PG detects and kills one.

12. DynamoDB PK design rule?

High-cardinality keys for even distribution. Avoid hot partitions.

13. jsonb_set() use?

Update a specific key within JSONB without replacing the entire value.

14. What's PgBouncer?

Connection pooler: multiplexes many app connections onto few PG backends.

15. N+1 query problem?

Fetch list (1 query), then details per item (N queries). Fix: JOIN or batch fetch.

16. Redis SET ... NX?

"Set only if Not eXists" — used for distributed locks.

17. Materialized view?

Cached query result stored as physical table. Must be manually refreshed.

18. ANALYZE (no VACUUM)?

Updates table statistics for better query planner decisions.

19. When is Seq Scan fine?

Small tables (<1K rows) or returning >10-15% of rows.

20. PostgreSQL CAP tradeoff?

CP — prioritizes Consistency over Availability during network partitions.
