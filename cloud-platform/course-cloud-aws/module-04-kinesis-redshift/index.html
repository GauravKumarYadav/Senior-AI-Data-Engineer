Kinesis Data Streams — Real-Time Ingestion

Kinesis Data Streams (KDS) captures gigabytes per second from hundreds of thousands of sources — POS terminals, clickstream, IoT sensors, application logs.

📥 Write: 1 MB/s per shard

Max 1,000 records/sec per shard. Record size max: 1 MB.

📤 Read: 2 MB/s per shard

Shared across all consumers (or 2 MB/s per consumer with Enhanced Fan-Out).

🕐 Retention: 24h → 365d

Default 24 hours. Extend up to 365 days. Replay from any point.

# Partition key = store_id → ordered processing per store import boto3, json kinesis = boto3.client("kinesis") def send_pos_batch(events: list[dict]): records = [ {"Data": json.dumps(e).encode(), "PartitionKey": str(e["store_id"])} for e in events ] resp = kinesis.put_records(StreamName="retailmart-pos-events", Records=records) if resp["FailedRecordCount"] > 0: # Implement exponential backoff retry for failed records ...
💡 Partition Key Strategy

❌ Bad: constant key → all records to 1 shard (bottleneck). ❌ Mediocre: store_id with skew (flagship store = 40%). ✅ Good: composite f"{store_id}-{register_id}" → 60K unique keys, even distribution. ✅ Best: uuid4() when ordering doesn't matter.

💡 How Many Shards?

Shards = max(incoming_MB/s ÷ 1, outgoing_MB/s ÷ 2). RetailMart: 8 MB/s peak → 8 shards min. 3 consumers × 8 MB/s ÷ 2 = 12 shards for fan-out. Add 20% headroom. Use On-Demand mode if traffic is unpredictable (auto-scales to 200 MB/s).

Kinesis Data Firehose — Managed Delivery

Fully managed delivery stream: captures, transforms, and loads data into S3, Redshift, OpenSearch, Splunk. Zero administration, auto-scaling.

📡 Kinesis Data Streams

  • ⚡ ~200ms latency (real-time)
  • 🔧 You manage shards
  • 👥 You build consumers (KCL, Lambda)
  • 🔄 Replay supported (re-read from shard)
  • 💰 Per shard-hour pricing

🚚 Kinesis Data Firehose

  • ⏱️ 60-900 sec buffer (near-real-time)
  • ✅ Fully managed, auto-scales
  • 📦 Built-in destinations (S3, Redshift…)
  • ❌ No replay (one-shot delivery)
  • 💰 Per GB ingested pricing

💡 Firehose Alone vs. Data Streams + Firehose

Firehose alone (Direct PUT): just loading into S3/Redshift. Simple, cheap.
Data Streams + Firehose: when you need (1) multiple consumers, (2) real-time processing alongside loading (Lambda fraud detection <200ms while Firehose buffers to S3), (3) replay capability. RetailMart uses KDS as the "spine" with 3 consumers: Firehose→S3, Lambda→fraud alerts, Flink→dashboards.

Kinesis vs. Kafka (MSK)

One of the most common interview questions for data engineering roles.

📡 Kinesis

  • ✅ Zero ops — AWS-managed shards
  • ✅ Tight AWS integration (Firehose, Lambda)
  • ✅ Budget predictability (per-shard)
  • ⚠️ 1 MB/s in, 2 MB/s out per shard
  • ⚠️ Shard split/merge takes minutes
  • ⚠️ AWS SDK only (not open standard)

📨 MSK (Kafka)

  • ✅ No per-partition throughput limit
  • ✅ Kafka Connect (100+ connectors)
  • ✅ Open standard, multi-cloud portable
  • ✅ Log compaction (latest per key)
  • ⚠️ Broker patching, disk, ZooKeeper/KRaft
  • ⚠️ Higher baseline cost ($4,800+/mo)
💡 Migrating from On-Prem Kafka to AWS

MSK in most cases — existing producers/consumers, Kafka Connect pipelines, team expertise make it a drop-in replacement. MirrorMaker 2 replicates from on-prem to MSK, then cut over. Kinesis would require rewriting every producer/consumer. Exception: going "all-in serverless" with Lambda + Firehose as a clean break.

🏪 RetailMart Chose Kinesis — Why?

Split into 3 streams (POS ~150 MB/s, clickstream ~200 MB/s, inventory ~50 MB/s). On-Demand mode auto-scales. Total: ~$3,200/mo vs. MSK $4,800/mo + ops. 33% cheaper, zero ops, 3-person team.

Enhanced Fan-Out & Flink Analytics

By default, all consumers share 2 MB/s per shard. Enhanced Fan-Out gives each consumer a dedicated 2 MB/s pipe via HTTP/2 push.

📡 Standard (Shared)

3 consumers on 1 shard = ~0.67 MB/s each. Polling-based. Good for 1-2 consumers.

🚀 Enhanced Fan-Out

3 consumers on 1 shard = 2 MB/s EACH (6 MB/s total). HTTP/2 push, ~70ms latency. Use for 3+ consumers or <200ms needs.

Flink SQL: Real-Time Fraud Detection

-- Detect fraud: 3+ transactions at different stores within 5 minutes INSERT INTO fraud_alerts SELECT customer_id, COUNT(DISTINCT store_id) AS distinct_stores, SUM(total_amount) AS total_spent, TUMBLE_START(event_time, INTERVAL '5' MINUTE) AS window_start FROM pos_events GROUP BY customer_id, TUMBLE(event_time, INTERVAL '5' MINUTE) HAVING COUNT(DISTINCT store_id) >= 3;
💡 Flink Window Types

Tumbling: fixed, non-overlapping (every 5 min). Sliding: overlapping (10-min window sliding every 1 min). Session: gap-based (closes after N min of inactivity). Watermarks handle late data — events up to the watermark threshold are included in the correct window.

Redshift — Columnar MPP Warehouse

Petabyte-scale, columnar, massively parallel processing (MPP) warehouse. Purpose-built for OLAP queries scanning billions of rows.

👑 Leader Node

Parses SQL, builds query plan, distributes work to compute nodes, aggregates results.

⚙️ Compute Nodes

Execute query fragments in parallel. Each node has multiple slices (1 CPU + memory + disk each).

📊 Columnar Storage

SELECT SUM(amount) reads ONLY the amount column (1/5 of data). 5-10x compression. Zone maps skip irrelevant blocks.

Distribution Styles

🔑 KEY

Same key value → same node. Best for large fact tables joined on that key. Avoid when key is skewed.

🔄 EVEN

Round-robin across nodes. For tables not joined, or no clear key. Causes redistribution on joins.

📋 ALL

Full copy on every node. For small dimension tables (<5M rows) joined frequently. Wastes storage for large tables.

🤖 AUTO

Redshift decides (starts EVEN, may change). Default — let Redshift optimize when you're unsure.

-- Fact table: KEY distribution on store_id, COMPOUND SORTKEY on timestamp CREATE TABLE warehouse.fact_daily_sales ( transaction_id VARCHAR(50) ENCODE zstd, store_id INTEGER ENCODE az64 DISTKEY, product_sku VARCHAR(30) ENCODE zstd, total_amount DECIMAL(10,2) ENCODE az64, transaction_ts TIMESTAMP ENCODE az64 SORTKEY ) DISTSTYLE KEY COMPOUND SORTKEY (transaction_ts, store_id); -- Dimension: ALL distribution (small, joined frequently) CREATE TABLE warehouse.dim_store ( store_id INTEGER PRIMARY KEY, store_name VARCHAR(100), region VARCHAR(30), state VARCHAR(2) ) DISTSTYLE ALL;
💡 DISTKEY Pitfall

KEY distribution on a skewed column (e.g., 1% of customers = 50% of transactions) drowns one node while others idle. Check: SELECT "table", skew_rows FROM svv_table_info. COMPOUND SORTKEY: first column dominates zone maps. INTERLEAVED: equal weight but expensive VACUUM REINDEX.

Redshift Spectrum & Serverless

Spectrum extends Redshift SQL to query data directly in S3 via Glue Data Catalog — no loading required. Bridge your warehouse and data lake in one SQL statement.

-- Create external schema pointing to Glue Data Catalog CREATE EXTERNAL SCHEMA spectrum_schema FROM DATA CATALOG DATABASE 'curated' IAM_ROLE 'arn:aws:iam::123456789:role/SpectrumRole'; -- Join local Redshift + S3 Spectrum tables in one query SELECT d.region, d.store_name, SUM(CASE WHEN s.year = '2026' THEN s.total_amount END) AS rev_2026, SUM(CASE WHEN h.year = '2025' THEN h.total_amount END) AS rev_2025 FROM warehouse.fact_daily_sales s -- Local Redshift JOIN spectrum_schema.historical_sales h ON s.store_id = h.store_id -- S3 JOIN warehouse.dim_store d ON s.store_id = d.store_id -- Local GROUP BY d.region, d.store_name;

Redshift Serverless

No clusters. Define a workgroup + namespace. Auto-scales RPUs. Scales to zero when idle — $0 cost.

🖥️ Provisioned

  • 💰 Per-node-hour (always on)
  • 📈 Resize cluster (minutes)
  • ✅ Reserved Instances (75% off)
  • 📊 Best: steady, predictable workloads

⚡ Serverless

  • 💰 Per RPU-second (pay for use)
  • 📈 Auto-scale in seconds
  • 🆓 Scales to zero ($0 idle)
  • 📊 Best: variable, bursty workloads
💡 Spectrum vs. Athena

Spectrum: you already have Redshift and need to join local tables with S3 data. Requires running cluster/serverless.
Athena: no Redshift needed, purely S3 queries, serverless per-query pricing. They complement — use Athena for exploration, Spectrum for warehouse+lake joins.

Data Sharing & Redshift vs. Athena vs. BigQuery

Data Sharing: share live data across Redshift clusters/accounts without copying. Consumers use their own compute, see real-time data, pay no storage.

-- PRODUCER: create and grant data share CREATE DATASHARE retailmart_share; ALTER DATASHARE retailmart_share ADD TABLE warehouse.fact_daily_sales; GRANT USAGE ON DATASHARE retailmart_share TO ACCOUNT '987654321012'; -- CONSUMER: mount and query CREATE DATABASE shared_warehouse FROM DATASHARE retailmart_share; SELECT region, SUM(total_amount) FROM shared_warehouse.warehouse.fact_daily_sales f JOIN shared_warehouse.warehouse.dim_store s ON f.store_id = s.store_id GROUP BY region;

The Three-Way Comparison

🏢 Redshift

MPP warehouse, sub-second cached queries, complex joins, materialized views, BI dashboards. Cluster or RPU pricing. Best for known DW workloads.

🔍 Athena

Serverless Trino/Presto, $5/TB scanned, zero idle cost. Best for ad-hoc S3 exploration, data quality checks, one-off reports.

☁️ BigQuery

Serverless warehouse, $6.25/TB on-demand, native ML, streaming ingest. Best for all-purpose serverless DW, especially multi-cloud.

💡 RetailMart: Redshift + Athena Together

Not either/or — they complement. Athena for exploration (raw/curated S3 zones). Redshift for production analytics warehouse powering executive dashboards. Spectrum bridges both — Redshift queries reach into S3 for historical data in a single SQL statement.

Streaming-to-Warehouse End-to-End

RetailMart's complete pipeline: POS → Kinesis → S3 → Redshift. Three paths: Firehose (S3), Flink (dashboards), Lambda (alerts).

-- COPY: fastest way to load S3 → Redshift COPY warehouse.fact_daily_sales FROM 's3://retailmart-data-lake-prod/curated/daily_sales/year=2026/' IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftCopyRole' FORMAT AS PARQUET; -- Streaming Ingestion: direct from Kinesis into materialized view CREATE EXTERNAL SCHEMA kinesis_schema FROM KINESIS IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftStreamRole'; CREATE MATERIALIZED VIEW warehouse.mv_realtime_sales AS SELECT approximate_arrival_timestamp AS event_time, JSON_EXTRACT_PATH_TEXT(kinesis_data, 'store_id')::INT AS store_id, JSON_EXTRACT_PATH_TEXT(kinesis_data, 'total_amount')::DECIMAL(10,2) AS amount FROM kinesis_schema."retailmart-pos-events";
💡 Near-Real-Time into Redshift — 3 Options

(1) Streaming Ingestion — materialized views read from Kinesis/MSK, refresh in seconds. Lowest latency. (2) Firehose → S3 → COPY — 1-5 min latency, most common. (3) Micro-batch with Glue/Lambda — custom COPY every N minutes. RetailMart: Streaming Ingestion for real-time dashboards, COPY from S3 for nightly refresh.

🔑 Key Takeaways

KDS = Real-Time Backbone

Per-shard: 1 MB/s in, 2 MB/s out, 1,000 rec/s. Memorize these limits.

Firehose = Managed Delivery

S3/Redshift with built-in Parquet. Near-real-time (60-900s buffer), not real-time.

Kinesis vs. Kafka

Kinesis for AWS-native simplicity. MSK for existing Kafka ecosystems & extreme throughput.

Enhanced Fan-Out

Dedicated 2 MB/s per consumer via HTTP/2. Use for 3+ consumers or <200ms latency.

Redshift = Columnar + MPP

DISTKEY (KEY/EVEN/ALL) and SORTKEY are the two biggest perf levers.

Spectrum Bridges Lake + DW

Query S3 from Redshift SQL. Join local tables with S3 data in one statement.

Serverless Scales to Zero

RPU-second billing, auto-scale, zero idle cost. Ideal for variable workloads.

Streaming Ingestion

Kinesis → Redshift materialized views. Lowest latency for real-time warehouse updates.
