🏛️ Lake Formation — Centralized Governance

RetailMart's data lake spans S3, Glue/Athena, EMR, Kinesis, and Redshift. But who can access what? The compliance team needs column-level PII restrictions, analysts need cross-account sharing, and the nightly ETL has 12 interdependent jobs. This module tackles governance and orchestration.

AWS Lake Formation is a centralized governance layer on top of the Glue Data Catalog. Instead of managing S3 bucket policies, IAM policies, and Glue resource policies separately — Lake Formation gives you a single pane of glass.

❌ Before Lake Formation

  • S3 Bucket Policy: 200 lines of JSON
  • IAM Policy for Role A: 50 lines
  • IAM Policy for Role B: 50 lines
  • Glue Resource Policy: 80 lines
  • KMS Key Policy: 40 lines
  • = 15 different places to manage

✅ After Lake Formation

  • One GRANT statement per role
  • ✓ S3 access (credential vending)
  • ✓ Glue Catalog permissions
  • ✓ Athena query filtering
  • ✓ Redshift Spectrum access
  • ✓ EMR access — all handled
# Step 1: Register S3 locations with Lake Formation aws lakeformation register-resource \ --resource-arn "arn:aws:s3:::retailmart-data-lake-prod" \ --use-service-linked-role # Step 2: Make Lake Formation the authority (THE critical step) aws lakeformation put-data-lake-settings \ --data-lake-settings '{"DataLakeAdmins":[{"DataLakePrincipalIdentifier":"arn:aws:iam::123456789:role/DataLakeAdminRole"}],"CreateDatabaseDefaultPermissions":[],"CreateTableDefaultPermissions":[]}' # Step 3: Revoke the default super-permission aws lakeformation batch-revoke-permissions \ --entries '[{"Id":"1","Principal":{"DataLakePrincipalIdentifier":"IAM_ALLOWED_PRINCIPALS"},"Resource":{"Database":{"Name":"curated"}},"Permissions":["ALL"],"PermissionsWithGrantOption":["ALL"]}]'
💡 Interview Gold

Q: "Why not just use IAM policies for data lake access control?" — IAM operates at the S3 object level (paths like s3://bucket/curated/*). But data access should be at the table and column level. IAM can't do column-level filtering. Lake Formation adds a semantic layer: permissions on databases, tables, columns, and rows — regardless of the underlying S3 layout. The trade-off: revoking IAMAllowedPrincipals trips up many teams during initial setup.

🔒 Fine-Grained Access — Column, Row & Cell

Column-Level Security

Restrict which columns a principal can see — the most common PII protection pattern.

# Grant analysts SELECT on specific columns only (PII excluded) aws lakeformation grant-permissions \ --principal '{"DataLakePrincipalIdentifier":"arn:aws:iam::123456789:role/AnalystRole"}' \ --resource '{"TableWithColumns":{"DatabaseName":"curated","Name":"daily_sales","ColumnNames":["store_id","product_sku","quantity","total_amount","transaction_ts"]}}' \ --permissions '["SELECT"]' # customer_id and payment_method are NOT granted — analysts simply cannot see PII

Row-Level Security (Data Filters)

Restrict which rows a principal can see — e.g., regional managers see only their region.

# Create a data cell filter for the Northeast region manager aws lakeformation create-data-cells-filter \ --table-data '{"TableCatalogId":"123456789012","DatabaseName":"curated","TableName":"daily_sales","Name":"northeast-only","RowFilter":{"FilterExpression":"region = '\''northeast'\''"},"ColumnNames":["store_id","product_sku","quantity","total_amount","transaction_ts","region"]}'

Cell-Level Security (Column + Row Combined)

🔓 Data Engineer Sees

  • store_id | cust_id | product | amount | card_last4 | region
  • 42 | C-99381 | TV-55 | 499.99 | 4242 | NE
  • 17 | C-44210 | Laptop | 899.00 | 1234 | SE
  • 103 | C-77512 | Headphones | 149.99 | 5678 | NE
  • 88 | C-11293 | Tablet | 329.00 | 9999 | MW

🔐 NE Manager Sees

  • store_id | product | amount | region
  • 42 | TV-55 | 499.99 | NE
  • 103 | Headphones | 149.99 | NE
  • 2 cols hidden + 2 rows filtered
  • = Cell-level security
💡 Interview Gold

Q: "How do you implement GDPR right-to-be-forgotten in a data lake?" — Multiple layers: (1) Column-level security via LF hides PII from analysts. (2) Row-level filtering excludes flagged customers. (3) For actual deletion, use Apache Iceberg DELETE FROM to physically remove records. (4) For pseudonymization, hash PII in a Glue ETL job. Lake Formation controls visibility — physical deletion requires a separate process.

🏷️ LF-Tags — Tag-Based Access at Scale

Managing individual table/column grants doesn't scale with 500+ tables across 20 teams. LF-Tags enable attribute-based access control (ABAC).

🏷️ Tag Taxonomy

sensitivity = public | internal | confidential | pii
domain = sales | inventory | hr | finance
environment = dev | staging | prod

📊 Resource Tagging

curated.daily_sales → sensitivity=internal, domain=sales
curated.daily_sales.cust_id → sensitivity=pii
curated.employee_data → sensitivity=confidential, domain=hr

👤 Principal Matching

AnalystRole → sensitivity≤internal, domain=sales ✅ daily_sales, ❌ cust_id column, ❌ employee_data
New sales table tagged correctly? → Auto-access, zero tickets.

# Create tag keys aws lakeformation create-lf-tag --tag-key "sensitivity" \ --tag-values '["public","internal","confidential","pii"]' # Tag PII columns separately aws lakeformation add-lf-tags-to-resource \ --resource '{"TableWithColumns":{"DatabaseName":"curated","Name":"daily_sales","ColumnNames":["customer_id","payment_method","card_last_four"]}}' \ --lf-tags '[{"TagKey":"sensitivity","TagValues":["pii"]}]' # Grant via tag-based policy — one statement covers all matching tables aws lakeformation grant-permissions \ --principal '{"DataLakePrincipalIdentifier":"arn:aws:iam::123456789:role/AnalystRole"}' \ --resource '{"LFTagPolicy":{"ResourceType":"TABLE","Expression":[{"TagKey":"sensitivity","TagValues":["public","internal"]},{"TagKey":"domain","TagValues":["sales"]}]}}' \ --permissions '["SELECT","DESCRIBE"]'
ApproachNamed ResourcesLF-Tags
ScaleO(tables × roles) grantsO(tags × roles) policies
New tableManual grant requiredAuto if tagged correctly
Best for< 50 tables, few roles50+ tables, many teams
🎯 Interview Tip

Q: "How do you manage data access for 200 tables across 15 teams?" — LF-Tags. Define a taxonomy, tag every table at creation time via CI/CD, create tag-based policies. When a new sales table is created and tagged domain=sales, sensitivity=internal, Role X automatically gets access. This scales from 200 to 2,000 tables without additional policies.

🔗 Cross-Account Data Sharing

RetailMart's analytics subsidiary operates in a separate AWS account. They need access to curated sales data without duplicating it.

📤 Producer Account

  • Lake Formation Admin
  • GRANT cross-account SELECT on curated.sales to Account B
  • Data stays here (S3) — no data movement
  • Via RAM Share or Direct LF grant

📥 Consumer Account

  • Create resource link to shared database
  • Grant local roles access to the link
  • Query with Athena/EMR/Redshift
  • Compute is consumer-side
# PRODUCER: Grant cross-account access aws lakeformation grant-permissions \ --principal '{"DataLakePrincipalIdentifier":"987654321098"}' \ --resource '{"Table":{"DatabaseName":"curated","Name":"daily_sales"}}' \ --permissions '["SELECT","DESCRIBE"]' \ --permissions-with-grant-option '["SELECT","DESCRIBE"]' # CONSUMER: Create resource link (pointer to shared database) aws glue create-database \ --database-input '{"Name":"shared_curated","TargetDatabase":{"CatalogId":"123456789012","DatabaseName":"curated"}}'
💡 Interview Gold

S3 bucket policies give path-level access — no schema, no column filtering. Lake Formation shares at the table/column level with full governance. LF handles credential vending so you don't expose your S3 bucket at all. The consumer doesn't even know your bucket name.

⚙️ Step Functions — State Machine Orchestration

AWS Step Functions coordinate multiple services into visual workflows using state machines. Think of them as Airflow, but serverless and AWS-native.

RetailMart Nightly ETL Pipeline

START → Check S3 for today's raw files (Lambda) │ Files > 0? YES │ NO → Alert & Wait 30m → Retry (max 3) ▼ Parallel Branch: ├── Glue Job: POS ETL (raw → curated) ├── Glue Job: Clickstream ETL └── Glue Job: Inventory ETL │ (wait for all 3) ▼ Data Quality Validation (Lambda: Great Expectations) │ DQ Pass? YES │ NO → Quarantine + Alert PagerDuty ▼ Redshift COPY → Update Glue Catalog (Crawler) → Notify Success (SNS) ▼ END
// ASL snippet — Parallel ETL with Retry + Catch { "RunParallelETL": { "Type": "Parallel", "Branches": [{ "StartAt": "POS_ETL", "States": { "POS_ETL": { "Type": "Task", "Resource": "arn:aws:states:::glue:startJobRun.sync", "Parameters": { "JobName": "retailmart-pos-etl", "Arguments": {"--job-bookmark-option": "job-bookmark-enable"} }, "Retry": [{"ErrorEquals": ["Glue.ConcurrentRunsExceededException"], "IntervalSeconds": 60, "MaxAttempts": 3, "BackoffRate": 2.0}], "Catch": [{"ErrorEquals": ["States.ALL"], "Next": "POS_ETL_Failed"}], "End": true } } }], "Next": "DataQualityCheck" } }
🎯 Interview Tip

Step Functions when: all-AWS, event-driven, visual debugging, zero infra. MWAA (Airflow) when: existing DAGs, complex scheduling/sensors/backfill, non-AWS integrations, Python-first API. RetailMart uses Step Functions for nightly ETL and MWAA for the monthly ML retraining pipeline.

📋 Standard (Nightly ETL)

  • 12 states, runs 45 minutes
  • Must be exactly-once
  • 1 execution per night
  • Cost: negligible

⚡ Express (Per-Order)

  • 4 states, 2-3 seconds each
  • 50,000 orders/hour at peak
  • At-least-once is acceptable
  • Cost: much cheaper at volume
💡 Interview Gold

Q: "100K executions/day, 5 states each — Standard or Express?"
Standard: 500K transitions/day × 30 = 15M/month × $0.025/1K = $375/mo.
Express: 100K × $0.000001 + duration = ~$153/mo. Express is ~60% cheaper. Rule of thumb: high-volume, short-lived → Express.

🛡️ Error Handling — Retry, Catch & Fallback

// Task error flow Task Executes ├── Success ──► Next State ├── Retryable Error ──► Retry Policy │ ├── Attempt 1 (wait 5s) │ ├── Attempt 2 (wait 10s) │ └── Attempt 3 (wait 20s) │ └── Recovered? YES → Next State │ NO → Catch Handler └── Non-retryable Error ──► Catch Handler ──► Fallback State
// Retry + Catch configuration { "Retry": [ {"ErrorEquals": ["Glue.ConcurrentRunsExceededException"], "IntervalSeconds": 60, "MaxAttempts": 5, "BackoffRate": 2.0}, {"ErrorEquals": ["Lambda.TooManyRequestsException"], "IntervalSeconds": 5, "MaxAttempts": 3, "BackoffRate": 2.0}, {"ErrorEquals": ["States.Timeout"], "IntervalSeconds": 30, "MaxAttempts": 2} ], "Catch": [ {"ErrorEquals": ["Glue.EntityNotFoundException"], "Next": "HandleMissingTable"}, {"ErrorEquals": ["States.ALL"], "Next": "GenericErrorHandler"} ] }
Error TypeMeaning
States.ALLCatches everything (wildcard)
States.TimeoutTask exceeded TimeoutSeconds
States.TaskFailedTask returned a failure
States.PermissionsInsufficient IAM permissions
States.HeartbeatTimeoutLong task missed heartbeat
💡 Interview Gold

Q: "Glue job intermittently fails with OOM. How do you handle it?" — (1) Retry with exponential backoff — OOM can be transient. (2) Catch after max retries → fallback state that re-runs with G.2X instead of G.1X. (3) If fallback fails → alert PagerDuty. The state machine encodes the escalation policy: retry → scale up → alert human. No silent failures.

🔑 IAM for Data Pipelines

Every AWS service in your pipeline needs an IAM role. Understanding the trust chain is critical for debugging "Access Denied" errors.

// IAM Role Chain for ETL Pipeline EventBridge Rule → Assumes: StepFunctions Execution Role Trust: events.amazonaws.com Permissions: states:StartExecution Step Functions Execution Role → Passes role to Glue Trust: states.amazonaws.com Permissions: glue:StartJobRun, lambda:InvokeFunction, iam:PassRole, sns:Publish Glue ETL Job Role → Reads/writes S3, accesses Catalog Trust: glue.amazonaws.com Permissions: s3:GetObject, s3:PutObject, glue:GetTable, kms:Decrypt Redshift Copy Role → Reads S3 curated data Trust: redshift.amazonaws.com Permissions: s3:GetObject, s3:ListBucket, kms:Decrypt

Least-Privilege Glue ETL Policy (Key Statements)

{ "Sid": "ReadRawData", "Effect": "Allow", "Action": ["s3:GetObject", "s3:ListBucket"], "Resource": ["arn:aws:s3:::retailmart-data-lake-prod", "arn:aws:s3:::retailmart-data-lake-prod/raw/*"] }, { "Sid": "KMSDecryptEncrypt", "Effect": "Allow", "Action": ["kms:Decrypt", "kms:GenerateDataKey"], "Resource": "arn:aws:kms:us-east-1:123456789:key/mrk-abcdef", "Condition": {"StringEquals": {"kms:ViaService": "s3.us-east-1.amazonaws.com"}} }
🎯 Interview Tip

Q: "Glue job gets Access Denied reading S3." — Debug order: (1) Which IAM role? (2) S3 bucket policy deny? (3) KMS kms:Decrypt on that key? (4) Lake Formation grants? (5) VPC endpoint policy? (6) Organization SCP? Use aws iam simulate-principal-policy to test without trial-and-error.

📋 Policy Evaluation & KMS Keys

// S3 Access Evaluation Flow Request to access S3 object │ ▼ Organization SCP ──── Deny? → DENIED │ ▼ IAM Policy ──── Explicit Deny? → DENIED │ ▼ S3 Bucket Policy ──── Explicit Deny? → DENIED │ ▼ Union of Allows ──── Any Allow? → ALLOWED │ ──── No Allow? → DENIED │ ▼ KMS Key Policy (if encrypted) ──── Allows? → ALLOWED

🏠 Same Account

  • IAM allows s3:GetObject → ALLOWED
  • Bucket policy doesn't need explicit allow
  • Union of IAM + Bucket applies

🌐 Cross Account

  • IAM in Account A allows → NOT ENOUGH
  • Bucket Policy in Account B MUST also allow
  • Both sides must explicitly allow
# KMS key policy — least-privilege for data pipeline roles { "Sid": "GlueETLAccess", "Effect": "Allow", "Principal": {"AWS": "arn:aws:iam::123456789:role/GlueETLRole"}, "Action": ["kms:Decrypt", "kms:GenerateDataKey"], "Resource": "*", "Condition": {"StringEquals": {"kms:ViaService": "s3.us-east-1.amazonaws.com"}} }
💡 Interview Gold

Q: "Cross-account Glue job reads S3 but gets 'Access Denied' on kms:Decrypt." — The KMS key policy in the source account must explicitly grant the cross-account role kms:Decrypt. Unlike S3 bucket policies, KMS key policies are the primary authority — if the key policy doesn't grant the external role, no amount of IAM policies will help. Add kms:ViaService condition for least privilege.

🔑 Key Takeaways

🏛️ Lake Formation

Centralizes governance — one place for database, table, column, and row-level permissions. Revoke IAMAllowedPrincipals or old IAM policies bypass everything.

🏷️ LF-Tags Scale

Tag-based policies mean new tables get automatic access grants. O(tags) policies vs O(tables × roles) grants.

⚙️ Step Functions

AWS-native orchestration with visual debugging. Standard for long ETL, Express for high-volume micro-workflows.

🔑 IAM Chain

EventBridge → Step Functions → Glue → S3 → KMS. Every link needs trust + permissions. Cross-account = both sides must allow.
