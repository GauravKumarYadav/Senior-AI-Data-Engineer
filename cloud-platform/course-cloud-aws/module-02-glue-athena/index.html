Glue Data Catalog — The Central Nervous System

Scenario: RetailMart's S3 data lake is filling up with raw POS transactions, clickstream logs, and inventory feeds. Now you need to catalog it, transform it, and let business users query it with SQL — all serverless.

The Glue Data Catalog is a fully managed metastore. Every service reads from this single source of truth.

💡 Interview Insight

Q: "How does Glue Data Catalog compare to a standalone Hive metastore?" — It's API-compatible with HMS (drop-in replacement for EMR). Key wins: (1) Serverless — no EC2/RDS to manage. (2) Cross-service — Athena, Redshift Spectrum, EMR share one catalog. (3) IAM-based access control. (4) Schema versioning built in. Limit: soft cap of 1M tables/account.

Crawlers — Automatic Schema Discovery

Crawlers scan data sources (S3, JDBC, DynamoDB), infer schemas, and register or update tables in the Data Catalog. They're the "auto-discovery" engine that saves you from writing DDL by hand.

🎯 Interview Tip

Q: "Your crawler keeps creating new tables instead of updating the existing one. Why?" — Most common cause: schema changes interpreted as a new table. Fix: (1) Set UpdateBehavior = UPDATE_IN_DATABASE. (2) Configure grouping behavior to merge compatible schemas. (3) Use table prefixes. (4) If raw data genuinely has inconsistent schemas, normalize with Glue ETL before crawling.

Glue ETL Jobs — Spark Transformations

Glue ETL jobs are managed Apache Spark environments. Write PySpark scripts, Glue handles cluster provisioning, scaling, and teardown.

# RetailMart: Raw CSV → Curated Parquet from awsglue.context import GlueContext from awsglue.job import Job from pyspark.sql.functions import col, year, month glueContext = GlueContext(SparkContext()) job = Job(glueContext) # Read from Data Catalog raw_dyf = glueContext.create_dynamic_frame \ .from_catalog( database="raw_zone", table_name="pos_transactions") # Resolve ambiguous types resolved = raw_dyf.resolveChoice( choice="match_catalog", database="raw_zone", table_name="pos_transactions") # Transform with Spark DataFrame df = resolved.toDF() df_clean = df \ .filter(col("total_amount") > 0) \ .filter(col("transaction_id").isNotNull()) # Write partitioned Parquet sink = glueContext.getSink( connection_type="s3", path="s3://retailmart/curated/", partitionKeys=["year","month"]) sink.setFormat("glueparquet") sink.writeFrame(output_dyf) job.commit()

Glue Bookmarks — Incremental Processing

Bookmarks track which data has already been processed, enabling incremental ETL without reprocessing the entire dataset.

# How Bookmarks Work Run 1 (Day 1): Reads: s3://raw/pos/2026/04/01/ → Processes 2.3 TB Saves bookmark: {path: "…/04/01/", ts: "2026-04-01T23:59:59"} Run 2 (Day 2): Reads bookmark → Knows Day 1 is done Reads: s3://raw/pos/2026/04/02/ → Processes 2.1 TB (NEW files only) Run 3 (Day 3): Reads: s3://raw/pos/2026/04/03/ → Only new data ...

Bookmark Caveats

ScenarioBookmark Behavior
New files in existing partition✅ Detected and processed
Modified existing files⚠️ Only if timestamp changes
Deleted files❌ Not tracked
Schema changes✅ Works (re-reads with new schema)
Job fails mid-run⟲ Bookmark NOT updated; full retry
Re-partition existing data❌ Must reset bookmark
# Enable bookmarks on a job run aws glue start-job-run \ --job-name retailmart-pos-etl \ --arguments '{"--job-bookmark-option": "job-bookmark-enable"}' # Reset bookmark (reprocess everything) aws glue reset-job-bookmark --job-name retailmart-pos-etl
🎯 Interview Tip

Q: "Bookmarks missed some late-arriving files. How?" — Bookmarks track by file timestamp and path. Late files written to an already-passed partition get missed. Fixes: (1) S3 events + SQS for explicit file tracking. (2) Lookback window — re-read last 2 days. (3) Iceberg/Delta Lake with merge operations. RetailMart uses a 24-hour lookback for late supplier feeds.

DynamicFrames, resolveChoice & Schema Registry

DynamicFrames handle messy real-world data with per-record schemas. They're your ingestion workhorse before converting to DataFrames for heavy transforms.

🔄 DynamicFrame

  • ✅ Per-record schema (flexible)
  • ✅ resolveChoice() for type mismatches
  • ✅ Built-in null handling
  • ✅ Direct Data Catalog R/W
  • ⚠️ Slight performance overhead
  • 🎯 Best for: ingestion, raw data

📊 Spark DataFrame

  • ✅ Enforced schema (rigid, fast)
  • ❌ Fails or casts silently on mismatch
  • ⚠️ Manual .na.fill()
  • ⚠️ Via Glue catalog API
  • ✅ Faster for large joins
  • 🎯 Best for: complex transforms, ML
# resolveChoice — handling messy POS data # (total_amount is sometimes string, sometimes double) # Option 1: Cast to specific type resolved = raw_dyf.resolveChoice(specs=[ ("total_amount", "cast:double"), ("quantity", "cast:int"), ]) # Option 2: Match the catalog definition resolved = raw_dyf.resolveChoice( choice="match_catalog", database="raw_zone", table_name="pos_transactions", ) # Option 3: Split into separate columns resolved = raw_dyf.resolveChoice(specs=[ ("total_amount", "make_cols"), ]) # → total_amount_double, total_amount_string

Q: "When should you convert DynamicFrame → DataFrame?" — When you need: (1) Complex joins. (2) Window functions (rank(), lag()). (3) ML feature engineering. (4) UDFs. The pattern: read with DynamicFrame → resolveChoice → .toDF() → transform → DynamicFrame.fromDF() → write via Glue sink.

Amazon Athena — Serverless SQL on S3

Athena is a serverless, interactive query service built on Trino. Point it at S3 data in the Glue Data Catalog and run standard SQL.

-- Top 10 stores by revenue, last 7 days SELECT store_id, COUNT(*) AS transaction_count, SUM(total_amount) AS total_revenue, AVG(total_amount) AS avg_transaction_value, APPROX_PERCENTILE(total_amount, 0.95) AS p95_transaction FROM curated.daily_sales WHERE year = '2026' AND month = '04' AND day BETWEEN '01' AND '07' GROUP BY store_id ORDER BY total_revenue DESC LIMIT 10;

Partition Projection — Skip MSCK REPAIR TABLE

Instead of storing partition metadata in the catalog, partition projection tells Athena to compute partitions on the fly — no crawler needed.

-- Partition projection: Athena computes paths, no catalog updates CREATE EXTERNAL TABLE curated.daily_sales_projected ( transaction_id STRING, store_id INT, total_amount DECIMAL(10,2) ) PARTITIONED BY (year STRING, month STRING, day STRING) STORED AS PARQUET LOCATION 's3://retailmart/curated/daily_sales/' TBLPROPERTIES ( 'projection.enabled' = 'true', 'projection.year.type' = 'integer', 'projection.year.range' = '2020,2030', 'projection.month.type' = 'integer', 'projection.month.range' = '1,12', 'projection.month.digits'= '2', 'projection.day.type' = 'integer', 'projection.day.range' = '1,31', 'projection.day.digits' = '2', 'storage.location.template' = 's3://retailmart/curated/daily_sales/year=${year}/month=${month}/day=${day}/' );
💡 Interview Insight

Q: "An Athena query scans 500 GB but only needs 1 day. How do you fix it?" — Three levers: (1) Partition pruning — filter on partition columns. (2) Columnar format — convert CSV to Parquet (selecting 5/50 columns scans ~10%). (3) Compression — Snappy/ZSTD. Combined: 500 GB → under 5 GB → saving $2.50/query. RetailMart cut Athena costs by 94%.

Athena Advanced — Federation, Iceberg & Notebooks

Query Federation

Athena queries data outside S3 using Lambda connectors — DynamoDB, RDS, Redshift, CloudWatch Logs, and custom sources.

-- Federated query: join S3 + DynamoDB + RDS in one query SELECT s.store_id, p.product_name, SUM(s.total_amount) AS revenue FROM curated.daily_sales s JOIN dynamodb.product_catalog p ON s.product_sku = p.sku JOIN rds.store_metadata m ON s.store_id = m.store_id GROUP BY s.store_id, p.product_name;

Apache Iceberg on Athena

Iceberg tables add ACID transactions, time travel, schema evolution, and row-level updates/deletes to S3.

-- UPSERT with Iceberg MERGE — handle late corrections MERGE INTO curated.daily_sales_iceberg target USING staging.corrections source ON target.transaction_id = source.transaction_id WHEN MATCHED THEN UPDATE SET total_amount = source.corrected_amount WHEN NOT MATCHED THEN INSERT (transaction_id, store_id, total_amount, transaction_ts) VALUES (source.transaction_id, source.store_id, source.corrected_amount, source.transaction_ts); -- Time travel — query data as it was 3 days ago SELECT COUNT(*), SUM(total_amount) FROM curated.daily_sales_iceberg FOR TIMESTAMP AS OF TIMESTAMP '2026-04-03 00:00:00';
💡 Interview Insight

Q: "When Iceberg vs. standard Hive-style Parquet?" — Use Iceberg when: (1) Row-level updates/deletes (GDPR, corrections). (2) Time travel for audit. (3) Concurrent writers (ACID). (4) Schema evolution without rewriting. Stick with Hive-style for: append-only, stable schema, max tool compatibility.
