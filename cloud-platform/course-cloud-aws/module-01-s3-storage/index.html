☁️ Amazon S3 Storage Classes

Scenario: RetailMart, a national retail chain with 4,000+ stores, is building a cloud-native data lake on AWS. Every day, 2.3 TB of POS transactions, inventory snapshots, clickstream logs, and supplier feeds land in S3. Your job: architect the storage layer.

S3 offers a spectrum of storage classes that trade retrieval speed for cost per GB. Choosing correctly is the difference between a $4K/month bill and a $40K/month bill.

Use Intelligent-Tiering when access patterns are unpredictable. For millions of tiny files (like clickstream JSON), manual lifecycle rules to Standard-IA are cheaper. For ad-hoc analyst uploads with unknown frequency, Intelligent-Tiering wins.

Lifecycle Policies — Automating Data Movement

Lifecycle policies define rules that automatically transition objects between storage classes or expire (delete) them. This is how you avoid the #1 S3 cost mistake: forgetting data exists.

// Lifecycle policy for POS transactions { "Rules": [{ "ID": "retailmart-pos-lifecycle", "Status": "Enabled", "Filter": { "Prefix": "raw/pos-transactions/" }, "Transitions": [ { "Days": 30, "StorageClass": "STANDARD_IA" }, { "Days": 90, "StorageClass": "GLACIER_IR" }, { "Days": 365, "StorageClass": "DEEP_ARCHIVE" } ], "Expiration": { "Days": 2555 } }] }

Key Transition Rules

Day 0 Day 30 Day 90 Day 365 Day 2555 │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ Standard ──► Standard-IA ──► Glacier Instant ──► Deep Archive ──► DELETE │ 7-year retention (tax compliance)

Important: You cannot transition from a colder class to a warmer one. Minimum object size for IA is 128 KB. Transitions only go one direction.

💡 Interview Insight

Q: "How do you handle the 128 KB minimum for Standard-IA with millions of small log files?" — Two strategies: (1) Aggregate before transitioning — compact small files into larger Parquet files. (2) Use Intelligent-Tiering instead — no minimum object size charge.

S3 Select — Query in Place

S3 Select lets you run SQL expressions directly against objects in S3 without downloading the entire file. Transformative when you need a needle from a haystack.

# S3 Select: Extract high-value txns aws s3api select-object-content \ --bucket retailmart-data-lake-prod \ --key "raw/pos/2026/04/05/store-0042.csv" \ --expression "SELECT s.transaction_id, s.total_amount FROM s3object s WHERE CAST(s.total_amount AS FLOAT) > 500.00" \ --expression-type SQL \ --input-serialization \ '{"CSV": {"FileHeaderInfo": "USE"}}' \ --output-serialization '{"JSON": {}}' \ output.json
📦 Run SQL directly on a single S3 object
 
🎯 Target a specific CSV file in the bucket

S3 Versioning — Protecting Your Data Lake

S3 Versioning keeps every version of every object. Combined with MFA Delete, it's your last line of defense against accidental deletions and overwrites.

# Enable versioning + MFA Delete aws s3api put-bucket-versioning \ --bucket retailmart-data-lake-prod \ --versioning-configuration Status=Enabled # MFA Delete (requires root credentials) aws s3api put-bucket-versioning \ --bucket retailmart-data-lake-prod \ --versioning-configuration Status=Enabled,MFADelete=Enabled \ --mfa "arn:aws:iam::123456789:mfa/root-device 123456"
# How versioning handles deletes and overwrites Object: raw/pos-transactions/2026/04/05/store-0042.parquet t=1: PUT (v1) ──► Version ID: abc111 ◄── Original upload t=2: PUT (v2) ──► Version ID: abc222 ◄── Schema change re-upload t=3: DELETE ──► Delete Marker added ◄── "Deleted" (v1, v2 remain!) t=4: DELETE v-abc222 ► Permanently removes v2

Noncurrent Version Lifecycle

Every version is a full copy. Without lifecycle rules, storage costs grow unbounded. Always pair versioning with noncurrent version expiration:

// Keep 3 most recent noncurrent versions, delete after 30 days { "NoncurrentVersionExpiration": { "NoncurrentDays": 30, "NewerNoncurrentVersions": 3 } }
🎯 Interview Tip

Q: "A junior engineer deleted the entire curated/ prefix. How do you recover?" — If versioning is on, S3 added delete markers. List all delete markers, remove them, and previous versions become current again. Script with list-object-versions + jq + batch delete. Prevention: MFA Delete + restrict s3:DeleteObject via IAM.

S3 Transfer Acceleration

Uses CloudFront edge locations to accelerate uploads over long distances — 40-60% faster for remote stores.

# Enable Transfer Acceleration aws s3api put-bucket-accelerate-configuration \ --bucket retailmart-data-lake-prod \ --accelerate-configuration Status=Enabled # Upload using the accelerated endpoint aws s3 cp store-recon-0042.parquet \ s3://retailmart-data-lake-prod/raw/store-recon/ \ --endpoint-url https://retailmart-data-lake-prod.s3-accelerate.amazonaws.com
💡 Interview Insight

Q: "Does CRR replicate existing objects?" — No! CRR only replicates objects uploaded after the rule is enabled. Use S3 Batch Replication for existing objects. Also: delete markers can be replicated, but permanent deletes are NOT — a safety feature against cascade deletion.

S3 Performance — Maximizing Throughput

S3 supports 5,500 GET/HEAD and 3,500 PUT/POST/DELETE requests per second per prefix. Prefix design is critical.

# Prefix design matters! ❌ BAD — All files under one prefix (bottleneck): s3://retailmart/data/2026-04-05-store-0042-txn-0001.parquet ✅ GOOD — Distributed across prefixes: s3://retailmart/raw/pos-transactions/2026/04/05/store=0042/part-0001.parquet s3://retailmart/raw/clickstream/2026/04/05/session=abc123/part-0001.json Each unique prefix path gets its own 5,500/3,500 RPS budget.

Multipart Upload

Objects > 100 MB should use multipart. Objects > 5 GB must use multipart.

# Python multipart upload with TransferConfig import boto3 from boto3.s3.transfer import TransferConfig s3 = boto3.client("s3") config = TransferConfig( multipart_threshold=64 * 1024 * 1024, # 64 MB max_concurrency=20, multipart_chunksize=16 * 1024 * 1024, # 16 MB use_threads=True, ) s3.upload_file( "daily-pos-aggregate.parquet", "retailmart-data-lake-prod", "curated/daily-pos/2026/04/05/aggregate.parquet", Config=config, )
🎯 Interview Tip

Q: "Your Spark job reading from S3 is slow. Diagnose it." — (1) Shared prefix bottleneck → distribute paths. (2) Small file problem → compact into 128 MB–1 GB files. (3) Check CloudWatch for 503 SlowDown errors. (4) Use s3a:// with fs.s3a.connection.maximum=100. (5) Enable Transfer Acceleration if cross-region.

S3 Access Points & Object Lambda

As the data lake grows, managing one bucket policy with dozens of conditions becomes unmanageable. S3 Access Points create named endpoints with dedicated access policies — one per team.

🔧 Data Engineering

Full R/W to raw/ and curated/. VPC-restricted to vpc-prod-123.

📊 Analysts

Read-only on curated/ and analytics/. VPC-restricted to vpc-analytics.

🧠 ML Team

Read curated/, write ml-features/. VPC-restricted to vpc-ml-123.

📦 Vendor Feed

Write-only to raw/supplier-feeds/. Internet-facing with IAM auth.

S3 Object Lambda

Transform data on the fly as it's retrieved — no copy needed. RetailMart uses this to redact PII from customer records when the analytics team reads them.

# Object Lambda flow — PII redaction on read Analyst GET ──► S3 Object Lambda Access Point ──► Lambda (redact PII) ──► Sanitized data │ ▼ S3 (original data with PII intact)
💡 Interview Insight

Q: "How do you give 15 teams access to the same bucket without a 500-line policy?" — S3 Access Points. Each team gets a dedicated access point with its own IAM policy, VPC restriction, and prefix scope. The bucket policy delegates to access points. Adding a team = new access point, not editing a monolithic policy.

SSE-KMS Bucket Key — Why It Matters

# KMS API call reduction with Bucket Key Without Bucket Key: With Bucket Key: PUT obj-1 → KMS call PUT obj-1 ──┐ PUT obj-2 → KMS call ├──► KMS call (1x) PUT obj-3 → KMS call PUT obj-2 ──┤ → Bucket Key generated PUT obj-4 → KMS call PUT obj-3 ──┤ → Reused for all objects ...10,000 objects = 10,000 calls PUT obj-4 ──┘ ...10,000 objects ≈ 1 KMS call
🎯 Interview Tip

Q: "You're hitting KMS throttling during a bulk load." — (1) Enable Bucket Key — 99% fewer KMS calls. (2) Request KMS quota increase. (3) Verify KMS key policy grants kms:GenerateDataKey and kms:Decrypt. (4) Consider SSE-S3 if you don't need per-key audit trails.
