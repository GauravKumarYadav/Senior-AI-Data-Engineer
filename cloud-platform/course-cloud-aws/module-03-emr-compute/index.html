EMR Deployment Models — Choose Your Fighter

Amazon EMR offers three deployment models, each with distinct operational characteristics. Picking the right one is one of the highest-leverage architecture decisions you'll make.

🖥️ EMR on EC2 (Traditional)

Master + Core + Task nodes. You manage instance types, scaling rules, bootstrap actions. Full customization. Startup: 5-10 min. Supports Spark, Hive, Presto, HBase, GPU instances.

☸️ EMR on EKS (Kubernetes)

Spark as pods on your existing EKS cluster. You manage EKS cluster & namespaces. Startup: 1-2 min. Spark & Hive only. Great for K8s-native orgs wanting to consolidate.

⚡ EMR Serverless (Fully Managed)

No cluster to manage — submit job, get results. Auto-provisions workers. Startup: 30 sec - 1 min. Spark & Hive only. No GPU. Pay per vCPU-hour + GB-hour.

💡 Interview Gold

Q: "How do you architect EMR for both nightly batch + ad-hoc data science?" — Hybrid: (1) EMR Serverless for nightly batch ETL — zero cluster mgmt, auto-scales, shuts down when done. (2) EMR on EC2 persistent cluster for data science — interactive notebooks, custom libs, GPU instances. (3) If already on EKS, consider EMR on EKS to consolidate the K8s footprint.

EMR on EC2 — The Workhorse

The original model: define a cluster with master, core, and task nodes. EMR provisions EC2 instances with Hadoop, Spark, Hive pre-installed.

👑 Master Node

YARN ResourceManager, Spark Driver, HDFS NameNode. Single node, never Spot. m5/m6i.xlarge.

💪 Core Nodes

YARN NodeManager + HDFS DataNode + Spark Executor. Scale out only. On-Demand preferred (holds HDFS data). r5/r6i.2xlarge.

🏃 Task Nodes

Compute only — no HDFS. Scale in/out freely. Perfect for Spot instances. Spark retries lost tasks automatically.

# Create a production EMR cluster with auto-scaling task nodes aws emr create-cluster \ --name "retailmart-spark-cluster" \ --release-label "emr-7.1.0" \ --applications Name=Spark Name=Hive Name=JupyterEnterpriseGateway \ --instance-groups '[ {"InstanceGroupType":"MASTER","InstanceType":"m6i.xlarge","InstanceCount":1}, {"InstanceGroupType":"CORE","InstanceType":"r6i.2xlarge","InstanceCount":5}, {"InstanceGroupType":"TASK","InstanceType":"r6i.2xlarge","InstanceCount":0, "Market":"SPOT","BidPrice":"OnDemandPrice"} ]' \ --auto-termination-policy '{"IdleTimeout": 3600}'
💡 Instance Fleets vs. Instance Groups

Instance Groups lock you into a single instance type per group. Instance Fleets let you specify a list of types (r5.2xl, r5.4xl, r6i.2xl…) and EMR picks the best available — critical for Spot. Fleets reduce Spot interruptions by 3-5x. Always use capacity-optimized allocation strategy.

EMR on EKS — Spark on Kubernetes

Run Spark workloads on your existing EKS cluster as Kubernetes pods. Spark shares nodes with your microservices — no separate EMR cluster needed.

☸️ How It Works

  • 📦 Register EKS cluster with EMR as a "virtual cluster"
  • 🏷️ Spark runs in a dedicated namespace
  • 🐳 Driver + Executor pods auto-created per job
  • ⚡ ~30 sec startup if nodes pre-provisioned (Karpenter)
  • 📊 Pod templates for fine-grained resource control

✅ Choose EKS When

  • 🏢 Org already runs EKS — consolidate infra
  • 🔒 Need multi-tenant isolation via namespaces
  • 🤝 Co-locate Spark with APIs/ML serving
  • ⚡ Need fast startup with pre-provisioned nodes
# Register EKS cluster, then submit a Spark job aws emr-containers start-job-run \ --virtual-cluster-id "vc-abc123def456" \ --name "daily-sales-aggregation" \ --release-label "emr-7.1.0-latest" \ --job-driver '{"sparkSubmitJobDriver":{ "entryPoint":"s3://retailmart-scripts/spark/daily_sales_agg.py", "sparkSubmitParameters":"--conf spark.executor.instances=10 --conf spark.executor.memory=8g --conf spark.executor.cores=4" }}'

EMR Serverless — Zero Infrastructure

Submit a Spark or Hive app, AWS provisions compute automatically. No clusters, no nodes, no scaling policies. Pay per vCPU-hour ($0.053) + GB-hour ($0.006).

📊 Example Cost

10 vCPU, 40 GB, 1 hour = (10 × $0.053) + (40 × $0.006) = $0.77. Compare to a full EMR cluster at $8-15/hr.

🔥 Pre-Initialized Capacity

Keep workers warm for sub-30s startup. You pay while idle, but auto-stop releases them after a configurable timeout.

💡 EMR Serverless vs. Glue ETL — Top Interview Q

Glue: Simple, catalog-centric ETL with job bookmarks, DynamicFrame, crawlers. May lag on Spark version.
EMR Serverless: Full Spark control, latest version (3.5+), custom JARs, Iceberg/Delta heavy usage, ~30s startup.
Rule of thumb: Glue for CSV→Parquet ingestion. EMR Serverless for complex aggregation pipelines with Iceberg merges.

# Create application + submit job aws emr-serverless create-application \ --name "retailmart-batch" --type "SPARK" \ --release-label "emr-7.1.0" \ --maximum-capacity '{"cpu":"200vCPU","memory":"400GB"}' \ --auto-stop-configuration '{"enabled":true,"idleTimeoutMinutes":5}' aws emr-serverless start-job-run \ --application-id "app-abc123" \ --name "daily-sales-etl" \ --job-driver '{"sparkSubmit":{ "entryPoint":"s3://retailmart-scripts/spark/daily_sales_etl.py", "sparkSubmitParameters":"--conf spark.sql.adaptive.enabled=true" }}'

EMR Studio & Bootstrap Actions

EMR Studio: Managed JupyterLab notebooks connected to EMR clusters (EC2, EKS, or Serverless). Git integration, S3 persistence, SSO auth.

# EMR Studio PySpark — Sales Feature Engineering from pyspark.sql import functions as F from pyspark.sql.window import Window df = spark.sql(""" SELECT store_id, day, SUM(total_amount) AS daily_revenue FROM glue_catalog.curated.daily_sales WHERE year = '2026' GROUP BY store_id, day """) window_7d = Window.partitionBy("store_id").orderBy("day").rowsBetween(-6, 0) df_features = df.withColumn("rolling_7d_avg", F.avg("daily_revenue").over(window_7d))

Bootstrap Actions

Shell scripts that run on every node during startup — install packages, download models, configure Spark.

#!/bin/bash — bootstrap_retailmart.sh sudo pip3 install pandas==2.2.1 pyarrow==15.0.0 scikit-learn==1.4.0 # Master-only: download ML model artifacts if grep -q isMaster /mnt/var/lib/info/instance.json; then aws s3 cp s3://retailmart-models/latest/ /opt/models/ --recursive fi
💡 Speed Up Bootstrap

20 Python packages = +5 min startup. Fix: (1) Custom AMI — pre-bake dependencies, zero startup penalty. (2) Private PyPI on S3 — faster than public. (3) For EKS: custom Docker image — fastest, most portable. Use custom AMI for prod, bootstrap for dev.

EMR vs. Glue — The Decision Framework

This is the #1 most common interview question in AWS data engineering.

🔧 AWS Glue ETL

  • ✅ Serverless — zero provisioning
  • ✅ Native Data Catalog integration
  • ✅ Job bookmarks (incremental)
  • ✅ DynamicFrame for messy data
  • ✅ Crawlers for schema discovery
  • ⚠️ May lag on Spark versions
  • ⚠️ Limited custom JARs/configs
  • ❌ No GPU, no non-Spark frameworks

⚡ EMR (EC2 / Serverless / EKS)

  • ✅ Latest Spark version (3.5+)
  • ✅ Full custom JARs, configs, libs
  • ✅ EMR Serverless: ~30s startup
  • ✅ GPU support (EC2 model)
  • ✅ Hive, Presto, HBase, Flink
  • ⚠️ Catalog integration = manual config
  • ⚠️ No job bookmarks (DIY)
  • ⚠️ EC2 model: higher ops overhead
💡 RetailMart's Hybrid Strategy ($5,900/mo)

Ingestion: Glue ETL — CSV→Parquet, bookmarks, crawlers ($800/mo).
Transform: EMR Serverless — Iceberg merges, AQE, custom Python ($1,200/mo).
Exploration: EMR on EC2 + Studio — notebooks, GPU, YARN queues ($3,500/mo).
Query: Athena — ad-hoc SQL, partition projection ($400/mo).

Cost Optimization — Spot, Scaling & Right-Sizing

A misconfigured cluster can cost 10x more than necessary. Cost is the #1 EMR concern.

👑 Master: Always On-Demand

If master dies, entire cluster dies. Never Spot.

💪 Core: On-Demand Preferred

Holds HDFS data. Spot termination = potential data loss. Use EBS for HDFS if mixing Spot.

🏃 Task: 100% Spot

Compute-only, no HDFS. Spot = 60-90% savings. Spark retries lost tasks automatically.

# Managed auto-scaling: 10-100 units, max 20 On-Demand aws emr put-managed-scaling-policy \ --cluster-id "j-abc123" \ --managed-scaling-policy '{"ComputeLimits":{ "UnitType":"InstanceFleetUnits", "MinimumCapacityUnits":10,"MaximumCapacityUnits":100, "MaximumOnDemandCapacityUnits":20,"MaximumCoreCapacityUnits":20 }}'
💡 "Cut EMR costs by 70%" — Interview Answer

(1) Spot on task nodes — move 40 of 50 instances to Spot. (2) Instance Fleets — 5-6 types, capacity-optimized. (3) Right-size — if YARN memory is 40%, halve the nodes. (4) Graviton — r6g/m6g = 20% cheaper, 40% better perf. (5) Auto-scaling — start small, scale during peak. (6) Transient clusters — don't run 24/7 for a 2-hour job. Auto-terminate after 1hr idle.

🔑 Key Takeaways

Three Models, One Framework

EMR on EC2 (max control), EMR on EKS (K8s-native), EMR Serverless (zero ops). Pick based on operational maturity.

Spot = Free Money

60-90% savings on task nodes. Never Spot the master. Use Instance Fleets for Spot diversity.

EMR vs. Glue: Use Both

Glue for catalog-centric ingestion + bookmarks. EMR Serverless for complex transforms + latest Spark. EMR on EC2 for interactive + GPU.

Right-Size First

Check YARN memory/CPU before anything. Most clusters are 2-3x over-provisioned. Highest-ROI optimization.

Auto-Terminate = Mandatory

Every persistent cluster needs idle timeout. Forgotten clusters are the #1 EMR cost leak.

Studio for Exploration

Data scientists get JupyterLab without managing infra. But always promote notebooks to production scripts.
