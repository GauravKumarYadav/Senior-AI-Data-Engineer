🔥 Architecture Design — "Draw Me a Pipeline"

No gentle explanations. No hand-holding. You're across from a senior data engineer. They want to see you reason through trade-offs, not just name services.

Q1: Design a Data Lake for a 10,000-Store Retail Chain

"5 TB/day. POS, clickstream, inventory, supplier feeds. Analysts want SQL. ML engineers need raw data. 7-year retention. PII controls."

Ingestion: POS → Kinesis Data Streams (10 shards) → Firehose (Parquet, partitioned by date/store) → S3 raw zone. Clickstream → Kinesis (On-Demand). Inventory → S3 Transfer Acceleration + Lambda validation. Supplier → S3 Access Points (write-only).

Storage (S3): raw/ (immutable) → curated/ (Parquet/Iceberg) → analytics/ (aggregated) → archive/. Lifecycle: Standard 30d → IA 90d → Glacier Instant 365d → Deep Archive 7yr. SSE-KMS with Bucket Key.

Processing: Glue ETL with bookmarks (raw → curated). Step Functions orchestrates nightly pipeline. EMR for heavy ML feature engineering.

Query: Athena for ad-hoc SQL (workgroup scan limits). Redshift Serverless for dashboards. Spectrum for joining warehouse + historical S3.

Governance: Lake Formation with LF-Tags. Column-level PII security. Cross-account sharing. KMS per-pipeline keys.

Q2: Design a Real-Time Fraud Detection Pipeline

"POS transactions analyzed for fraud within 10 seconds. Card used at 3+ stores in 5 minutes = alert."

POS → Kinesis Data Streams (partition key = card_hash for ordering) → Kinesis Data Analytics (Flink) with 5-min sliding window, 1-min slide → GROUP BY card_id HAVING COUNT(DISTINCT store) ≥ 3 → Alert Kinesis stream → Lambda → PagerDuty.

Key decisions: Enhanced Fan-Out for parallel consumers. Watermark = 30s. Firehose parallel consumer for S3 audit trail. Latency budget: < 5 seconds (POS→Kinesis 200ms, Flink 1-2s, Lambda 500ms).

Q3: Design a CDC Pipeline from RDS to Data Lake

"RDS PostgreSQL. Need change data capture — inserts, updates, deletes."

Recommended: RDS → AWS DMS (CDC mode, WAL-based logical replication) → Kinesis Data Stream → Firehose → S3 (raw CDC events) → Glue ETL → Iceberg table (MERGE INTO on primary key).

Iceberg is essential for DELETE operations (Hive tables are append-only). DMS handles full load → CDC transition automatically. Schema evolution detectable if configured.

Q4: Multi-Region Disaster Recovery

"Data lake is in us-east-1. If the region goes down?"

Tier 1 (RPO 15min, RTO 1hr): S3 CRR for curated/. Redshift cross-region snapshots hourly. Kinesis producer DNS failover.
Tier 2 (RPO 4hr, RTO 24hr): S3 CRR for raw/. Manual Redshift restore.
Cost: DR uses Redshift Serverless (scales to zero). S3 replication to Standard-IA. Only replicate curated/ (raw is re-derivable) — reduces CRR by 60%.

Firehose: Managed bulk delivery to S3/Redshift/OpenSearch.

💰 Cost & Security — "Keep It Cheap and Safe"

Q: "Athena costs $5,000/month. Reduce by 80%."

1. CSV/JSON → Parquet + Snappy (90% scan reduction)
2. Date partitioning (prune entire directories)
3. Partition projection (instant query start)
4. Workgroup scan limits (BytesScannedCutoffPerQuery)
5. CTAS for frequently-run queries
6. Column pruning: never SELECT *
Result: $5,000 → ~$500/month

Q: "Redshift $8K/month and 70% idle. Fix it."

1. Redshift Serverless → pay active time only → ~$2,400/mo
2. Or: Pause cluster evenings/weekends → 60% savings
3. Materialized views pre-compute heavy queries
4. WLM queues to throttle expensive ad-hoc queries

Q: "$15K/month analytics spend → reduce 50%"

Redshift dc2×4 ($5,400) → Serverless ($1,800). Glue full reprocess ($4,200) → bookmarks + Flex ($1,500). Athena raw CSV ($3,000) → Parquet + partitions ($300). S3 all Standard ($2,400) → lifecycle ($1,200). Total: $15K → $4,800 (68% reduction).

🛡️ "How do you prevent a junior from exposing PII?"

Layer 1-2

Lake Formation column-level security (PII excluded from grants). S3 Block Public Access (account-level).

Layer 3-4

VPC endpoints for S3 (no internet). KMS encryption (useless without key).

Layer 5-6

SCPs (prevent disabling encryption). AWS Config rules (alert in 1 hour).

Layer 7-8

CloudTrail + Athena (audit who accessed what). Macie (ML-based PII detection).

🔧 Debugging — "It's Broken, Fix It"

Q1: "Nightly Glue ETL failed at 2 AM."

1. CloudWatch /aws-glue/jobs/error for stack trace
2. Common: OOM → scale to G.2X. Access Denied → IAM/bucket/KMS. Schema mismatch → crawler or resolveChoice.
3. Check Spark UI for data skew/straggler tasks.
4. Bookmark corrupted? Reset with job-bookmark-pause.

Q2: "Athena returns zero results but data exists in S3."

1. SHOW PARTITIONS — is partition registered?
2. S3 path mismatch: must be year=2026/month=04/ not 2026/04/
3. Case sensitivity: Athena case-insensitive but S3 keys are case-sensitive
4. File format mismatch: table says PARQUET but files are JSON
5. Fix: MSCK REPAIR TABLE or ALTER TABLE ADD PARTITION

Q3: "Kinesis consumer falling behind — iterator age growing."

1. Scale shards: UpdateShardCount
2. Enhanced Fan-Out if multiple consumers compete for 2 MB/s
3. Increase Lambda BatchSize + ParallelizationFactor
4. Switch to On-Demand mode if shard management is the bottleneck

Q4: "Redshift COPY 100 GB takes 2 hours."

1. Split files = number_of_slices (parallel COPY)
2. Compress source files (GZIP/ZSTD = 3-5x less I/O)
3. Parquet/ORC > CSV. COMPUPDATE OFF + STATUPDATE OFF for bulk.
4. Check DISTKEY skew. Same region as S3. Use manifest file.

Q5: "Step Functions 'States.Permissions' on Glue state."

Root cause: execution role lacks iam:PassRole for the Glue role. Fix: add iam:PassRole with condition iam:PassedToService = glue.amazonaws.com. Also check glue:StartJobRun on the specific job ARN.

Q6: "S3 PUT requests returning 503 SlowDown."

Exceeding 3,500 PUT/sec on a single prefix. Fix: distribute writes across more prefixes (hash keys), add exponential backoff retry. S3 auto-partitions but starts at 3,500/sec and ramps up.

Q7: "LF permissions granted but analyst still gets 'Access Denied' in Athena."

1. IAMAllowedPrincipals still active? — Most common issue. Revoke it.
2. S3 location registered with LF?
3. Need DESCRIBE on database AND SELECT on table.
4. Column-level grant? SELECT * fails — must select only granted columns.

⚡ Rapid Fire — 20 Quick Q&A

Answer each in ≤ 3 sentences. These come up constantly in phone screens.

1. Max record size in Kinesis?

1 MB. For larger payloads, store in S3 and send the S3 key (claim-check pattern).

2. COMPOUND vs INTERLEAVED sort keys in Redshift?

COMPOUND prioritizes first column (fast for leading-column filters). INTERLEAVED = equal weight (better for ad-hoc, but VACUUM REINDEX is expensive).

3. Can Firehose write to Redshift?

Yes. Stages in S3 first, then auto-issues COPY command. Configure cluster, table, and COPY options in the delivery stream.

4. Min DPUs for a Glue Spark job?

2 DPUs (Glue 4.0). Each DPU = 4 vCPU + 16 GB. Python Shell = 1/16 DPU (0.0625).

5. How does Athena charge?

$5/TB scanned. Cancelled queries charged for data scanned before cancellation. Cached results are free.

6. Redshift Concurrency Scaling?

Auto-spins additional capacity when queries queue. First 1 hour/day free. Beyond = per-second billing.

7. Kinesis On-Demand vs Provisioned?

Provisioned: manage shards (1 MB/s each). On-Demand: auto-scales to 200 MB/s, ~15-20% more expensive at steady state.

8. Redshift materialized view?

Pre-computed query stored on disk. Refreshed manually or auto. Supports incremental refresh for append-only tables.

9. Schema evolution in Glue?

Crawlers detect new columns. For production: Glue Schema Registry (Avro/JSON with compatibility modes). Iceberg handles it natively.

10. S3 Events vs EventBridge for S3?

S3 Events: free, prefix/suffix filtering, Lambda/SQS/SNS only. EventBridge: $1/M events, content-based filtering, 20+ targets, archive/replay.

11. Can Redshift Spectrum write to S3?

No — read-only. Use UNLOAD from Redshift or CTAS in Athena to write to S3.

12. Glue Bookmarks vs Iceberg?

Bookmarks track which files were processed (ETL state). Iceberg = table format with ACID, time travel, row-level mutations. Different problems.

13. Kinesis shard split?

Divides one shard into two for more throughput. Parent stops accepting writes; two children take over. Takes minutes, is disruptive.

14. What's AQUA in Redshift?

Hardware-accelerated cache on RA3 nodes. Pushes scan + aggregation to storage layer. Only on RA3.4xl and RA3.16xl.

15. Kinesis data at rest encryption?

SSE-KMS. Enable: start-stream-encryption --encryption-type KMS. Encrypted before writing to disk.

16. Max concurrent Standard executions?

1,000,000 open per account/region (soft limit). Express: 100,000 start/sec. Standard can run up to 1 year.

17. Redshift RA3 vs DC2?

DC2 = compute + local SSD (coupled). RA3 = compute + S3-backed managed storage with SSD cache (independent scaling). RA3 is the modern choice.

18. Lake Formation + EMR?

Yes. EMR clusters integrate via instance role. LF vends temporary credentials scoped to granted tables/columns. Requires EMR 5.31+.

19. Kinesis Video Streams?

Separate service for streaming video/audio for ML. Not related to Kinesis Data Streams — don't confuse them.

20. DELETE from Iceberg on Athena?

Writes a delete file marking rows deleted. Original data untouched. Run OPTIMIZE to compact and physically remove.

The STAR-T Answer Framework

S — Situation: Restate the problem (shows listening) T — Technology: Name AWS services, justify each choice A — Architecture: Describe data flow (ingestion → storage → processing → serving) R — Risks: Proactively mention failure modes and mitigations T — Trade-offs: Acknowledge what you sacrifice (cost vs latency, simplicity vs flexibility) Example: "So you need real-time fraud detection under 10 seconds" [S] "I'd use Kinesis + Flink" [T] "POS → Kinesis → Flink (5-min window) → Alert → Lambda → PagerDuty" [A] "Risk: if Flink falls behind, we'd monitor iterator age and auto-scale" [R] "Trade-off: Flink adds complexity vs Lambda, but Lambda can't do stateful windows" [T]

❌ Common Mistakes to Avoid

"I'd use Glue for everything"

Shows you don't know when EMR, Athena, or Redshift is better.

"We'd use Kinesis AND Kafka"

Pick one and justify. Using both signals confusion.

"Just use IAM policies"

For data lakes, you NEED Lake Formation for column/row security.

Forgetting cost or failure handling

Always mention lifecycle policies, Spot, Parquet, scan limits. Every pipeline must address: what happens when it fails?

🔑 Final Key Takeaways

The 8 Commandments

1. Know the "why" not just the "what" — interviewers follow up with "why not X?"
2. Draw diagrams — describe architecture as boxes and arrows.
3. Numbers matter — "1 MB/s per shard" beats "a lot of throughput."
4. Cost is a first-class design criterion — mention it proactively.
5. Security is not an afterthought — weave it into your design.
6. Failure handling differentiates seniors — retry, DLQ, alerting, idempotency.
7. Acknowledge trade-offs — "I chose X over Y because… but if Z, I'd reconsider."
8. Practice aloud — reading ≠ articulating. Record yourself.
