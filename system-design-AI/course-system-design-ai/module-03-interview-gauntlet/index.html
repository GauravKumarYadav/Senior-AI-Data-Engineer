HOT Design a RAG system for 10M internal documents

Requirements: Document types (PDFs, wikis, Slack)? Access controls? Latency — internal tool (2-3s) or customer-facing (sub-second)?

Architecture — Two pipelines: Ingestion: Connectors (Confluence, SharePoint, S3) → parse → semantic chunking (~512 tokens, 50-token overlap) → embed via e5-large-v2 → write to Qdrant (self-hosted for data sovereignty). Metadata store tracks permissions, source URLs, freshness. Query: Auth check → query embedding → hybrid retrieval (dense + BM25 via reciprocal rank fusion) → cross-encoder re-ranking (top 100 → top 5) → LLM generation with context → response with citations.

Key Components: Document-level auth filtering happens before re-ranking so the LLM never sees unauthorized content. Semantic cache (cosine > 0.95) catches 30-40% near-duplicate queries in enterprise settings, saving significant LLM costs.

Tradeoffs: Self-hosted embedding (control, cost) vs API-based (simpler ops). Cross-encoder adds ~200ms but dramatically improves precision. Full re-indexing vs incremental — at 10M docs, incremental is mandatory with periodic full rebuilds.

Scale: Vector DB needs sharding with HNSW across nodes. LLM is the throughput bottleneck — request queuing, multiple instances, aggressive caching. Feedback loop: thumbs-up/down → retrieval quality dashboards → periodic embedding fine-tuning.

SENIOR Design a real-time fraud detection pipeline at 50K TPS

Requirements: Latency budget (<100ms for most processors)? Fraud rate? Cost asymmetry between false negatives and false positives?

Architecture — Three paths: Training: Historical transactions + feature store (point-in-time correct) → XGBoost/LightGBM → MLflow registry → shadow deploy. Serving: Transaction → Kafka → feature enrichment (online store for batch features + Flink for streaming features like tx_count_last_5min) → Triton model server → score → decision engine → approve/decline/review. Feedback: Analyst labels → label store → periodic retraining.

Components: Flink partitioned by user ID with RocksDB state backends. Redis Cluster for online features (sub-ms latency, sharded by user ID). XGBoost chosen deliberately — sub-ms inference, handles tabular features well, interpretable for compliance.

Data Flow: Three-way decision split — auto-approve (score < 0.1), auto-decline (score > 0.95), manual review (between). Review queue prioritized by amount × probability.

Scale: Tiered scoring — lightweight rule-based filter catches 80% of obvious cases, only 20% hit the ML model, reducing serving costs 5x.

TRICKY Design a search ranking system for e-commerce

Requirements: Catalog size? Query volume? Optimizing for clicks, add-to-cart, or revenue?

Architecture — Multi-stage funnel: Stage 1 — Candidate Retrieval (~10ms): Bi-encoder embeds query, HNSW retrieves top 1,000, BM25 in parallel, reciprocal rank fusion. Optimizes for recall. Stage 2 — Re-ranking (~20ms): LambdaMART scores each (query, candidate) pair with BM25 score, embedding similarity, CTR, conversion rate, freshness, seller rating. Top 50. Stage 3 — Personalization (~5ms): User features adjust scores. Business rules enforce diversity, boost promoted products, filter availability. Final top 10-20.

Tradeoffs: NDCG improvements offline don't always translate to conversion gains online — always validate with live experiments. BERT cross-encoders beat LambdaMART on accuracy but cost 100x latency — reserve for re-ranking top 20 at most. Head query caching is essential (top 1% queries = 30%+ traffic).

HOT Design an AI chatbot that handles 80% of tickets

Requirements: The 80% target is the key constraint. Must be good enough to resolve most queries autonomously, smart enough to know when to escalate the remaining 20%.

Architecture: User message → Conversation manager → Intent classifier → LangGraph state machine routing to specialized nodes: retrieve_knowledge (RAG over help docs), call_order_api, process_refund, generate_response, escalate_to_human. Conditional edges route by intent.

Components: Comprehensive, fresh knowledge base (weekly sync). Tool-calling — read-only actions launch immediately; write actions have guardrails (spending limits, confirmation). Confidence-based escalation when context has low relevance or user expresses frustration twice.

Guardrails: PII redaction, topic restrictions, output validation, hard "talk to a human" escape hatch. Every conversation logged. Human agents label escalated responses → weekly retraining. Track automation rate and CSAT per channel.

Tradeoffs: Start with read-only actions, expand to write actions as confidence grows. Streaming improves perceived latency but complicates guardrail checking — run async checks and retract if needed.

SENIOR Design an LLM gateway for 500 internal teams

Requirements: Without a gateway, 500 teams = 500 API keys, zero cost tracking, no fallback when providers go down.

Architecture: Client request → API Gateway (auth, rate limiting) → Router (complexity analysis, model selection) → Cache check (exact + semantic) → LLM provider call → Response logging (model, tokens, cost) → Cache write → Response.

Components: Router sends simple queries to GPT-4o-mini, complex reasoning to GPT-4o/Claude. Start rule-based, evolve to ML-based. Rate limiting at three levels: per-user, per-team (budgets with 80% alerts, 100% hard caps), global. Track request count and token count.

Data Flow: Fallback chain: primary model → secondary → cached response → error, with circuit breakers. Cost tracking aggregates by team, project, user. Semantic caching (cosine > 0.95) provides 30-40% hit rates.

Tradeoffs: Centralization adds latency and SPOF. Mitigate with multi-region deployment and circuit breakers. The visibility benefits far outweigh the complexity.

TRICKY Design a content moderation system for social media

Requirements: What modalities? What volume? Async (post-upload) or must it block publishing?

Architecture — Tiered pipeline: Stage 1 — Fast classifier (<50ms): Lightweight models (distilled BERT, MobileNet) handle 85-90%. Safe → approve. Violating → block. Stage 2 — LLM (~500ms): Nuanced cases — sarcasm, context-dependent. 8-12%. Stage 3 — Human review: Ambiguous cases, 1-3%. Queue prioritized by severity × reach × confidence gap.

Components: Multi-modal: text classifiers for hate speech/spam/self-harm, vision models for NSFW/violence, video via keyframe sampling + audio transcription. Content hashing (perceptual hashes) catches re-uploads in O(1).

Tradeoffs: Category taxonomy needs severity levels, not binary labels. Child safety demands near-100% recall; humor/satire detection prioritizes precision. Human decisions retrain Stage 1 weekly. User appeals enter the review queue.

SENIOR Design an ML feature platform for 200 models

Requirements: At 200 models, ad-hoc feature computation is chaos. The feature platform centralizes everything.

Architecture — Four layers: Computation: Batch (Spark, daily/hourly) + streaming (Flink, real-time). Storage: Offline store (BigQuery/Snowflake) for training; online store (Redis Cluster) for sub-ms serving. Serving: Feature server API returns feature vectors by entity ID. Registry: Feature catalog with owners, definitions, freshness SLAs, lineage.

Key Challenge: Point-in-time correctness — training data must use feature values as they existed at event time, not current values. Getting this wrong causes feature leakage. Offline-online consistency is the hardest problem — same definition must produce identical values in batch and real-time.

Tradeoffs: Feast (open-source, operational investment) vs Tecton (managed, vendor dependency). Freshness vs cost: only compute in real-time what genuinely changes the decision at serving time. Redis Cluster with read replicas, federated team ownership over shared infra.

SENIOR How does a feature store achieve point-in-time correctness?

The offline store maintains timestamped feature values. When building training data, you pass an entity DataFrame where each row has an entity ID and event timestamp. The store performs an asof-join: for each row, it finds the most recent feature value with a timestamp before the event timestamp. This prevents feature leakage — e.g., a fraud model training on a January transaction gets spending patterns as of January, not April.

Without this, features from future data leak information about the label, causing inflated offline metrics that don't transfer to production. Feast implements this via get_historical_features(). Key detail: feature values must be stored with computation timestamps, and join logic must handle late-arriving data and out-of-order events.

SENIOR How would you design a model registry for 50 ML engineers?

Four core capabilities: versioning, metadata, lineage, and promotion workflows. Every model gets an immutable version with its artifact, hyperparameters, training data version, eval metrics, and git commit. Lineage traces from production prediction → model version → training run → dataset version → feature definitions.

Promotion follows stage gates: Development → Staging → Production → Archived. Staging requires automated tests (latency, schema validation, metric thresholds). Production requires shadow deployment + peer approval. For 50 engineers: ownership per model, audit logs, automated alerts for stale staging models. Build on MLflow with custom CI/CD — PR merging triggers validation, shadow deploy, and Slack notification for approval.

SENIOR Describe the architecture of a semantic cache for LLMs

Three components: embedding service (~5ms per query), vector index (HNSW in Redis, ~5-10ms search), and response store (Redis, keyed by cache entry ID). New queries are embedded and similarity-searched against the index. If cosine > 0.95, the cached response is returned without calling the LLM.

The similarity threshold is the critical tuning knob — too low (0.85) gives false matches ("return an item" ≠ "return a library book"); too high (0.99) rarely hits. TTL-based expiration prevents staleness. Version tagging invalidates entries when underlying data changes. Total overhead: ~10-50ms, but hits save 500-2000ms of LLM inference. At enterprise scale, 30-40% hit rates = massive cost savings.

SENIOR How does HNSW work and when to choose it over IVF-PQ?

HNSW builds a multi-layer graph. Bottom layer has all vectors; upper layers have progressively fewer ("express lanes"). Search starts at top, greedily navigates to nearest node, drops down, repeats. Parameters M (connections/node) and efConstruction (search width) control accuracy-memory tradeoff. >95% recall with microsecond queries.

IVF-PQ partitions into clusters (IVF) and compresses via product quantization (10-50x memory reduction). Disk-friendly, handles billions, but lower recall (85-93%) and higher latency. Choose HNSW for latency-critical apps under ~100M vectors. Choose IVF-PQ when data exceeds RAM or cost demands compression. Many systems use HNSW for hot tier, IVF-PQ for cold tier.

SENIOR Design a model monitoring system for concept drift

Three layers: Data collection — log every prediction with input features, output, and ground truth (when available). Metrics computation — scheduled statistical tests: PSI/KS tests on feature distributions (data drift), output score distribution comparison (prediction drift), performance metrics on labeled subsets (concept drift). Alerting — INFO for minor shifts, WARNING for moderate drift triggering investigation, CRITICAL for severe degradation triggering automated rollback.

Hardest part: ground truth labels arrive with delay (fraud labels take weeks). Rely on proxy signals — prediction distribution shifts, feature drift on high-importance features, business metric anomalies — as early warning while waiting for labels.

SENIOR How would you build an embedding refresh pipeline?

Four phases: Trigger — new model registered (quarterly or on quality regression). Batch re-embedding — distributed Spark/Ray job reads all docs, chunks, generates new embeddings. For 10M docs (~50M chunks), takes hours on GPU clusters. Atomic swap — write to a fresh vector index (not live), validate retrieval quality on a golden test set, swap alias. Cleanup — keep old index 48 hours as rollback target.

Incremental re-embedding (only new/modified docs) saves compute but creates a mixed-version index where vector spaces aren't aligned — degrades search quality. Full re-embedding is cleaner and worth the cost at reasonable corpus sizes.

SENIOR Explain the architecture of LLM guardrails

Two-stage filter: Input guardrails — prompt injection detection (fine-tuned classifier or patterns), topic boundary enforcement (intent classifier blocks off-topic queries), PII detection/redaction, rate limiting. Output guardrails — PII scanning of response, factuality grounding checks for RAG (verify citations aren't hallucinated), toxicity classification, format validation (valid JSON if requested).

Pipeline: User Input → Input Guardrails → LLM → Output Guardrails → Response. Each guardrail returns pass/block/redact. Blocked requests get a safe fallback. All triggers logged for auditing. NeMo Guardrails (dialogue flow) and Guardrails AI (schema validation) provide building blocks, but production systems customize heavily.

Both options have merit. The right answer depends on context — and saying so is the point.

A) Always pre-filter — you only want to search relevant documents

D) Avoid metadata filtering entirely — it degrades ANN quality

A) Always synchronous — callers need immediate results
B) Sync for user-facing hot paths (search, checkout); async for batch-like workloads where results can be consumed later, enabling request batching for GPU efficiency

A) Always use a feature store — consistency is non-negotiable
B) Ad-hoc is fine for 1-2 models; the tipping point is ~5-10 models / 3+ teams — above that, the consistency and reuse benefits compound rapidly
C) Ad-hoc is always better — feature stores add unnecessary complexity

Shadow deployment vs canary deployment for a new ML model?

A) Always canary — you need real user signal to measure impact
B) Always shadow — it's zero-risk and you can compare outputs offline
C) They're sequential, not competing — shadow first to catch catastrophic failures, then canary to measure real business impact

⚡ Rapid Fire

Short questions, short answers. If you can't answer in one breath, review Modules 1 & 2.

What is HNSW?
Hierarchical Navigable Small World — a multi-layer graph-based ANN index. High recall, high memory, best when data fits in RAM.
What is PSI?
Population Stability Index — measures distribution shift between two time periods. <0.1 stable, 0.1-0.25 moderate, >0.25 significant drift.
Data drift vs concept drift?
Data drift = input distributions change. Concept drift = input→output relationship changes. Data drift may be harmless; concept drift always breaks the model.
What is a cold start problem?
Insufficient data for good predictions — new users, new items, or new systems. Solve with content-based features, popularity defaults, or onboarding flows.
What is interleaving in A/B testing?
Merge results from two ranking models into one list. Users implicitly vote by clicking. 10-100x more sensitive than traditional split tests.
What is a data flywheel?
More users → more data → better models → better product → more users. The strongest competitive moat in AI.
What is point-in-time correctness?
Training data uses feature values as they existed at event time, not current values. Prevents feature leakage. Implemented via temporal asof-joins.
What is product quantization?
Split high-dim vectors into subvectors, quantize each with a learned codebook. 10-50x memory reduction with moderate recall loss. Used in IVF-PQ.
What does a model registry store?
Artifacts, hyperparameters, training data version, eval metrics, lineage (code commit, dataset, features), deployment stage, and ownership metadata.
What is a feature store's online store?
Low-latency key-value store (Redis, DynamoDB) holding latest feature values for real-time inference. Sub-ms reads, updated by materialization pipelines.
What is Thompson Sampling?
Bayesian bandit algorithm — maintains probability distributions over reward rates, samples to decide which option to try. Balances explore vs exploit naturally.
What is a guardrail metric?
A metric that must not degrade during an experiment (latency p99, crash rate, safety violations). If it trips, experiment stops regardless of primary gains.
What is shadow mode deployment?
New model runs on production traffic but predictions aren't served to users. Logged and compared offline. Zero user risk — use before canary.
What is semantic caching?
Cache LLM responses indexed by query embedding. If a similar query exists (cosine > 0.95), return cached response. Catches paraphrases that exact-match misses.
What is BM25?
Sparse retrieval based on TF-IDF. Excels at exact keyword matching, acronyms, proper nouns. Often combined with dense vectors in hybrid retrieval.
What is NDCG?
Normalized Discounted Cumulative Gain — ranking metric that rewards placing relevant results near the top. Accounts for position and graded relevance.
What is a cross-encoder?
Scores a (query, document) pair jointly — more accurate than bi-encoders but O(n). Use only for re-ranking small candidate sets, not retrieval.
What is model lineage?
Full provenance: prediction → model version → training run → dataset → features → code commit. Essential for debugging, compliance, reproducibility.
What is feature leakage?
Training data accidentally includes future info or label-derived info. Causes inflated offline metrics that don't transfer to production.
What is the LLM cascading pattern?
Try cheapest model first. If confidence is low (log-probs or quality classifier), escalate to a more expensive model. Reduces costs 50-70%.

🧠 The 6-Step Framework (State It First!)

1. Clarify Requirements — business objective, users, latency, scale, compliance (3-5 min)
2. Data — sources, labeling, volume, schema, privacy, batch vs streaming
3. Model — right model for the job, accuracy vs latency vs cost triangle
4. Serving — real-time/batch/streaming, latency SLAs, autoscaling
5. Monitoring — data drift, concept drift, prediction drift, alerting
6. Iteration — feedback loops, retraining cadence, A/B testing, data flywheel

🔥 Top 5 Patterns Across All Problems

Tiered processing — cheap tier for easy cases, expensive tier for hard cases (fraud, moderation, search, LLM routing).
Offline-online duality — feature stores, evaluation, models all live in two worlds: batch learning and real-time serving.
Human-in-the-loop — every production AI system needs an escape hatch. Start conservative, expand autonomy.
Caching is architecture — semantic cache in RAG, result cache in search, content hash in moderation, exact+semantic in LLM gateways.
Feedback loops power the flywheel — user interactions = implicit labels, escalations = gold labels. The system that learns fastest wins.
