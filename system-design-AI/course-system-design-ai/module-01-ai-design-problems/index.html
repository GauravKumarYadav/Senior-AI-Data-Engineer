🏛️ AI System Design Problems

This module covers nine foundational AI system design problems you'll encounter in real-world production environments. These are the "greatest hits" — the patterns that show up again and again across companies, domains, and use cases.

The goal isn't to memorize blueprints. It's to internalize the reasoning behind design decisions so you can adapt when the requirements inevitably shift.

1. Enterprise RAG Q&A

Ground LLMs in private data with retrieval-augmented generation.

2. Fraud Detection

Real-time ML scoring with feature stores and streaming.

3. Search Ranking

Multi-stage funnel from millions to top-10 results.

4. AI Support Agent

LLM-powered agents that can look up, act, and escalate.

5. LLM Gateway

Centralized routing, caching, rate limiting for LLM calls.

6. Content Moderation

Tiered pipeline for text, image, and video moderation at scale.

7. Feature Platform

Shared infrastructure for feature engineering and serving.

8. Model Monitoring

Detect data drift, concept drift, and prediction drift.

9. Multi-Agent Docs

Orchestrated agents for document classification and extraction.

🎯 The 6-Step Framework for Every Problem

1. Clarify requirements — functional vs. non-functional, scale numbers.
2. Define the API / interface — what goes in, what comes out.
3. High-level architecture — draw the boxes and arrows.
4. Deep-dive key components — the 2-3 hardest parts.
5. Discuss tradeoffs — why this choice over that one?
6. Address scale — what breaks at 10x, 100x, 1000x?

Problems 1–2: RAG & Fraud

📚 1. Enterprise RAG Q&A System

RAG is the workhorse pattern for grounding LLMs in private enterprise data. Instead of fine-tuning (expensive, stale), you retrieve relevant chunks at query time and feed them as context. The LLM generates answers from those chunks, with citations.

Architecture — two pipelines:

Ingestion: Data sources → Document parser → Chunking engine → Embedding model → Vector DB + metadata store.

Query: User question → Auth check → Query embedding → Hybrid retrieval (dense + BM25) → Re-ranker → LLM generation → Response with citations → Semantic cache write.

🧩 Chunking Engine

Deceptively critical. Naive fixed-size chunking breaks semantic boundaries. Use recursive splitting with overlap or semantic chunking that respects document structure. Too small = lost context. Too large = diluted relevance.

🔍 Hybrid Search

Pure dense retrieval struggles with exact keywords, acronyms, proper nouns. BM25 handles these well. Combine with reciprocal rank fusion for best of both worlds.

🏆 Re-Ranker

Cross-encoder (e.g., Cohere Rerank) scores each (query, chunk) pair jointly. More expensive than bi-encoder retrieval but dramatically improves precision for the final top-5 chunks.

🔒 Document-Level Auth

Every chunk inherits source permissions. Filter results by user's access groups before re-ranking. Never send unauthorized content to the LLM — even if you don't show the answer, the model "saw" it.

# Hybrid search: combine dense + sparse retrieval def hybrid_search(query: str, k: int = 20) -> list[Document]: dense_results = vector_db.search(embed(query), top_k=k) sparse_results = bm25_index.search(query, top_k=k) return reciprocal_rank_fusion(dense_results, sparse_results, k=60)

Option A

  • 📦 Fixed-size chunking — simple, fast
  • ☁️ OpenAI ada-002 — easy, hosted
  • 🔧 Managed Pinecone — less ops
  • ⏭️ Skip re-ranking — fast, less precise

Option B

  • 🧠 Semantic chunking — better quality, complex
  • 🏠 Self-hosted e5-large — control, cost
  • 🔧 Self-hosted Qdrant — more control
  • 🏆 Cross-encoder — accurate, slower
💡 Interview Gold

The biggest tradeoff is retrieval quality vs. latency. Every stage you add (hybrid search, re-ranking, query expansion) improves answer quality but adds latency. For internal Q&A, 2-3s is fine. For customer-facing chat, you need <1s for retrieval. At 10M docs, you need ANN indices (HNSW) and sharding. At 1000 concurrent users, semantic cache becomes critical — 30-40% of enterprise questions are near-duplicates.

🛡️ 2. Real-Time Fraud Detection

The canonical ML systems problem — it touches everything: feature engineering, real-time serving, class imbalance, feedback loops, and the tension between precision and recall. Block a legit transaction → lose a customer. Miss fraud → lose money.

Training path: Historical transactions → Feature engineering (Spark batch) → Feature store (offline) → Model training → Model registry → Deploy.
Serving path: Incoming transaction → Feature enrichment → Model server → Score → Decision engine (approve/decline/review) → Response (<50ms).
Feedback path: Fraud analyst labels → Periodic retraining → Shadow deploy → Champion/challenger → Promote.

🗄️ Feature Store (Feast)

Dual-store pattern: offline store (data warehouse) for training with point-in-time correct joins. Online store (Redis/DynamoDB) for single-digit ms inference lookups.

⚡ Batch vs. Streaming

Batch features (daily Spark): avg monthly spend, account age. Streaming features (Flink): tx count in last 5 min, geographic velocity. The interplay between these is where model power comes from.

🌳 Model Choice

XGBoost/LightGBM dominate. Fast inference (sub-ms), handle tabular data, interpretable for compliance. Deep learning might gain 0.5% AUC at 10x latency cost.

⚖️ Decision Engine

Model outputs probability, but business logic decides action. Separate component with configurable thresholds — tune without redeploying the model.

# Point-in-time correct feature retrieval for training training_df = feast_store.get_historical_features( entity_df=transactions_with_timestamps, features=[ "user_features:avg_spend_30d", "user_features:tx_count_7d", "merchant_features:risk_score", ], ).to_df()
🎯 Interview Tip

Precision vs. recall is the existential tradeoff. Most systems optimize for high recall with a manual review queue for borderline cases — the "approve / decline / review" trichotomy. At 10x volume, streaming feature computation becomes the bottleneck. At 100x, use tiered scoring: lightweight rules catch 80% of obvious cases, ML handles only borderline ones.

Problems 3–4: Search & Agents

🔎 3. Search Ranking System

Search ranking is a multi-stage funnel: you can't run an expensive model on millions of documents, so you progressively narrow the candidate set while increasing model complexity at each stage.

Query flow: User query → Query understanding (spell check, expansion, intent) → Stage 1: Candidate retrieval (ANN, top-1000) → Stage 2: Re-ranking (LambdaMART, top-50) → Stage 3: Personalization + business rules (final top-10).

🎯 Stage 1 — Retrieval

Embedding-based ANN (HNSW). Bi-encoder encodes query + docs into same vector space. Prioritizes recall — good results must be in the set. Budget: ~10ms for top-1000.

🏆 Stage 2 — Re-ranking

LambdaMART (GBM optimized for ranking loss) is the industry workhorse. Uses richer features: BM25 score, CTR, freshness, authority. BERT cross-encoders are more accurate but 10-100x slower.

👤 Stage 3 — Personalization

User-level features (history, department, role). Business rules: boost promoted content, enforce diversity, apply recency bias. This is where business value meets ML.

🧪 A/B Testing

Interleaving experiments merge results from two rankers into one list and measure clicks. 10-100x more sensitive than traditional 50/50 A/B splits.

# LambdaMART feature vector per (query, doc) pair features = { "bm25_score": 12.4, "embedding_similarity": 0.82, "doc_click_rate": 0.15, "doc_freshness_days": 3, "query_doc_token_overlap": 0.4, "doc_authority_score": 0.91, }
💡 Interview Gold

Online vs. offline metrics often diverge. A model can improve NDCG on held-out data but hurt CTR in production because it doesn't account for position bias. Always validate with online experiments. At 100x query volume, caching is essential — top 1% of queries account for 30%+ of traffic.

🤖 4. AI Customer Support Agent

LLM-powered agents that can actually do things — look up orders, process refunds, answer from a knowledge base. The key challenge is reliability and safety. An agent that confidently gives wrong answers or processes unauthorized refunds is worse than no agent at all.

Runtime flow: User message → Conversation manager → LangGraph agent → Tool selection → Tool execution (RAG / API / escalation) → Guardrail check → Stream response.

🔀 LangGraph State Machine

Each node = a step (retrieve, call API, generate). Edges = transitions based on intent classification. Superior to simple ReAct loops because you can enforce structure.

🧠 Conversation Memory

Short-term: last N turns. Long-term: user preferences stored in DB, retrieved per session. Summarize older turns instead of stuffing full history into context.

🛡️ Guardrails

Topic restriction, PII detection/redaction, escalation triggers (frustration detection), output validation (no hallucinated order numbers). Non-negotiable.

🙋 Human-in-the-Loop

When confidence is low or case is sensitive (large refund, legal threat), package conversation summary and route to human agent queue with full context.

# LangGraph agent with conditional routing from langgraph.graph import StateGraph graph = StateGraph(AgentState) graph.add_node("classify_intent", classify_intent) graph.add_node("retrieve_knowledge", retrieve_from_kb) graph.add_node("call_order_api", lookup_order) graph.add_node("process_refund", handle_refund) graph.add_node("escalate", escalate_to_human) graph.add_conditional_edges("classify_intent", route_by_intent, { "question": "retrieve_knowledge", "order_status": "call_order_api", "refund": "process_refund", "complex": "escalate", })

⚡ High Autonomy

  • ✅ Processes refunds, updates orders
  • ✅ Higher deflection rate, more value
  • ⚠️ Higher risk of errors
  • ⚠️ Needs spending limits + guardrails

🔒 Conservative Start

  • ✅ Read-only: lookups, knowledge retrieval
  • ✅ Lower risk, build trust first
  • ⚠️ Lower deflection, less value
  • ✅ Gradually expand as system proves itself

Problems 5–6: Gateway & Moderation

🚪 5. LLM Gateway / Router

As organizations adopt LLMs, you end up with chaos: different teams, different providers, different API keys, no cost visibility, no fallback when a provider goes down. The LLM Gateway is your centralized control plane.

Request flow: Client → API Gateway (auth, rate limiting) → Router (complexity analysis, model selection) → Cache check (exact + semantic) → LLM provider call → Response processing (logging, cost tracking) → Cache write → Response.

🔀 Model Router

Routes to optimal model. Simple queries → GPT-4o-mini (cheap). Complex reasoning → GPT-4o or Claude. Router can be rule-based or ML-based (classifier on prompt complexity).

🚦 Rate Limiting

Three tiers: per-user, per-team, global. Track both request count and token count — one long prompt costs more than 100 short ones. Token bucket or sliding window.

🔄 Fallback Chain

Claude → GPT-4 → GPT-4-mini → cached response → error. Each step has its own timeout. Resilience without manual intervention.

💰 Cost Tracking

Every request logs model, token counts, computed cost. Aggregate by team/project/user. Budget alerts at 80%, hard caps at 100%. Prevent runaway scripts burning $50K overnight.

# Smart model routing based on prompt complexity def route_request(prompt: str, metadata: dict) -> str: if metadata.get("force_model"): return metadata["force_model"] complexity = estimate_complexity(prompt) if complexity < 0.3: return "gpt-4o-mini" elif complexity < 0.7: return "gpt-4o" else: return "claude-3-5-sonnet"
💡 Interview Gold

Semantic caching is a game-changer. Beyond exact-match caching, embed the prompt and check for similar recent queries (cosine > 0.95). "What's the capital of France?" and "What is France's capital city?" should hit the same cache. At 10x traffic, a 30-40% cache hit rate = 30-40% fewer LLM calls. At 100x, the gateway itself must be horizontally scalable — stateless processing with shared Redis cache.

🛡️ 6. Content Moderation System

Every platform that accepts user-generated content needs moderation. The challenge: doing it at scale, across modalities (text, images, video), with acceptable latency, and without drowning your human review team.

Pipeline: Content upload → Stage 1: Fast classifier (<50ms) → If clear, auto-action → Stage 2: LLM analysis (~500ms, nuanced cases) → Stage 3: Human review queue (ambiguous cases, prioritized by severity × reach).

🎭 Multi-Modal Analysis

Text: fine-tuned BERT for hate speech, spam, toxicity. Images: vision models for NSFW, violence. Video: sample key frames + audio transcription. Each modality needs its own pipeline.

📊 Tiered Pipeline

The core design insight. Fast classifier handles 85-90% (obvious). LLM handles 8-12% (nuanced). Humans handle 1-3% (genuinely ambiguous). Single model can't do millions/day.

📋 Priority Queue

Not all pending reviews are equal. Score by: severity × content reach × classifier confidence gap. Violent post on public page with millions of followers → top of queue.

🏷️ Category Taxonomy

Categories aren't binary. "Violence" ranges from boxing (allowed) to graphic gore (blocked). Multi-label predictions with confidence scores per category + severity level.

🔍 Pre-Upload Screening

  • ✅ Bad content never goes live
  • ⚠️ Adds latency to upload flow
  • 📸 Best for images/video

📤 Post-Upload Screening

  • ✅ Non-blocking upload experience
  • ⚠️ Window where bad content is live
  • 📝 Best for text (fast enough to be near-sync)
🎯 Interview Tip

Different categories get different thresholds. For child safety: near-100% recall, accept high false positives. For satire misclassified as hate speech: higher precision, over-moderation kills UX. At 10x scale, use perceptual hashes for images and fingerprinting for videos to instantly catch re-uploads of previously blocked content.

Problems 7–8: Features & Monitoring

🏗️ 7. ML Feature Platform

The feature platform makes feature engineering a first-class, reusable concern rather than ad-hoc code scattered across notebooks. If the model is the brain, the feature platform is the nervous system — it determines what signals reach the model and how fast.

⏱️ Point-in-Time Correctness

The #1 most important concept. Training data must use features as they existed at the event time, not today. Getting this wrong = feature leakage → silently inflated offline metrics.

🔄 Offline–Online Consistency

Same feature definition must produce identical values in batch (training) and real-time (serving). Feast solves this with a single definition materialized to both stores.

📖 Feature Registry

Catalog of all features: name, description, owner, data type, computation logic, freshness SLA, downstream consumers. Prevents three teams independently computing the same feature.

⚙️ Materialization

Computing features and writing to online store. Batch runs on schedule (daily 2 AM). Streaming writes near-real-time. Must handle backfills when logic changes.

# WRONG: uses current feature values for historical events features = feature_store.get_online_features(entity_ids) # RIGHT: uses feature values as of each event's timestamp features = feature_store.get_historical_features( entity_df=events_with_timestamps, # each row has event_timestamp features=["user:avg_spend_30d", "user:tx_count_7d"], )

🔧 Build (Feast OSS)

  • ✅ Open-source, flexible
  • ✅ No vendor dependency
  • ⚠️ Requires operational investment
  • ⚠️ Team must manage infra

💳 Buy (Tecton / Databricks)

  • ✅ Reduced ops burden
  • ✅ Enterprise support
  • ⚠️ Vendor lock-in
  • ⚠️ Higher cost
💡 Interview Gold

Freshness vs. cost is the key tradeoff. Streaming features (updated in seconds) are expensive. Batch features (updated daily) are cheap and reliable. Most features don't need real-time freshness — only compute in real-time what genuinely changes the model's decision at serving time. At 10x features, the registry prevents sprawl. At 10x models, Redis cluster with read replicas handles online store throughput.

📡 8. Model Monitoring & Drift Detection

Deploying a model isn't the finish line — it's the starting line. Models degrade silently. Data distributions shift, user behavior changes, upstream pipelines break. Without monitoring, you won't know your model is broken until a business metric craters weeks later.

📊 Data Drift

Input feature distributions change. Measured via KL divergence, PSI, or KS tests. Example: fraud model trained on US transactions starts receiving 40% international transactions.

🔄 Concept Drift

The relationship between inputs and outputs changes. Example: COVID shifted spending patterns — pre-COVID features became unreliable. Harder to detect because you need ground truth labels (which arrive late).

📈 Prediction Drift

Model output distribution shifts. If your fraud model flags 20% of transactions instead of the usual 2%, something's wrong — even without knowing the root cause yet.

🚨 Alerting Strategy

Not every blip is actionable. Severity levels: INFO (minor shift, log it), WARNING (moderate, investigate), CRITICAL (severe, trigger rollback). Windowed comparisons smooth out noise.

from scipy.stats import ks_2samp def detect_drift(reference: np.ndarray, current: np.ndarray, threshold: float = 0.05) -> bool: """KS test for distribution drift on a single feature.""" statistic, p_value = ks_2samp(reference, current) return p_value < threshold # True = drift detected
🎯 Interview Tip

Automated retraining on drift sounds appealing but can be dangerous — if drift is caused by a data pipeline bug, retraining on bad data makes things worse. Safer pattern: automated detection and alerting, but human-approved response (retrain, rollback, or investigate). At 100x features per model, use feature importance to prioritize — monitor top-20 features intensively, sample the rest.

Problem 9 + Cross-Cutting Patterns

📄 9. Multi-Agent Document Processing

When documents require complex processing — classification, extraction, validation, routing — a single monolithic model struggles. Multi-agent architectures decompose the problem: each agent specializes in one task, and an orchestrator coordinates the workflow. Think: insurance claims, legal review, invoice handling, healthcare intake.

Pipeline: Document upload → OCR/parsing → Agent 1: Classification (type, urgency) → Agent 2: Extraction (key fields, entities) → Agent 3: Validation (business rules, cross-referencing) → Agent 4: Routing (approve, reject, escalate) → Output + human review queue for low-confidence items.

🏷️ Classification Agent

Determines document type (invoice, contract, claim) and metadata (urgency, department). Classification accuracy is critical — a misclassified doc enters the wrong downstream pipeline.

📋 Extraction Agent

Extracts structured fields from unstructured text. Uses LLM with structured output (JSON mode / function calling). Validates against expected field types.

✅ Validation Agent

Cross-references extracted data against business rules and external systems. Vendor exists? Amount in expected range? Duplicate invoice check? Catches extraction errors and fraud.

🎯 Supervisor Orchestrator

LangGraph supervisor coordinates agents. Maintains shared state accumulating results. On failure or low confidence: retry, invoke fallback agent, or route to human review.

# Shared state object across agents class DocumentState(TypedDict): raw_text: str doc_type: str | None confidence: float extracted_fields: dict validation_results: list[ValidationResult] action: str | None needs_human_review: bool
# Extraction prompt with structured output extraction_prompt = """ Extract the following fields from this invoice: - vendor_name: str - invoice_number: str - total_amount: float - due_date: str (YYYY-MM-DD) - line_items: list[{description, quantity, unit_price}] Return as JSON. If a field is not found, use null. """

🧠 LLM Agents

  • ✅ Flexible, handles novel document types
  • ⚠️ Slower, more expensive per call
  • 📋 Best for long-tail / rare documents

⚡ Specialized Models

  • ✅ Faster, cheaper, more accurate on known types
  • ⚠️ Rigid, requires retraining for new types
  • 📋 Best for high-volume, well-defined doc types
💡 Interview Gold

Confidence thresholds are a business decision. Too low → humans review everything (defeats the purpose). Too high → errors slip through. Start conservative (30-40% to humans), measure error rates, and tighten. At 100x volume, you need a job queue (Celery, Temporal) with priority levels and fine-tuned models for top-5 document types.

🔗 Cross-Cutting Patterns

After studying all nine problems, these patterns emerge again and again:

📊 Tiered Processing

Cheap-fast tier for easy cases, expensive-slow tier for hard cases. Fraud detection, content moderation, search ranking, document processing — all follow this pattern.

🗄️ Feature Stores Everywhere

Fraud detection and search ranking both rely on offline/online feature store pattern. If building multiple ML systems, invest in the feature platform early.

🙋 Human-in-the-Loop

Every production AI system needs an escape hatch to humans. Start conservative and expand as you build trust. This is not optional.

📡 Monitoring First

Drift detection isn't a nice-to-have. Models degrade silently. Build monitoring from day one, not as an afterthought.

💾 Caching as Architecture

Semantic caching in RAG & LLM Gateway, result caching in search, content hash caching in moderation. Caching appears at every layer and often gives the biggest bang for your infra buck.

🎯 The Big Takeaway

These nine problems give you the vocabulary and structural intuition to tackle most AI system design challenges. The specific technologies will change — the architectural patterns won't.
