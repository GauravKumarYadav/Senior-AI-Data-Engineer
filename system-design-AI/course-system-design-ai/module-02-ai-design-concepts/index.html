The 6-Step ML Design Framework

The single most important tool in your AI system design toolkit. Without a framework, you'll ramble. With one, you'll be structured, thorough, and confident. This works for virtually every ML system design question.

1️⃣ Clarify Requirements

Ask: business objective, users, latency constraints, scale (RPS, data volume), fairness/compliance. Spend 3–5 min here. Common pitfall: jumping straight to modeling.

2️⃣ Data

Where does data come from? Labeling strategy (human, implicit, weak supervision)? Batch vs streaming ingestion? Class imbalance? PII masking? Data versioning?

3️⃣ Model

Right model for the right job. Tradeoff triangle: accuracy vs latency vs cost. Single model vs multi-stage pipeline? Pretrained vs train-from-scratch? Define offline eval metrics.

4️⃣ Serving

Real-time (API), batch (precompute), or streaming? Latency SLAs, throughput, model format (ONNX, TorchScript), hardware (CPU/GPU), autoscaling, load balancing.

5️⃣ Monitoring

Models rot. Data drifts. Monitor: prediction distributions, feature distributions, latency, error rates, business metrics. Alerts for drift (PSI, KL divergence).

6️⃣ Iteration

ML is never "done." Feedback loops → retrain on fresh data → A/B test → iterate features. Connects back to Step 2 (new data) and Step 3 (better models).

💡 Interview Gold

State this framework up front: "I'll walk through clarification, data, modeling, serving, monitoring, and iteration." It gives the interviewer a roadmap and shows you think in systems, not just models. A senior engineer asks "what does success look like?" before writing a line of code.

Feature Stores + Model Registries

🗄️ Feature Stores

A centralized system for managing, storing, and serving ML features. Without one, every team computes features independently — duplicated work, inconsistent definitions, and training-serving skew.

Imagine three teams all computing "user's average order value over 30 days." Without a feature store, each writes its own SQL and inevitably gets slightly different numbers. A feature store gives you one definition, one pipeline, one source of truth.

📦 Offline Store

  • 🗃️ Data warehouse (BigQuery, S3 + Parquet)
  • 📚 Holds historical feature values
  • 🎯 Used for training
  • ⏰ Time-travel: "features as of March 15th"
  • 🔗 Point-in-time correct joins

⚡ Online Store

  • 🔴 Low-latency KV store (Redis, DynamoDB)
  • 📌 Holds latest feature values only
  • 🎯 Used for real-time inference
  • ⏱️ Sub-millisecond reads
  • 🔄 Materialized from batch + streaming

Point-in-Time Correctness is the subtle killer. When training, you need features as they existed at the time of each example. Using future data (e.g., lifetime value to predict first purchase) is feature leakage — offline metrics look amazing while production falls flat.

# Feast feature definition — one definition, two stores from feast import Entity, FeatureView, Field from feast.types import Float64, Int64 user = Entity(name="user_id", join_keys=["user_id"]) user_features = FeatureView( name="user_features", entities=[user], schema=[ Field(name="avg_spend_30d", dtype=Float64), Field(name="tx_count_7d", dtype=Int64), Field(name="account_age_days", dtype=Int64), ], source=bq_source, # BigQuery table as offline source ttl=timedelta(days=1), # Freshness SLA )

📋 Model Registries

A versioned catalog of trained models with metadata — hyperparameters, training data version, evaluation metrics, lineage, and deployment status. Without one, "which model is in production?" becomes an archaeology exercise involving model_v3_final_FINAL.pkl.

🔢 Versioning

Every model gets a version. You can diff v7 vs v8 — what changed in data, hyperparams, metrics?

🔗 Lineage

Trace from a production prediction back to the exact training run, dataset, and code commit.

📊 Metadata

What data was it trained on? Hyperparameters? AUC? F1? All stored and queryable.

🚀 Promotion Workflow

None → Staging → Production → Archived. Promotion requires approval, passing tests, meeting metric thresholds.

# MLflow model registry — log, register, promote import mlflow with mlflow.start_run(): mlflow.log_params({"learning_rate": 0.01, "epochs": 50}) mlflow.log_metrics({"auc": 0.92, "f1": 0.87}) mlflow.sklearn.log_model( model, "model", registered_model_name="fraud-detector" )
💡 Interview Gold

Mentioning feature stores signals you understand production ML. Say: "I'd use a feature store to avoid training-serving skew and ensure point-in-time correctness." For model registries, say: "Governance and reproducibility are first-class concerns — the registry lets us trace any prediction back to its training run."

A/B Testing + Online vs Offline Eval

🧪 A/B Testing for ML

A/B testing ML systems is harder than A/B testing a button color. ML changes are diffuse — a new ranking model affects every search result, every session, in complex ways. Effects are often small, noisy, and take time to materialize.

📊 Metrics Selection

Business metrics: revenue, engagement, retention. ML metrics: NDCG, precision@k, accuracy. Danger: optimizing ML metrics that don't correlate with business metrics.

🔀 Interleaving

For ranking: show results from both models interleaved in one list. Users "vote" by clicking. 10x more sensitive than traditional 50/50 splits — needs far fewer samples.

🎰 Multi-Arm Bandits

Dynamically allocate more traffic to the winning variant. Thompson Sampling (Bayesian) or epsilon-greedy. Great for minimizing regret during the test period.

🛡️ Guardrail Metrics

Metrics that must NOT degrade: latency p99, crash rate, safety violations. If a guardrail trips, the experiment stops — even if the primary metric improved.

Statistical rigor matters: decide on significance thresholds (p < 0.05), minimum detectable effect, and sample size upfront. Underpowered tests → false negatives. Peeking early → inflated false positives. Watch for novelty effects — engagement lifts in week 1 that vanish by week 3.

📏 Online vs Offline Evaluation

📋 Offline Evaluation

  • 📊 Static dataset, before deploying
  • 🏷️ Classification: Precision, Recall, F1, AUC
  • 📈 Ranking: NDCG, MAP, MRR
  • 📝 Generation: BLEU, ROUGE, BERTScore
  • ⏰ Always use temporal splits for time-series
  • ✅ Fast iteration, cheap to run

🌐 Online Evaluation

  • 👥 Real users, real traffic
  • 👻 Shadow mode: run alongside, don't show
  • 🐤 Canary: 1-5% traffic, monitor for regressions
  • 🔬 A/B test: full statistical comparison
  • 🔀 Interleaving: merged rankings, implicit votes
  • ⚠️ Expensive, slow, but ground truth
⚠️ The Offline-Online Gap

A model can improve offline metrics by 5% and show zero lift online. Why? 1) Distribution mismatch — test set ≠ real traffic. 2) Feedback effects — model changes user behavior. 3) Latency — more accurate but slower model causes abandonment. 4) Business context — a 2% AUC gain is irrelevant if the bottleneck is inventory, not prediction. Always say: "I'd validate offline first, then shadow deploy, then canary, then full A/B test."

Flywheel, Feedback & Cold Start

🔄 The Data Flywheel

The most powerful competitive moat in AI: more users → more data → better models → better product → more users. It's a compounding loop.

🔍 Google Search

Billions of queries → click data → better ranking models → better search → more users.

🎬 Netflix

Every play, pause, skip, rewatch → trains recommendation models → better content surfacing → more engagement.

🚗 Tesla

Every mile driven → camera/sensor data → better self-driving models → more compelling product → more drivers.

⚠️ When It Stalls

More data ≠ always better. Diminishing returns when: model is architecture-limited, data is noisy/biased, or you're optimizing the wrong metric.

🔁 Feedback Loops

How ML systems learn from the real world after deployment. Get them wrong and your model either stagnates or spirals into dysfunction.

🤫 Implicit Feedback

  • 👆 Clicks (abundant but click-bait inflates)
  • ⏱️ Dwell time (longer = more engaged)
  • 📜 Scroll depth, watch percentage
  • 🛒 Conversions, purchases, add-to-cart
  • ⏭️ Skip, dismiss, back-button (negative)
  • 📊 Abundant but noisy

🗣️ Explicit Feedback

  • 👍 Thumbs up/down, star ratings
  • 📝 Text reviews, survey responses
  • 🚫 "Not relevant" or "report" buttons
  • 🎛️ User preference settings
  • 💬 Direct complaints or praise
  • ✨ Clean but sparse

Dangerous loops to watch: Popularity bias (popular items get more clicks → recommended more → monopolize attention). Filter bubbles (users appear to narrow → model doubles down → diversity drops). Runaway loops (fraud model flags legit user → odd behavior → more fraud signals). Mitigate with exploration, debiasing, and diversity metrics.

🥶 Cold Start Problem

What happens when your ML system has no data — for a new user, a new item, or an entirely new system.

👤 New Users

Demographic priors, onboarding flows (select interests), content-based features, popularity-based defaults. Collaborative filtering is useless here.

📦 New Items

Content-based features (metadata, embeddings of similar items), exploration strategies (boosted exposure), active learning (show to most informative users).

🆕 New Systems

Transfer learning from pretrained models, synthetic data, seed data from public datasets, rule-based v1 systems. This is "cold start for the flywheel."

💡 Interview Gold

When designing a recommendation system, the interviewer will ask about cold start. Have a crisp answer for each scenario. For the flywheel, explicitly name it: "User interactions generate implicit labels that feed back into retraining, creating a data flywheel." For feedback loops: "I'd log all impressions and interactions for continuous learning, while monitoring for popularity bias and filter bubbles."

Embeddings + Vector Search at Scale

🧬 Embedding Infrastructure

Embeddings are dense vector representations encoding semantic meaning — the lingua franca of modern ML. Used in search, recommendations, clustering, classification, and RAG.

📝 Text Embeddings

Sentence-transformers (self-hosted), OpenAI text-embedding-3 (API). 384–3072 dimensions. Map text to dense vectors.

🖼️ Image Embeddings

CLIP (shared text-image space), ResNet feature extractors. Enable cross-modal search.

🌐 Multi-Modal

CLIP, ImageBind — encode different modalities into a shared vector space. Search images with text queries.

🔄 Versioning

When you update the embedding model, ALL vectors become incompatible. Re-embed entire corpus and swap atomically. Map model_version → index_version.

# Embedding generation pipeline from sentence_transformers import SentenceTransformer model = SentenceTransformer("all-MiniLM-L6-v2") # 384 dims # Batch: embed entire corpus via Spark/Dataflow corpus_embeddings = model.encode(documents, batch_size=256) # Real-time: embed queries on-the-fly query_embedding = model.encode(["How do I reset my password?"])

🔍 Vector Search at Scale

Exact nearest-neighbor is O(n) — fine for 10K vectors, useless for 100M. Enter ANN search.

📊 HNSW

  • 📐 Graph-based, in-memory
  • ✅ Excellent recall (>95%) at high speed
  • ⚠️ High memory (full vectors + graph)
  • 🎯 Best for: fits in RAM, latency-critical

📦 IVF-PQ

  • 🗂️ Partition into clusters + quantize
  • ✅ Much lower memory footprint
  • ⚠️ Slightly lower recall than HNSW
  • 🎯 Best for: billion-scale, disk-friendly

🔗 Hybrid Search

Combine dense vectors (semantic) with sparse vectors (BM25 / keywords). Handles exact keywords (SKUs, proper nouns) alongside semantic understanding.

# Hybrid search scoring # α controls the balance between semantic and keyword matching final_score = α × dense_score + (1 - α) × sparse_score # Metadata filtering approaches: # Pre-filter: filter first → search within subset (precise but slow if selective) # Post-filter: search first → discard non-matching (fast but may return fewer results)

🌲 Pinecone

Managed, easy to use, scales well. Great for teams that don't want to manage infra.

🔷 Weaviate

Open-source, hybrid search built-in. Good community and flexibility.

🐋 Milvus

Open-source, billion-scale, GPU support. Built for serious scale.

🐘 pgvector

Postgres extension. Good for small-medium scale when you already use Postgres.

💡 Interview Gold

When designing retrieval, always specify: index type (HNSW vs IVF-PQ), metadata filtering (pre vs post), and whether hybrid search is warranted. For embeddings, discuss: which model, dimensionality, storage, and how you handle model updates (full re-embed on update vs incremental for new items). This shows depth beyond "just use a vector DB."

Caching, Routing, Guardrails & HITL

💾 Caching for LLMs

LLM inference is expensive — $0.01–$0.10 per call adds up fast. At 1M daily queries without caching: $20K/day. With 40% cache hit rate: $12K/day. That's $240K/month in savings.

#️⃣ Exact Match Cache

  • 🔑 Hash the prompt → Redis/Memcached lookup
  • ⚡ Simple, fast, effective for FAQ bots
  • ❌ "What's the weather?" ≠ "Tell me the weather"
  • 📏 Zero fuzzy matching

🧠 Semantic Cache

  • 🔍 Embed query → vector search cached queries
  • ✅ Catches paraphrases, near-duplicates
  • ⏱️ Adds ~10-50ms for embed + search
  • 💰 Saves ~500-2000ms of LLM inference
# Exact match caching with TTL import hashlib, redis def get_or_generate(prompt: str) -> str: cache_key = hashlib.sha256(prompt.encode()).hexdigest() cached = redis_client.get(cache_key) if cached: return cached.decode() response = llm.generate(prompt) redis_client.setex(cache_key, ttl=3600, value=response) return response

🔀 Prompt Routing & Model Selection

Not every query needs GPT-4. Smart routing sends simple queries to cheap, fast models and reserves expensive models for complex ones. Can reduce costs by 50-70% with minimal quality degradation.

🧠 Complexity-Based

Lightweight classifier estimates query complexity. Simple factual → small model. Complex reasoning → large model.

🏷️ Topic-Based

Route to specialized models. Medical → fine-tuned medical model. Code → code model. General → general model.

💰 Cost-Based

Set a budget. Route to the cheapest model that meets quality threshold for each query type.

🔄 Cascading

Try cheapest model first. If confidence is low (log-probs, self-consistency), escalate to expensive model. Most queries are simple.

🛡️ Guardrails Architecture

LLMs are powerful but unpredictable. Guardrails are the safety net between the model and the user.

🚫 Input: Injection Detection

Classify if user tries to override system instructions. Fine-tuned classifier or pattern matching.

🎯 Input: Off-Topic Detection

Is this query within scope? A customer service bot shouldn't answer nuclear physics questions.

🔒 Output: PII Detection

Scan for SSNs, credit cards, emails, phone numbers. Redact or block before user sees response.

✅ Output: Factuality Check

For RAG: verify response is grounded in retrieved docs (not hallucinated). Check format compliance (valid JSON?).

# Guardrails architecture flow [User Input] → [Input Guardrails] → block / pass → [LLM / RAG Pipeline] → [Output Guardrails] → block / redact / pass → [User Response] # Frameworks: NeMo Guardrails (NVIDIA), Guardrails AI

🙋 Human-in-the-Loop Design

No ML system is perfect. The best systems know when to admit uncertainty and escalate to a human. This isn't a failure — it's a feature.

When to escalate: Low confidence (probability < 0.7), sensitive topics (medical, legal, financial), user requests ("talk to a human"), guardrail triggers that can't be auto-resolved.

The gold: Every human decision becomes a training label. Over time, escalation rate decreases as the model learns from human expertise. Track this rate as a key system health metric.

# Human-in-the-loop feedback cycle [AI handles query] → [Low confidence] → [Escalate to human] → [Human resolves] → [Log decision as training label] → [Retrain model] → [AI handles similar queries next time] # Key UX: preserve context, warm handoff, no dead ends
💡 Interview Gold

In any LLM system design, dedicate a section to guardrails (both input and output). Always mention caching — the cost savings are too significant to ignore. For human-in-the-loop, frame it as: "The system knows what it doesn't know, and uses human expertise to both serve the user and improve itself over time." This shows you think about safety, cost, and continuous improvement — not just the happy path.
