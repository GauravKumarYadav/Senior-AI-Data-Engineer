🧩 Jinja in SQL — The Template Engine

Every .sql file in dbt is a Jinja template, not plain SQL. Before dbt sends anything to the warehouse, the Jinja engine processes your file — resolving variables, executing control flow, and expanding macros into raw SQL.

{{ expression }}

Output: evaluates and prints the result. Example: {{ ref('stg_orders') }} → full table path.

{% statement %}

Logic: control flow, no output. Example: {% if target.name == 'prod' %}. For if/else, loops, set.

{# comment #}

Comment: completely stripped from output. Not visible in compiled SQL. Use for TODOs.

Jinja Template vs Compiled SQL

{# Core orders fact table #} {{ config(materialized='table') }} SELECT order_id, customer_id, order_amount_cents / 100.0 AS amount, {% if target.name == 'prod' %} customer_email, {% else %} MD5(customer_email) AS customer_email, {% endif %} CURRENT_TIMESTAMP() AS loaded_at FROM {{ ref('stg_orders') }}
💬 Jinja comment — stripped from output
⚙️ Config — sets materialization
 
📊 Normal SQL columns
 
🔒 Conditional: show real email in prod…
 
🔒 …mask it everywhere else
 
 
🔗 ref() resolves to full table path
# View compiled SQL (what the warehouse actually sees) $ dbt compile --select fct_orders # Output in: target/compiled/your_project/models/marts/fct_orders.sql
💡 Interview Insight

"Explain how Jinja works in dbt." — "Every dbt SQL file is a Jinja template processed before execution. {{ }} for expressions that output values, {% %} for control flow, and {# #} for comments stripped from compiled SQL. I use dbt compile to inspect generated SQL during development."

🔧 Built-in Macros — ref, source, config

These macros form the backbone of every dbt project. They're what makes dbt's DAG, documentation, and environment awareness possible.

{{ ref('model_name') }} — The Heart of dbt

-- What ref() does: -- 1. Resolves model name → full table/view path -- 2. Registers a dependency in the DAG -- 3. Handles environment-specific schemas SELECT * FROM {{ ref('stg_orders') }} -- Dev: → SELECT * FROM DEV_SCHEMA.stg_orders -- Prod: → SELECT * FROM ANALYTICS.staging.stg_orders -- Cross-project ref (dbt Mesh): SELECT * FROM {{ ref('other_project', 'shared_model') }}

{{ var('variable') }} — Project Variables

{% set lookback = var('lookback_days', 3) %} SELECT * FROM {{ ref('stg_orders') }} WHERE order_date >= DATEADD(day, -{{ lookback }}, CURRENT_DATE()) -- Override at runtime: -- $ dbt run --vars '{"lookback_days": 7}'

Key Jinja Variables

VariableWhat It Contains
{{ ref('model') }}Resolved table reference + DAG dependency
{{ source('src','tbl') }}Resolved source reference + freshness
{{ this }}Current model's table (for incremental)
{{ target.name }}Environment name (dev/prod/ci)

"What does ref() do beyond resolving table names?" — "Three things: (1) resolves to environment-specific paths. (2) Registers DAG dependencies — dbt knows execution order. (3) Creates documentation lineage for dbt docs generate."

⚡ Custom Macros — Reusable SQL Functions

Macros are Jinja functions in macros/ that accept arguments and return SQL strings — the dbt equivalent of Python functions.

-- macros/cents_to_dollars.sql {% macro cents_to_dollars(column_name, decimal_places=2) %} ROUND({{ column_name }} / 100.0, {{ decimal_places }}) {% endmacro %}
-- Using it in a model SELECT order_id, {{ cents_to_dollars('order_amount_cents') }} AS order_amount_dollars, {{ cents_to_dollars('shipping_cost_cents', 2) }} AS shipping_cost_dollars, {{ cents_to_dollars('tax_cents') }} AS tax_dollars FROM {{ source('ecommerce', 'raw_orders') }} -- Compiles to: -- ROUND(order_amount_cents / 100.0, 2) AS order_amount_dollars, -- ROUND(shipping_cost_cents / 100.0, 2) AS shipping_cost_dollars, -- ROUND(tax_cents / 100.0, 2) AS tax_dollars

More Practical Macros

-- macros/safe_divide.sql {% macro safe_divide(numerator, denominator, default=0) %} CASE WHEN {{ denominator }} = 0 OR {{ denominator }} IS NULL THEN {{ default }} ELSE {{ numerator }}::FLOAT / {{ denominator }} END {% endmacro %}
-- macros/generate_schema_name.sql (override dbt default) {% macro generate_schema_name(custom_schema_name, node) -%} {%- set default_schema = target.schema -%} {%- if custom_schema_name is none -%} {{ default_schema }} {%- elif target.name == 'prod' -%} {{ custom_schema_name | trim }} {%- else -%} {{ default_schema }}_{{ custom_schema_name | trim }} {%- endif -%} {%- endmacro %}
💡 Interview Insight

"Give me an example of a custom macro." — "A cents_to_dollars macro standardizing currency conversion across all models. If we change rounding logic, we update one file. I also wrote safe_divide for metrics and a generate_schema_name override for clean prod schemas vs prefixed dev schemas."

🔀 Control Flow — if, for, set

Jinja's control flow lets you generate dynamic SQL — conditionally including columns, looping over lists, and setting variables for reuse.

{% for %} — Looping to Generate SQL

{% set statuses = ['placed', 'shipped', 'completed', 'returned', 'cancelled'] %} SELECT customer_id, COUNT(*) AS total_orders, {% for status in statuses %} SUM(CASE WHEN status = '{{ status }}' THEN 1 ELSE 0 END) AS {{ status }}_count {% if not loop.last %},{% endif %} {% endfor %} FROM {{ ref('stg_orders') }} GROUP BY customer_id

Loop Variables

VariableDescription
loop.indexCurrent iteration (1-indexed)
loop.index0Current iteration (0-indexed)
loop.firstTrue if first iteration
loop.lastTrue if last iteration
loop.lengthTotal number of iterations

Dynamic UNION ALL

{% set months = ['2024_01', '2024_02', '2024_03', '2024_04'] %} {% for month in months %} SELECT event_id, user_id, event_type, event_timestamp, '{{ month }}' AS source_month FROM {{ source('events', 'events_' ~ month) }} {% if not loop.last %}UNION ALL{% endif %} {% endfor %}
💡 Interview Insight

"How do you avoid repetitive SQL in dbt?" — "Three techniques: (1) {% for %} loops — generate CASE WHENs from a list. Adding a new status = adding one string. (2) Custom macros like cents_to_dollars(). (3) {% if %} for environment-specific logic like PII masking in dev."

🌐 adapter.dispatch() — Cross-Database

When your project runs on multiple warehouses, function syntax differs. adapter.dispatch() routes to the correct implementation based on the active adapter.

-- macros/date_trunc_month.sql {% macro date_trunc_month(date_column) %} {{ return(adapter.dispatch('date_trunc_month')(date_column)) }} {% endmacro %} -- Snowflake implementation {% macro snowflake__date_trunc_month(date_column) %} DATE_TRUNC('month', {{ date_column }}) {% endmacro %} -- BigQuery implementation {% macro bigquery__date_trunc_month(date_column) %} DATE_TRUNC({{ date_column }}, MONTH) {% endmacro %} -- Default fallback {% macro default__date_trunc_month(date_column) %} DATE_TRUNC('month', {{ date_column }}) {% endmacro %}
Dispatch Flow: 1. Check: does snowflake__date_trunc_month exist? └── If adapter is Snowflake: YES → use it 2. Check: does bigquery__date_trunc_month exist? └── If adapter is BigQuery: YES → use it 3. Fallback: use default__date_trunc_month Naming convention: {adapter}__macro_name
💡 Interview Insight

"How do you handle SQL syntax differences across warehouses?" — "With adapter.dispatch(). I write a base macro that delegates to snowflake__, bigquery__, etc. implementations. It auto-routes based on the active adapter. Especially useful for date functions, string functions, and type casting."

📋 Dependency Management — packages.yml

Packages extend dbt with reusable macros, models, and tests. Managing them properly is essential.

Version Pinning

✅ GOOD — Pin to a range (allows patches): version: [">=1.1.0", "<2.0.0"] ✅ GOOD — Exact version (maximum reproducibility): version: 1.1.1 ❌ BAD — No version constraint (breaks on updates): - package: dbt-labs/dbt_utils ❌ BAD — Git ref to a branch (not reproducible): revision: main
💡 Interview Insight

"How do you manage dbt packages?" — "Version ranges like >=1.1.0, <2.0.0 for Hub packages — allows patches, prevents major breaks. Git packages pinned to tags, never branches. Packages install to dbt_packages/ (gitignored), fetched fresh with dbt deps in CI."

🚀 Advanced — run_query & execute

For power users: query the warehouse at compile time, run admin operations, and build truly dynamic models.

-- macros/get_max_date.sql {% macro get_max_order_date() %} {% set query %} SELECT MAX(order_date)::VARCHAR AS max_date FROM {{ ref('fct_orders') }} {% endset %} {% set results = run_query(query) %} {% if execute %} ← CRITICAL guard! {% set max_date = results.columns[0].values()[0] %} {{ return(max_date) }} {% else %} {{ return('2020-01-01') }} {% endif %} {% endmacro %}

The execute variable: During dbt compile, Jinja processes templates but doesn't actually run queries. execute is True only during dbt run / dbt build. Always guard run_query() with {% if execute %}.

Custom Operation Macros

-- macros/operations/drop_old_models.sql {% macro drop_old_models(days=30) %} {% set query %} SELECT table_name FROM information_schema.tables WHERE table_schema = '{{ target.schema }}' AND last_altered < DATEADD(day, -{{ days }}, CURRENT_TIMESTAMP()) {% endset %} {% set results = run_query(query) %} {% if execute %} {% for table in results.columns[0].values() %} {{ log("Dropping: " ~ table, info=true) }} {% do run_query("DROP TABLE IF EXISTS " ~ target.schema ~ "." ~ table) %} {% endfor %} {% endif %} {% endmacro %} -- Run: $ dbt run-operation drop_old_models --args '{"days": 60}'
💡 Interview Insight

"What is run_query() and when would you use it?" — "run_query() executes SQL at compile time and returns results as a Jinja object. I use it for dynamic SQL — like fetching distinct values for pivot columns. Critical: always guard with {% if execute %} because during parse phase the query can't run. I also use it in operation macros for admin tasks."
