📸 Why Track Historical Changes?

In a typical e-commerce database, when a customer upgrades from "bronze" to "gold" tier, the source system overwrites the old value. Yesterday they were bronze; today they're gold. The old value is gone — mutated in place.

But your analytics team needs to answer: "How much revenue did gold-tier customers generate last quarter?" If a customer upgraded mid-quarter, should their earlier purchases count as bronze or gold?

This is the Slowly Changing Dimension (SCD) problem. Dimensions — customers, products, stores — change over time, and your analytics warehouse needs to preserve that history even when the source system doesn't.

SOURCE DATABASE (raw_customers): ┌─────────────┬────────┬──────────────┬─────────────────────┐ │ customer_id │ tier │ email │ updated_at │ ├─────────────┼────────┼──────────────┼─────────────────────┤ │ 101 │ gold │ ali@shop.com │ 2024-06-15 10:30:00 │ └─────────────┴────────┴──────────────┴─────────────────────┘ ↑ Was "bronze" before June 15 — but that info is GONE. WHAT WE WANT (snapshot table): ┌─────────────┬────────┬──────────────┬────────────┬────────────┐ │ customer_id │ tier │ email │ valid_from │ valid_to │ ├─────────────┼────────┼──────────────┼────────────┼────────────┤ │ 101 │ bronze │ ali@shop.com │ 2024-01-01 │ 2024-06-15 │ │ 101 │ gold │ ali@shop.com │ 2024-06-15 │ NULL │ └─────────────┴────────┴──────────────┴────────────┴────────────┘ ↑ Both versions preserved! NULL valid_to = current record.

SCD Types — A Quick Taxonomy

TypeBehaviorUse Case
Type 0Never update — keep originalRarely used (original signup date)
Type 1Overwrite — no historySmall/unimportant dimensions
Type 2Add new row — full historydbt snapshots — the standard
Type 3Add column (current + previous)When only one prior value matters
💡 Interview Insight

"What is SCD Type 2 and how does dbt implement it?" — "SCD Type 2 preserves full history by adding a new row every time a dimension changes, with validity timestamps marking each version. dbt implements this through snapshot blocks that compare the current source state against the snapshot table. When a row changes, dbt sets dbt_valid_to on the old row and inserts a new row with dbt_valid_from = now and dbt_valid_to = NULL."

🧱 Snapshot Blocks — Anatomy & Syntax

Snapshots live in the snapshots/ directory (not models/) and use a special {% snapshot %} block syntax.

-- snapshots/snap_customers.sql {% snapshot snap_customers %} {{ config( target_database='ANALYTICS_DB', target_schema='snapshots', unique_key='customer_id', strategy='timestamp', updated_at='updated_at', invalidate_hard_deletes=True ) }} SELECT customer_id, first_name, last_name, email, subscription_tier, shipping_country, lifetime_spend_cents, updated_at FROM {{ source('ecommerce', 'raw_customers') }} {% endsnapshot %}

Key Elements

🔑 unique_key

The natural key identifying each entity. Must be truly unique in the source.

📐 strategy

How dbt detects changes: timestamp or check.

🗄️ target_schema

Where the snapshot table is created. Best practice: isolate in own schema.

🗑️ invalidate_hard_deletes

Whether to track rows deleted from the source.

# Running snapshots $ dbt snapshot # Run all snapshots $ dbt snapshot --select snap_customers # Run a specific one $ dbt build # Runs models, tests, snapshots, seeds in DAG order

What dbt Generates (First Run)

CREATE TABLE ANALYTICS_DB.snapshots.snap_customers AS ( SELECT customer_id, first_name, last_name, email, subscription_tier, shipping_country, lifetime_spend_cents, updated_at, -- dbt adds these meta columns: MD5(customer_id || '|' || updated_at) AS dbt_scd_id, updated_at AS dbt_valid_from, NULL::TIMESTAMP AS dbt_valid_to, CURRENT_TIMESTAMP() AS dbt_updated_at FROM RAW_DB.ecommerce.raw_customers );
💡 Interview Insight

"Where do snapshots live and how are they different from models?" — "Snapshots live in snapshots/, use {% snapshot %} blocks, and always query source tables directly — not ref() to other models. They're run with dbt snapshot or dbt build. The output table includes dbt_valid_from, dbt_valid_to, and dbt_scd_id columns automatically."

🔍 Strategies — timestamp vs check

dbt offers two strategies for detecting changes. The wrong strategy can miss changes or create unnecessary versions.

Strategy 1: timestamp (preferred)

{% snapshot snap_products %} {{ config( target_schema='snapshots', unique_key='product_id', strategy='timestamp', updated_at='updated_at' ) }} SELECT product_id, product_name, category, price_cents, is_active, updated_at FROM {{ source('ecommerce', 'raw_products') }} {% endsnapshot %}

Strategy 2: check (column comparison)

{% snapshot snap_products_check %} {{ config( target_schema='snapshots', unique_key='product_id', strategy='check', check_cols=['product_name', 'category', 'price_cents', 'is_active'] ) }} SELECT product_id, product_name, category, price_cents, is_active FROM {{ source('ecommerce', 'raw_products') }} {% endsnapshot %}

⏱️ timestamp

  • ✅ Fast — single column compare
  • ✅ dbt_valid_from = source timestamp
  • ⚠️ Requires reliable updated_at
  • ⚠️ False positives if timestamp changes w/o data change
  • 🏆 DEFAULT CHOICE

✔️ check

  • ✅ No timestamp column needed
  • ✅ Checks actual values — no false positives
  • ⚠️ Slower — multi-column compare each run
  • ⚠️ May miss changes if columns excluded
  • Use only when no reliable timestamp
The In-Between Problem: Actual changes: bronze → silver → gold (Jan 1) (Jan 5) (Jan 10) Snapshot runs: Jan 1 Jan 15 ↓ ↓ Captured: bronze ──────→ gold (silver state is LOST forever) Solution: Run snapshots frequently (daily or more)
💡 Interview Insight

"When would you use check vs timestamp?" — "I default to timestamp — it's faster. I use check when the source has no updated_at column, or when the timestamp is unreliable. With check, I specify exactly which columns to monitor so non-meaningful changes don't trigger new versions. The tradeoff is performance."

📋 Understanding the Snapshot Table

Every snapshot table includes four dbt-managed columns alongside your source columns. Understanding these is critical for downstream dimension models.

snap_customers — After Multiple Runs ┌─────────────┬────────┬──────────────┬──────────────────┬──────────────────┐ │ customer_id │ tier │ email │ dbt_valid_from │ dbt_valid_to │ ├─────────────┼────────┼──────────────┼──────────────────┼──────────────────┤ │ 101 │ bronze │ ali@shop.com │ 2024-01-01 06:00 │ 2024-03-15 06:00 │ │ 101 │ silver │ ali@shop.com │ 2024-03-15 06:00 │ 2024-06-15 06:00 │ │ 101 │ gold │ ali@b.com │ 2024-06-15 06:00 │ NULL │ │ 102 │ gold │ bob@shop.com │ 2024-02-01 06:00 │ NULL │ │ 103 │ bronze │ cat@shop.com │ 2024-01-15 06:00 │ 2024-09-01 06:00 │ └─────────────┴────────┴──────────────┴──────────────────┴──────────────────┘
ColumnTypeDescription
dbt_scd_idVARCHARUnique surrogate key for each snapshot row
dbt_valid_fromTIMESTAMPWhen this version became active
dbt_valid_toTIMESTAMPWhen superseded (NULL = current)
dbt_updated_atTIMESTAMPWhen dbt last processed this row

Key Queries Against Snapshots

-- Get CURRENT state (latest version) SELECT * FROM snap_customers WHERE dbt_valid_to IS NULL; -- Point-in-time lookup SELECT * FROM snap_customers WHERE customer_id = 101 AND dbt_valid_from <= '2024-04-01' AND (dbt_valid_to > '2024-04-01' OR dbt_valid_to IS NULL); -- Count tier changes per customer SELECT customer_id, COUNT(*) - 1 AS tier_changes FROM snap_customers GROUP BY customer_id HAVING COUNT(*) > 1;
💡 Interview Insight

"How do you get the current version from a snapshot table?" — "Filter for WHERE dbt_valid_to IS NULL. The current version always has NULL dbt_valid_to. For point-in-time queries, filter where dbt_valid_from <= target_date AND (dbt_valid_to > target_date OR dbt_valid_to IS NULL)."

🗑️ Hard Deletes — invalidate_hard_deletes

When a row is deleted from the source, what happens in the snapshot? By default, dbt does nothing — the row looks "current" forever. invalidate_hard_deletes=True fixes this.

Hard Delete Detection Run 1 (Jan 1): Source has customers 101, 102, 103 Snapshot: 101 (valid), 102 (valid), 103 (valid) ── Customer 103 is deleted from source ── Run 2 (Jan 15): Source has customers 101, 102 (no 103!) invalidate_hard_deletes=False (default): → 103 stays with dbt_valid_to=NULL (looks current!) ❌ invalidate_hard_deletes=True: → 103 gets dbt_valid_to=CURRENT_TIMESTAMP() → Row is properly closed out ✅

Performance note: With invalidate_hard_deletes=True, dbt must perform a full anti-join between source and snapshot every run. For 100M+ row tables, this can be expensive.

Detecting Deletes in Downstream Models

-- models/marts/dim_customers.sql SELECT dbt_scd_id AS customer_key, customer_id, first_name, last_name, email, subscription_tier, dbt_valid_from AS effective_from, dbt_valid_to AS effective_to, CASE WHEN dbt_valid_to IS NULL THEN TRUE ELSE FALSE END AS is_current, CASE WHEN dbt_valid_to IS NOT NULL AND customer_id NOT IN ( SELECT customer_id FROM {{ ref('snap_customers') }} WHERE dbt_valid_to IS NULL ) THEN TRUE ELSE FALSE END AS is_deleted FROM {{ ref('snap_customers') }}
💡 Interview Insight

"What happens when a source row is deleted?" — "By default, nothing — the deleted row stays with dbt_valid_to = NULL, looking current. With invalidate_hard_deletes=True, dbt detects missing rows via an anti-join and closes them out. I enable this for entity dimensions where deletions matter; I disable it for huge tables where the anti-join cost is high."

🏗️ Building Dimension Tables from Snapshots

Snapshots are raw SCD Type 2 tables. In practice, you build polished dimension models on top — adding business logic, derived columns, and proper naming.

-- models/marts/dim_customers.sql {{ config(materialized='table') }} WITH snapshot_data AS ( SELECT * FROM {{ ref('snap_customers') }} ), enriched AS ( SELECT dbt_scd_id AS customer_version_key, customer_id, first_name || ' ' || last_name AS full_name, LOWER(email) AS email, COALESCE(subscription_tier, 'free') AS subscription_tier, lifetime_spend_cents / 100.0 AS lifetime_spend_dollars, CASE WHEN subscription_tier = 'gold' THEN 3 WHEN subscription_tier = 'silver' THEN 2 WHEN subscription_tier = 'bronze' THEN 1 ELSE 0 END AS tier_rank, dbt_valid_from AS effective_from, dbt_valid_to AS effective_to, dbt_valid_to IS NULL AS is_current_version, ROW_NUMBER() OVER ( PARTITION BY customer_id ORDER BY dbt_valid_from ) AS version_number FROM snapshot_data ) SELECT * FROM enriched

Joining Facts to SCD Type 2 Dimensions

You must join on the natural key AND the validity period to get correct attributes at the time of each fact event.

-- Join orders to customer dim valid at order time SELECT o.order_id, o.order_date, o.order_amount_dollars, c.full_name, c.subscription_tier AS tier_at_order_time FROM {{ ref('fct_orders') }} o LEFT JOIN {{ ref('dim_customers') }} c ON o.customer_id = c.customer_id AND o.order_date >= c.effective_from AND (o.order_date < c.effective_to OR c.effective_to IS NULL)
💡 Interview Insight

"How do you join a fact table to an SCD Type 2 dimension?" — "Join on the natural key AND the validity period: ON fact.customer_id = dim.customer_id AND fact.event_date >= dim.effective_from AND (fact.event_date < dim.effective_to OR dim.effective_to IS NULL). The NULL check handles the current version."

⚖️ Snapshots vs Manual SCD

Not every SCD scenario calls for dbt snapshots. Understanding when to use each is a mark of senior dbt proficiency.

✅ Use dbt Snapshots

  • Source has no change history (mutations overwrite)
  • Table is reasonably sized (< 50M rows)
  • Changes are infrequent (dimensions change slowly)
  • Daily-granularity tracking is acceptable
  • Standard timestamp/check strategy suffices

🔧 Build Manual SCD

  • Source already has CDC/event stream history
  • Need sub-daily change granularity
  • Source table is huge (100M+ rows)
  • Need SCD Type 3 or Type 6 (hybrid)
  • Complex merge / conditional update logic

Manual SCD from CDC Source

-- models/marts/dim_customers_from_cdc.sql {{ config(materialized='incremental', unique_key='customer_version_key') }} WITH cdc_events AS ( SELECT customer_id, first_name, last_name, email, subscription_tier, operation, event_timestamp, LEAD(event_timestamp) OVER ( PARTITION BY customer_id ORDER BY event_timestamp ) AS next_event_timestamp FROM {{ ref('stg_customer_cdc_events') }} {% if is_incremental() %} WHERE event_timestamp > (SELECT MAX(effective_from) FROM {{ this }}) {% endif %} ) SELECT {{ dbt_utils.generate_surrogate_key(['customer_id', 'event_timestamp']) }} AS customer_version_key, customer_id, first_name, last_name, email, subscription_tier, event_timestamp AS effective_from, next_event_timestamp AS effective_to, next_event_timestamp IS NULL AS is_current_version FROM cdc_events
💡 Interview Insight

"When would you NOT use dbt snapshots?" — "Three scenarios: (1) Source already provides CDC history — I'd build an incremental model instead. (2) Source is 100M+ rows — full-table comparison gets expensive. (3) I need sub-daily tracking — snapshots only capture state at run time, so intermediate states are lost."

⚠️ Real-World Gotchas & Best Practices

After running snapshots in production for months, these are the issues that bite teams.

Gotcha 1: Always Snapshot source(), Never ref()

-- ❌ WRONG — Don't snapshot a staging model SELECT * FROM {{ ref('stg_customers') }} -- Tracks transformation changes! -- ✅ CORRECT — Snapshot the raw source directly SELECT * FROM {{ source('ecommerce', 'raw_customers') }} -- Tracks source changes!

Gotcha 2: Never Change unique_key After Deployment

Changing the unique_key corrupts the table. dbt can't match old rows to new ones. If you must change it, drop the table and rebuild from scratch.

Gotcha 3: Timezone Mismatches

-- ✅ Always normalize timezones in the snapshot SELECT SELECT customer_id, CONVERT_TIMEZONE('UTC', updated_at) AS updated_at FROM {{ source('ecommerce', 'raw_customers') }}

Best Practices Checklist

1️⃣ source() not ref()

Capture raw state, not transformed state.

2️⃣ Run Daily+

Changes between runs are lost forever.

3️⃣ Prefer timestamp

Faster. Use check only when no reliable timestamp.

4️⃣ Test dbt_scd_id

Add unique + not_null tests on the surrogate key.

5️⃣ Dedicated Schema

Store snapshots in their own schema (e.g., snapshots).

6️⃣ Monitor Growth

SCD2 tables grow fast. Watch version counts per entity.

💡 Interview Insight

"Common issues with dbt snapshots in production?" — "Four big ones: (1) Snapshotting a staging model instead of the raw source. (2) Timezone mismatches between updated_at and dbt_valid_from. (3) Not enabling invalidate_hard_deletes. (4) Running snapshots too infrequently. I also monitor version counts to detect false positives."
