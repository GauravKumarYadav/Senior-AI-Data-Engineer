What Are Materializations?

A materialization tells dbt how to persist the results of a model's SELECT statement in the warehouse. You write a SELECT вҖ” dbt decides whether to wrap it in a CREATE VIEW, CREATE TABLE AS SELECT, a MERGE statement, or inline it as a CTE.

рҹ‘ҒпёҸ view

CREATE VIEW AS (SELECT ...) вҖ” no data stored, re-executes every query

рҹ“Ұ table

CREATE TABLE AS SELECT вҖ” full rebuild, data stored and precomputed

рҹ“Ҳ incremental

INSERT/MERGE into existing table вҖ” only new or changed rows processed

рҹ‘» ephemeral

Inlined as CTE вҖ” no database object created at all

The right choice depends on: (1) complexity of the transformation, (2) frequency of access, and (3) data volume.

-- Setting materialization in a model {{ config(materialized='table') }} SELECT order_id, customer_id, order_amount_cents / 100.0 AS order_amount_dollars FROM {{ ref('stg_orders') }}
рҹ’Ў Interview Insight

"Views for simple transformations queried infrequently, tables for complex transforms queried often, incremental for large datasets where you only want to process new rows, and ephemeral for intermediate logic you don't need to persist. The key tradeoff is build time vs. query time vs. storage cost."

Views вҖ” The Lightweight Default

A view materialization creates a database view. No data is stored вҖ” the view is just a saved SQL query. Every time someone queries the view, the underlying SQL executes against the source tables.

-- models/staging/stg_orders.sql {{ config(materialized='view') }} SELECT id AS order_id, user_id AS customer_id, order_date, status, CAST(amount AS DECIMAL(10,2)) AS order_amount_cents FROM {{ source('ecommerce', 'raw_orders') }} WHERE status != 'test'
рҹ“„ Staging model for orders
рҹ‘ҒпёҸ Materialized as a view вҖ” instant build
 
вңҸпёҸ Rename columns to clean conventions
 
 
рҹ”ў Explicit type casting
рҹ”— source() points to raw table
рҹ—‘пёҸ Filter out junk data

вң… Advantages

  • вҡЎ Fast to build вҖ” creating a view is nearly instant
  • рҹ”„ Always fresh вҖ” queries always read latest source data
  • рҹ’ҫ No storage cost вҖ” no duplicate data stored
  • рҹҡҖ Instant deployment вҖ” no waiting for table rebuilds

вқҢ Disadvantages

  • рҹҗў Slow to query вҖ” every query re-executes the SQL
  • рҹ’Ҙ Cascading compute вҖ” downstream queries multiply
  • рҹ’ё Warehouse load вҖ” complex views are expensive
рҹ’Ў Interview Insight

"Staging models are 1:1 with source tables and perform lightweight transformations вҖ” renaming columns, casting types, filtering test data. These are cheap SQL operations that don't benefit from table materialization. Views keep staging builds instant, and since staging models are only queried by downstream dbt models (not dashboards), the compute cost is manageable."

Tables вҖ” Precomputed for Performance

A table materialization creates a physical table using CREATE TABLE AS SELECT. The query runs once during dbt run, data is stored, and subsequent queries read precomputed results вҖ” no re-execution.

-- models/marts/dim_customers.sql {{ config(materialized='table') }} WITH customer_orders AS ( SELECT customer_id, MIN(order_date) AS first_order_date, MAX(order_date) AS most_recent_order_date, COUNT(*) AS number_of_orders, SUM(order_amount_cents) / 100.0 AS lifetime_value FROM {{ ref('stg_orders') }} GROUP BY customer_id ) SELECT c.customer_id, c.first_name, c.last_name, COALESCE(co.number_of_orders, 0) AS total_orders, COALESCE(co.lifetime_value, 0) AS lifetime_value_dollars, CASE WHEN co.lifetime_value >= 1000 THEN 'platinum' WHEN co.lifetime_value >= 500 THEN 'gold' WHEN co.lifetime_value >= 100 THEN 'silver' ELSE 'bronze' END AS customer_tier FROM {{ ref('stg_customers') }} AS c LEFT JOIN customer_orders AS co ON c.customer_id = co.customer_id
Table vs. View Decision Matrix в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ Factor в”Ӯ Use View в”Ӯ Use Table в”Ӯ в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө в”Ӯ Transform cost в”Ӯ Low в”Ӯ High в”Ӯ в”Ӯ Query frequency в”Ӯ Low в”Ӯ High в”Ӯ в”Ӯ Data freshness в”Ӯ Critical в”Ӯ Can wait в”Ӯ в”Ӯ Downstream users в”Ӯ Few models в”Ӯ Dashboards в”Ӯ в”Ӯ Data volume в”Ӯ Small в”Ӯ Any в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
рҹ’Ў Interview Insight

"When the transformation is expensive and gets queried frequently. For example, a dim_customers model that joins three tables, aggregates order history, and calculates customer tiers вҖ” this should be a table because it would be wasteful to re-compute on every dashboard query. I also switch to table when I notice slow dashboard performance traced to view cascading."

Incremental вҖ” Process Only New Data

Incremental is dbt's most powerful materialization. Instead of rebuilding the entire table from scratch, it only processes new or changed rows since the last run. For a 500M-row table with 100K daily arrivals, this means processing 100K instead of 500M.

-- models/marts/fct_orders.sql {{ config( materialized='incremental', unique_key='order_id', incremental_strategy='merge' ) }} SELECT o.order_id, o.customer_id, o.order_date, o.status, o.order_amount_cents / 100.0 AS order_amount_dollars, CURRENT_TIMESTAMP() AS dbt_updated_at FROM {{ ref('stg_orders') }} AS o {% if is_incremental() %} -- Only process rows newer than the latest existing row WHERE o.order_date > (SELECT MAX(order_date) FROM {{ this }}) {% endif %}
Incremental Model Execution First Run: SELECT * FROM source в”Җв”Җв–ә CREATE TABLE (full build) Next Runs: SELECT * FROM source WHERE date > max(existing_table.date) в”Җв”Җв–ә MERGE INTO existing table Full Refresh: dbt run --full-refresh в”Җв”Җв–ә DROP + CREATE TABLE (rebuild)

The {{ this }} variable refers to the current model's existing table. It's only valid inside an {% if is_incremental() %} block. We'll cover incremental models in depth in Module 3.

рҹ’Ў Interview Insight

"Performance and cost. A table materialization rebuilds from scratch every run вҖ” fine for small datasets, but a 500M-row fact table would take hours. Incremental processes only new/changed rows, reducing build time from hours to minutes and cutting warehouse compute costs significantly. The tradeoff is added complexity in handling edge cases like late-arriving data."

Ephemeral вҖ” The Invisible Model

An ephemeral model doesn't create any database object. Instead, dbt inlines the model's SQL as a Common Table Expression (CTE) in every model that references it.

-- int_order_totals.sql (ephemeral) {{ config(materialized='ephemeral') }} SELECT customer_id, COUNT(*) AS total_orders, SUM(order_amount_cents) / 100.0 AS total_spent_dollars FROM {{ ref('stg_orders') }} GROUP BY customer_id
рҹ“„ Intermediate model вҖ” NOT persisted
рҹ‘» Becomes a CTE in downstream models
 
рҹ“Ҡ Aggregate order data per customer
 
 
 
рҹ”„ This SQL gets copy-pasted as a WITH clause

What dbt compiles downstream to:

-- Compiled dim_customers.sql WITH __dbt__cte__int_order_totals AS ( SELECT customer_id, COUNT(*) AS total_orders, SUM(order_amount_cents) / 100.0 AS total_spent_dollars FROM ECOMMERCE_DEV.dbt_jsmith.stg_orders GROUP BY customer_id ) SELECT c.customer_id, c.first_name, c.last_name, COALESCE(ot.total_orders, 0) AS total_orders FROM ECOMMERCE_DEV.dbt_jsmith.stg_customers AS c LEFT JOIN __dbt__cte__int_order_totals AS ot ON c.customer_id = ot.customer_id
рҹ’Ў Interview Insight

"Ephemeral models are inlined as CTEs вҖ” no database object is created. I'd avoid them when the model is referenced by many downstream models, because the CTE gets duplicated in each compiled query, multiplying compute. They're best for simple, focused logic referenced by one or two downstream models."

Model Layers & Naming Conventions

A well-organized dbt project uses a layered architecture with clear naming conventions that communicate purpose at a glance.

dbt Model Layers в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ STAGING в”Ӯ в”Ӯ INTERMEDIATE в”Ӯ в”Ӯ MARTS в”Ӯ в”Ӯ stg_* в”Ӯв”Җв”Җв”Җв–әв”Ӯ int_* в”Ӯв”Җв”Җв”Җв–әв”Ӯ fct_* / dim_* в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ 1:1 with в”Ӯ в”Ӯ Business в”Ӯ в”Ӯ Business-ready в”Ӯ в”Ӯ sources в”Ӯ в”Ӯ logic joins в”Ӯ в”Ӯ fact & dim в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ tables в”Ӯ в”Ӯ view в”Ӯ в”Ӯ ephemeral в”Ӯ в”Ӯ table / в”Ӯ в”Ӯ в”Ӯ в”Ӯ or view в”Ӯ в”Ӯ incremental в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

stg_ вҖ” Staging

1:1 source mapping. Rename, cast, filter. Always views. No joins, no aggregations.

int_ вҖ” Intermediate

Business logic: joins, pivots, window functions. Not queried by end users.

fct_ вҖ” Facts

Events/transactions. One row per event (order, click, payment). Table or incremental.

dim_ вҖ” Dimensions

Entities. One row per entity (customer, product, store). Table materialization.

-- models/staging/stg_customers.sql {{ config(materialized='view') }} SELECT id AS customer_id, LOWER(TRIM(first_name)) AS first_name, LOWER(TRIM(last_name)) AS last_name, LOWER(TRIM(email)) AS email, CAST(created_at AS TIMESTAMP) AS created_at FROM {{ source('ecommerce', 'raw_customers') }} WHERE id IS NOT NULL
рҹ’Ў Interview Insight

"I follow the staging вҶ’ intermediate вҶ’ marts pattern. Staging is 1:1 with sources вҖ” renaming, casting, filtering вҖ” always views. Intermediate holds reusable business logic вҖ” typically ephemeral. Marts are business-ready fct_ and dim_ tables. Naming conventions make it immediately clear where a model sits in the pipeline and who should be querying it."

The config() Block вҖ” Model-Level Config

Every dbt model can include a {{ config() }} block to set model-specific configurations, overriding directory-level settings from dbt_project.yml.

рҹ’Ў Interview Insight

"I use the schema config to assign models to logical schemas вҖ” staging, intermediate, marts. By default, dbt prepends the target schema, so I customize generate_schema_name. In dev, I keep the prefix for developer isolation. In prod, I override to use exact schema names."

Custom Schemas & Multi-Schema Deploys

In production, you want staging models in a staging schema, marts in marts, and PII models in a restricted sensitive schema. dbt's custom schema handling is commonly misunderstood.

-- macros/generate_schema_name.sql (prod-aware) {% macro generate_schema_name(custom_schema_name, node) -%} {%- set default_schema = target.schema -%} {%- if target.name == 'prod' -%} {# In production, use the custom schema directly #} {%- if custom_schema_name is not none -%} {{ custom_schema_name | trim }} {%- else -%} {{ default_schema }} {%- endif -%} {%- else -%} {# In dev, prefix with target schema for isolation #} {%- if custom_schema_name is not none -%} {{ default_schema }}_{{ custom_schema_name | trim }} {%- else -%} {{ default_schema }} {%- endif -%} {%- endif -%} {%- endmacro %}
