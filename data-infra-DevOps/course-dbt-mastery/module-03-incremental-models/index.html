📈 Why Incremental Models Matter

In toy dbt projects, you can rebuild every table from scratch on every run. In production — with billions of rows, tight SLAs, and real money riding on warehouse compute costs — that's simply not possible. Incremental models are the single most important materialization for production dbt projects.

Your fct_orders table has 800 million rows spanning 3 years. Every day, 200K new orders arrive. A table materialization scans all 800M rows every run — 45 minutes, $50+ in compute. An incremental model processes only the 200K new rows in under 2 minutes.

❌ TABLE Materialization

  • 📊 Scan 800,000,000 rows every run
  • ⏱️ Build time: ~45 minutes
  • 💰 Daily compute: ~$50
  • 📅 Monthly cost: ~$1,500

✅ INCREMENTAL Materialization

  • 📊 Scan 200,000 new rows per run
  • ⏱️ Build time: ~2 minutes
  • 💰 Daily compute: ~$0.50
  • 📅 Monthly cost: ~$15

The fundamental idea: An incremental model keeps its existing data and only processes rows that are new or changed since the last run. dbt determines what's "new" based on a filter you provide inside an {% if is_incremental() %} block.

The tradeoff: Incremental models are more complex to write, harder to debug, and introduce edge cases around late-arriving data, schema changes, and data corrections.

💡 Interview Insight

"When the data volume makes full rebuilds impractical or expensive. I use incremental models for fact tables with millions+ rows where new data arrives in a predictable, append-like pattern — event logs, orders, clickstream data. I'd use a table for smaller datasets or dimension tables that need full refreshes. The key decision factor is: does the cost/time of a full rebuild justify the added complexity of incremental logic?"

The Core Pattern — config + is_incremental()

Every incremental model has two parts: the config block and the incremental filter.

-- models/marts/fct_orders.sql -- Part 1: The config block {{ config( materialized='incremental', unique_key='order_id', incremental_strategy='merge', on_schema_change='append_new_columns' ) }} -- Part 2: The base SELECT (always runs) SELECT o.order_id, o.customer_id, o.order_date, o.status, o.order_amount_cents / 100.0 AS order_amount_dollars, CURRENT_TIMESTAMP() AS dbt_loaded_at FROM {{ ref('stg_orders') }} AS o -- Part 3: The incremental filter (only on subsequent runs) {% if is_incremental() %} WHERE o.loaded_at > (SELECT MAX(dbt_loaded_at) FROM {{ this }}) {% endif %}

How dbt Executes This

First run — is_incremental() returns False: dbt builds the full table with no WHERE clause. Subsequent runs — returns True: dbt selects only new rows, then MERGEs them into the existing table.

🔑 unique_key

Column(s) that uniquely identify each row. Used by merge strategy to UPDATE (exists) or INSERT (new).

📌 {{ this }}

A reference to the current model's existing table. Only valid inside {% if is_incremental() %}.

🔄 is_incremental()

Returns True when: (1) configured as incremental, (2) target table exists, AND (3) no --full-refresh flag.

💡 Interview Insight

"is_incremental() returns True only when three conditions are met: the model is configured as incremental, the target table already exists in the warehouse, and the run doesn't include --full-refresh. On the first run, it's False (no existing table), so dbt builds the full table. On subsequent runs, it's True, so the WHERE clause filters to only new rows."

Incremental Strategies

dbt supports four incremental strategies. The strategy determines how new rows are combined with existing data.

📥 append

INSERT new rows. Never updates existing rows. Fastest. Can create duplicates if rerun. Use for immutable events (clickstream, logs).

🔀 merge

MERGE (upsert) using unique_key. Updates existing rows, inserts new ones. Most common. The workhorse of production dbt.

🗑️ delete+insert

DELETE matching rows, then INSERT. Alternative when MERGE is slow. Same behavior, different execution path.

📦 insert_overwrite

Overwrite entire partitions containing new data. Best for BigQuery partitioned tables. Requires partition_by config.

-- Append: immutable event data {{ config(materialized='incremental', incremental_strategy='append') }} SELECT event_id, user_id, event_type, event_timestamp FROM {{ ref('stg_clickstream_events') }} {% if is_incremental() %} WHERE event_timestamp > (SELECT MAX(event_timestamp) FROM {{ this }}) {% endif %}
-- Merge: mutable data (order status can change) {{ config(materialized='incremental', unique_key='order_id', incremental_strategy='merge') }} SELECT order_id, customer_id, status, updated_at FROM {{ ref('stg_orders') }} {% if is_incremental() %} WHERE updated_at > (SELECT MAX(updated_at) FROM {{ this }}) {% endif %}
-- Insert_overwrite: partition-level replacement (BigQuery) {{ config( materialized='incremental', incremental_strategy='insert_overwrite', partition_by={"field": "order_date", "data_type": "date", "granularity": "day"} ) }} SELECT * FROM {{ ref('stg_orders') }} {% if is_incremental() %} WHERE order_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 3 DAY) {% endif %}
💡 Interview Insight

"Merge, with unique_key='order_id'. When an order status changes from 'shipped' to 'delivered', the merge strategy updates the existing row. I'd filter on updated_at to capture both new orders and status changes. For immutable event data like clickstream, I'd use append since rows never change."

unique_key — Deduplication & Composite Keys

The unique_key tells dbt how to identify whether an incoming row already exists. Get this wrong, and you'll either create duplicates or lose data.

-- Single column (most common) {{ config(unique_key='order_id') }} -- Composite key (no single column is unique) {{ config(unique_key=['customer_id', 'activity_date']) }} -- Surrogate key (when you need to create a unique key) SELECT {{ dbt_utils.generate_surrogate_key(['order_id', 'line_item_id']) }} AS surrogate_key, order_id, line_item_id, product_id, quantity FROM {{ ref('stg_order_items') }}

⚠️ Common Gotchas

❌ Non-Unique Column

unique_key='customer_id' on an orders table → Only keeps one order per customer!

❌ Forgetting Composite

unique_key='date' on a per-customer-per-day table → Only one customer's data survives per day!

✅ Match the Grain

One row per order → 'order_id'. One per customer per day → ['customer_id', 'date']. Key = grain.

💡 Interview Insight

"Through the unique_key config, which must match the table's grain. For an orders table, it's order_id. For a daily customer activity table, it's ['customer_id', 'activity_date']. When I don't have a natural key, I generate a surrogate key using dbt_utils.generate_surrogate_key(). The unique_key drives the MERGE statement's ON clause — get it wrong and you get duplicates or data loss."

on_schema_change — Evolving Schemas

In production, source schemas change. A product team adds a discount_amount column. Your incremental model's SELECT now returns more columns than the target table has. The on_schema_change config determines what dbt does.

🙈 ignore (default)

New columns silently dropped. Use when you control schema tightly. Risky — you lose data!

➕ append_new_columns

New columns added as nullable. Old rows get NULL for new columns. Good when schema evolves naturally.

🔄 sync_all_columns

Add new columns, remove old ones. Most aggressive — can remove columns you still need!

🛑 fail

Raise an error on mismatch. Forces explicit handling. Safest for critical production tables.

-- For critical fact tables, fail on schema changes -- Forces explicit handling via --full-refresh or schema migration {{ config( materialized='incremental', unique_key='order_id', on_schema_change='fail' ) }} -- When 'fail' triggers, you have two options: -- 1. Full refresh: dbt run --select fct_orders --full-refresh -- 2. Manual ALTER: Add the column manually, then run normally
💡 Interview Insight

"For critical tables, I use 'fail' to force explicit handling — I don't want schema changes sneaking in silently. For event tables with evolving properties, I use 'append_new_columns' since new fields are expected. I avoid 'ignore' in production because it silently drops data. When a schema change is detected, I do a --full-refresh to rebuild with the correct schema."

Full Refresh — The Escape Hatch

dbt run --full-refresh is your emergency reset button. It forces dbt to drop the existing table and rebuild from scratch.

# Full refresh a single model $ dbt run --select fct_orders --full-refresh # Full refresh all incremental models $ dbt run --full-refresh # Full refresh a model and all downstream dependencies $ dbt run --select fct_orders+ --full-refresh

🔧 Schema Change

New columns need to be added to the target table.

🐛 Bug Fix

Historical data was calculated wrong — recompute all rows.

📥 Backfill

Source data was retroactively corrected.

📋 Logic Change

Business rule changed; need to recompute everything.

┌──────────────────────────────────────────────────────┐ │ Typical Production Schedule │ │ │ │ Mon-Sat: dbt run │ │ (incremental — process new rows only) │ │ │ │ Sunday: dbt run --full-refresh │ │ (complete rebuild — catch any drift) │ │ │ │ Cost: 6 fast runs + 1 slow run per week │ │ Benefit: Guaranteed data consistency │ └──────────────────────────────────────────────────────┘

Design for full-refresh safety: Your incremental model should produce the exact same results whether run incrementally or as a full refresh. If your incremental logic diverges from a full rebuild, you have a bug.

💡 Interview Insight

"I run incremental daily for cost efficiency and schedule weekly full refreshes as a safety net. This catches late-arriving data, retroactive corrections, and any incremental drift. For critical models, I also validate that incremental and full-refresh produce identical row counts as a data quality check."

Gotchas — Late-Arriving Data

Late-arriving data is the #1 production issue with incremental models. Your filter says "give me everything after the max timestamp," but data can arrive after the cutoff because of processing delays, timezone issues, or batch retries.

-- The Problem: Late data is filtered out forever! -- Run at 6am: MAX(loaded_at) = 5:59am → processes all rows ✓ -- At 7am: batch from 4am-5am finally arrives -- Next run: MAX is now today's 5:59am -- Late rows (4am-5am yesterday) are OLDER → filtered out forever! ✗

Solution 1: Lookback Window

{% if is_incremental() %} -- Look back 3 days to catch late-arriving data WHERE loaded_at >= ( SELECT DATEADD(day, -3, MAX(loaded_at)) FROM {{ this }} ) {% endif %} -- merge + unique_key ensures no duplicates when reprocessing

Solution 2: Monotonically Increasing Column

{% if is_incremental() %} -- _fivetran_synced = when row was loaded, not event time -- Monotonically increasing → no late-arrival issues WHERE _fivetran_synced > (SELECT MAX(_fivetran_synced) FROM {{ this }}) {% endif %}

Backfill Pattern with dbt vars

{% if is_incremental() %} {% if var('backfill_start_date', none) is not none %} WHERE order_date BETWEEN '{{ var("backfill_start_date") }}' AND '{{ var("backfill_end_date") }}' {% else %} WHERE loaded_at > (SELECT MAX(loaded_at) FROM {{ this }}) {% endif %} {% endif %} -- Normal: dbt run --select fct_orders -- Backfill: dbt run --select fct_orders \ -- --vars '{"backfill_start_date": "2024-01-15", "backfill_end_date": "2024-01-20"}'
💡 Interview Insight

"Three strategies: (1) A lookback window — reprocess the last 3 days with merge so existing rows get updated and late arrivals are captured. (2) Use a monotonically increasing column like _fivetran_synced instead of event time. (3) On BigQuery, use insert_overwrite to rewrite recent partitions entirely. I also schedule periodic full refreshes as a safety net."

Real-World Production Patterns

Three patterns that come up in every real dbt project and every senior-level interview.

Pattern 1: High-Volume Event Table

-- fct_page_views.sql — immutable events {{ config( materialized='incremental', incremental_strategy='append', -- Events never change partition_by={"field": "event_date", "data_type": "date"}, cluster_by=['user_id'] ) }} SELECT event_id, user_id, page_url, session_id, CAST(event_timestamp AS DATE) AS event_date, event_timestamp FROM {{ ref('stg_page_views') }} {% if is_incremental() %} WHERE event_timestamp > (SELECT MAX(event_timestamp) FROM {{ this }}) {% endif %}

Pattern 2: Daily Snapshot (Aggregated Metrics)

-- fct_daily_order_metrics.sql — recompute last 3 days {{ config(materialized='incremental', unique_key=['metric_date'], incremental_strategy='merge') }} SELECT DATE(order_date) AS metric_date, COUNT(DISTINCT order_id) AS total_orders, SUM(order_amount_dollars) AS total_revenue FROM {{ ref('fct_orders') }} {% if is_incremental() %} WHERE DATE(order_date) >= (SELECT DATEADD(day, -3, MAX(metric_date)) FROM {{ this }}) {% endif %} GROUP BY DATE(order_date)

Pattern 3: SCD Type 2 with dbt Snapshots

-- snapshots/snap_customers.sql {% snapshot snap_customers %} {{ config( target_schema='snapshots', unique_key='customer_id', strategy='timestamp', updated_at='updated_at', invalidate_hard_deletes=True ) }} SELECT customer_id, first_name, last_name, email, subscription_tier, updated_at FROM {{ source('ecommerce', 'raw_customers') }} {% endsnapshot %} -- Produces: dbt_valid_from, dbt_valid_to, dbt_scd_id columns -- customer 101: bronze (Jan-Jun) → gold (Jun-present)
💡 Interview Insight

"I built an incremental fact table processing ~500K rows daily from a 300M+ row table. Merge strategy with order_id as unique key and a 3-day lookback for late-arriving data. Partitioned by order_date, clustered by customer_id. Weekly full refreshes caught drift. We also used dbt snapshots on the customer dimension for SCD Type 2 historical tracking."
