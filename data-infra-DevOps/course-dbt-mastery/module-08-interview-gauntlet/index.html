🔥 Architecture & Design — "Design Me a dbt Project"

This module is different. No lectures — just scenario-based questions that mirror real dbt interviews. Work through each question, formulate your answer, then reveal the response.

Q1: You're joining an e-commerce company with 50 raw source tables, 3 analysts, and Snowflake. Design the dbt project structure.

ecommerce-dbt/ ├── models/ │ ├── staging/ # 1:1 with source tables, materialized as views │ │ ├── shopify/ │ │ │ ├── _shopify__sources.yml │ │ │ ├── stg_shopify__orders.sql │ │ │ └── stg_shopify__customers.sql │ │ └── stripe/ │ │ ├── _stripe__sources.yml │ │ └── stg_stripe__payments.sql │ ├── intermediate/ # Business logic joins, pre-aggregation │ │ └── int_orders_with_payments.sql │ └── marts/ # Business-facing, consumption-ready │ ├── core/ (fct_orders, dim_customers) │ ├── finance/ (fct_revenue, rpt_monthly_revenue) │ └── marketing/ (fct_customer_acquisition) ├── snapshots/ tests/ macros/ seeds/

Key decisions: Staging is 1:1 with sources (views), organized by source system. Intermediate handles complex joins. Marts organized by business domain. Custom generate_schema_name macro so dev uses prefixed schemas while prod uses clean names.

Q2: Your fct_orders has 2 billion rows and takes 90 minutes to build. How do you optimize it?

1. Materialize as incremental with merge strategy, unique_key='order_id', 3-day lookback

2. Partition by order_date (BigQuery) or cluster by order_date, customer_id (Snowflake)

3. Reduce upstream scanning — ensure staging views filter appropriately

4. Right-size warehouse — use MEDIUM for this model, SMALL for lighter models via {{ config(snowflake_warehouse='TRANSFORM_MEDIUM') }}

5. Schedule full refresh weekly (Sunday 2am), incremental daily

6. Consider splitting — if 90 min is due to complex joins, pre-compute intermediate models

Q3: Two teams (Finance and Marketing) both need customer data but with different definitions. How?

Shared foundation, domain-specific marts:

• dim_customers in marts/core/ — the canonical customer dimension with shared attributes

• fct_customer_revenue in marts/finance/ — Finance's view with LTV, payment terms

• fct_customer_engagement in marts/marketing/ — Marketing's view with acquisition channel, campaigns

Both domain marts ref('dim_customers') for shared attributes. Business-specific logic stays in domain models. Document the "official" definition in dim_customers description.

Q4: When would you use ephemeral vs. view vs. table vs. incremental?

Q6: How do you handle different environments (dev, staging, prod) in dbt?

Three-pronged approach:

1. profiles.yml — different targets per environment (dev uses personal schema, prod uses production database)

2. generate_schema_name macro override — dev: dev_alice_staging, prod: staging

3. target.name conditionals — mask PII in dev, use different warehouse sizes

CI gets its own target with isolated schemas: ci_pr_{{ var('pr_number') }}. Each PR builds into its own schema, cleaned up after merge.

🐛 Incremental Model Debugging — "My Model Has a Bug"

These scenarios test your ability to diagnose and fix common incremental model issues — situations you will face in production.

Q1: Your incremental fct_orders has duplicate rows. Orders from yesterday appear twice. What happened?

Diagnosis checklist:

1. Missing unique_key — Without it, merge behaves like append

2. Wrong unique_key — Using customer_id instead of order_id

3. Upstream duplicates — stg_orders itself has duplicates (source issue)

4. Overlapping lookback + append strategy — Reprocessed rows get inserted again

Fix: Verify unique_key matches the grain, ensure staging is deduplicated, use merge strategy, then --full-refresh.

Q2: After deploying a logic change to fct_orders, new rows use the new formula but historical rows show old values. Fix?

This is expected behavior for incremental models — changes only apply to newly processed rows.

Fix: dbt run --select fct_orders --full-refresh. This drops and rebuilds the table with the new logic applied to ALL rows. Validate before the next incremental run resumes.

Q3: Your is_incremental() block filters WHERE updated_at > MAX(updated_at). Late-arriving orders from 3 days ago are missed. Why?

The MAX(updated_at) in the target is from today. Late-arriving orders have updated_at from 3 days ago — older than MAX, so they're filtered out.

-- Fix: Add a lookback window WHERE updated_at >= ( SELECT DATEADD(day, -3, MAX(updated_at)) FROM {{ this }} )

The merge strategy with unique_key ensures reprocessed rows update in place (no duplicates).

Q4: You add a new column discount_amount to your incremental model. After running, all values are NULL. What's wrong?

Check on_schema_change config:

• ignore (default): new column is silently dropped — won't be added to target

• append_new_columns: column is added, new rows have values, old rows are NULL

Fix: Set on_schema_change='append_new_columns' and re-run. If ALL rows need the column populated, run --full-refresh.

Q5: Your incremental model processed 0 rows. No errors. What do you check?

1. Source freshness — Is new data arriving?

2. Incremental filter — Is MAX(updated_at) in the future? (timezone bug)

3. is_incremental() behavior — Does the target table exist?

4. Compiled SQL — dbt compile --select fct_orders and inspect the actual WHERE clause

5. Timezone mismatch — Source in UTC but {{ this }} has local time values

Q6: Model works perfectly with --full-refresh but produces different row counts incrementally. How to diagnose?

Causes of incremental/full-refresh parity bugs:

1. Late-arriving data without lookback — some rows never captured

2. Different filter logic — the {% if is_incremental() %} block differs from full-load

3. Source mutations — rows change after initial load, no merge on unique_key

4. Aggregation windows — incremental processes a slice but aggregation depends on full dataset

Fix: Run dbt_audit_helper.compare_relations to find exact differing rows, then adjust incremental logic.

📊 Testing & Data Quality — "The Dashboard Shows Wrong Numbers"

Q1: dim_customers.total_orders doesn't match COUNT in fct_orders for some customers. Investigate.

-- tests/assert_customer_order_count_consistency.sql SELECT c.customer_id, c.total_orders, COUNT(o.order_id) AS actual FROM {{ ref('dim_customers') }} c LEFT JOIN {{ ref('fct_orders') }} o ON c.customer_id = o.customer_id GROUP BY c.customer_id, c.total_orders HAVING c.total_orders != COUNT(o.order_id)

Then check: refresh schedule differences, filter differences (cancelled orders?), incremental drift.

Q2: Your accepted_values test on orders.status fails because a new status 'backordered' appeared. What do you do?

1. Immediate: Change severity to warn so production doesn't halt

2. Investigate: Confirm 'backordered' is legitimate from source system

3. Update: Add 'backordered' to accepted values list

4. Process: Ensure downstream CASE WHEN statements handle the new status

5. Prevent: Consider dbt_utils.get_column_values() to dynamically fetch statuses

Q3: Source freshness shows raw_orders last loaded 36 hours ago (error: 24h). dbt build is about to run. What do you do?

1. Don't build — stale source data means incomplete results

2. Alert the ingestion team — Fivetran/Airbyte may have a sync failure

3. Design: Configure Airflow DAG to check source freshness BEFORE dbt build — skip build and alert on error

4. Communicate: Notify stakeholders that today's data may be stale

Q4: Ensure daily revenue never drops more than 50% compared to the previous day. Write the test.

-- tests/assert_revenue_no_cliff_drop.sql {{ config(severity='warn') }} WITH daily AS ( SELECT DATE(order_date) AS order_day, SUM(order_amount_dollars) AS revenue FROM {{ ref('fct_orders') }} WHERE order_date >= DATEADD(day, -30, CURRENT_DATE()) GROUP BY DATE(order_date) ), with_prev AS ( SELECT *, LAG(revenue) OVER (ORDER BY order_day) AS prev_revenue FROM daily ) SELECT * FROM with_prev WHERE prev_revenue IS NOT NULL AND revenue < prev_revenue * 0.5

Q5: dbt tests vs warehouse constraints vs model contracts. What's your position?

Use all three — defense in depth:

• Model contracts (dbt 1.5+): Compile-time schema validation — catches structural issues before SQL runs

• Warehouse constraints: Prevent bad data from being written (not all warehouses enforce them)

• dbt tests: Post-load validation, flexible (cross-model relationships), severity levels, store_failures

Stack them: contracts (compile) → constraints (write) → tests (validate).

🧩 Jinja/Macro Challenges — "Write the Macro"

Q1: Write a macro that generates a CASE WHEN mapping status codes to labels.

-- macros/status_label.sql {% macro status_label(column_name) %} case {{ column_name }} when 'PL' then 'Placed' when 'SH' then 'Shipped' when 'DL' then 'Delivered' when 'RT' then 'Returned' when 'CA' then 'Cancelled' else 'Unknown' end {% endmacro %} -- Usage: SELECT {{ status_label('status_code') }} AS status_label

Q2: Write a macro that generates a UNION ALL of monthly partitioned tables.

-- macros/union_monthly_tables.sql {% macro union_monthly_tables(source_name, prefix, start_date, end_date) %} {% set start = modules.datetime.datetime.strptime(start_date, '%Y-%m') %} {% set tables = [] %} {% for i in range(100) %} {% if current <= end %} {% do tables.append(prefix ~ current.strftime('%Y_%m')) %} {% endif %} {% endfor %} {% for table in tables %} select *, '{{ table }}' as source_table from {{ source(source_name, table) }} {% if not loop.last %}union all{% endif %} {% endfor %} {% endmacro %}

Q3: Write a macro that grants SELECT on all models in a schema to a role. Use it as a post-hook.

-- macros/grant_select.sql {% macro grant_select(schema_name, role_name) %} grant usage on schema {{ target.database }}.{{ schema_name }} to role {{ role_name }}; grant select on all tables in schema {{ target.database }}.{{ schema_name }} to role {{ role_name }}; grant select on all views in schema {{ target.database }}.{{ schema_name }} to role {{ role_name }}; {% endmacro %} -- dbt_project.yml: on-run-end: "{{ grant_select('marts', 'analyst_role') }}"

Q4: This Jinja code has a bug. Find it:

{% set payment_methods = ['credit_card', 'paypal', 'apple_pay'] %} SELECT order_id, {% for method in payment_methods %} SUM(CASE WHEN payment_method = '{{ method }}' THEN amount ELSE 0 END) AS {{ method }}_total, {% endfor %} FROM {{ ref('stg_payments') }} GROUP BY order_id

Bug: Trailing comma after the last {% endfor %} — the , after apple_pay_total, creates invalid SQL (comma before FROM).

Fix: Use {% if not loop.last %},{% endif %}:

{% for method in payment_methods %} SUM(CASE WHEN payment_method = '{{ method }}' THEN amount ELSE 0 END) AS {{ method }}_total {% if not loop.last %},{% endif %} {% endfor %}

🚨 Production Operations — "It's 3am and the Pipeline Broke"

Q1: Daily dbt build fails at 4am. Slack alert: fct_orders test unique_order_id failed (47 duplicate rows). Walk me through incident response.

1. Triage severity: 47 duplicates in millions of rows = low impact but needs fixing

2. Check downstream: Did dbt build gate downstream models? (Yes, if error severity)

3. Investigate: SELECT order_id, COUNT(*) FROM fct_orders GROUP BY 1 HAVING COUNT(*) > 1

4. Check source: Are duplicates coming from stg_orders? (Source system issue?)

5. Fix: Add dedup logic in staging (ROW_NUMBER) or verify unique_key config

6. Remediate: --full-refresh on fct_orders after fix

7. Prevent: Add unique test to stg_orders.order_id to catch source duplicates earlier

Q2: A model that normally takes 5 minutes suddenly takes 90 minutes. No code changes. What do you investigate?

1. Warehouse load — saturated with concurrent queries?

2. Data volume spike — source table suddenly grew?

3. Incremental regression — is_incremental() returning False? (table dropped → full rebuild)

4. Warehouse sizing — someone changed the warehouse size?

5. Query plan — check warehouse query profile for spills to disk, bad join strategies

6. Clustering — check if micro-partition pruning is effective

Q3: Deploy a breaking change — rename order_total → order_amount_dollars. Three dashboards consume this table. Rollout plan?

1. Announce: 2-week notice to all consumers

2. Deprecation period: Add both columns: order_amount_dollars, order_amount_dollars AS order_total -- DEPRECATED

3. Track usage: Query INFORMATION_SCHEMA.ACCESS_HISTORY to see who still uses order_total

4. Remove: After consumers migrate, drop the deprecated column

5. Contracts: Add model contract enforcement to prevent future accidental renames

Q4: Reduce the dbt project's warehouse compute costs by 40%. What levers do you pull?

1. Convert tables to views — staging models don't need to be tables (biggest savings)

2. Convert large tables to incremental — stop rebuilding 100M+ row tables daily

3. Right-size warehouses — SMALL for staging, MEDIUM for marts, LARGE only for full refreshes

4. Auto-suspend warehouses — 60-second auto-suspend between runs

5. Eliminate unused models — query ACCESS_HISTORY for models nobody queries; disable them

6. Use Slim CI — stop rebuilding entire project on every PR

⚡ Rapid Fire — 20 Quick Q&A

Click any question to reveal the answer. Know these cold.

1. What does dbt build do that dbt run doesn't?
→ Runs models, tests, snapshots, AND seeds in DAG order with test gating. dbt run only runs models.
2. What's the difference between {{ ref() }} and {{ source() }}?
→ ref() references dbt models (within the project). source() references external raw tables (entry point to dbt).
3. When does is_incremental() return False?
→ First run (table doesn't exist), --full-refresh flag, or model not configured as incremental.
4. What is {{ this }}?
→ A reference to the current model's existing target table. Only meaningful inside {% if is_incremental() %}.
5. Name the 4 built-in dbt tests.
→ unique, not_null, accepted_values, relationships.
6. severity: warn vs severity: error?
→ warn logs the failure but allows downstream models to proceed. error halts downstream execution in dbt build.
7. Snapshot timestamp vs check strategies?
→ timestamp compares updated_at column (faster). check compares actual column values (no timestamp needed).
8. How do you identify the current record in a snapshot?
→ WHERE dbt_valid_to IS NULL.
9. What file stores the DAG/project graph?
→ target/manifest.json.
10. What does state:modified+ do?
→ Builds only models that changed since the last production run, plus all downstream dependencies.
11. What's the + operator in selectors?
→ model+ = model and all downstream. +model = model and all upstream. +model+ = both.
12. What does store_failures=true do?
→ Saves failing test rows to a table in the warehouse for investigation.
13. How do you mask PII in dev but keep it in prod?
→ {% if target.name == 'prod' %}customer_email{% else %}MD5(customer_email){% endif %}.
14. What does dbt deps do?
→ Installs packages defined in packages.yml into the dbt_packages/ directory.
15. Purpose of generate_schema_name macro?
→ Controls schema names. Override so dev uses dev_alice_marts while prod uses marts.
16. Name three must-have dbt packages.
→ dbt_utils, dbt_expectations, codegen.
17. What's a model contract?
→ Enforced schema definition (column names, types, constraints) that must match the model's output. Compile-time check.
18. What does on_schema_change='fail' do?
→ Raises an error when an incremental model's SELECT returns different columns than the existing target table.
19. How do you run a one-time backfill on an incremental model?
→ dbt run --select my_model --vars '{"backfill_start": "2024-01-01"}' with var-driven WHERE clause. Or --full-refresh for complete rebuild.
20. dbt compile vs dbt run?
→ compile resolves Jinja and writes compiled SQL to target/compiled/ without executing. run compiles AND executes against the warehouse.
