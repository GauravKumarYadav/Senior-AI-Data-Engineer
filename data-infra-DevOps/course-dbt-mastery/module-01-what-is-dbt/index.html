🔨 The T in ELT — What dbt Actually Does

In the modern ELT paradigm, raw data is first extracted and loaded into a cloud data warehouse (Snowflake, BigQuery, Redshift, Databricks), and then dbt handles the T — the transformation — using pure SQL executed inside the warehouse itself.

dbt (data build tool) does one thing exceptionally well: it lets analytics engineers write modular SQL SELECT statements that dbt compiles into DDL/DML and executes against your warehouse. You write a SELECT, dbt wraps it in a CREATE TABLE AS SELECT or CREATE VIEW AS, and runs it.

┌─────────────────────────────────────────────────────────┐ │ Modern ELT Pipeline │ ├─────────────┬─────────────────────┬─────────────────────┤ │ EXTRACT │ LOAD │ TRANSFORM │ │ Fivetran │ Fivetran/Airbyte │ dbt │ │ Airbyte │ Stitch │ SELECT statements │ │ Sources ──►│──► Raw Tables ─────►│──► Clean Models │ └─────────────┴─────────────────────┴─────────────────────┘

🧩 Modularity

Break complex transformations into small, reusable models

🔗 Dependency Management

Automatically builds a DAG of model execution order

✅ Testing

Write assertions about your data (uniqueness, not-null, referential integrity)

📖 Documentation

Auto-generate a searchable data catalog from your models

🔀 Version Control

All transformations live in Git — pull requests, code review, CI/CD

🌍 Environments

Dev vs. staging vs. production with the same codebase

💡 Interview Insight

"dbt is a transformation framework that lets analytics engineers write modular SQL models inside the warehouse, following the ELT pattern. It compiles SELECT statements into DDL/DML, builds a dependency DAG, runs tests, and generates documentation — essentially bringing software engineering practices like version control, testing, and modularity to the analytics layer."

dbt Core vs dbt Cloud

dbt Core is the open-source CLI tool. You install it locally, write models in a code editor, and run commands like dbt run, dbt test, and dbt build from the terminal. You're responsible for orchestration (Airflow, Dagster, Prefect), environment management, and CI/CD pipelines.

dbt Cloud is the managed SaaS platform built on top of dbt Core. It provides a web-based IDE, job scheduling, environment management, CI/CD with pull-request builds, a hosted documentation site, and features like the Semantic Layer and dbt Explorer.

🔧 dbt Core (OSS)

  • 💰 Free
  • 📝 Your own editor
  • ⏰ BYO scheduling (Airflow, etc.)
  • 🔄 BYO CI/CD (GH Actions)
  • 📖 Self-host documentation
  • 🔑 You manage versions

☁️ dbt Cloud (Managed)

  • 💰 Paid (free tier available)
  • 📝 Web IDE + VS Code extension
  • ⏰ Built-in scheduler
  • 🔄 Slim CI on pull requests
  • 📖 Hosted auto-docs
  • 🔑 Managed upgrades + RBAC/SSO

Important: The SQL you write and the project structure are identical between Core and Cloud. You can migrate between them without rewriting any models.

💡 Interview Insight

"The core SQL and project structure are identical — the difference is the operational layer. For teams with existing orchestration like Airflow, dbt Core gives full control. For teams wanting faster onboarding and built-in CI/CD, dbt Cloud reduces operational overhead. The transformation code itself is portable between them."

Anatomy of a dbt Project

Every dbt project follows a standard directory layout. Understanding this structure is essential — interviewers expect you to describe where things live and why they're separated.

my_ecommerce_project/ ├── dbt_project.yml # Project configuration ├── profiles.yml # Warehouse connection (~/.dbt/) ├── packages.yml # External packages (dbt-utils) │ ├── models/ # SQL models — the core │ ├── staging/ # 1:1 with source tables │ │ ├── stg_orders.sql │ │ └── _stg_sources.yml │ ├── intermediate/ # Business logic joins │ │ └── int_order_items_pivoted.sql │ └── marts/ # Business-ready tables │ ├── fct_orders.sql │ └── dim_customers.sql │ ├── tests/ # Custom singular tests ├── macros/ # Reusable Jinja functions ├── seeds/ # Static CSV files ├── snapshots/ # SCD Type 2 tracking ├── analyses/ # Ad-hoc SQL (compiled only) └── target/ # Compiled output (gitignored)

📁 models/

SQL SELECT statements. Each .sql file becomes a table or view. 90% of your time is here.

🧪 tests/

Custom data quality assertions. SQL queries that should return zero rows if passing.

🔧 macros/

Reusable Jinja2 template functions — DRY up repeated logic across models.

📸 snapshots/

Track source data changes over time using SCD Type 2 patterns.

💡 Interview Insight

"Models are organized into staging, intermediate, and marts layers. Staging models are 1:1 with sources and handle renaming and type casting. Intermediate models contain business logic. Marts are the final business-ready tables. Tests validate data quality, macros keep SQL DRY, seeds handle static reference data, and snapshots track historical changes."

dbt_project.yml — The Heartbeat

This file tells dbt your project's name, where to find models, how to materialize them by default, and how to apply configurations across entire directories.

# dbt_project.yml name: 'ecommerce_analytics' version: '1.0.0' config-version: 2 profile: 'ecommerce' model-paths: ["models"] test-paths: ["tests"] seed-paths: ["seeds"] macro-paths: ["macros"] models: ecommerce_analytics: staging: +materialized: view # All staging = views +schema: staging intermediate: +materialized: ephemeral # CTEs, not persisted marts: +materialized: table # Full tables +schema: marts vars: payment_methods: ['credit_card', 'coupon', 'bank_transfer']

Key Concepts

📂 Cascading Config

Settings at a directory level cascade to all models within. Individual models can override with their own config() block.

➕ The + Prefix

The + before a config key means "apply to all resources in this directory." Without it, dbt interprets it as a subdirectory name.

🔐 Profile Reference

The profile key links to profiles.yml with warehouse credentials. Your project config (in Git) never contains secrets.

📦 Variables

vars defines project-wide values accessible via {{ var('payment_methods') }}. Override at runtime with --vars.

💡 Interview Insight

"In dbt_project.yml, I use directory-level configuration with the + prefix. Staging models default to views, intermediate models are ephemeral (compiled as CTEs), and mart models are tables. Individual models can override with an inline config() block — for example, a high-traffic staging model might override to table for performance."

Profiles — Connecting to Your Warehouse

The profiles.yml file defines how dbt connects to your data warehouse. It lives in ~/.dbt/profiles.yml by default — outside your project directory — because it contains credentials that should never be committed to Git.

# ~/.dbt/profiles.yml ecommerce: target: dev outputs: dev: type: snowflake user: "{{ env_var('DBT_USER') }}" password: "{{ env_var('DBT_PASSWORD') }}" database: ECOMMERCE_DEV schema: dbt_jsmith # Developer-specific schema threads: 4 prod: type: snowflake user: "{{ env_var('DBT_PROD_USER') }}" database: ECOMMERCE_PROD schema: analytics threads: 8

🎯 Target

The active environment. dbt run uses the default target (dev). Switch with dbt run --target prod.

🧵 Threads

How many models dbt runs in parallel. Independent models execute concurrently for faster runs.

📁 Schema

In dev, each developer gets their own schema (dbt_jsmith) to avoid conflicts. In prod, models land in shared schemas.

🔐 env_var()

Use env_var() to inject secrets from the environment. Never hardcode credentials.

💡 Interview Insight

"Through profiles.yml targets. Each developer has a personal schema in a dev database. The same dbt code runs against prod by switching the target. Credentials are injected with env_var() — never hardcoded. This gives us identical transformation logic across environments with isolated execution contexts."

ref() — The Function That Builds the DAG

The ref() function is arguably the most important concept in dbt. Every time you reference another model, you use ref() instead of hardcoding a table name. This does two critical things:

(1) Resolves the correct table/view name for the current environment
(2) Registers a dependency in the DAG so dbt knows execution order

-- models/marts/fct_orders.sql SELECT o.order_id, o.customer_id, c.first_name, c.last_name, o.order_amount_cents / 100.0 AS order_amount_dollars FROM {{ ref('stg_orders') }} AS o LEFT JOIN {{ ref('stg_customers') }} AS c ON o.customer_id = c.customer_id
📄 Mart model with business logic
 
📊 Select business-ready columns
 
💰 Convert cents to dollars
🔗 ref() resolves to the correct schema
🔗 Also registers this dependency in DAG
🔄 dbt guarantees stg_ runs before fct_
dbt DAG Example raw_orders ──► stg_orders ──┐ ├──► fct_orders raw_customers──►stg_customers┘ │ └──────────► dim_customers

Why not hardcode table names? (1) Environment portability breaks, (2) dbt can't determine execution order, and (3) you lose lineage tracking in docs.

💡 Interview Insight

"ref() serves two purposes: it resolves model names to the correct database.schema.table for the current environment, and it registers a dependency in the DAG so dbt knows the execution order. Without ref(), dbt can't build the lineage graph, can't parallelize independent models, and your code isn't portable across environments."

source() — Referencing Raw Data

While ref() references other dbt models, source() references raw tables loaded by your ingestion tools (Fivetran, Airbyte, Stitch). Sources are defined in YAML and give you a clean abstraction over raw table names.

# models/staging/_stg_sources.yml version: 2 sources: - name: ecommerce schema: shopify freshness: warn_after: {count: 12, period: hour} error_after: {count: 24, period: hour} loaded_at_field: _fivetran_synced tables: - name: raw_orders columns: - name: id tests: [unique, not_null] - name: raw_customers
-- models/staging/stg_orders.sql SELECT id AS order_id, user_id AS customer_id, order_date, status FROM {{ source('ecommerce', 'raw_orders') }} -- Compiles to: RAW_DATABASE.shopify.raw_orders

Source freshness: Run dbt source freshness to check whether raw tables are being updated on schedule. Critical for production monitoring.

💡 Interview Insight

"source() references raw tables loaded by external tools — it provides an abstraction over physical table names and enables freshness monitoring. ref() references models within dbt itself. The pattern is: staging models use source() to read raw data, and all downstream models use ref() to read from staging."

Putting It All Together

Let's trace a complete data flow through an e-commerce dbt project:

FT
Fivetran just synced raw_orders and raw_customers into the warehouse.
stg
Staging models read source() — renaming columns, casting types. Created as views (instant).
DAG
dbt parsed all ref() calls and built the dependency graph. stg_ models run first!
fct
fct_orders joins stg_orders + stg_customers. Materialized as a table — precomputed for dashboards!
dim
dim_customers calculates lifetime value, order counts, customer tiers. Another table for Looker.
tst
dbt test runs assertions: unique order_ids ✅, not-null customer_ids ✅, valid statuses ✅. All green!

💡 Interview Insight

"First, define sources pointing to raw tables. Then staging models with 1:1 source mappings. Next, intermediate models for complex joins. Finally, mart-layer fact and dimension tables. Add tests at each layer and document columns in YAML. Then dbt build runs everything — models in DAG order, then tests."
