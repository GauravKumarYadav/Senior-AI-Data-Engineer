From dbt run to Production

Running dbt run on your laptop is development. Running dbt reliably, on schedule, with tests, alerts, documentation, and rollback capability — that's production. This module covers everything between "it works on my machine" and "it runs at 4am and wakes someone up if it breaks."

The Production dbt Stack

Git Repo ──▶ CI/CD (Actions) ──▶ Orchestrator (Airflow) ──▶ Warehouse (Snowflake) │ │ │ │ PR opened Slim CI: Scheduled: Executes SQL ↓ dbt build dbt build Pre-commit --select (daily 4am) hooks: state:modified+ ↓ SQLFluff ↓ Alert on failure YAML valid Artifacts: ↓ manifest.json dbt docs serve run_results.json

The Evolution of dbt Execution Commands

CommandWhat It DoesUse When
dbt runBuild models onlyDevelopment (legacy)
dbt testRun tests onlyAd-hoc validation
dbt snapshotRun snapshots onlyStandalone snapshot job
dbt seedLoad CSV seed filesReference data loading
dbt buildModels + tests + snapshots + seeds in DAG orderProduction standard
💡 Interview Gold

"What command do you use in production and why?" — "dbt build. It executes models, tests, snapshots, and seeds in DAG order with test gating. If stg_orders tests fail, fct_orders is skipped — bad data never reaches marts. This is superior to running dbt run + dbt test separately, which builds all models first (allowing bad data to propagate) and only then discovers test failures."

# Production run — the one command $ dbt build # Production run with selector $ dbt build --select tag:daily # CI run — only changed models and their downstream $ dbt build --select state:modified+ # Full refresh (weekly maintenance) $ dbt build --full-refresh

Slim CI: state:modified+ & Artifacts

In a project with 500+ models, you don't want every PR to rebuild everything. Slim CI uses dbt artifacts to identify which models changed and only builds those (plus their downstream dependents).

Slim CI Flow 1. Production run generates manifest.json (saved as artifact) 2. Developer opens PR with changes to stg_orders.sql 3. CI pipeline: a. dbt compile (generates new manifest.json) b. dbt build --select state:modified+ --state ./prod-artifacts/ c. Compares new manifest vs prod manifest d. Finds: stg_orders was modified e. Builds: stg_orders + all downstream (fct_orders, etc.) f. Runs tests on those models Result: Only 5 models built instead of 500+ CI time: 2 minutes instead of 45 minutes

State Comparison Selectors

SelectorWhat It Matches
state:modifiedModels that changed (SQL or YAML)
state:modified+Changed models + downstream
state:newModels that are new (not in state)
state:modified.bodyOnly SQL body changes
state:modified.configsOnly config/YAML changes
result:errorModels that errored last run
result:failTests that failed last run
🎯 Interview Answer

"Slim CI using dbt build --select state:modified+. On every PR, the CI pipeline downloads the production manifest.json, compares it against the current branch, and only builds modified models plus their downstream dependencies. This turns a 45-minute full build into a 2-minute targeted build. The CI target uses an isolated schema (e.g., ci_pr_123) so PR builds don't affect production data."

Artifacts: manifest.json & run_results.json

dbt generates metadata artifacts in the target/ directory after every run. These files are the foundation for Slim CI, documentation, and monitoring.

# target/ directory after a dbt run target/ ├── manifest.json # Complete project graph (models, tests, sources, macros) ├── run_results.json # Results of the latest run (timing, status, rows) ├── catalog.json # Column-level metadata (after dbt docs generate) ├── sources.json # Source freshness results └── compiled/ # Compiled SQL for every model

manifest.json — The Project Graph

// manifest.json — each model's identity, dependencies, and checksum { "nodes": { "model.ecommerce.fct_orders": { "unique_id": "model.ecommerce.fct_orders", "depends_on": { "nodes": ["model.ecommerce.stg_orders", "model.ecommerce.stg_customers"] }, "config": { "materialized": "incremental", "unique_key": "order_id" }, "checksum": { "name": "sha256", "checksum": "abc123..." } } } }

run_results.json — Execution Metrics

// run_results.json — timing, status, row counts per model { "results": [ { "unique_id": "model.ecommerce.fct_orders", "status": "success", "execution_time": 12.45, "rows_affected": 245000 }, { "unique_id": "test.ecommerce.unique_fct_orders_order_id", "status": "pass", "execution_time": 0.8, "failures": 0 } ], "elapsed_time": 142.3 }

Artifact Usage Matrix

ArtifactUsed For
manifest.jsonSlim CI (state comparison), dbt docs (lineage graph), external tools
run_results.jsonPost-run monitoring & alerting, performance trending, retry failed models
catalog.jsonColumn-level documentation, schema change detection
sources.jsonSource freshness monitoring, ingestion SLA tracking
💡 Interview Gold

"manifest.json contains the full project graph — every model, test, source, and their dependencies. I use it for Slim CI and documentation. run_results.json captures execution results — timing, status, row counts. I parse it post-run to detect failures, alert on slow models, and track performance trends. In production, I upload both artifacts to S3 after every successful run."

dbt with Airflow — Orchestration Patterns

dbt handles transformation, but it doesn't handle scheduling, dependencies on external systems, or alerting. That's where an orchestrator like Airflow comes in.

Pattern 1: BashOperator (simple but brittle)

# dags/dbt_daily.py — Basic approach from airflow import DAG from airflow.operators.bash import BashOperator with DAG(dag_id="dbt_daily_build", schedule_interval="0 6 * * *") as dag: dbt_build = BashOperator( task_id="dbt_build", bash_command="cd /opt/dbt/ecommerce && dbt build --target prod", ) # Problem: one opaque task — all-or-nothing retry

Pattern 2: Cosmos (recommended)

# dags/dbt_cosmos_daily.py — Production-grade approach from cosmos import DbtTaskGroup, ProjectConfig, ProfileConfig with DAG(dag_id="dbt_cosmos_daily", schedule_interval="0 6 * * *") as dag: # Each dbt model becomes an Airflow task! dbt_tasks = DbtTaskGroup( group_id="dbt_build", project_config=ProjectConfig(dbt_project_path="/opt/dbt/ecommerce"), profile_config=profile_config, ) # Per-model visibility, retry, and parallelism in Airflow UI

BashOperator vs Cosmos

📦 BashOperator

  • ⚠️ One task = entire dbt build
  • ⚠️ Retries rebuild everything
  • ⚠️ One green/red box in UI
  • ⚠️ All-or-nothing on failure
  • ✅ Very simple setup

🚀 Cosmos

  • ✅ One task = one model
  • ✅ Retries just the failed model
  • ✅ Full DAG visibility
  • ✅ Partial success supported
  • ⚠️ Medium setup complexity
# Production DAG flow [wait_for_fivetran_sync] │ ▼ [dbt_source_freshness] │ ▼ [dbt_build_staging] ──→ [dbt_build_marts] │ │ ▼ ▼ [dbt_snapshot] [dbt_build_reports] │ ▼ [upload_artifacts] → [notify_slack]
🎯 Interview Answer

"I use Airflow with Cosmos (astronomer-cosmos). Cosmos maps each dbt model to an individual Airflow task, giving per-model visibility, retry, and parallelism. The DAG flow is: wait for ingestion → check source freshness → dbt build staging → marts → reports → upload artifacts → notify. I store dbt artifacts after each successful run for Slim CI."

SQLFluff — SQL Linting for dbt

SQLFluff is the standard linter for dbt SQL files. It enforces consistent formatting, catches style violations, and integrates with CI pipelines and pre-commit hooks.

# .sqlfluff (project root) [sqlfluff] templater = dbt dialect = snowflake max_line_length = 120 [sqlfluff:indentation] tab_space_size = 4 indented_ctes = true [sqlfluff:rules:capitalisation.keywords] capitalisation_policy = lower [sqlfluff:rules:aliasing.table] aliasing = explicit [sqlfluff:rules:aliasing.length] min_alias_length = 2

Common SQLFluff Rules

RuleWhat It Catches
LT01Trailing whitespace
LT02Indentation not a multiple of configured spaces
AM01Non-ANSI-92 joins (implicit joins)
ST06SELECT * in production models
CP01–04Capitalization (keywords, identifiers, functions)
AL01–03Aliasing rules (explicit AS, min length)

Before & After SQLFluff Fix

❌ Before

SELECT o.Order_ID, c.first_name, SUM(o.amount) total FROM {{ ref('stg_orders') }} o join {{ ref('stg_customers') }} c ON o.customer_id = c.customer_id GROUP BY 1,2

✅ After

select o.order_id, c.first_name, sum(o.amount) as total from {{ ref('stg_orders') }} as o join {{ ref('stg_customers') }} as c on o.customer_id = c.customer_id group by 1, 2
💡 Interview Gold

"SQLFluff with the dbt templater. I configure .sqlfluff at the project root — consistent keyword casing, explicit aliases with AS, 4-space indentation. It runs in CI as a PR check (fail on violations) and locally via pre-commit hooks. sqlfluff fix auto-corrects most violations. Consistent SQL style makes code reviews faster."

Pre-commit Hooks for dbt Projects

Pre-commit hooks catch issues before code reaches the repo — linting, YAML validation, and dbt compilation happen automatically on every git commit.

# .pre-commit-config.yaml repos: - repo: https://github.com/pre-commit/pre-commit-hooks hooks: - id: trailing-whitespace - id: check-yaml - id: no-commit-to-branch args: ['--branch', 'main'] - repo: https://github.com/sqlfluff/sqlfluff hooks: - id: sqlfluff-lint args: ['--dialect', 'snowflake'] files: ^models/ - repo: https://github.com/dbt-checkpoint/dbt-checkpoint hooks: - id: check-model-has-tests args: ['--test-cnt', '2', '--'] - id: check-model-has-description - id: check-model-name-contract args: ['--pattern', '(stg|int|fct|dim|rpt)_.*', '--']

What dbt-checkpoint Validates

HookWhat It Enforces
check-model-has-testsEvery model has ≥N tests
check-model-has-descriptionEvery model has a description
check-source-has-freshnessSources have freshness config
check-model-name-contractNames match pattern (stg_, fct_…)
check-model-parents-schemaMarts don't ref sources directly
check-script-has-no-table-nameNo hardcoded table references
# Install hooks (one-time per clone) $ pre-commit install # Now every git commit runs these checks automatically # Run manually against all files $ pre-commit run --all-files # Skip hooks for emergency commits (use sparingly!) $ git commit --no-verify -m "hotfix: emergency production fix"
🎯 Interview Answer

"I use pre-commit with several hooks: SQLFluff for SQL linting, dbt-checkpoint for structural validation (every model has tests, descriptions, proper naming), YAML schema validation, and standard checks (trailing whitespace, no direct commits to main). This catches 90% of PR review issues before code even reaches GitHub."

Documentation: dbt docs

dbt auto-generates a documentation website from your project — model descriptions, column definitions, lineage graphs, and test coverage. It's one of dbt's most underrated features.

# Generate documentation $ dbt docs generate # Serve locally $ dbt docs serve --port 8081 # Avoid port 8080 (Teams uses it)

📊 Lineage Graph

Visual DAG showing all models and dependencies. Click any model to see upstream/downstream.

📋 Model Details

Description, SQL, compiled code, columns, types, tests, and config per model.

🔗 Source Details

Database, schema, table names, freshness configuration, and column descriptions.

📈 Test Coverage

Which models have tests, which columns are tested, and test types/severity.

Exposures — End-to-End Lineage

# models/exposures.yml exposures: - name: weekly_revenue_dashboard type: dashboard url: https://lookerstudio.google.com/your-dashboard description: "Executive revenue dashboard. Refreshed daily at 7am." depends_on: - ref('fct_orders') - ref('dim_customers') owner: name: Analytics Team - name: customer_churn_ml_model type: ml description: "ML model predicting customer churn risk" depends_on: - ref('fct_customer_activity') owner: name: Data Science Team

Exposures show up in the lineage graph, creating end-to-end visibility from source → staging → marts → dashboards/ML models.

💡 Interview Gold

"Every model and column gets descriptions in YAML — including grain, update frequency, business rules, and caveats. I use {% docs %} blocks for rich markdown descriptions. I define exposures for dashboards and ML models — this creates end-to-end lineage from source to consumer. dbt docs generate produces a browsable website with lineage graphs."

Putting It All Together: The Production Workflow

🔨 Development

Feature branch → write models/tests → dbt build --select state:modified+ locally → commit → pre-commit hooks run

🔍 CI (Pull Request)

GitHub Actions Slim CI → dbt build --select state:modified+ → SQLFluff lint → PR comment with results

🚀 Merge → Deploy

PR merged to main → production dbt build → upload artifacts to S3 → deploy docs

⏰ Scheduled Production

Airflow DAG at 6am: wait for ingestion → source freshness → dbt build → upload artifacts → Slack alert on failure

Essential dbt_project.yml Configs

# dbt_project.yml name: ecommerce version: '1.0.0' models: ecommerce: staging: +materialized: view +schema: staging +tags: ['staging'] marts: +materialized: table +schema: marts +tags: ['marts'] reports: +materialized: table +schema: reports +tags: ['reports'] on-run-end: - "{{ grant_select_on_schemas(['staging','marts','reports'], 'analyst_role') }}"
🎯 Interview Answer

"Development: feature branch → local dbt build → pre-commit hooks. CI: PR triggers Slim CI with state:modified+ against the production manifest. On merge: production deploy runs full dbt build. Scheduled: Airflow DAG at 6am — waits for ingestion, checks source freshness, runs dbt build, uploads artifacts, alerts Slack on failures. Weekly: full refresh."
