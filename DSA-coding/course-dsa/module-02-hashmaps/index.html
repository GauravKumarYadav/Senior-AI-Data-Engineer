🗺️ Hash Map Internals

Hash maps are the single most frequently used data structure in coding interviews. If you master nothing else, master this. Let's open the hood and see how they actually work.

A hash map stores key-value pairs in an underlying array. To decide where a key goes: compute hash(key) → reduce to an index: index = hash(key) % array_size → store the pair there. Lookup follows the same path. This is O(1) average.

Collision Handling

⛓️ Chaining (Separate Chaining)

  • 📦 Each slot holds a linked list
  • ➕ Colliding keys appended to the list
  • ✅ Average: O(1) per operation
  • ⚠️ Worst: O(n) if all keys hash to same slot
  • ❌ Pointer chasing hurts cache locality

🏠 Open Addressing (Linear Probing)

  • 🔍 If slot occupied, probe the next slot
  • 📦 All entries live directly in the array
  • ✅ Better cache locality (no pointers)
  • ⚠️ Degrades badly at high load factors
  • ⚠️ Clustering can create hot spots

Python's dict — Under the Hood

🏗️ Open Addressing

Python uses open addressing with perturbation (not pure linear probing) to spread clusters.

📐 Load Factor ~0.66

Resizes when 2/3 full. Since Python 3.7, dict preserves insertion order (guaranteed by spec).

💾 Compact Layout

Since 3.6: dense array of (hash, key, value) + sparse index table. ~25% less memory, faster iteration.

🔐 Hash Randomization

PYTHONHASHSEED since 3.3 prevents adversarial O(n) attacks. Always say "O(1) average, O(n) worst case."

💡 Sets vs Dicts

A set is essentially a dict without values — same hash table, same O(1) lookup. Use a set when you only need membership testing ("have I seen this before?"). Clearer intent, slightly less memory.

📊 Memory Comparison

dict with string keys: ~100 bytes/entry. Fixed-size list of 26 ints: ~280 bytes total. For character frequency, [0]*26 is a 10× memory reduction — not premature optimization!

📊 Frequency Counting & Two Sum

Frequency Counting — Bread and Butter

from collections import Counter freq = Counter("hello") # Counter({'l': 2, 'h': 1, 'e': 1, 'o': 1}) freq.most_common(2) # [('l', 2), ('h', 1)] # For small key spaces, use a fixed-size array (faster!): freq = [0] * 26 for c in s: freq[ord(c) - ord('a')] += 1

Top-K Pattern: Heap + HashMap

For "top K frequent elements": count frequencies with hashmap → O(n), then use a min-heap of size K → O(n log k). Better than sorting when k ≪ n. Or use bucket sort for O(n).

The Two Sum Pattern

Arguably the most important single pattern in interviews. It embodies the core idea of hashmaps: trade space for time by pre-storing complements.

# The core idea — O(n²) → O(n) for each number: complement = target - number if complement in seen: return answer seen[number] = index
💡 Why One Pass Works

"Don't you need two passes?" No! When you reach the second number of a valid pair, the first is already in the hashmap. You can't double-count because you check before inserting. This subtlety is worth mentioning in interviews.

Two Sum Variations

🔢 Unsorted → HashMap

Store complement → index. Classic O(n).

📊 Sorted → Two Pointers

No extra space needed. O(n) time, O(1) space.

🔱 3Sum → Sort + Two Ptr

Fix one element, two-pointer on rest. O(n²).

🔄 Count Pairs → Freq Map

HashMap of complement → frequency count.

📦 Grouping & Window + HashMap

Group Anagrams — The Canonical Key Trick

To group anagrams, create a canonical key that's the same for all anagrams. Sort each string → all anagrams produce the same sorted key.

from collections import defaultdict groups = defaultdict(list) for s in strs: key = "".join(sorted(s)) # "eat" → "aet" groups[key].append(s) return list(groups.values())
💡 General Grouping Pattern

This pattern applies any time you bucket items by a shared property: group files by extension, employees by department, logs by error code. Always: defaultdict(list) + compute key + append.

Sliding Window + HashMap

Many sliding window problems use a hashmap to track window state. The combo is powerful: HashMap tells you what's in the window, sliding window efficiently adds/removes at the edges.

Minimum Window Template

def min_window_template(s, target_freq): window_freq = {} have, need = 0, len(target_freq) left = 0 result = "" for right, char in enumerate(s): # Expand: add char to window window_freq[char] = window_freq.get(char, 0) + 1 if char in target_freq \ and window_freq[char] == target_freq[char]: have += 1 # Shrink: minimize while valid while have == need: # Update result (shortest → here!) window_size = right - left + 1 if not result or window_size < len(result): result = s[left:right + 1] # Remove leftmost left_char = s[left] window_freq[left_char] -= 1 if left_char in target_freq \ and window_freq[left_char] < target_freq[left_char]: have -= 1 left += 1 return result
🎯 The have/need Optimization

Comparing two frequency maps every step would be O(26) or O(|charset|). By tracking have (count of satisfied characters), we reduce each check to O(1). This trick appears in many sliding-window-plus-hashmap problems.

🏗️ LRU Cache Design

The LRU Cache is a design problem — it tests data structure composition, O(1) constraints, and clean API thinking. Appears at Google, Meta, Amazon, and Microsoft with high frequency.

Why One Data Structure Isn't Enough

❌ HashMap Alone

  • ✅ O(1) lookup by key
  • ❌ No access order tracking
  • ❌ Can't find "least recently used"
  • ❌ No concept of oldest/newest

❌ Linked List Alone

  • ✅ Tracks insertion/access order
  • ✅ O(1) add/remove at ends
  • ❌ Finding a node by key is O(n)
  • ❌ Must traverse entire list

The Architecture: HashMap + Doubly-Linked List

🗺️ HashMap

key → Node pointer. O(1) lookup to find any node instantly.

🔗 Doubly-Linked List

head ↔ node ↔ … ↔ tail. Head = most recent, tail = LRU. O(1) reorder.

📥 get(key)

Look up node via hashmap (O(1)), move it to head of list (O(1)).

📤 put(key, val)

If exists: update + move to head. If new + at capacity: evict tail, then insert at head.

💡 Sentinel Nodes — Eliminate Edge Cases

Dummy head and tail nodes mean every real node always has a valid prev and next. No special cases for empty list, single node, or boundary removal. Learn this pattern once, use it everywhere.

Full Implementation

class Node: __slots__ = ("key", "value", "prev", "next") def __init__(self, key=0, value=0): self.key, self.value = key, value self.prev = self.next = None class LRUCache: def __init__(self, capacity): self.capacity = capacity self.cache = {} # key → Node self.head = Node() # sentinel (most recent) self.tail = Node() # sentinel (least recent) self.head.next = self.tail self.tail.prev = self.head def _remove(self, node): node.prev.next = node.next node.next.prev = node.prev def _add_to_front(self, node): node.prev = self.head node.next = self.head.next self.head.next.prev = node self.head.next = node def get(self, key): if key not in self.cache: return -1 node = self.cache[key] self._remove(node) self._add_to_front(node) return node.value def put(self, key, value): if key in self.cache: self._remove(self.cache[key]) self.cache[key].value = value self._add_to_front(self.cache[key]) else: if len(self.cache) >= self.capacity: lru = self.tail.prev self._remove(lru) del self.cache[lru.key] new_node = Node(key, value) self._add_to_front(new_node) self.cache[key] = new_node

🧩 Solutions: Problems 1–3

Problem 1: Two Sum (Unsorted — HashMap)

Given an unsorted array and target, return indices of two numbers summing to target. Each input has exactly one solution.

def two_sum(nums, target): seen = {} # value → index for i, num in enumerate(nums): complement = target - num if complement in seen: return [seen[complement], i] seen[num] = i return []
📌 HashMap: value → its index
 
🔄 Walk through each number
🔍 What number would complete the pair?
✅ Seen before! Return both indices
 
📝 Store this number for future lookups
 
💨 O(n) time, O(n) space

Problem 2: Group Anagrams

Group strings that are anagrams of each other. Use sorted string as canonical key.

from collections import defaultdict def group_anagrams(strs): groups = defaultdict(list) for s in strs: key = "".join(sorted(s)) groups[key].append(s) return list(groups.values()) # Alt: use tuple(freq) as key for O(n·k)
 
 
📌 Bucket: sorted_key → list of originals
 
🔄 For each string:
🔑 Sort it → "eat" becomes "aet"
📦 All anagrams land in same bucket
 
📤 Return all groups
 
⚡ O(n·k log k) — fast enough for interviews

Problem 3: Top K Frequent Elements

Find the k most frequent elements. Two approaches: heap O(n log k) or bucket sort O(n).

from collections import Counter def top_k_frequent(nums, k): freq = Counter(nums) # Bucket sort: index = frequency buckets = [[] for _ in range( len(nums) + 1)] for num, count in freq.items(): buckets[count].append(num) result = [] for i in range(len(buckets)-1, 0, -1): for num in buckets[i]: result.append(num) if len(result) == k: return result return result
 
 
📊 Count how often each number appears
 
🪣 Create buckets indexed by frequency
(max freq = n, so n+1 buckets)
 
📦 Place each number in its freq bucket
 
⬇️ Collect from highest frequency down
 
 
✅ Stop once we have k elements
⚡ O(n) time — beats heap for large n!

🧩 Solutions: Problems 4–6

Problem 4: Longest Substring Without Repeating Characters

Variable sliding window + hashmap tracking character positions. The bridge between arrays and hashmaps.

def length_of_longest_substring(s): last_seen = {} left = 0 max_len = 0 for right, char in enumerate(s): if char in last_seen and last_seen[char] >= left: left = last_seen[char] + 1 # jump past duplicate last_seen[char] = right max_len = max(max_len, right - left + 1) return max_len # Critical: last_seen[char] >= left prevents # moving left BACKWARD to a stale position! # Example: "abba" — without the check, left # would jump back when hitting the second 'a'.

Problem 5: Minimum Window Substring

Find the smallest substring of s containing all characters of t. Hard — one of the most asked hard problems.

def min_window(s, t): target_freq = Counter(t) window_freq = {} have, need = 0, len(target_freq) result = "" result_len = float("inf") left = 0 for right, char in enumerate(s): window_freq[char] = window_freq.get(char, 0) + 1 if char in target_freq \ and window_freq[char] == target_freq[char]: have += 1 while have == need: window_size = right - left + 1 if window_size < result_len: result_len = window_size result = s[left:right + 1] left_char = s[left] window_freq[left_char] -= 1 if left_char in target_freq \ and window_freq[left_char] < target_freq[left_char]: have -= 1 left += 1 return result # "ADOBECODEBANC", "ABC" → "BANC"

Problem 6: LRU Cache — OrderedDict Shortcut

Know the full implementation (Screen 4), but also know the Pythonic shortcut:

from collections import OrderedDict class LRUCacheSimple(OrderedDict): def __init__(self, capacity): super().__init__() self.capacity = capacity def get(self, key): if key not in self: return -1 self.move_to_end(key) # mark as recently used return self[key] def put(self, key, value): if key in self: self.move_to_end(key) self[key] = value if len(self) > self.capacity: self.popitem(last=False) # evict oldest
🎯 Interview Strategy

Always implement the full doubly-linked-list version first — it shows you understand the mechanics. Then mention OrderedDict to show Python stdlib knowledge. Many interviewers require the manual implementation.

🗺️ When to Reach for a HashMap

✅ Use a HashMap When…

  • 🔍 You need O(1) lookup by key
  • 📊 Counting frequencies
  • 🔗 Finding complements (Two Sum)
  • 📦 Grouping items by property
  • 🔄 Detecting duplicates
  • 💾 Caching/memoizing results
  • 🪟 Tracking sliding window state

❌ Don't Use When…

  • 📊 Input is sorted → two pointers
  • 📈 Need ordering → sorted array/BST
  • 📐 Range queries → segment tree
  • 🔤 Keys are 0–26 → fixed array (faster)
  • 💾 Memory constrained → bit manipulation
