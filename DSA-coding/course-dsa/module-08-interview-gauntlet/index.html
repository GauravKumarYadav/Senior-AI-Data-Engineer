🧮 Algorithms: Part 1

Problem 1: Product of Array Except Self

Return an array where result[i] = product of all elements except nums[i]. O(n) time, no division.

def product_except_self(nums: list[int]) -> list[int]: n = len(nums) result = [1] * n # Left pass: result[i] = product of nums[0..i-1] left = 1 for i in range(n): result[i] = left left *= nums[i] # Right pass: multiply by product of nums[i+1..n-1] right = 1 for i in range(n - 1, -1, -1): result[i] *= right right *= nums[i] return result
MetricValue
TimeO(n) — two linear passes
SpaceO(1) extra — output doesn't count

Problem 2: Merge Intervals

Sort by start time, then greedily merge overlapping intervals.

def merge(intervals: list[list[int]]) -> list[list[int]]: intervals.sort(key=lambda x: x[0]) merged: list[list[int]] = [] for start, end in intervals: if merged and start <= merged[-1][1]: merged[-1][1] = max(merged[-1][1], end) else: merged.append([start, end]) return merged # Time: O(n log n) | Space: O(n)

Problem 3: Binary Tree Right Side View

BFS level-order — take the last node at each level.

from collections import deque def right_side_view(root) -> list[int]: if not root: return [] result: list[int] = [] queue: deque = deque([root]) while queue: level_size = len(queue) for i in range(level_size): node = queue.popleft() if i == level_size - 1: # last node in level result.append(node.val) if node.left: queue.append(node.left) if node.right: queue.append(node.right) return result # Time: O(n) | Space: O(w) — tree width
💡 BFS = Swiss Army Knife for Trees

Any question mentioning "level," "depth," or "layer" is almost certainly BFS. Use deque for O(1) popleft — never list.pop(0) which is O(n).

🧮 Algorithms: Part 2

Problem 4: Word Break

1-D DP: dp[i] is True if s[0:i] can be segmented into dictionary words.

def word_break(s: str, word_dict: list[str]) -> bool: word_set = set(word_dict) # O(1) lookup n = len(s) dp = [False] * (n + 1) dp[0] = True # empty string is valid for i in range(1, n + 1): for j in range(i): if dp[j] and s[j:i] in word_set: dp[i] = True break # key optimization! return dp[n] # Time: O(n²) avg with break | Space: O(n)

Problem 5: Clone Graph

BFS with a hashmap that maps original nodes → clones. The hashmap is both "visited" set and clone registry.

def clone_graph(node) -> \"Node | None\": if not node: return None clones = {node: Node(node.val)} queue = deque([node]) while queue: current = queue.popleft() for neighbor in current.neighbors: if neighbor not in clones: clones[neighbor] = Node(neighbor.val) queue.append(neighbor) clones[current].neighbors.append(clones[neighbor]) return clones[node] # Time: O(V + E) | Space: O(V)

Problem 6: Decode Ways

Linear DP — Climbing Stairs in disguise, but the "zero" handling is the trap.

def num_decodings(s: str) -> int: if not s or s[0] == "0": return 0 prev2, prev1 = 1, 1 # dp[i-2], dp[i-1] for i in range(1, len(s)): current = 0 if s[i] != "0": # single digit valid current += prev1 two_digit = int(s[i-1:i+1]) if 10 <= two_digit <= 26: # two digits valid current += prev2 prev2, prev1 = prev1, current return prev1 # Time: O(n) | Space: O(1)
🚨 The Zero Trap

Half of candidates fail on inputs like "10", "20", or "30". Walk through these edge cases explicitly before submitting. "30" → 0 ways (invalid). "10" → 1 way.

🗃️ SQL Challenges: All 6 Problems

Problem 1: Consecutive Numbers

Find numbers appearing at least 3 times consecutively. Two approaches:

Self-Join (if IDs are contiguous)

SELECT DISTINCT l1.num FROM logs l1 JOIN logs l2 ON l1.id = l2.id - 1 JOIN logs l3 ON l2.id = l3.id - 1 WHERE l1.num = l2.num AND l2.num = l3.num;

Gap & Island (robust)

WITH grouped AS ( SELECT num, id - ROW_NUMBER() OVER ( PARTITION BY num ORDER BY id) AS grp FROM logs ) SELECT DISTINCT num FROM grouped GROUP BY num, grp HAVING COUNT(*) >= 3;

Problem 2: Department Top 3 Salaries

WITH ranked AS ( SELECT d.name AS department, e.name AS employee, e.salary, DENSE_RANK() OVER ( PARTITION BY e.department_id ORDER BY e.salary DESC ) AS dr FROM employees e JOIN departments d ON e.department_id = d.id ) SELECT department, employee, salary FROM ranked WHERE dr <= 3;

Problem 3: Trips & Users Cancellation Rate

SELECT t.request_at AS day, ROUND( SUM(CASE WHEN t.status LIKE 'cancelled%' THEN 1.0 ELSE 0 END) / COUNT(*), 2 ) AS cancellation_rate FROM trips t JOIN users c ON t.client_id = c.users_id AND c.banned = 'No' JOIN users d ON t.driver_id = d.users_id AND d.banned = 'No' WHERE t.request_at BETWEEN '2025-10-01' AND '2025-10-03' GROUP BY t.request_at ORDER BY t.request_at;

Problem 4: Exchange Seats

SELECT CASE WHEN id % 2 = 1 AND id = (SELECT MAX(id) FROM seat) THEN id WHEN id % 2 = 1 THEN id + 1 ELSE id - 1 END AS id, student FROM seat ORDER BY id;

Problem 5: Human Traffic of Stadium

WITH filtered AS ( SELECT *, id - ROW_NUMBER() OVER (ORDER BY id) AS grp FROM stadium WHERE people >= 100 ), streaks AS ( SELECT grp FROM filtered GROUP BY grp HAVING COUNT(*) >= 3 ) SELECT f.id, f.visit_date, f.people FROM filtered f JOIN streaks s ON f.grp = s.grp ORDER BY f.id;

Problem 6: Median Employee Salary

WITH counted AS ( SELECT *, ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary) AS rn, COUNT(*) OVER (PARTITION BY department) AS cnt FROM employees ) SELECT department, salary AS median_salary FROM counted WHERE rn IN (FLOOR((cnt + 1) / 2.0), CEIL((cnt + 1) / 2.0));
💡 Double Join = Key Insight

In the Trips problem, joining users twice (once for client, once for driver) is the key trick. Both sides must be unbanned. Using LIKE 'cancelled%' handles both cancelled_by_driver and cancelled_by_client.

⚙️ System Design: Code It

Problem 1: LRU Cache

Doubly linked list + hashmap. O(1) get and put.

class Node: __slots__ = ("key", "val", "prev", "next") def __init__(self, key=0, val=0): self.key, self.val = key, val self.prev = self.next = None class LRUCache: def __init__(self, capacity: int): self.capacity = capacity self.cache: dict[int, Node] = {} self.head, self.tail = Node(), Node() # sentinels! self.head.next = self.tail self.tail.prev = self.head def _remove(self, node): node.prev.next = node.next node.next.prev = node.prev def _add_to_front(self, node): node.next = self.head.next node.prev = self.head self.head.next.prev = node self.head.next = node def get(self, key: int) -> int: if key not in self.cache: return -1 node = self.cache[key] self._remove(node) self._add_to_front(node) return node.val def put(self, key: int, value: int): if key in self.cache: self._remove(self.cache[key]) node = Node(key, value) self._add_to_front(node) self.cache[key] = node if len(self.cache) > self.capacity: lru = self.tail.prev self._remove(lru) del self.cache[lru.key]
💡 Sentinel Nodes = Secret Weapon

The head and tail sentinel nodes eliminate four if-statements. Without them, every _remove and _add needs null checks for empty lists and boundary conditions. Sentinels make the code dramatically cleaner.

Problem 2: Rate Limiter (Sliding Window)

import time from collections import defaultdict class RateLimiter: def __init__(self, max_requests: int, window_seconds: int): self.max_requests = max_requests self.window = window_seconds self.requests: dict[str, list] = defaultdict(list) def allow(self, client_id: str) -> bool: now = time.time() ts = self.requests[client_id] while ts and ts[0] <= now - self.window: ts.pop(0) # evict expired if len(ts) < self.max_requests: ts.append(now) return True return False # Production: use Redis sorted set (ZADD + ZRANGEBYSCORE)

Problem 3: Event Processor with Dedup

import heapq class EventProcessor: def __init__(self, max_lateness: int = 300): self.seen_ids: set[str] = set() self.buffer: list = [] # min-heap by timestamp self.max_lateness = max_lateness self.watermark: float = 0.0 def ingest(self, event_id, timestamp, payload): if event_id in self.seen_ids: return # dedup if timestamp < self.watermark - self.max_lateness: return self.seen_ids.add(event_id) heapq.heappush(self.buffer, (timestamp, event_id, payload)) def flush(self, current_time): self.watermark = current_time - self.max_lateness results = [] while self.buffer and self.buffer[0][0] <= self.watermark: ts, eid, payload = heapq.heappop(self.buffer) results.append({"event_id": eid, "ts": ts, **payload}) return results
💡 Watermark = Flink's Core Concept

The watermark says "I believe all events before this time have arrived." This mirrors how Apache Flink handles late data. Explaining this concept shows system design maturity beyond just writing code.

Problem 4: Min Stack — O(1) Everything

class MinStack: def __init__(self): self.stack: list[int] = [] self.min_stack: list[int] = [] # parallel min tracker def push(self, val: int): self.stack.append(val) min_val = min(val, self.min_stack[-1]) if self.min_stack else val self.min_stack.append(min_val) def pop(self): self.stack.pop(); self.min_stack.pop() def top(self): return self.stack[-1] def get_min(self): return self.min_stack[-1]

Problem 5: Hit Counter

Simple (deque)

class HitCounter: def __init__(self): self.hits = deque() def hit(self, ts): self.hits.append(ts) def get_hits(self, ts): while (self.hits and self.hits[0] <= ts - 300): self.hits.popleft() return len(self.hits)

Scalable (circular buffer)

class HitCounterScalable: def __init__(self): self.times = [0]*300 self.counts = [0]*300 def hit(self, ts): idx = ts % 300 if self.times[idx] != ts: self.times[idx] = ts self.counts[idx] = 1 else: self.counts[idx] += 1 def get_hits(self, ts): return sum( c for i, c in enumerate(self.counts) if ts - self.times[i] < 300)
💡 Simple → Scalable Progression

Start with the deque (simple, correct), then optimize to the circular buffer when asked about scale. This show-then-optimize progression is exactly what interviewers want to see.

🧮 Algorithm Questions

Q1: Time complexity of binary search? ▼
O(log n). Each comparison halves the search space.
Q2: When use a heap over a sorted array? ▼
When you need efficient insert + extract-min/max. Heap: O(log n) both. Sorted array: O(n) insert, O(1) extract.
Q3: O(1) insert, delete, AND random access? ▼
Array + hashmap (index lookup). Swap element with last, then pop. This is the RandomizedSet pattern.
Q4: BFS vs DFS for shortest path? ▼
BFS finds shortest path in unweighted graphs (level by level). DFS finds a path but doesn't guarantee shortest.
Q5: Why is quicksort O(n²) worst-case? ▼
Pivot is always smallest/largest element (sorted input + bad pivot). Random pivot makes this astronomically unlikely.

🐍 Python Questions

Q6: What is the GIL and why does it matter? ▼
Global Interpreter Lock prevents parallel thread execution for Python bytecode. CPU-bound → use multiprocessing, not threads.
Q7: Generator vs list comprehension? ▼
Generator: lazy, O(1) memory, for large/infinite sequences. List comprehension: when you need random access or multiple passes.
Q8: list vs tuple? ▼
Lists are mutable, tuples are immutable. Tuples are hashable (dict keys), slightly faster, signal "don't change this."
Q9: deepcopy() vs copy()? ▼
copy() creates new outer object but shares nested references (shallow). deepcopy() recursively copies everything — fully independent.
Q10: How does dict achieve O(1) lookup? ▼
Hash table with open addressing. Keys hashed to buckets; collisions resolved by probing. Resizes when load factor exceeds ~2/3.

🗃️ SQL Questions

Q11: HAVING vs WHERE? ▼
WHERE filters rows before grouping. HAVING filters groups after aggregation. Can't use aggregates in WHERE.
Q12: UNION vs UNION ALL? ▼
UNION deduplicates (slower, requires sort/hash). UNION ALL keeps all rows (faster). Default to UNION ALL unless you need dedup.
Q13: Types of indexes and when to use each? ▼
B-tree (default, range queries), hash (exact lookups), GIN/GiST (full-text, JSON), bitmap (low-cardinality OLAP). Composite indexes follow leftmost-prefix rule.
Q14: What's a correlated subquery? ▼
A subquery referencing the outer query's columns. Re-executes per outer row → O(n × m) unless optimized by the engine.
Q15: ROWS vs RANGE in window frames? ▼
ROWS counts physical rows. RANGE groups by logical value — same ORDER BY value = one unit. Use ROWS for deterministic totals.

🔧 Data Engineering Questions

Q16: Parquet vs CSV — why Parquet? ▼
Columnar storage, built-in compression, schema enforcement, predicate pushdown, splittable for parallel reads. CSV has none.
Q17: What is exactly-once semantics? ▼
Every message processed exactly one time — no dupes, no losses. Hard because retries cause dupes. Achieved via idempotent writes + transactional commits.
Q18: What does "idempotent" mean in pipelines? ▼
Same input → same output, no matter how many runs. Via MERGE/upsert, partition overwrites, or dedup-on-write. Essential for safe re-runs.
Q19: Backfill strategy? ▼
Re-process historical data after pipeline changes. Key: partition-based execution, idempotency, rate limiting upstream, validate counts before/after.
Q20: Schema evolution — how to handle? ▼
When source schemas change. Handle via: schema registries (Avro/Protobuf), mergeSchema in Spark, backward/forward compatibility, defaults for new fields.
