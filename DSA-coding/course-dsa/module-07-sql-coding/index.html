🏆 Top 3 Salaries & Self-Joins

Problem 1: Top 3 Salaries per Department

Find the top 3 distinct salaries in each department. Use DENSE_RANK() — not RANK() or ROW_NUMBER() — because the problem asks for distinct salary tiers.

WITH ranked AS ( SELECT emp_id, name, department, salary, DENSE_RANK() OVER ( PARTITION BY department ORDER BY salary DESC ) AS dr FROM employees ) SELECT emp_id, name, department, salary FROM ranked WHERE dr <= 3 ORDER BY department, salary DESC;

How it works: Alice & Bob (same salary) both get dr = 1. Carol gets dr = 2. Dave gets dr = 3. The filter dr <= 3 keeps all three tiers.

💡 DENSE_RANK vs RANK vs ROW_NUMBER

If the interviewer says "top 3 employees" (not distinct salaries), use ROW_NUMBER(). If they say "top 3 salaries allowing ties", use RANK(). Always clarify before you code.

Problem 2: Employees Earning More Than Their Manager

Classic self-join — join the table to itself on emp.manager_id = mgr.emp_id.

SELECT e.name AS employee, e.salary AS employee_salary, m.name AS manager, m.salary AS manager_salary FROM employees e JOIN employees m ON e.manager_id = m.emp_id WHERE e.salary > m.salary;

Key insight: INNER JOIN naturally excludes the CEO (NULL manager_id). This is often the first SQL question in a screen — solve it in under 2 minutes to set a strong tone.

🚫 Anti-Joins & Running Totals

Problem 3: Customers Who Never Ordered

Two valid approaches — both are anti-join patterns:

LEFT JOIN + IS NULL

SELECT c.customer_id, c.name FROM customers c LEFT JOIN orders o ON c.customer_id = o.customer_id WHERE o.order_id IS NULL;

NOT EXISTS (preferred)

SELECT c.customer_id, c.name FROM customers c WHERE NOT EXISTS ( SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id );
🚨 NEVER Use NOT IN!

NOT IN breaks silently when the subquery contains NULLs — the entire expression returns no rows. NOT EXISTS is always safe. This is one of the top SQL gotchas in interviews. Mention it unprompted to impress.

Problem 4: Running Total of Sales

Window function with an explicit frame specification:

SELECT sale_date, amount, SUM(amount) OVER ( ORDER BY sale_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_total FROM daily_sales ORDER BY sale_date; -- Result: 100, 350, 525, 825
💡 ROWS vs RANGE — The Subtle Gotcha

Without explicit frame, the default with ORDER BY is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, which groups ties together. ROWS is deterministic — it counts physical rows. Always specify the frame explicitly in interviews to show you understand this difference.

📈 MoM Growth & Pivoting

Problem 5: Month-over-Month Revenue Growth

LAG() to access previous month + NULLIF to prevent division by zero:

SELECT month, revenue, LAG(revenue) OVER (ORDER BY month) AS prev_revenue, ROUND( 100.0 * (revenue - LAG(revenue) OVER (ORDER BY month)) / NULLIF(LAG(revenue) OVER (ORDER BY month), 0), 1 ) AS mom_pct_change FROM monthly_revenue ORDER BY month; -- Jan=NULL, Feb=+20.0%, Mar=−4.2%, Apr=+13.0%

Key details: NULLIF(..., 0) prevents division-by-zero. ROUND(..., 1) gives clean output. First row is naturally NULL. Mention LEAD() works the same but looks forward.

Problem 6: Pivot Sales by Quarter

Conditional aggregation — the most portable pivot approach (works in every SQL engine):

SELECT product, SUM(CASE WHEN quarter = 'Q1' THEN revenue ELSE 0 END) AS Q1, SUM(CASE WHEN quarter = 'Q2' THEN revenue ELSE 0 END) AS Q2, SUM(CASE WHEN quarter = 'Q3' THEN revenue ELSE 0 END) AS Q3, SUM(CASE WHEN quarter = 'Q4' THEN revenue ELSE 0 END) AS Q4 FROM sales GROUP BY product ORDER BY product;
💡 Dynamic Pivot?

If the interviewer asks "What if we don't know the quarters in advance?" — that's dynamic pivot. PostgreSQL uses crosstab(), BigQuery uses scripting, app code builds the SQL string dynamically. There's no clean ANSI-standard way.

🧹 Dedup & Gap-and-Island

Problem 7: Dedup Records — Keep Latest

CDC (change data capture) creates duplicates. ROW_NUMBER() partitioned by key, ordered by timestamp DESC. Keep rn = 1.

WITH numbered AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY user_id ORDER BY updated_at DESC ) AS rn FROM user_profiles ) SELECT user_id, email, updated_at FROM numbered WHERE rn = 1;

This is strictly better than GROUP BY + MAX(updated_at) because it returns ALL columns. In production, use this inside a MERGE / INSERT OVERWRITE for SCD Type 1.

Problem 8: Consecutive Login Days (Gap & Island)

Find users who logged in for 3+ consecutive days. The insight: subtract ROW_NUMBER() from the date — consecutive dates produce the same island ID.

WITH islands AS ( SELECT user_id, login_date, login_date - INTERVAL '1 day' * ROW_NUMBER() OVER ( PARTITION BY user_id ORDER BY login_date ) AS island_id FROM logins ) SELECT user_id, MIN(login_date) AS streak_start, MAX(login_date) AS streak_end, COUNT(*) AS streak_length FROM islands GROUP BY user_id, island_id HAVING COUNT(*) >= 3 ORDER BY user_id, streak_start;

🔄 Sessionization & Recursive CTEs

Problem 9: Sessionize User Clicks

A new session starts when there's a gap > 30 minutes between clicks. This is the SQL version of the exact same pattern from Module 6:

-- The 4-step pipeline: -- 1. LAG → previous timestamp -- 2. GAP → time difference -- 3. FLAG → new session? 1 or 0 -- 4. CUMSUM → session ID
Same pattern, three languages:
📊 Pandas: shift() → diff() → flag → cumsum()
⚡ PySpark: lag() → gap → when() → sum().over()
🗃️ SQL: LAG() → EXTRACT → CASE → SUM() OVER()
Being fluent in all three = superpower
WITH with_gap AS ( SELECT *, LAG(click_time) OVER ( PARTITION BY user_id ORDER BY click_time ) AS prev_click, EXTRACT(EPOCH FROM ( click_time - LAG(click_time) OVER ( PARTITION BY user_id ORDER BY click_time ) )) / 60.0 AS gap_minutes FROM clicks ), with_flag AS ( SELECT *, CASE WHEN gap_minutes IS NULL OR gap_minutes > 30 THEN 1 ELSE 0 END AS new_session_flag FROM with_gap ) SELECT user_id, click_time, page, SUM(new_session_flag) OVER ( PARTITION BY user_id ORDER BY click_time ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS session_id FROM with_flag ORDER BY user_id, click_time; -- BigQuery: use TIMESTAMP_DIFF(click_time, prev_click, MINUTE)

Trace: Flag values [1, 0, 0, 1, 0] become cumulative sums [1, 1, 1, 2, 2] → two sessions.

Problem 10: Org Chart Hierarchy (Recursive CTE)

WITH RECURSIVE org_tree AS ( -- Anchor: start at the root SELECT emp_id, name, manager_id, 0 AS level FROM employees WHERE manager_id IS NULL UNION ALL -- Recursive: find children of current level SELECT e.emp_id, e.name, e.manager_id, t.level + 1 FROM employees e JOIN org_tree t ON e.manager_id = t.emp_id ) SELECT emp_id, name, level FROM org_tree ORDER BY level, name;
🚨 Cycle Detection — Chef's Kiss Moment

Recursive CTEs can infinite-loop if there's a cycle (A manages B manages A). In production, add WHERE level < 20 as a safety guard, or use CYCLE detection (PostgreSQL 14+). Mentioning this unprompted is a chef's kiss moment in interviews. Note: BigQuery doesn't support WITH RECURSIVE — you'd use iterative scripting.

🎯 Funnel Analysis & Nth Salary

Problem 11: Funnel Conversion Rates

Compute conversion rates across: page_view → add_to_cart → checkout → purchase

WITH funnel AS ( SELECT COUNT(DISTINCT CASE WHEN event_type = 'page_view' THEN user_id END) AS step1_views, COUNT(DISTINCT CASE WHEN event_type = 'add_to_cart' THEN user_id END) AS step2_cart, COUNT(DISTINCT CASE WHEN event_type = 'checkout' THEN user_id END) AS step3_checkout, COUNT(DISTINCT CASE WHEN event_type = 'purchase' THEN user_id END) AS step4_purchase FROM user_events ) SELECT step1_views, step2_cart, step3_checkout, step4_purchase, ROUND(100.0 * step2_cart / NULLIF(step1_views, 0), 1) AS view_to_cart_pct, ROUND(100.0 * step3_checkout / NULLIF(step1_views, 0), 1) AS view_to_checkout_pct, ROUND(100.0 * step4_purchase / NULLIF(step1_views, 0), 1) AS view_to_purchase_pct FROM funnel;

How it works: COUNT(DISTINCT CASE WHEN ...) counts unique users per step. The CASE returns NULL for non-matching events, and COUNT(DISTINCT) ignores NULLs.

💡 Strict vs Simple Funnel

A more rigorous funnel requires sequential steps (cart only counts if the user also viewed). Enforce ordering by joining on user_id with timestamp conditions: cart.event_time > view.event_time. Mentioning this distinction shows analytical maturity.

Problem 12: Second Highest Salary Without LIMIT

Two approaches — show both for maximum interview points:

🅰️ Subquery (Classic)

SELECT MAX(salary) AS second_highest FROM employees WHERE salary < ( SELECT MAX(salary) FROM employees );

Fast to write. Only works for "second." Inner query finds max, outer finds max below that.

🅱️ Window Function (Flexible)

WITH ranked AS ( SELECT salary, DENSE_RANK() OVER ( ORDER BY salary DESC ) AS dr FROM employees ) SELECT DISTINCT salary FROM ranked WHERE dr = 2;

More general — change dr = N for any Nth value. Start with A, offer B to show range.

⚠️ Common Pitfalls & Study Plan

7 Pitfalls That Trip Up Even Senior Engineers

1. NOT IN with NULLs

If the subquery contains ANY NULL, the entire NOT IN returns empty. Use NOT EXISTS instead.

2. Missing GROUP BY Cols

Every non-aggregated column in SELECT must appear in GROUP BY. MySQL is lenient; PostgreSQL & BigQuery are strict.

3. Window Frame Defaults

SUM() OVER (ORDER BY x) uses RANGE by default, which groups ties. Use ROWS for deterministic behavior.

4. Integer Division

SELECT 1 / 3 returns 0 in many engines. Always cast: 1.0 / 3 or CAST(1 AS DECIMAL) / 3.

5. Forgetting DISTINCT

COUNT(user_id) counts all rows. COUNT(DISTINCT user_id) counts unique users. Huge difference in funnels.

6. Self-Join Double Counting

When comparing pairs, use e1.id < e2.id to avoid counting (A,B) and (B,A) as separate pairs.

7. Recursive CTE Infinite Loops

Always add a depth limit (WHERE level < 100) or ensure your data is acyclic. No guard = runaway query.
