🏗️ GPT Decoder-Only Architecture

Modern large language models — GPT-4, Claude, Llama 3 — share a common ancestor: the decoder-only Transformer. Understanding this architecture is non-negotiable for anyone building on top of these models.

The Core Idea: Next-Token Prediction

A decoder-only model does exactly one thing during training: given a sequence of tokens, predict the next token. Every stunning capability — writing poetry, debugging code, answering customer queries — emerges from this deceptively simple objective.

Input: "The customer wants to return their" Output: " order" (predicted next token)

Causal Self-Attention

The key mechanism is causal (masked) self-attention. Unlike the encoder (which sees the full sequence bidirectionally), a decoder masks future tokens so position i can only attend to positions 0..i.

Causal Attention Mask ┌─────────────────────────────┐ │ The cust wants to return│ The │ 1 0 0 0 0 │ cust │ 1 1 0 0 0 │ wants │ 1 1 1 0 0 │ to │ 1 1 1 1 0 │ return│ 1 1 1 1 1 │ └─────────────────────────────┘ 1 = can attend, 0 = masked

Each attention head computes Query, Key, Value projections: Attention(Q, K, V) = softmax(QK^T / √d_k) · V. Multi-head attention runs this in parallel across h heads, each learning different relationship patterns.

Autoregressive Generation

# For our customer support system, autoregressive generation means # every token is conditioned on ALL previous tokens def generate_support_response(client: openai.OpenAI, query: str) -> str: """Each token is generated sequentially — the model 'writes' left to right.""" response = client.chat.completions.create( model="gpt-4o", messages=[ {"role": "system", "content": "You are a helpful e-commerce support agent."}, {"role": "user", "content": query}, ], stream=True, ) full_response = "" for chunk in response: token = chunk.choices[0].delta.content or "" full_response += token return full_response
💡 Interview Insight

Q: Why decoder-only instead of encoder-decoder? — "Decoder-only won for three reasons: simplicity (one model for understanding + generation), flexibility (classification to open-ended generation without architectural changes), and scalability (next-token prediction on massive unsupervised corpora). Encoder-decoder models like T5 require constructing explicit input-output pairs."

Pre-train → SFT → RLHF/DPO Alignment

Raw pre-training produces a model that completes text — not one that follows instructions helpfully and safely. The alignment pipeline transforms a base model into a useful assistant.

📚 Stage 1: Pre-training

Next-token prediction on ~15T tokens from the internet, books, code. The most expensive stage (~$5–100M compute). Learns language, world knowledge, and reasoning.

🎯 Stage 2: SFT

Fine-tuned on ~100K high-quality (instruction, response) pairs. Cost: ~$50K. For our support system, this would be exemplary agent-customer conversations.

🏆 Stage 3: RLHF / DPO

Human annotators rank model outputs. RLHF trains a reward model + PPO. DPO skips the reward model entirely — simpler, cheaper, increasingly preferred. Cost: ~$200K+.

DPO: Direct Preference Optimization

# Conceptual DPO loss — directly optimize from preference pairs def dpo_loss(policy, reference, preferred, dispreferred, beta=0.1): """ DPO eliminates the reward model entirely. Loss = -log σ(β · (log π(y_w|x)/π_ref(y_w|x) - log π(y_l|x)/π_ref(y_l|x))) """ log_ratio_w = policy.log_prob(preferred) - reference.log_prob(preferred) log_ratio_l = policy.log_prob(dispreferred) - reference.log_prob(dispreferred) return -torch.log(torch.sigmoid(beta * (log_ratio_w - log_ratio_l))).mean()
💡 Interview Insight

When DPO over RLHF? — "DPO is my default. It eliminates the reward model and PPO's instability. For support, collect agent ratings on outputs, create preference pairs, run DPO directly. RLHF is still relevant for fine-grained reward shaping. DPO gets 90% of the benefit at 30% of the complexity."

KV-Cache — Inference Speed & Memory

When your support system handles thousands of concurrent conversations, inference speed becomes a production concern. The KV-cache is the single most important optimization to understand.

The Problem Without KV-Cache

To produce token t_n, the model computes attention over all previous tokens. Without caching, generating the next token requires recomputing Q, K, V for every previous token — O(n²) per token, O(n³) for the full sequence.

Without KV-Cache (generating token 5): Compute Q,K,V for tokens 1,2,3,4,5 → attend → output token 5 Generating token 6: Compute Q,K,V for tokens 1,2,3,4,5,6 → attend → output token 6 ← REDUNDANT! With KV-Cache: Token 5: Compute Q5, K5, V5. Cache K5,V5. Attend Q5 to [K1..K5] → output Token 6: Compute Q6, K6, V6. Cache K6,V6. Attend Q6 to [K1..K6] → output ↑ cached!

Memory Cost & GQA

For a 70B model (80 layers, 64 heads, d_head=128) with FP16 at 4K context: ~10.7 GB per request.

Standard MHA: 64 Q heads, 64 K heads, 64 V heads → full KV-cache GQA (Llama 3): 64 Q heads, 8 K heads, 8 V heads → 8× smaller KV-cache MQA: 64 Q heads, 1 K head, 1 V head → 64× smaller KV-cache
💡 Interview Insight

Optimizing inference costs? — "Three layers: (1) Use a GQA model like Llama 3 for 8× KV-cache reduction, (2) continuous batching (vLLM) keeps GPU utilization above 80%, (3) PagedAttention allocates KV-cache in non-contiguous blocks eliminating fragmentation. Together: 3-5× cost reduction."

Chinchilla Scaling Laws

When deciding which model to use — or whether to train your own — understanding scaling laws tells you how to allocate your compute budget.

The Chinchilla Insight

DeepMind's Chinchilla paper (2022) showed most LLMs were over-parameterized and under-trained. The compute-optimal ratio: Optimal tokens ≈ 20 × parameters

ModelParametersTraining TokensChinchilla Optimal?
GPT-3175B300B❌ Under-trained
Chinchilla70B1.4T✅ Optimal
Llama 2 70B70B2T✅ Over-trained
Llama 3 8B8B15T✅ Deliberately over-trained

The Post-Chinchilla Shift

Teams now deliberately over-train smaller models beyond the optimum. Training compute is a one-time cost, but inference is ongoing. A smaller model trained on more data gives better performance-per-FLOP at inference time.

Decision Framework for ShopAssist: Budget: $10K/month inference ──▶ Use Llama 3 8B / Mistral 7B Budget: $50K/month inference ──▶ Use GPT-4o-mini / Claude 3.5 Haiku Budget: $200K/month inference ──▶ Use GPT-4o / Claude 3.5 Sonnet Need custom domain knowledge ──▶ Fine-tune a small model on domain data Need maximum quality, any cost ──▶ GPT-4o / Claude 3.5 Opus
💡 Interview Insight

Scaling laws & model selection? — "For inference-heavy workloads, prefer smaller models trained on more data. Llama 3 8B trained on 15T tokens gives 80-90% of GPT-4's quality at 1/50th the inference cost. Start small, measure quality gaps, upgrade only where needed."

Models above ~60B parameters can "reason" step-by-step when prompted with "Let's think step by step." Dramatically improves complex scenarios.

Inference Parameters — Controlling Generation

Every customer support response is shaped by inference parameters. Misconfigure them and you get either robotic repetition or chaotic hallucination.

Temperature

# Temperature 0.0 — Deterministic, best for factual support responses response_factual = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": "What is your return policy?"}], temperature=0.0, # Always picks highest-probability token ) # Temperature 1.2 — Creative, useful for marketing copy response_creative = client.chat.completions.create( model="gpt-4o-mini", messages=[{"role": "user", "content": "Write a fun apology for a late delivery"}], temperature=1.2, # Flattens distribution, more variety )

Context Windows & Tokenization

LLMs don't see characters or words — they see tokens. Most modern models use Byte-Pair Encoding (BPE), learned during training.

import tiktoken # GPT-4o uses the o200k_base tokenizer (~200K vocab) enc = tiktoken.encoding_for_model("gpt-4o") text = "Customer wants to return order #A1234-XZ for a refund" tokens = enc.encode(text) print(f"Token count: {len(tokens)}") # Token count: 13 # Cost estimation for our support system avg_input_tokens = 150 avg_output_tokens = 200 queries_per_day = 50_000 daily_cost_gpt4o_mini = ( (avg_input_tokens * queries_per_day / 1_000_000) * 0.15 + (avg_output_tokens * queries_per_day / 1_000_000) * 0.60 ) print(f"Daily cost (GPT-4o-mini): ${daily_cost_gpt4o_mini:.2f}") # Daily cost: $7.13/day — very affordable!

The Lost-in-the-Middle Problem

Stanford research (2023) showed LLMs perform worse when relevant information is in the middle of long contexts. Performance is best when key info is at the beginning or end.

🎯 Design Implication

When stuffing retrieved context into the prompt, put the most relevant documents at the beginning and end. Don't bury critical return policy details in the middle of 10 retrieved chunks.

💡 Interview Insight

Tokenization impact on cost & quality? — "Every token costs money. Non-English and code tokenize 2-3× less efficiently, shrinking effective context. The lost-in-the-middle problem means arrangement matters as much as content. I place most relevant context at the top, instructions next, user query at the end."

LLM APIs — ChatCompletion, Function Calling, Streaming

The ChatCompletion Format

from openai import OpenAI client = OpenAI() messages = [ {"role": "system", "content": "You are ShopAssist, an e-commerce support agent."}, {"role": "user", "content": "I received a damaged item. Order #9912."}, {"role": "assistant", "content": "I'm sorry! Let me look up order #9912."}, {"role": "user", "content": "It's a laptop with a cracked screen."}, ] response = client.chat.completions.create( model="gpt-4o-mini", messages=messages, temperature=0.1, )

Function Calling / Tool Use

tools = [ { "type": "function", "function": { "name": "lookup_order", "description": "Look up order details by order ID", "parameters": { "type": "object", "properties": { "order_id": {"type": "string"} }, "required": ["order_id"], }, }, }, ] response = client.chat.completions.create( model="gpt-4o-mini", messages=messages, tools=tools, tool_choice="auto", ) # The model returns a tool call instead of text tool_call = response.choices[0].message.tool_calls[0] print(f"Function: {tool_call.function.name}") print(f"Args: {tool_call.function.arguments}") # Function: lookup_order # Args: {"order_id": "#9912"}
💡 Interview Insight

Rate limits & reliability? — "Three-layer resilience: (1) client-side rate limiting with token bucket, (2) exponential backoff + jitter on 429/5xx, (3) model fallback (GPT-4o → GPT-4o-mini → Claude Haiku). Add a circuit breaker that serves cached responses when all providers are down."

Multi-Modal LLMs — Vision, Audio & Beyond

Modern LLMs process more than text. For customer support, multi-modal capabilities unlock powerful use cases like damage assessment from photos.

Vision: Processing Customer Images

import base64 from openai import OpenAI client = OpenAI() def assess_damage_from_photo(image_path: str) -> dict: """Use vision model to assess product damage from customer photo.""" with open(image_path, "rb") as f: image_b64 = base64.b64encode(f.read()).decode() response = client.chat.completions.create( model="gpt-4o", messages=[{ "role": "user", "content": [ {"type": "text", "text": "Assess the damage in this photo:"}, {"type": "image_url", "image_url": { "url": f"data:image/jpeg;base64,{image_b64}" }}, ], }], temperature=0.1, response_format={"type": "json_object"}, ) return response.choices[0].message.content

Audio: Voice-Based Support

# Pipeline: Voice call → Whisper transcription → LLM analysis → Action def transcribe_support_call(audio_path: str) -> str: """Transcribe customer support call for analysis.""" with open(audio_path, "rb") as audio_file: transcript = client.audio.transcriptions.create( model="whisper-1", file=audio_file, response_format="verbose_json", ) return transcript
💡 Interview Insight

Multi-modal in support? — "Build it as an extension of the text pipeline. GPT-4o's vision assesses damage and extracts details from photos. Whisper transcribes voice calls. The key decision: use a unified model like GPT-4o for text+images natively, rather than stitching separate models — reduces latency and improves cross-modal reasoning."

Key Takeaways

🏗️ Decoder-Only

Causal self-attention + next-token prediction. The simplicity of this objective enables scaling to emergent capabilities.

🎯 Alignment Pipeline

Pre-train → SFT → RLHF/DPO transforms a text completer into a useful assistant. DPO is increasingly preferred for simplicity.

⚡ KV-Cache

Critical inference optimization. GQA reduces memory by 8×, enabling practical long-context serving.

📏 Scaling Laws

Smaller models trained on more data are more inference-efficient. Prefer over-trained small models for production.

🎛️ Inference Params

Temperature, top-p, penalties dramatically affect quality. For support: low temperature, moderate frequency penalty.

🔧 APIs & Multimodal

Function calling bridges LLM reasoning and real-world actions. Vision + audio enable rich customer interactions.
