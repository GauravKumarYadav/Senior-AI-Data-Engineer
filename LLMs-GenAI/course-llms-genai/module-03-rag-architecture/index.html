🔍 Why RAG?

Your GPT-4o-powered support agent is great at language, but it doesn't know your business: your return policy (updated last Tuesday), order #7788's tracking info, or your 50K-product catalog.

Fine-tuning is expensive & stale. RAG gives the model access to current, authoritative information at query time.

📄 Document Loading

Before you retrieve anything, you need to load, parse, and clean your documents. Quality extraction is foundational — bad extraction → bad chunks → bad retrieval.

PDF Loading with PyMuPDF

import pymupdf from pathlib import Path def load_pdf(file_path: str) -> list[dict]: doc = pymupdf.open(file_path) pages = [] for page_num, page in enumerate(doc): text = page.get_text("text") tables = page.find_tables() pages.append({ "text": text, "tables": [t.extract() for t in tables], "page_number": page_num + 1, "source": Path(file_path).name, }) doc.close() return pages
💡 Interview Insight

"I build a document loader registry — a factory pattern that routes files to the right parser based on MIME type. The critical step most teams skip is quality validation: sample 5% of loaded docs and compare extracted text to the original visually."

✂️ Chunking Strategies

Chunking is where many RAG systems fail. Too large → imprecise retrieval. Too small → lost context.

Recursive Character Splitting

splitter = RecursiveCharacterTextSplitter( chunk_size=512, chunk_overlap=50, separators=["\\n\\n", "\\n", ". ", " ", ""], # Try larger splits first )

Parent-Child Chunking (Best of Both Worlds)

🔎 Child Chunks (256 tok)

Small chunks used for RETRIEVAL — precise semantic matching in vector search.

📖 Parent Chunks (1024 tok)

Large chunks sent to the LLM — full context for generating complete answers.

# Small chunks for retrieval, large for generation child_splitter = RecursiveCharacterTextSplitter(chunk_size=256, chunk_overlap=30) parent_splitter = RecursiveCharacterTextSplitter(chunk_size=1024, chunk_overlap=100) retriever = ParentDocumentRetriever( vectorstore=vectorstore, docstore=InMemoryStore(), child_splitter=child_splitter, parent_splitter=parent_splitter, )

📐 Similarity Search

Understanding how vectors are compared and indexed is essential for tuning retrieval quality.

def cosine_similarity(a, b): """Most common for normalized embeddings. Range: -1 to 1.""" return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) query_emb = embed_texts(["How do I return electronics?"])[0] doc1_emb = embed_texts(["Electronics return policy: 30 days"])[0] print(cosine_similarity(query_emb, doc1_emb)) # ~0.85

ANN Algorithms: HNSW vs IVF-PQ

🕸️ HNSW

  • ✅ ~99% recall@10, <5ms for 1M vectors
  • ✅ Multi-layer graph, greedy descent
  • ⚠️ High memory (stores graph + vectors)
  • Best for: <10M vectors, quality-critical

📦 IVF-PQ

  • ✅ ~95% recall@10, <10ms for 1B vectors
  • ✅ Cluster + compress vectors
  • ⚠️ Lower recall than HNSW
  • Best for: >10M vectors, memory-tight
💡 Interview Insight

"Three levers for retrieval quality: (1) embedding model — biggest impact, benchmark on real queries; (2) index params — tune ef_construction and ef_search for HNSW; (3) metadata filtering — pre-filter by category before vector search."

🚀 Advanced RAG

Naive RAG works for 70% of cases. These techniques handle the other 30%.

🔀 Hybrid Search

Combine vector similarity (semantic) with BM25 keyword search (lexical). Catches exact IDs + paraphrased queries.

🏆 Re-Ranking

Retrieve top-20 fast, then re-rank with a cross-encoder. Improves precision@5 by 15-25%.

🌀 HyDE

Generate a hypothetical answer, embed THAT instead of the query. Closer in embedding space to real docs.

🔄 Multi-Query

LLM rephrases query from multiple angles, retrieves for each, deduplicates results.

Hybrid Search Code

def search(self, query, query_embedding, alpha=0.7): # Dense scores (cosine similarity) dense_scores = np.dot(self.embeddings, q_emb) / norms # Sparse scores (BM25) sparse_scores = self.bm25.get_scores(query.lower().split()) # Normalize both to [0, 1], then weighted fusion combined = alpha * dense_norm + (1 - alpha) * sparse_norm

"I categorize failures first: retrieval miss → try smaller chunks, hybrid search. Ranking issue → add cross-encoder re-ranker. Query-doc style mismatch → try HyDE. Each change evaluated on 50+ query-answer test pairs."

📊 RAG Evaluation

You can't improve what you can't measure. RAG evaluation has unique challenges because failures occur at multiple points.

RAGAS Framework

📋 Context Relevance

Are the retrieved documents relevant to the question? Measures retrieval quality.

🎯 Faithfulness

Is the answer grounded in the retrieved context? Detects hallucination.

✅ Answer Relevance

Does the answer actually address the question? End-to-end quality.

RAG Failure Taxonomy

1. Retrieval Miss

Correct doc isn't in results. Fix: add missing docs, improve chunking, try hybrid search.

2. Context Stuffing

Too many loosely related chunks. Fix: reduce top_k, add re-ranking, raise threshold.

3. Hallucination

Model ignores or misreads context. Fix: better prompts, faithfulness checks.

4. Wrong Context

Retrieved related but wrong category. Fix: metadata filters, multi-query retrieval.

💡 Interview Insight

"I evaluate RAG at three levels: (1) retrieval recall@5 weekly on 200 query-doc pairs; (2) RAGAS metrics for generation quality; (3) production sampling — 50 conversations/day through LLM-as-Judge. The failure taxonomy guides interventions."
