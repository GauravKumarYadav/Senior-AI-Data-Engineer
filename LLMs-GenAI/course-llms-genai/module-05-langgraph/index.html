🧠 LLMs & GenAI Mastery

🔗 Why LangGraph?

ShopAssist's AgentExecutor works for simple queries, but it can't handle complex workflows: multi-step returns requiring approval, routing tickets to specialists, or retrying failed tool calls. LangGraph gives you a graph-based runtime with full control.

🔄 AgentExecutor

  • ❌ LLM decides everything — no control
  • ❌ No human approval gates
  • ❌ No branching or routing
  • ❌ Sequential tool execution only
  • ❌ Opaque "agent_scratchpad"
  • ❌ Crash = lose entire state

🔗 LangGraph

  • ✅ You control edges & transitions
  • ✅ interrupt() for human approval
  • ✅ Conditional edges for routing
  • ✅ Send() for parallel fan-out
  • ✅ Full state inspection & time-travel
  • ✅ Checkpointing for crash recovery
💡 Interview Gold

"I start with AgentExecutor for prototyping — it's simple and covers single-agent tool use. I move to LangGraph when I need: (1) human-in-the-loop — refund confirmation before processing; (2) deterministic routing — billing issues to the billing specialist; (3) persistence — crash recovery via checkpointing; (4) multi-agent collaboration — supervisor delegates to specialists. The mental model shift is from 'loop' to 'graph.'"

🏗️ StateGraph Fundamentals

Everything in LangGraph starts with three things: a state schema, node functions, and edges.

from langgraph.graph import StateGraph, START, END from typing import TypedDict # 1. Define state schema class SupportState(TypedDict): customer_message: str category: str response: str # 2. Define node functions def classify(state): msg = state["customer_message"].lower() if "order" in msg: return {"category": "shipping"} elif "refund" in msg: return {"category": "returns"} return {"category": "general"} # 3. Build graph + edges graph = StateGraph(SupportState) graph.add_node("classify", classify) graph.add_edge(START, "classify") graph.add_conditional_edges("classify", route) app = graph.compile()
📦 Import graph building blocks
 
 
📋 State is a TypedDict — the data that flows through the graph
 
 
 
⚙️ Nodes are functions: receive state → return partial update
 
 
 
 
 
 
🔗 Wire it up: add nodes, add edges (fixed or conditional)
 
 
 
🚀 compile() validates & freezes the graph

1️⃣ Nodes = Functions

(state) → partial state update. Each node only returns the keys it changes.

2️⃣ Edges = Transitions

Fixed (add_edge) or conditional (add_conditional_edges) — a routing function returns the next node name.

3️⃣ Cycles Allowed

Node A → B → A for retry loops. Impossible with simple chains!

4️⃣ START → … → END

Every graph starts at START and terminates at END. compile() validates this.

💡 Interview Gold

"A StateGraph is a finite state machine where nodes are Python functions and edges define transitions. State is a TypedDict that flows through — each node receives full state, returns a partial update. Conditional edges inspect state and return the next node name. The critical insight is that cycles are allowed — enabling retry loops, iterative refinement, and ReAct-style agent loops."

🧬 State Management: Reducers

Without reducers, each node overwrites state keys. For chat messages, you want to append, not replace.

# ❌ Default: node_b's message REPLACES node_a's class State(TypedDict): messages: list # Each node replaces the entire list! # ✅ Fix: Annotated + operator.add = APPEND from typing import Annotated import operator class SupportState(TypedDict): messages: Annotated[list[AnyMessage], operator.add] # Append! customer_id: str # Overwrite (default)

MessagesState — Built-in Chat State

from langgraph.graph import MessagesState # Pre-built TypedDict with messages: Annotated[list, add_messages] # add_messages: appends, updates by ID, deduplicates class ShopAssistState(MessagesState): customer_id: str order_id: str | None escalated: bool
💡 Interview Gold

"By default, new values overwrite old ones — fine for scalars like customer_id. But for message lists, you need Annotated[list, operator.add] to append. LangGraph provides MessagesState with a smart add_messages reducer that handles appending, updating by ID, and deduplication. Multiple nodes writing to the same key without proper reducers = data loss."

🔄 Building a ReAct Agent

The ReAct pattern is a two-node cyclic graph: agent (LLM) ↔ tools. The LLM decides when to stop.

# Hand-built ReAct agent def call_model(state: MessagesState) -> dict: response = llm.invoke([system] + state["messages"]) return {"messages": [response]} tool_node = ToolNode(tools) def should_continue(state) -> str: if state["messages"][-1].tool_calls: return "tools" return END graph = StateGraph(MessagesState) graph.add_node("agent", call_model) graph.add_node("tools", tool_node) graph.add_edge(START, "agent") graph.add_conditional_edges("agent", should_continue) graph.add_edge("tools", "agent") # ← THE CYCLE app = graph.compile()

One-Liner Alternative

# create_react_agent builds the same graph! from langgraph.prebuilt import create_react_agent agent = create_react_agent( model=ChatOpenAI(model="gpt-4o"), tools=tools, prompt="You are ShopAssist.", )

"A ReAct agent is a two-node cyclic graph: agent (LLM call) and tools (execution). If the LLM returns tool_calls, it routes to tools; otherwise → END. The cycle is the key: tools → agent → tools until done. I prefer hand-building when I need custom control flow — like a validation step between agent and tools."

🎯 Agent Patterns: Plan, Reflect, Self-RAG

Pattern 1: Plan-and-Execute

Create a plan → execute each step → replan if something goes wrong. Great for multi-step workflows.

# Planner creates steps, Executor runs them, Replanner adjusts def planner(state) -> dict: plan = planner_llm.invoke(["Create step-by-step plan..."]) return {"plan": json.loads(plan.content), "current_step": 0} def executor(state) -> dict: step = state["plan"][state["current_step"]] result = executor_chain.invoke({"step": step}) return {"results": [result], "current_step": state["current_step"] + 1} # Graph: planner → executor ↔ replanner → END

Pattern 2: Reflection

Generate → Critique → Revise loop. Ideal for quality-critical outputs like customer emails.

# Generate-Critique-Revise loop (cap at 3 iterations) graph.add_edge(START, "generate") graph.add_edge("generate", "critique") graph.add_conditional_edges("critique", should_revise) # If "APPROVED" → END, else → "generate" (loop back)

Pattern 3: Self-RAG

Decide if retrieval is needed → retrieve → grade relevance → generate → check hallucination.

# Smart retrieval pipeline decide_retrieval → retrieve → grade_documents → generate → check_hallucination # If hallucinated → regenerate; if grounded → END
💡 Interview Gold

"Plan-and-Execute for multi-step tasks where sequence matters — the replanner adjusts if something fails. Reflection for quality-critical outputs — generate-critique-revise catches errors (cap at 3 iterations). Self-RAG adds intelligence to retrieval — decides if retrieval is needed, grades relevance, and checks hallucination. For ShopAssist: Plan-and-Execute for returns, Reflection for VIP emails, Self-RAG for knowledge base Q&A."

🤖 Multi-Agent Architectures

When one agent has too many tools or must reason across unrelated domains, split into specialists.

# Supervisor Pattern: router → specialists → router → END billing_agent = create_react_agent(model, billing_tools) shipping_agent = create_react_agent(model, shipping_tools) def supervisor(state) -> dict: # Routes to "billing", "shipping", "product", or "DONE" response = supervisor_llm.invoke([...]) return {"next": response.content.strip().lower()} graph.add_edge(START, "supervisor") graph.add_conditional_edges("supervisor", route_supervisor) # After each specialist → back to supervisor graph.add_edge("billing", "supervisor") graph.add_edge("shipping", "supervisor")

👔 Supervisor

Router agent delegates to domain specialists. Most controllable. Best for clear specialization.

🏢 Hierarchical

Nested supervisors manage sub-teams. Best for large orgs with team structure.

🤝 Collaborative

Agents pass work to each other via shared state. Best for creative brainstorming.

🐝 Swarm

Any agent can hand off to any other dynamically. No central coordinator. Flexible but harder to debug.

💡 Interview Gold

"I'd use the Supervisor pattern — it's the most controllable. The supervisor uses a cheaper model (GPT-4o-mini) for routing; specialists use GPT-4o for reasoning. All agents share MessagesState so context isn't lost. I cap at 3 specialist calls per conversation to prevent infinite delegation loops."

🙋 Human-in-the-Loop & Checkpointing

interrupt() pauses the graph, saves state, and waits for a human decision. Command(resume=...) continues.

from langgraph.types import interrupt, Command from langgraph.checkpoint.memory import MemorySaver def request_approval(state) -> dict: amount = state["refund_amount"] approval = interrupt( # ⏸️ PAUSES HERE f"Refund of ${amount:.2f} requires approval." ) return {"approved": approval == "yes"} # Compile with checkpointer app = graph.compile(checkpointer=MemorySaver()) config = {"configurable": {"thread_id": "refund-001"}} # Run → pauses at interrupt result = app.invoke(input, config) # Resume with human decision result = app.invoke(Command(resume="yes"), config)

Checkpointer Options

💾 MemorySaver

In-memory. Dev/testing only. Gone on restart.

📁 SqliteSaver

SQLite file. Single-server persistence.

🐘 PostgresSaver

Production. Multi-server. Crash recovery.

Time Travel

# Replay from any checkpoint — undo/redo! checkpoints = list(app.get_state_history(config)) earlier = checkpoints[2] # Go back 2 steps app.update_state(config, earlier.values) result = app.invoke(None, config) # Continue from there
💡 Interview Gold

"When a node calls interrupt(), the graph pauses, saves complete state to the checkpointer, and returns the interrupt value. The caller presents this to a human. When they respond, Command(resume="yes") resumes exactly where it paused. The graph can pause for minutes, hours, or days. thread_id isolates conversations — thousands of paused conversations coexist."

⚡ Subgraphs, Parallelism & Streaming

Subgraphs — Nested Graphs

# Complex agents = composition of smaller subgraphs return_subgraph = return_graph.compile() refund_subgraph = refund_graph.compile() parent = StateGraph(MessagesState) parent.add_node("returns", return_subgraph) # Subgraph as node! parent.add_node("refunds", refund_subgraph) # Subgraph as node!

Send() — Parallel Fan-Out

from langgraph.types import Send def fan_out_items(state) -> list[Send]: return [ Send("process_item", {"item": item}) for item in state["items"] ] # All items processed in parallel, results merged via reducer

Streaming Modes

📸 values

Full state snapshot after each node. Good for debugging.

📝 updates

Delta changes per node. Good for observability.

💬 messages

Token-by-token LLM output. Perfect for UX typing indicators.

💡 Interview Gold

"Send() enables fan-out parallelism — like processing a multi-item return where each item is checked independently. For streaming, I use messages mode for the frontend (typing indicators) and updates mode for our observability stack. Streaming works through the entire graph — even subgraphs and tool calls."
