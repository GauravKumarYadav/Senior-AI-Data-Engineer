⛓️ LCEL — Expression Language

The | pipe operator creates a RunnableSequence where each step's output becomes the next step's input.

# The | operator composes components into a chain chain = ( ChatPromptTemplate.from_messages([ ("system", "You are ShopAssist. Answer concisely."), ("human", "{question}"), ]) | llm | StrOutputParser() ) # Streaming works end-to-end automatically! for chunk in chain.stream({"question": "How do I track my order?"}): print(chunk, end="")

LCEL Building Blocks

🔗 RunnableSequence

A | B | C — sequential pipeline, output flows forward.

🔀 RunnableParallel

{"a": A, "b": B} — fan-out, run concurrently, merge results.

➡️ RunnablePassthrough

Pass input unchanged. Identity function for forwarding originals.

🔧 RunnableLambda

Wrap any function as a Runnable. Gets .invoke/.batch/.stream free.

🔀 RunnableBranch

Conditional routing — if/else chains based on input conditions.

➕ .assign()

Add new keys to state while keeping existing ones. Accumulate context.

RAG Chain with LCEL

rag_chain = ( { "context": retriever | format_docs, # question → docs → text "question": RunnablePassthrough(), # question unchanged } | prompt | llm | StrOutputParser() )

🔎 Vector Stores & Retrievers

# Chroma (dev), FAISS (fast), PGVector (production) chroma = Chroma.from_documents(chunks, embeddings, persist_directory="./db") faiss = FAISS.from_documents(chunks, embeddings) # Basic retriever retriever = chroma.as_retriever(search_type="mmr", search_kwargs={"k": 5})

Retriever Types

🔍 VectorStore

Basic similarity or MMR search. MMR balances relevance with diversity.

🔄 MultiQuery

LLM generates query variants, retrieves for all, deduplicates results.

🏷️ SelfQuery

Extracts metadata filters from natural language: "laptops under $500" → semantic + price filter.

🔀 Ensemble

Combines BM25 (keyword) + vector (semantic). Catches both exact IDs and paraphrases.

👪 ParentDocument

Small 400-char chunks for search, returns parent 2000-char chunks for context.

RunnableWithMessageHistory (Modern Way)

# Step 1: Base chain (no memory baked in) chain = prompt | ChatOpenAI(model="gpt-4o") | StrOutputParser() # Step 2: Session store (prod: Redis/PostgreSQL) sessions: dict[str, InMemoryChatMessageHistory] = {} def get_history(session_id): return sessions.setdefault(session_id, InMemoryChatMessageHistory()) # Step 3: Wrap with history agent = RunnableWithMessageHistory(chain, get_history, input_messages_key="question", history_messages_key="chat_history") # Step 4: Invoke with session_id agent.invoke({"question": "Where's ORD-7788?"}, config={"configurable": {"session_id": "cust-123"}})
💡 Interview Insight

"RunnableWithMessageHistory separates chain logic from history. In prod: last 10 messages verbatim + rolling summary of older ones via GPT-4o-mini. Session ID maps to support ticket ID. Max history window prevents prompt injection."

🔧 Tools & Tool Calling

An LLM can talk about tracking orders, but it can't actually track them. Tools bridge language → action.

Defining Tools

from langchain_core.tools import tool @tool def track_order(order_id: str) -> dict: """Look up current status and ETA for a customer order.""" return {"status": "shipped", "eta": "2026-04-08"} @tool def process_return(order_id: str, reason: str) -> dict: """Initiate a return for a customer order.""" return {"return_id": "RET-001", "status": "initiated"}

Binding & Executing Tools

llm_with_tools = llm.bind_tools([track_order, process_return]) response = llm_with_tools.invoke([HumanMessage("Where is ORD-7788?")]) # response.tool_calls → [{'name': 'track_order', 'args': {'order_id': 'ORD-7788'}}] # Execute tool → send ToolMessage back → LLM synthesizes final answer

AgentExecutor — The Tool Loop

agent = create_tool_calling_agent(llm, tools, prompt) executor = AgentExecutor(agent=agent, tools=tools, max_iterations=5, verbose=True) # Handles the full loop: LLM → tool call → result → LLM → ... result = executor.invoke({"input": "Return ORD-7788, item was defective"})
🎯 Interview Tip

"The LLM doesn't execute tools — our code does. Failure modes: wrong tool (fix descriptions), hallucinated params (validate with Pydantic), infinite loops (max_iterations). Add confirmation steps before destructive actions."

🔬 LangSmith — Observability

Without observability, you're flying blind with probabilistic systems.

❌ Without LangSmith

  • "Wrong answer"… but why?
  • "It's slow"… which step?
  • "New prompt better?"… no comparison
  • "Production-ready?"… vibes only

✅ With LangSmith

  • Full trace: prompt → LLM → tool → result
  • Latency per step: retrieval 200ms, LLM 1.2s
  • A/B: v1 72% correct → v2 89% correct
  • 100 test cases, scored automatically
