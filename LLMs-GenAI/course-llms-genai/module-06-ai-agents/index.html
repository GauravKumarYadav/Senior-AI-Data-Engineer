🤖 What Is an AI Agent?

An AI agent is an LLM that can observe, reason, plan, and act in a loop — using tools to interact with the real world until a goal is achieved.

👁️ Observe

Take in environment info — tool results, user messages, system state.

🧠 Reason

LLM thinks about what to do next based on observations.

📋 Plan

Break the goal into steps and select the right tool.

🔧 Act

Call a tool or return a final response. Loop back if not done.

Single-Agent Tool Loop

# The simplest agent: one LLM, tools, a loop def run_agent(user_message: str) -> str: messages = [system_msg, {"role": "user", "content": user_message}] while True: response = client.chat.completions.create( model="gpt-4o", messages=messages, tools=tools ) msg = response.choices[0].message if not msg.tool_calls: # No tools → done! return msg.content messages.append(msg) # Add assistant msg for tc in msg.tool_calls: result = execute_tool(tc.function.name, tc.function.arguments) messages.append({"role": "tool", "tool_call_id": tc.id, "content": json.dumps(result)})

"I follow the 'cognitive load' heuristic: if a single agent needs >10 tools or must reason across unrelated domains, it makes poor tool choices. At that point, I split into specialists with a supervisor pattern — one routing agent delegates to domain experts, each with 3-5 focused tools."

🔧 Tool Use: Function Calling & MCP

Function calling lets the LLM output structured JSON mapping to a tool. The LLM never executes — it just says which tool with what arguments. You execute.

# Anthropic function calling response = client.messages.create( model="claude-sonnet-4-20250514", tools=[{ "name": "lookup_order", "description": "Look up order details by ID", "input_schema": {"type": "object", "properties": { "order_id": {"type": "string"} }} }], messages=[{"role": "user", "content": "Where is order #5521?"}] ) # Response: tool_use block → {name: "lookup_order", input: {"order_id": "5521"}}

MCP — Model Context Protocol

An open standard (by Anthropic) that lets agents discover and use tools from external servers — like a USB port for AI tools.

📋 Planning & Reasoning

Complex goals must be broken into subtasks. The LLM acts as a planner.

# Plan-and-Execute with replanning plan = {"steps": [ {"id": 1, "action": "Look up customer profile", "tool": "lookup_customer"}, {"id": 2, "action": "Fetch all transactions", "tool": "get_transactions"}, {"id": 3, "action": "Identify duplicate charges", "tool": "analyze_duplicates"}, {"id": 4, "action": "Check payment gateway logs", "tool": "query_payment_logs"}, {"id": 5, "action": "Determine root cause", "tool": "llm_reasoning"}, {"id": 6, "action": "Process refund if confirmed", "tool": "process_refund"} ]}

Self-Reflection & Critique

# After each step, evaluate and potentially replan def execute_with_reflection(plan, tools): completed, remaining = [], list(plan) while remaining: step = remaining.pop(0) result = tools[step["tool"]](step["action"]) reflection = llm_reflect(step, result) if reflection["success"]: completed.append({"step": step, "result": result}) else: # REPLAN from current state remaining = llm_replan(completed, step, reflection) return synthesize_answer(completed)
💡 Interview Gold

"I implement plan-and-execute with dynamic replanning. After each step, a reflection module evaluates: did the tool return useful data? If a step fails — say, the payment API times out — the agent replans: maybe it can get the info from the transaction DB instead. I set a max step count (10-15) to prevent infinite loops, and implement backtracking — revert to last known good state. This is where LangGraph's checkpointing shines."

🧠 Agent Memory

Agents need different types of memory for different purposes. It's more than just chat history.

⚡ Short-Term

Current conversation buffer. Last N messages with sliding window + summarization. Cleared each session.

💾 Long-Term

Vector store of past interactions. Semantic search. Survives across sessions. "Have we helped this user before?"

📖 Episodic

Specific experiences with outcomes. "Last time this approach worked." Task traces with success/fail metadata.

📚 Semantic

Domain facts and policies. "Refund policy is 30 days." Retrieved via similarity search from knowledge base.

⚙️ Procedural

Learned workflow rules. "For international returns, always check customs first." Updated as agent discovers better patterns.

# Multi-tier memory system class AgentMemory: conversation_buffer: list[dict] # Short-term: last 20 msgs vector_store: ChromaDB # Long-term: semantic search episodes: list[dict] # Episodic: past experiences def add_message(self, role, content): self.conversation_buffer.append({...}) # Sliding window: summarize old msgs before discarding if len(self.conversation_buffer) > self.max_buffer_size: summary = self._summarize(self.conversation_buffer[:5]) self.conversation_buffer = [{"role": "system", "content": f"Earlier: {summary}"}] + self.conversation_buffer[5:] def recall_similar(self, task, k=3): return self.vector_store.query(query_texts=[task], n_results=k)
💡 Interview Gold

"I use a tiered memory architecture. Short-term: conversation buffer with last 15-20 messages, summarizing older ones. Long-term: vector store — every completed interaction gets embedded. Episodic: task traces with outcomes — 'last time a double-charge investigation failed because we didn't check payment logs early.' The most overlooked is procedural memory — learned rules injected into system prompts for relevant tasks."

⚔️ Multi-Agent Frameworks Compared

🔗 LangGraph

Graph-based. Maximum control. Built-in persistence, HITL, streaming. Steep learning curve. Best for production.

🚢 CrewAI

Role-based crews. High-level, easy setup. Define agents by role & backstory. Best for rapid prototyping.

💬 AutoGen

Conversation-based. Agents talk to each other. UserProxyAgent can execute code. Best for research & code gen.

🐝 OpenAI Swarm

Lightweight handoff-based. Experimental. Minimal state. Best for simple routing demos.

💡 Interview Gold

"LangGraph for production — explicit control, persistence, HITL. CrewAI for rapid prototyping — role-based agents in 20 lines. AutoGen for code gen & research — agents converse naturally, execute code in sandbox. Swarm is experimental — good for understanding handoffs, not production. For ShopAssist in production, LangGraph wins: deterministic routing, approval workflows, crash recovery."

📊 Agent Evaluation

Measuring agent quality across five dimensions — "does it work?" is only the beginning.

✅ Completion Rate

Did the agent finish the task? Target: >85% for production.

🎯 Tool-Use Accuracy

Precision: right tools only? Recall: all needed tools called? Unnecessary calls waste tokens.

⚡ Efficiency

Steps taken, total tokens, latency. A 3-step solution beats a 10-step one for the same task.

🛡️ Safety

Zero unauthorized actions. Log every tool call, audit for boundary violations.

📝 Answer Quality

Factual accuracy, format adherence, helpfulness of the final response.

# Agent evaluation harness def evaluate_agent(agent, test_cases): results = [] for case in test_cases: result = agent.run(case["input"]) eval_result = AgentEvalResult( completed=result.completed, correct=judge_correctness(result.output, case["expected"]), tools_called=result.tool_log, expected_tools=case["expected_tools"], ) results.append(eval_result) return { "completion_rate": sum(r.completed for r in results) / len(results), "accuracy": sum(r.correct for r in results) / len(results), "safety_violations": sum(not r.is_safe for r in results), }
💡 Interview Gold

"I evaluate across five dimensions. Task completion (>85% for prod). Tool-use accuracy — precision and recall. Efficiency — fewer steps = better. Safety — zero unauthorized actions is non-negotiable. Answer quality — correctness and format. I build a test suite of ~50 representative tasks with expected tool sequences, run it on every agent change like a regression test suite."

🛡️ Agent Safety & Reliability

Production agents need defense-in-depth. Five safety layers, from the inside out.

1. 📦 Sandboxing

Code execution in Docker/E2B containers. No filesystem, no network unless whitelisted.

2. ⏱️ Rate Limiting

Max 5 tool calls/minute, max 15 steps/task. Prevents runaway loops and cost explosions.

3. 🔑 Permissions

Role-based: read-only agent vs. read-write agent. Each agent only calls authorized tools.

4. 🙋 Human-in-the-Loop

Require approval for high-risk actions: refunds >$100, account deletions, data exports.

5. 📋 Audit Logging

Every tool call, every decision → immutable audit log. Full replay capability.

# Defense-in-depth: all layers combined async def safe_tool_call(tool_name, args, agent_role, rate_limiter): try: rate_limiter.check_rate() # Layer 1: Rate limit rate_limiter.check_steps() # Prevent infinite loops check_permission(agent_role, tool_name) # Layer 2: Permission await maybe_require_approval(tool_name, args) # Layer 3: HITL result = await execute_tool(tool_name, args) return {"status": "success", "result": result} except RateLimitExceeded: return {"status": "rate_limited"} except PermissionDenied as e: return {"status": "denied", "message": str(e)} except ActionDenied: return {"status": "human_denied"}
💡 Interview Gold

"Defense-in-depth with five layers: Sandboxing (Docker/E2B), Rate limiting (prevent cost explosions), Permissions (role-based tool access), HITL (approval for high-risk actions), Audit logging (immutable replay). The key principle is graceful degradation: if any check fails, the agent gets a structured error and adapts its plan — it doesn't crash."

Agent Design Checklist

🏛️ Architecture

Single-agent for simple, multi-agent (supervisor) for complex. Don't over-engineer.

🔧 Tools

Well-defined JSON schemas. MCP for reusability across agents.

📋 Planning

Task decomposition with replanning on failure. Max step limits.

🧠 Memory

Short-term buffer + long-term vector store + episodic recall.

🛡️ Safety

Sandboxing → Rate limits → Permissions → HITL → Audit logs.

📊 Evaluation

Regression suite: completion rate, tool accuracy, safety metrics.

💡 Interview Gold

"I start with task analysis: what tools? What failure modes? Build a single-agent prototype with 5-6 tools, test on 50 queries, measure completion rate. If tool accuracy drops below 80%, split into supervisor + specialists. Add memory incrementally: buffer → vector store → episodic. Safety from day one. Build an evaluation harness. Deploy behind a feature flag for gradual rollout."
