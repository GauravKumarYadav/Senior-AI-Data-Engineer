📝 Text Preprocessing — Tokenization

Raw text is messy. Models need structured numerical input. Preprocessing bridges that gap.

Raw Text → Cleaned → Tokenized "I can't BELIEVE it's "i cannot believe it is ["i", "cannot", "believe", only $9.99!!!" only 9.99" "it", "is", "only", "9.99"]

Tokenization Strategies

📖 Word-Level

  • ✅ Simple, interpretable
  • ❌ Huge vocabulary, can't handle OOV words

🔤 Character-Level

  • ✅ Tiny vocabulary (26 letters + special)
  • ❌ Sequences too long, loses word meaning

✂️ Subword (The Sweet Spot)

  • ✅ Handles any word, 30K-50K vocab
  • ✅ Rare words decomposed, common stay whole

Stemming vs Lemmatization

✂️ Stemming

  • Crude, rule-based, fast
  • "studies" → "studi" ❌ not a word!
  • "flies" → "fli" ❌

📚 Lemmatization

  • Dictionary-based, produces real words
  • "studies" → "study" ✅
  • "better" → "good" ✅
💡 Interview Insight

"Why did the industry move to subword tokenization?" — Word-level creates massive vocabularies (100K+) and can't handle unseen words. Character-level makes sequences too long. Subword (BPE, WordPiece) hits the sweet spot: ~30K-50K vocab, handles any word including code/URLs/multilingual, and preserves meaning for common words.

🧮 Word Embeddings — Word2Vec, GloVe & FastText

One-Hot (Sparse)

  • "cat" → [0, 0, 1, 0, …, 0]
  • dim = vocab_size (50,000)
  • cat·dog = 0 (no similarity!)

Dense Embedding

  • "cat" → [0.21, -0.35, 0.72, …]
  • dim = 100-300
  • cat·dog = 0.95 (similar!)

Word2Vec: Skip-gram vs CBOW

Skip-gram

  • Given: center word → Predict: context
  • Better for rare words
  • More commonly used

CBOW

  • Given: context words → Predict: center
  • Better for frequent words
  • Faster training
from gensim.models import Word2Vec sentences = [["the", "cat", "sat"], ["the", "dog", "ran"]] model = Word2Vec(sentences, vector_size=100, window=5, min_count=1, sg=1) # Semantic operations! # king - man + woman ≈ queen result = model.wv.most_similar(positive=['king', 'woman'], negative=['man'])

GloVe & FastText

🌐 GloVe

Factorizes the global co-occurrence matrix. Learns vectors whose dot product = log probability of co-occurrence. Unlike Word2Vec's local context windows.

⚡ FastText

Represents each word as a bag of character n-grams. "where" → <wh, whe, her, ere, re>. Can generate vectors for OOV words!

Static vs Contextual Embeddings

Static (Word2Vec, GloVe)

  • "bank" → same vector always
  • One vector per word type

Contextual (BERT, GPT)

  • "river bank" ≠ "bank account"
  • One vector per word token in context
💡 Interview Insight

"What's the fundamental limitation of Word2Vec/GloVe?" — They produce static embeddings — one vector per word regardless of context. Contextual embeddings (BERT, GPT) solve this by computing embeddings as a function of surrounding context. Static embeddings are still useful for fast similarity lookups and as initialization for neural networks.

📐 Sentence & Document Embeddings

📊 Average Word Embeddings

Simple but weak. Loses word order — "dog bites man" = "man bites dog" ❌

🏷️ [CLS] from BERT

Common but not ideal. [CLS] wasn't trained for sentence similarity ❌

🎯 Sentence-BERT

Purpose-built for sentence similarity. Semantically meaningful distance ✅

from sentence_transformers import SentenceTransformer from sklearn.metrics.pairwise import cosine_similarity model = SentenceTransformer('all-MiniLM-L6-v2') # Fast, good quality sentences = [ "The cat sat on the mat", "A kitten rested on a rug", "The stock market crashed today" ] embeddings = model.encode(sentences) # shape: (3, 384) sim_matrix = cosine_similarity(embeddings) # cat/kitten: 0.72 ✅ kitten/stock: 0.05 ✅

Bi-Encoders vs Cross-Encoders

⚡ Bi-Encoder

  • ✅ Encode once, compare many
  • ✅ Pre-compute & index (FAISS)
  • ✅ 1M comparisons in ~seconds
  • Use for: retrieval (first stage)

🎯 Cross-Encoder

  • ✅ More accurate (sees both at once)
  • ❌ Must run for EVERY pair
  • ❌ 1M comparisons in ~hours
  • Use for: re-ranking (second stage)
💡 Interview Insight

"How would you build a semantic search system?" — Two-stage: (1) Bi-encoder encodes all documents offline → FAISS index → ANN search → top-K candidates. (2) Cross-encoder re-ranks the top-K for higher accuracy. Speed of bi-encoders + accuracy of cross-encoders. This is exactly how modern RAG systems work.

🏗️ Transformer NLP — BERT vs GPT

🔍 BERT (Encoder)

  • Bidirectional — sees full context
  • Pre-train: Masked Language Model
  • ✅ Classification, NER, QA
  • ✅ Sentence embeddings
  • base: 12 layers, 768 hidden, 110M params

✍️ GPT (Decoder)

  • Causal — sees only left context
  • Pre-train: Next Token Prediction
  • ✅ Text generation, chatbots
  • ✅ Summarization, translation
  • Causal mask prevents future-peeking
# BERT: Get contextual embeddings from transformers import AutoTokenizer, AutoModel import torch tokenizer = AutoTokenizer.from_pretrained('bert-base-uncased') model = AutoModel.from_pretrained('bert-base-uncased') inputs = tokenizer("Machine learning is fascinating", return_tensors='pt') with torch.no_grad(): outputs = model(**inputs) # outputs.last_hidden_state: (batch, seq_len, 768) cls_embedding = outputs.last_hidden_state[:, 0, :] # [CLS] token

Encoder-Decoder: T5 & BART

🔄 Text-to-Text

T5 treats every task as text-to-text: "translate English to French: Hello" → "Bonjour" | "classify: I love this movie" → "positive" | "summarize: ..." → "...". Encoder understands input bidirectionally, decoder generates output autoregressively via cross-attention.

💡 Interview Insight

"When would you use BERT vs GPT?" — BERT for understanding: classification, NER, embeddings for search. GPT for generation: chatbots, code, summaries. Modern GPT-scale models handle understanding too via prompting, but BERT-family models are far more efficient for embedding and classification at scale.

🛠️ NLP Tasks — Classification, NER, QA

# Zero-shot classification — no training needed! from transformers import pipeline classifier = pipeline("zero-shot-classification", model="facebook/bart-large-mnli") result = classifier( "The new iPhone has an incredible camera", candidate_labels=["technology", "sports", "politics"] ) # → technology, 0.97 # Named Entity Recognition ner = pipeline("ner", model="dslim/bert-base-NER", aggregation_strategy="simple") results = ner("Apple CEO Tim Cook announced the new iPhone in San Francisco.") # Apple → ORG (0.99), Tim Cook → PER (0.99), San Francisco → LOC (0.99)

Extractive vs Generative QA

📋 Extractive (BERT)

  • Answer = span from the context
  • ✅ Grounded, no hallucination
  • ❌ Answer must be in text
  • Great for RAG systems

✨ Generative (GPT/T5)

  • Answer = generated from model
  • ✅ Flexible, can synthesize
  • ❌ May hallucinate
  • Modern chatbot approach
# Extractive QA qa = pipeline("question-answering", model="deepset/roberta-base-squad2") result = qa( question="What is the capital of France?", context="France is a country in Europe. Paris is the capital of France." ) # Answer: Paris (confidence: 0.9876)
💡 Interview Insight

"How do you choose between extractive and generative QA?" — Extractive when answers must be grounded in a document — no hallucination, easy to verify, great for RAG. Generative when you need synthesized answers or multi-hop reasoning. For production: extractive QA + RAG is safer. For user-facing chatbots: generative QA with citation.

AutoModel Pattern

from transformers import AutoTokenizer, AutoModelForSequenceClassification tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased") model = AutoModelForSequenceClassification.from_pretrained( "bert-base-uncased", num_labels=3 ) inputs = tokenizer("I love this movie", return_tensors="pt", truncation=True) outputs = model(**inputs) predicted_class = outputs.logits.argmax(dim=-1).item() probabilities = torch.softmax(outputs.logits, dim=-1)
💡 Interview Insight

"How does the tokenizer handle variable-length inputs?" — padding=True pads shorter sequences with [PAD] tokens. truncation=True cuts at max_length. An attention_mask marks real tokens (1) vs padding (0). Use padding='longest' instead of padding='max_length' to save computation.

🔄 End-to-End Fine-Tuning Pipeline

from datasets import load_dataset from transformers import ( AutoTokenizer, AutoModelForSequenceClassification, TrainingArguments, Trainer ) from sklearn.metrics import accuracy_score, f1_score # 1. Load dataset & tokenizer dataset = load_dataset("imdb") tokenizer = AutoTokenizer.from_pretrained("distilbert-base-uncased") def preprocess(examples): return tokenizer(examples['text'], truncation=True, padding='max_length', max_length=256) tokenized = dataset.map(preprocess, batched=True, num_proc=4) tokenized = tokenized.rename_column("label", "labels") tokenized.set_format("torch", columns=["input_ids", "attention_mask", "labels"]) # 2. Model + Metrics model = AutoModelForSequenceClassification.from_pretrained("distilbert-base-uncased", num_labels=2) def compute_metrics(eval_pred): preds = np.argmax(eval_pred.predictions, axis=-1) return {'accuracy': accuracy_score(eval_pred.label_ids, preds), 'f1': f1_score(eval_pred.label_ids, preds, average='macro')} # 3. Train! trainer = Trainer( model=model, args=TrainingArguments( output_dir="./results", num_train_epochs=3, learning_rate=2e-5, per_device_train_batch_size=16, evaluation_strategy="epoch", save_strategy="epoch", load_best_model_at_end=True, metric_for_best_model="f1", fp16=True, warmup_ratio=0.1, ), train_dataset=tokenized["train"], eval_dataset=tokenized["test"], compute_metrics=compute_metrics, ) trainer.train()

💡 Interview Insight

"Walk me through fine-tuning a pre-trained model." — (1) Choose model (DistilBERT for speed, RoBERTa for accuracy). (2) Add classification head. (3) Tokenize with model's tokenizer. (4) Small LR (2e-5) with warmup to avoid catastrophic forgetting. (5) Train 2-5 epochs with early stopping on validation F1. (6) fp16 for 2x speed. (7) Evaluate on held-out test set.

Token Counting Matters

💡 Interview Insight

"How do you handle documents longer than the context window?" — (1) Chunking: split into overlapping chunks (512 tokens, 50-token overlap), embed each, retrieve relevant chunks. (2) Hierarchical: summarize sections, embed summaries. (3) Long-context models (128K+). (4) Map-reduce: process chunks independently, aggregate. Chunking with overlap is the RAG standard — chunk size (256-1024 tokens) is a critical hyperparameter.
