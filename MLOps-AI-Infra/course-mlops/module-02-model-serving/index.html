рҹ“Ұ Batch Inference

Process large volumes on a schedule вҖ” hourly, daily, weekly. Predictions are precomputed and stored. Use for: recommendation engines, credit scoring, churn prediction.

вҡЎ Real-Time API

REST or gRPC endpoint. Each request gets an immediate prediction (ms-level). Use for: fraud detection, search ranking, dynamic pricing, chatbots.

рҹҢҠ Streaming Inference

Process events as they arrive via Kafka/Kinesis. Model sits in the stream pipeline. Use for: real-time fraud alerts, IoT anomaly detection.

рҹ“ұ Embedded / Edge

Model compiled and shipped inside the client вҖ” mobile, browser (ONNX.js), IoT. Use for: offline apps, ultra-low latency (<1ms), privacy-sensitive scenarios.

REST vs gRPC Trade-offs

рҹҢҗ REST (HTTP/JSON)

  • вң… Native browser support
  • вң… Easy debugging (curl, Postman)
  • вҡ пёҸ Higher latency (text serialization)
  • вҡ пёҸ Limited streaming (SSE, WebSockets)
  • Best for: External APIs, simple cases

вҡЎ gRPC (HTTP/2 + Protobuf)

  • вң… Lower latency (binary serialization)
  • вң… Native bidirectional streaming
  • вҡ пёҸ Requires grpc-web proxy for browsers
  • вҡ пёҸ Harder debugging (need grpcurl)
  • Best for: Internal microservices, perf

рҹ’Ў Interview Insight

Q: "Your team has a recommendation system. When would you choose batch vs. real-time serving?"

Use batch for the homepage "top picks" вҖ” precompute nightly for all users, store in Redis/DynamoDB, serve with sub-ms lookup. Use real-time for "similar items" on the product page вҖ” the context (which product the user is viewing) isn't known until request time. A hybrid approach is common: batch for personalization features, real-time for contextual re-ranking.

TorchServe (PyTorch Native)

TorchServe is the official serving solution for PyTorch models. It uses a handler-based architecture.

рҹ’Ў Interview Insight

Q: "You need to serve a 70B parameter LLM with low latency. What framework and hardware?"

I'd use vLLM with AWQ INT4 quantization. A 70B model in FP16 needs ~140GB VRAM. With INT4: ~35GB + overhead вүҲ 40GB. I'd use 2Г— A100 80GB GPUs with tensor parallelism (--tensor-parallel-size 2). vLLM's PagedAttention and continuous batching maximize throughput. For production, front it with KServe on Kubernetes for autoscaling and canary deployments.

Shadow deployments are the safest way to test a new model in production. The new model receives real traffic but its responses are discarded вҖ” only logged for offline comparison.

рҹ’Ў Interview Insight

Q: "How do you decide between A/B testing and canary deployment?"

They serve different purposes. Canary is a risk mitigation strategy вҖ” validating a model won't break things (latency, errors, crashes). A/B testing is a measurement strategy вҖ” running a controlled experiment to measure business impact (CTR, conversion, revenue). In practice: canary first (validate safety at 5%), then A/B test (50/50 for statistical power), then full rollout. Shadow deployment comes before both.

Q: "A 7B parameter model in FP16 вҖ” how much GPU memory? Can you run it on a single RTX 4090?"

Q: "Your model's accuracy dropped 5% over the past month, but no code changes were made. What's your debugging process?"

Systematic approach: (1) Check data drift first вҖ” compare recent input distributions vs training set using PSI. If PSI > 0.2, data drift is likely. (2) Inspect prediction distribution вҖ” plot recent vs historical histograms. (3) Check upstream data issues вҖ” broken pipeline? Null features? Feature store staleness? (4) Evaluate on recent labeled data вҖ” compute per-segment metrics. (5) Remediation вҖ” data drift: retrain. Concept drift: may need new features or architecture. Set up automated monitoring.

Interview Scenario: Fraud Detection System

Q: "Design a real-time fraud detection system that scores credit card transactions with <100ms latency at 10,000 TPS."

1пёҸвғЈ Serving Pattern

Real-time API. Cannot be batch (need instant decision). Cannot be embedded (model updates frequently, needs centralized features).

2пёҸвғЈ Architecture

API Gateway вҶ’ Feature enrichment from Feature Store (user patterns, merchant risk) вҶ’ ONNX Runtime on GPU with dynamic batching вҶ’ fraud score + decision.

3пёҸвғЈ Scaling

10K TPS with batch-32 = ~312 calls/sec. Budget: 30ms features, 50ms inference, 20ms overhead. Need ~4 GPU pods with HPA on queue depth.

4пёҸвғЈ Deployment

Shadow first, then canary (5% вҶ’ 25% вҶ’ 100%). Auto-rollback if error rate >0.5% or P99 >200ms. Monitor prediction distribution and false positive rate.

Interview Scenario: LLM Chat Application

Q: "Design the serving infrastructure for an internal chatbot powered by a 70B parameter LLM."

рҹ§® Model Optimization

70B FP16 = 140GB вҶ’ INT4 AWQ вҶ’ ~35GB. Use 2Г— A100 80GB with tensor parallelism for headroom and fast generation.

рҹҡҖ Framework

vLLM with continuous batching, PagedAttention, prefix caching for shared system prompts. OpenAI-compatible API.

рҹ“Ҳ Scaling

Each pod handles ~20-30 concurrent requests. For 200 concurrent users вҶ’ 7-10 pods. Redis semantic cache for common questions.

рҹӣЎпёҸ Guardrails

Input: PII detection, prompt injection filtering. Output: toxicity detection, hallucination checks. Rate limiting per user.

рҹ’Ў Interview Insight

Q: "What's the difference between tensor parallelism and pipeline parallelism?"

Tensor parallelism splits individual layers across GPUs вҖ” each GPU computes a slice of every layer. This reduces per-token latency (all GPUs work on same token simultaneously). Best for latency-sensitive serving. Pipeline parallelism assigns different layers to different GPUs and processes micro-batches in pipeline fashion. Maximizes throughput but adds latency. For LLM serving: tensor parallelism. For training: pipeline parallelism.
