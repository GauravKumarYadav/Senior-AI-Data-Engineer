The Multi-Agent System

Meet the four agents, understand shared state, and see how the orchestrator conducts the orchestra.

🎭 Meet the Cast of Characters

Think of this system like a hospital emergency room. A patient walks in (the user's question). They don't go straight to surgery — first, a triage nurse decides what they need.

🔮
Query Agent Triage
The first responder. Looks at every incoming message and decides: Is this chitchat? A question? A math problem? A request to list documents? Then rewrites the query to be more searchable.
🔍
Retrieval Agent Search Team
The research librarian. Searches both the client's private documents AND the global shared documents. Uses BM25 + vector search, fuses with RRF, reranks with a cross-encoder.
🔧
Tool Agent Specialist
The on-call specialist. Handles specific tasks like calculations (SafeCalculator) and datetime queries. Uses a shunting-yard algorithm — never eval().
✍️
Synthesis Agent Doctor
The doctor who delivers the final answer. Takes all the evidence (retrieved docs, tool results, conversation context) and generates a cited, formatted response.
💡
Aha! Agents ≠ Autonomous AI
In this context, "agent" doesn't mean a sentient AI. It means a specialized function with a clear job that receives state, processes it, and returns updated state. Like a factory assembly line — each station does one thing well.

📋 The Shared Clipboard: AgentState

All four agents read from and write to one shared object — AgentState. Think of it like a medical chart that gets passed from triage to lab to doctor. Each adds their notes.

Python — state.py
class AgentState(TypedDict, total=False):
    # Input
    message: str
    client_id: str
    conversation_id: str

    # Memory
    conversation_summary: str
    recent_messages: List[Dict]

    # Query Processing
    intent: str
    needs_retrieval: bool
    rewritten_query: str

    # Retrieval
    retrieved_chunks: List[RetrievalHit]

    # Tools
    tool_name: Optional[str]
    tool_result: Optional[str]

    # Output
    response: str
    sources: List[Dict]
Plain English
The shared clipboard has these sections:
 
📥 INPUT — What the user said, who they are
→ The user's message
→ Which client they belong to
→ Which conversation this is
 
🧠 MEMORY — What we remember
→ Summary of earlier conversation
→ Last few messages (sliding window)
 
🔮 QUERY AGENT WRITES:
→ Intent: "question", "chitchat", "tool", etc.
→ Does it need document search? true/false
→ The improved version of the query
 
🔍 RETRIEVAL AGENT WRITES:
→ The document chunks it found
 
🔧 TOOL AGENT WRITES:
→ Which tool was used + its result
 
✍️ SYNTHESIS AGENT WRITES:
→ The final answer + cited sources
🔑
Why total=False?
The total=False in TypedDict makes every field optional. This is crucial because each agent only updates its own fields. The Query Agent doesn't write retrieved_chunks, and the Retrieval Agent doesn't write intent. Partial updates to shared state.

🎼 The Orchestrator: LangGraph in Action

The Orchestrator is the conductor of this orchestra. It decides the order agents play in, using a StateGraph — a directed graph where nodes are agents and edges are transitions.

Python — orchestrator.py
def _build_graph(self) -> StateGraph:
    graph = StateGraph(AgentState)

    # Add nodes (agents)
    graph.add_node("query", self._query_node)
    graph.add_node("retrieval", self._retrieval_node)
    graph.add_node("tool", self._tool_node)
    graph.add_node("synthesis", self._synthesis_node)
    graph.add_node("document_list",
                   self._document_list_node)

    # Entry point
    graph.set_entry_point("query")

    # Conditional routing
    graph.add_conditional_edges(
        "query",
        self._route_after_query,
        {"tool": "tool",
         "retrieval": "retrieval",
         "synthesis": "synthesis",
         "document_list": "document_list"})

    graph.add_edge("tool", "synthesis")
    graph.add_edge("retrieval", "synthesis")
    graph.add_edge("synthesis", END)

    return graph.compile()

🔀 The Router: How Decisions Are Made

Python — orchestrator.py
def _route_after_query(self, state):
    # Check for tool usage
    if state.get("tool_name"):
        return "tool"

    intent = state.get("intent", "question")

    if intent == "chitchat":
        return "synthesis"
    if intent == "document_list":
        return "document_list"
    if state.get("needs_retrieval", True):
        return "retrieval"

    return "synthesis"
Plain English
After the Query Agent has analyzed the message:
 
🔧 First check: Did it detect a tool?
→ Yes? Send to Tool Agent (math, datetime)
 
Read what the Query Agent classified the intent as
 
💬 Chitchat? Skip retrieval, go straight to answer
📋 Wants doc list? Go to the file clerk
🔍 Needs documents? Go to Retrieval Agent
 
🤷 Anything else? Default to synthesis
Simple, predictable, debuggable!

🔮 Deep Dive: The Query Agent

The Query Agent does three things in sequence: classify, detect tools, rewrite.

Python — query_agent.py
async def process(self, state):
    message = state.get("message", "")
    summary = state.get("conversation_summary")

    # Step 1: Classify intent via LLM
    intent, needs_retrieval = \
        await self._classify_intent(
            message, summary)

    # Step 2: Detect tools (overrides LLM)
    tool_name, tool_params = \
        self._detect_tool(message)
    if tool_name:
        intent = "tool"
        needs_retrieval = False

    # Step 3: Rewrite if retrieval needed
    rewritten = message
    if needs_retrieval:
        rewritten = await self._rewrite_query(
            message, summary)

    return {**state,
        "intent": intent,
        "needs_retrieval": needs_retrieval,
        "tool_name": tool_name,
        "rewritten_query": rewritten}
Plain English
When a message arrives at Query Agent:
Read what the user said + any conversation history
 
📋 Step 1: Ask the LLM "what kind of message is this?"
→ Returns: intent (chitchat/question/tool/etc.)
→ Returns: does it need document search? (true/false)
 
🔧 Step 2: Check for tools using regex patterns
→ Found a math expression? Override to "tool"
→ Pattern matching is faster and more reliable than LLM for this
 
✏️ Step 3: If search is needed, improve the query
→ "Can you tell me about the refund thing we discussed?"
→ Becomes: "refund policy details" (standalone, searchable)
 
Write all findings to the shared clipboard
The {**state, ...} pattern merges new data into existing state
💡
Aha! Two-Layer Tool Detection
The system uses pattern matching first (fast, reliable regex for "15% of 200"), then LLM as fallback. This is a common production pattern: cheap/fast first, expensive/smart second. The pattern match can override the LLM's classification because regex is more deterministic for structured inputs.

🔭 Observability: Tracing Every Step

Every agent is wrapped with @traceable — a LangSmith decorator that records inputs, outputs, timing, and token usage.

Python — orchestrator.py
@traceable(name="agent.query")
async def _query_node(self, state):
    return await self.query_agent.process(state)

@traceable(name="agent.retrieval")
async def _retrieval_node(self, state):
    return await self.retrieval_agent.process(state)

@traceable(name="agent.synthesis")
async def _synthesis_node(self, state):
    return await self.synthesis_agent.process(state)
Plain English
Each agent call is wrapped with a tracer:
 
🏷️ "agent.query" — traces the entire Query Agent
→ LangSmith records: inputs, outputs, duration, tokens
 
🏷️ "agent.retrieval" — traces document search
→ Shows which collections were searched, how many results
 
🏷️ "agent.synthesis" — traces response generation
→ Shows the full prompt sent to the LLM, the response
 
When something goes wrong, you can see EXACTLY
which step failed and why — like a flight recorder.

🎤 Interview Questions: Agent Reasoning & Planning

Q21 How do you implement a "reflection" step where the agent evaluates whether retrieved context is sufficient? ▼

In our system, reflection is implicit in the Synthesis Agent:

  • If retrieved_chunks is empty, the prompt explicitly states "No relevant documents found."
  • The system prompt instructs: "If the context doesn't contain enough information, say so clearly."
  • Source attribution ([Source: filename]) forces the agent to verify claims against evidence

For explicit reflection: Add a "reflection" node in LangGraph between retrieval and synthesis that evaluates relevance scores from RetrievalHit.score and rerank_score metadata.

Q22 How do you build an agent that decides at runtime whether to retrieve, call an API, query SQL, or respond from parametric memory? ▼

This is exactly what our system does (minus SQL). The QueryAgent.process() makes runtime decisions:

intent, needs_retrieval = await self._classify_intent(message)
tool_name, tool_params = self._detect_tool(message)
if tool_name:
    intent = "tool"
    needs_retrieval = False

The orchestrator then routes:

Q23 What is chain-of-thought prompting in an agentic context, and how do you ensure the reasoning trace stays grounded? ▼

Grounding is ensured through our agent pipeline structure:

  1. Structured reasoning: Instead of one big CoT prompt, reasoning is decomposed — QueryAgent outputs JSON ({"intent": "...", "needs_retrieval": ...}), RetrievalAgent adds source metadata, SynthesisAgent cites sources
  2. LangSmith tracing: Every step traced with @traceable — full reasoning trail without embedding it in prompts
  3. Low temperature classification: temperature=0.1 for deterministic intent classification
  4. Structured prompt layout: [conversation context] + [document context] + [tool results] + [question]
Q24 How do you prevent an agent from entering infinite retrieval loops? ▼

Our system prevents this through several mechanisms:

  1. max_steps = 10 — limits total graph traversals
  2. DAG structure (no cycles): The LangGraph is a directed acyclic graph: query → [retrieval|tool|doc_list] → synthesis → END. Infinite loops are structurally impossible.
  3. Intent-based routing: _route_after_query() always terminates at one of: "tool", "retrieval", "synthesis", or "document_list". No "retry" option.
  4. Exception handling: The orchestrator wraps execution in try/except, returning a fallback response on error
Q25 Explain "tool selection" in agentic RAG — how does the LLM decide which tool to invoke? ▼

Tool selection happens in two layers:

Layer 1 — Pattern matching (fast, reliable):

def _detect_tool(self, message):
    math_pattern = r'[\d\.\s\+\-\*\/\(\)\^%]+'
    percent_pattern = r'(\d+)\s*%\s*of\s*(\d+)'
    datetime_keywords = ["date", "time", "today"]

Layer 2 — LLM classification (fallback): The intent classifier can output "tool" intent, but explicit pattern matching overrides for reliability.

Why two layers: Pattern matching is O(n) regex with no LLM call needed. SafeCalculator uses a shunting-yard algorithm (no eval()!), rejecting suspicious input like import, exec, __.

Q26 How do you handle conflicting information from multiple sources? ▼

Our system retrieves from two collections (client + global) and handles conflicts:

  1. Client priority: _merge_results() adds client results first. Client-specific docs take precedence.
  2. Deduplication: seen_ids = set() prevents the same chunk appearing twice
  3. Source attribution: Citations ([Source: filename]) let users see which source each claim comes from
  4. Multi-source context: The synthesis agent can present conflicting views: "According to [Source: policy-2024], X. However, [Source: policy-2023] states Y."
Q27 What techniques give the agent "memory" of prior interactions in multi-turn RAG? ▼

Our memory system has multiple layers:

  1. Session Buffer: Redis lists with 24h TTL, circuit breaker fallback to in-memory
  2. Auto-Summarization: When tokens exceed 4000, older messages are summarized by LLM and stored in Redis
  3. Sliding Window: 10 recent messages kept in full; older ones compressed into summary
  4. Memory in agent state: conversation_summary + recent_messages flow through all agents
  5. Query rewriting with context: The QueryAgent receives conversation summary to resolve references like "tell me more about that"

2. Why does AgentState use total=False in its TypedDict?
It's a Python bug workaround for LangGraph compatibility
Each agent only writes its own fields — partial state updates need all fields to be optional
It makes the state object immutable for thread safety
✅ With total=False, an agent can return {**state, "intent": "question"} without providing all other fields. The Query Agent doesn't set retrieved_chunks, and the Retrieval Agent doesn't set intent. Each agent is responsible only for its own fields — clean separation of concerns.
