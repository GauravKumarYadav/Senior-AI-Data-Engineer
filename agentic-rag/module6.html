✂️ Chunking: Cutting Knowledge Into Digestible Pieces

If you split a book by page count, you might cut a sentence in half. Our chunker is like a careful editor — it splits at headings and sentence boundaries, preserving meaning.

Python — config.py chunking settings
class ChunkingSettings(BaseModel):
    chunk_size: int = 1000
    chunk_overlap: int = 200
    respect_headings: bool = True
    respect_sentences: bool = True
    min_chunk_tokens: int = 50
    use_content_hash_ids: bool = True
Plain English
📏 chunk_size = 1000: ~1000 characters per chunk
🔗 overlap = 200: 200 chars shared between adjacent chunks — like overlapping puzzle pieces, ensuring no idea falls between the cracks
📑 respect_headings: Break at Markdown # headers. A "Refund Policy" section stays in one chunk.
✍️ respect_sentences: Never cut mid-sentence. Finds the nearest period, question mark, or exclamation.
🚫 min_chunk_tokens = 50: Drop tiny, meaningless chunks (e.g., a lone page number)
#️⃣ content_hash_ids: ID = hash of content. Same text → same ID → automatic dedup!
⚠️
Pitfall: Chunk Size Sweet Spot
Too small (200 chars): Chunks lack context — "it refers to the policy" (what policy?). Too large (5000 chars): Embedding averages out meaning — the chunk is "about everything and nothing." The 1000-char default with 200-char overlap is a well-tested balance.

Document-level Content-hash chunk IDs hashlib digest of chunk content → same text = same ID. ChromaDB upsert() is idempotent.
Passage-level Cross-document hash match Two different PDFs with the same paragraph → identical hash → stored once.
Retrieval-level seen_ids = set() _merge_results() deduplicates across client + global collections.
Diversity MMR selector Even if IDs differ, semantically duplicate chunks are penalized by Maximal Marginal Relevance.

🎤 Interview Questions: Data Pipeline

Q46 How do you design an ingestion pipeline that keeps the knowledge base fresh? ▼

Freshness mechanisms in our pipeline:

  1. Hot reload: Documents are immediately available after upload — no batch job or restart required.
  2. Content-hash IDs: Re-uploading a document generates the same IDs for unchanged chunks. ChromaDB's upsert() deduplicates automatically — only new/changed chunks are added.
  3. Dual-index sync: HybridSearch.add_documents() updates both BM25 and vector indices in a single call.
  4. Delete + re-add: Document updates delete old chunks by doc_id, then add new ones.
  5. Per-client isolation: Updating one client's documents doesn't affect others.
Q47 What pre-processing steps do you perform before embedding documents? ▼

Four-stage pipeline:

  1. Format conversion (Docling): PDF/DOCX/Images → clean Markdown with OCR support.
  2. Chunking: Heading-aware + sentence-boundary-respecting splits. 1000 chars with 200-char overlap. Min 50 tokens to filter noise.
  3. Metadata enrichment (ChunkMetadata): doc_id, client_id, source_filename, page_number, section_heading, embedding_fingerprint, detected_entities.
  4. Content-hash ID generation: Deterministic IDs via hashlib → idempotent upserts, automatic dedup.
Q48 How do you handle multi-modal data (tables, images, charts inside PDFs)? ▼

Docling handles multi-modal content:

  1. PDF tables: Extracted and converted to Markdown table format. The chunker's respect_headings keeps tables with their section context.
  2. Images in PDFs: OCR via PdfPipelineOptions + AcceleratorOptions. Supports PNG, JPG, TIFF, BMP, GIF, WEBP.
  3. Charts: Text (labels, legends, titles) extracted via OCR. Chart interpretation limited to extractable text.
  4. Vision-language model: Default LLM is Qwen3-VL-30B-Instruct — the "-VL" indicates future image understanding capabilities.
Q49 Describe your approach to document-level and passage-level deduplication. ▼

Document-level: Content-hash chunk IDs (hashlib) → re-uploading produces identical IDs → upsert() is idempotent.

Passage-level: Two chunks from different documents with identical text → same hash ID → stored once. At retrieval, _merge_results() uses seen_ids = set() to deduplicate across client and global collections.

Why it matters: Without dedup, the same passage consumes multiple top-K slots, reducing diversity. MMR further penalizes semantically similar chunks.

Q50 How do you detect and handle data drift in the knowledge base? ▼

Detection:

  1. Embedding fingerprinting: EmbeddingFingerprint detects model changes that cause drift between old and new embeddings.
  2. Retrieval quality monitoring: HybridSearchResult stats track candidate counts — a drop in candidates suggests index drift.
  3. LangSmith trends: Monitoring answer quality over time surfaces gradual degradation.

Handling:

  1. Timestamp metadata: uploaded_at enables recency-based filtering.
  2. Idempotent re-ingestion: Content-hash IDs make periodic re-ingestion safe.
  3. BM25 index rebuild: Full rebuild recalibrates term frequencies for the current corpus.
🧪 Module 6 Quiz
1. You upload the same PDF twice. What happens the second time?
Duplicate chunks are created — the collection doubles in size
Content-hash IDs are identical, so ChromaDB's upsert() silently deduplicates — no change in collection size
An error is thrown rejecting the duplicate upload
✅ Because use_content_hash_ids = True, chunk IDs are derived from the content itself via hashlib. Re-uploading the same document generates the same IDs, and ChromaDB's upsert() is idempotent — it simply overwrites with identical data.
2. Why does the chunker use 200 characters of overlap between adjacent chunks?
To make chunks larger for better embeddings
To increase the total number of chunks for better BM25
To ensure no idea falls between chunk boundaries — context at the edges is preserved in both adjacent chunks
✅ The 200-char overlap is like overlapping puzzle pieces. If an important concept spans the boundary between two chunks, the overlap ensures it appears (at least partially) in both chunks, so retrieval can find it regardless of which chunk matches the query.
