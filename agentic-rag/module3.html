The heart of RAG — how the system finds the right documents using BM25, vectors, fusion, and reranking.

🔍 Why Two Search Methods?

Imagine you're searching a library. You could search by exact words on the spine (keyword search) or by meaning and topic (semantic search). Each misses things the other catches.

💡
Aha! Why Not Just Use Vectors?
Vector search converts "What is policy XYZ-789?" into numbers. But "XYZ-789" is a rare code — its embedding doesn't capture its importance. BM25 scores it highly because rare terms get high weight (TF-IDF principle). That's why hybrid search outperforms either alone in real-world benchmarks.

⚖️ Reciprocal Rank Fusion (RRF)

RRF is like combining election results from two voting districts. Each search method "votes" for documents by ranking them. RRF converts ranks into scores and adds them up.

Python — hybrid_search.py
def _rrf_fusion(self, bm25_hits, vector_hits):
    scores = {}
    k = 60  # RRF constant

    for rank, hit in enumerate(bm25_hits):
        scores[hit.id] = scores.get(hit.id, 0)
        scores[hit.id] += bm25_weight * (
            1.0 / (k + rank + 1))

    for rank, hit in enumerate(vector_hits):
        scores[hit.id] = scores.get(hit.id, 0)
        scores[hit.id] += vector_weight * (
            1.0 / (k + rank + 1))

    # Normalize to 0-1 range
    # Convert to distance (lower=better)
Plain English
Combining two ranked lists into one:
k=60 is a smoothing constant (standard in RRF)
 
For each BM25 result (by rank):
→ Score = 0.4 × (1 / (60 + rank))
→ Rank 1 gets 0.4 × 1/61 = 0.00656
→ Rank 10 gets 0.4 × 1/70 = 0.00571
 
For each vector result (by rank):
→ Score = 0.6 × (1 / (60 + rank))
→ Higher weight = vector search has more influence
 
Documents found by BOTH methods
get scores from both → ranked highest!
This is the magic: agreement = high confidence.
💡
Aha! Why RRF Instead of Score Addition?
BM25 scores and vector distances are on completely different scales. BM25 might give scores of 5-25, while vector distances are 0.1-0.9. You can't just add them. RRF only uses rank position (1st, 2nd, 3rd...), making it scale-independent. No calibration needed!

🎯 Cross-Encoder Reranking

Think of retrieval as a talent show audition. BM25 and vector search are the first round — they pick 30 candidates quickly. The cross-encoder is the judges' panel that carefully evaluates each finalist.

AspectBi-Encoder (Embedding Search)Cross-Encoder (Reranker)
How it worksEncodes query and document separatelyEncodes query+document together
SpeedVery fast (pre-computed embeddings)Slow (runs model on each pair)
AccuracyGood (misses nuances)Excellent (sees full context)
Use caseInitial retrieval (30 candidates)Final reranking (top 5)
Our modelnomic-embed-text-v1.5ms-marco-MiniLM-L-6-v2
Python — reranker.py
class Reranker:
    def rerank(self, query, hits, top_k):
        # Create query-document pairs
        pairs = [(query, hit.content)
                 for hit in hits]

        # Get cross-encoder scores
        scores = self.model.predict(pairs)

        # Sort by score (higher = better)
        scored.sort(
            key=lambda x: x.score,
            reverse=True)

        return scored[:top_k]
Plain English
The reranker:
 
📋 Step 1: Pair each candidate with the query
→ ("refund policy?", "Our refund policy states...")
→ ("refund policy?", "Company history was...")
 
🧠 Step 2: Feed each pair to the cross-encoder
→ It reads both TOGETHER, understanding context
→ Returns a relevance score for each pair
 
🏆 Step 3: Sort by relevance score
→ Most relevant at the top
 
✂️ Step 4: Keep only the top 5
30 candidates in → 5 best out

🌈 MMR: Preventing Duplicate Results

Without diversity control, all 5 results might be slight variations of the same paragraph. MMR (Maximal Marginal Relevance) balances relevance with diversity.

Concept — MMR Formula
# MMR balances two goals:
MMR = λ × relevance
    - (1-λ) × max_similarity_to_selected

# λ = 1.0 → pure relevance (no diversity)
# λ = 0.0 → pure diversity (ignore relevance)
# λ = 0.5 → balanced (our default)

# Two modes in our system:
# 1. Text-based: Jaccard word similarity
# 2. Embedding-based: cosine similarity
Plain English
The diversity formula:
Score = How relevant is this document?
       MINUS How similar is it to docs we already picked?
 
🎚️ λ controls the tradeoff:
→ λ=1: "Give me the most relevant, even if they overlap"
→ λ=0: "Give me the most diverse, even if less relevant"
→ λ=0.5: "Balance both" (our default)
 
Two ways to measure "similar":
→ Count shared words (Jaccard) — fast, no model needed
→ Compare embeddings (cosine) — semantic similarity

🏢 Dual-Collection Search & Merge

Python — retrieval_agent.py
async def _retrieve_with_global(self, query, client_id):
    client_results = []
    global_results = []

    # Search client collection
    if client_id != "global":
        client_results = await self._search(
            query, client_id)

    # Search global collection
    global_results = await self._search(
        query, "global")

    # Merge: client first (priority)
    merged = self._merge_results(
        client_results, global_results)

    return merged[:self.top_k]
Plain English
Search two libraries, combine results:
 
📁 First: Search the client's private library
→ Only if they're not the "global" client
→ Full hybrid search (BM25 + vector + rerank)
 
🌍 Then: Search the shared global library
→ Documents available to everyone
 
🔀 Merge with client priority:
→ Client docs added first (they take precedence)
→ Global docs fill remaining slots
→ Duplicates removed by chunk ID
 
✂️ Keep only top 5 results

🎤 Interview Questions: Retrieval Strategies

Q8 When an agent's first retrieval attempt returns low-relevance results, what self-correction strategies can it employ? ▼

Self-correction is partially implemented in our system:

  1. Query rewriting (pre-retrieval): QueryAgent._rewrite_query() optimizes the query before searching
  2. Cross-encoder reranking (post-retrieval): Even noisy initial candidates get accurately rescored
  3. Dual-collection fallback: If client collection returns nothing, global results act as fallback
  4. Graceful admission: The synthesis prompt says: "If the context doesn't contain enough information, say so clearly"

Not yet implemented: Iterative retrieval (re-query with sub-queries), HyDE (hypothetical document embedding)

Q9 Compare dense vector, sparse keyword, and hybrid retrieval. When would an agent dynamically switch? ▼

Our system implements all three via configuration:

MethodImplementationStrengths
Dense vectorChromaDB + nomic-embed-text-v1.5Semantic similarity, paraphrases
Sparse BM25rank_bm25.BM25OkapiExact keywords, rare terms, codes
Hybrid (RRF)hybrid_search.pyBest of both, no score calibration

The RetrievalAgent dynamically selects:

if self.use_bm25:
    hybrid = get_hybrid_search(client_id)  # BM25 + Vector + RRF
else:
    retriever = get_retriever()  # Vector-only

Dynamic switching scenarios: Queries with proper nouns/codes → BM25 excels. Semantic/conceptual queries → vector dominates. Our hybrid with bm25_weight=0.4, vector_weight=0.6 gives vector a slight edge while preserving BM25's keyword strength.

Q10 How do you implement adaptive chunking, and how does chunk size impact agent reasoning? ▼

Our chunking config in config.py:

chunk_size: int = 1200         # Target characters
chunk_overlap: int = 200       # Overlap between chunks
chunk_method: str = "char"     # "char" or "token"
respect_headings: bool = True  # Break at markdown headers
respect_sentences: bool = True # Break at sentence boundaries
min_chunk_tokens: int = 50     # Drop tiny chunks

Adaptive strategies:

  1. Heading-aware: Chunks align with markdown section headers — topically coherent
  2. Sentence-boundary respect: No mid-sentence splits that confuse reranker
  3. Content-hash IDs: Deterministic chunks enable deduplication during re-ingestion

Impact on reasoning: Too small (<50 tokens) → lose context. Too large → dilute relevance. Our 1200 chars (~300 tokens) balances context for synthesis with precision for retrieval.

Q11 Explain query decomposition in agentic RAG. ▼

Query decomposition breaks complex multi-part questions into simpler sub-queries.

Our system: The QueryAgent._rewrite_query() performs a simpler form — it rewrites for retrieval but doesn't explicitly decompose.

How full decomposition would work:

  1. QueryAgent generates multiple sub-queries from a complex question
  2. Each sub-query goes to RetrievalAgent._search_collection() independently
  3. Results merged using existing _merge_results() deduplication
  4. SynthesisAgent receives all chunks and synthesizes across them

The AgentState.rewritten_query (currently a single string) would need to become List[str].

Q12 What is Hypothetical Document Embedding (HyDE)? ▼

HyDE asks the LLM to generate a hypothetical answer, embeds THAT instead of the raw query, and searches with it.

Why it helps: The hypothetical document is in the same "language" as indexed documents (declarative, detailed), so its embedding is closer to relevant chunks than a short question.

Our system uses alternatives instead:

  • Query rewriting bridges the question-document gap
  • Hybrid search catches both semantic and lexical matches
  • Cross-encoder reranking rescores based on joint understanding

These three together achieve much of what HyDE offers without the extra LLM call.

Q13 How would you handle multi-hop retrieval? ▼

Multi-hop = chaining facts across documents ("Who manages the person who signed contract X?").

Current approach:

  1. Retrieval Agent searches client + global, pulling from multiple documents
  2. Synthesis Agent receives up to 6 chunks and reasons across them
  3. The knowledge graph module (graph_query.py) supports traversal for entity relationships

For true multi-hop: Need an iterative loop (retrieve → extract intermediate → re-retrieve). The knowledge graph with entity types (PERSON, ORG, DOCUMENT) and relationships (AUTHORED, REFERENCES) is specifically designed for this. Enable with RAG__KNOWLEDGE_GRAPH_ENABLED=true.

Q14 Describe a re-ranking strategy after initial retrieval. How do cross-encoders fit in? ▼

Our system uses cross-encoder reranking in reranker.py:

  • Bi-encoders encode query and document separately — fast but less accurate
  • Cross-encoders (ms-marco-MiniLM-L-6-v2, ~22M params) encode jointly — slower but significantly more accurate
  • We use bi-encoders for retrieval (fetch_k=30) and cross-encoders for reranking (top_k=5)

Additionally: MMR (Maximal Marginal Relevance) prevents returning 5 near-duplicate chunks using Jaccard text similarity or embedding cosine similarity.

Q15 What is metadata filtering, and how can an agent apply filters dynamically? ▼

Our system: Client isolation is the primary filter — separate ChromaDB collections per client (physical separation, not metadata filtering).

Chunk metadata includes: client_id, source_filename, page_number, section_heading, doc_id.

Dynamic filtering could work by:

  1. QueryAgent extracts filter hints from query ("in the 2024 report" → {"source_filename": {"$contains": "2024"}})
  2. Pass as where parameter through AgentState
  3. RetrievalAgent applies to both vector_store.query() and bm25_index.search()
