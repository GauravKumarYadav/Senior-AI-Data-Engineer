Foundations & Architecture

What this app does, how everything connects, and why agentic RAG changes the game.

🎯 What This App Actually Does

Imagine you upload a PDF of your company's policies. Then you ask: "What's our refund policy?" A normal chatbot guesses. This chatbot reads your actual document and answers from it — citing the exact page.

💡
The Big Idea: RAG = Retrieval-Augmented Generation
Instead of relying on what the AI "memorized" during training, RAG retrieves relevant documents first, then generates an answer grounded in those documents. Think of it like giving the AI an open-book exam instead of a closed-book one.

But this isn't basic RAG. This is Agentic RAG — the AI doesn't just retrieve and answer. It thinks about what to do:

🤔
"Is this a question or chitchat?"
The Query Agent classifies intent first. Saying "Hello!" doesn't trigger a document search.
🔧
"Does this need a tool?"
"What is 15% of 200?" goes to the calculator, not the document search.
🔍
"Which documents to search?"
Searches your client-specific docs first, then global docs, combines and reranks.
✍️
"How should I answer?"
The Synthesis Agent generates a response with source citations, formatting it in Markdown.

⚡ Traditional RAG vs. Agentic RAG

Think of traditional RAG like a vending machine — you push one button, get one result. Agentic RAG is like a personal assistant who thinks about what you need before acting.

AspectTraditional RAGAgentic RAG (Our System)

Decision Making None — always retrieves LLM classifies intent, decides whether to retrieve, use tools, or chat
Query Optimization Raw query → embedding Query rewritten for better retrieval using conversation context
Search Method Vector search only BM25 + Vector → RRF Fusion → Cross-encoder Reranking
Memory Usually none Redis session buffer + auto-summarization
Multi-tenant Usually single collection Per-client collections + global collection

LLM Calls 1 call 2–3 calls (classify, optional rewrite, generate)

Why Local LLM?
The default config uses LM Studio running locally. This means zero data leaves your machine — critical for privacy-sensitive document Q&A. But you can switch to Groq or OpenAI with a single config change.

🚀 A Request's Journey (End-to-End)

Follow a single question — "What is our refund policy?" — through the entire system:

1
User sends message via Frontend
The React+Vite frontend sends the message through a WebSocket connection (or REST fallback) to FastAPI. JWT token authenticates the user.
2
Memory loaded from Redis
The SessionBuffer fetches the conversation summary and recent messages from Redis. This gives the agents context about prior turns.
3
Query Agent classifies intent
An LLM call with temperature=0.1 classifies the message as {"intent": "question", "needs_retrieval": true}. The query is rewritten for better retrieval.
4
Orchestrator routes to Retrieval Agent
The LangGraph's _route_after_query() sees needs_retrieval=true and routes to the Retrieval Agent.
5
Hybrid search across two collections
BM25 keyword search + vector semantic search run on both the client collection and global collection. Results are fused with RRF, then reranked by a cross-encoder.
6
Synthesis Agent generates response
Top 5 chunks (with source labels) are passed to the LLM. It generates a Markdown-formatted response with [Source: filename] citations.
7
Memory updated
The message + response are saved to Redis. If the conversation exceeds 4000 tokens, older messages are auto-summarized by the LLM.
8
Response streamed to user
Via WebSocket, the response streams in real-time. Via REST, it's returned as a JSON payload with the response + sources list.

💻 Code Walkthrough: The Entry Point

The entire backend starts from backend/app/main.py. Here's how it wires everything together:

Python — main.py
@asynccontextmanager
async def lifespan(app: FastAPI):
    start_pruning_scheduler()
    buffer = get_session_buffer()
    yield
    stop_pruning_scheduler()

app = FastAPI(
    title="Agentic RAG Chatbot",
    version="2.0.0",
    lifespan=lifespan,
)

app.include_router(chat.router)
app.include_router(documents.router)
app.include_router(websocket.router)
Plain English
When the server starts up:
→ Start a background scheduler that prunes old memory
→ Connect to Redis for session storage
→ Then keep running until shutdown...
→ On shutdown, stop the memory pruner
 
Create the API server named "Agentic RAG Chatbot"
Version 2.0.0, with the startup/shutdown hooks above
 
Wire up all the URL routes:
→ /chat for sending messages
→ /documents for uploading files
→ /chat/ws/{client_id} for real-time streaming
💡
Aha! The Lifespan Pattern
FastAPI's lifespan is like a setup/teardown bracket. Everything before yield runs at startup, everything after runs at shutdown. This is a universal pattern in modern Python — you'll see it in database connections, job schedulers, and resource managers.

⚙️ The Configuration System

Every setting in the system is defined in backend/app/config.py using Pydantic Settings — a type-safe configuration system that reads from environment variables.

Python — config.py
class LLMSettings(BaseModel):
    provider: str = "lmstudio"
    lmstudio: LMStudioSettings
    ollama: OllamaSettings
    groq: GroqSettings
    openai: OpenAISettings
    temperature: float = 0.35
    max_tokens: int = 4096

    def get_active_provider(self):
        if self.provider == "ollama":
            return self.ollama
        if self.provider == "groq":
            return self.groq
        return self.lmstudio

🎤 Interview Questions: Foundations & Architecture

Q1 How does an agentic RAG system differ from a traditional (single-shot) RAG pipeline, and when would you choose one over the other? ▼

Traditional RAG: Query → Retrieve → Generate. One pass, no decision-making.

Agentic RAG (our system): The LLM is wrapped inside an agent loop. In our project, the Orchestrator builds a LangGraph StateGraph with specialized nodes:

  1. Query Agent — classifies intent, rewrites queries
  2. Retrieval Agent — searches client + global collections
  3. Tool Agent — executes calculator/datetime
  4. Synthesis Agent — generates grounded responses

The graph uses conditional edges (_route_after_query) to decide at runtime whether to retrieve, call a tool, list documents, or go straight to synthesis.

When to choose which:

  • Single-shot RAG: Simple Q&A over a single knowledge base with predictable query types
  • Agentic RAG: Heterogeneous queries (chitchat, tools, multi-collection retrieval, follow-ups) requiring dynamic routing and query rewriting
Q2 Walk me through the end-to-end architecture of an agentic RAG system you have built. ▼

Our system's end-to-end flow:

User → Frontend (React+Vite) → Nginx → FastAPI (Chat-API)
  ├── Memory: SessionBuffer (Redis) → conversation_summary + recent_messages
  ├── Orchestrator.run() (LangGraph StateGraph)
  │   ├── QueryAgent → classify intent, detect tools, rewrite query
  │   ├── Route: chitchat → SynthesisAgent
  │   ├── Route: tool → ToolAgent → SynthesisAgent
  │   ├── Route: document_list → _document_list_node → END
  │   └── Route: question → RetrievalAgent (BM25+Vector→RRF→Rerank) → SynthesisAgent
  └── _post_turn() → update memory, trigger auto-summarization
→ Response to user

Q3 What role does the "agent loop" (observe → think → act → reflect) play in improving retrieval quality? ▼

In our system, the agent loop manifests as the LangGraph state machine:

  1. Observe: The QueryAgent receives the user message + conversation_summary + recent_messages from Redis
  2. Think: The QueryAgent classifies intent (chitchat, question, tool, etc.) and rewrites the query for better retrieval
  3. Act: The orchestrator's _route_after_query() conditionally routes to the right agent
  4. Reflect (implicit): The synthesis agent evaluates retrieved context — if no docs found, it admits the gap rather than hallucinating

This improves retrieval quality because:

  • Query rewriting resolves references like "tell me more about that" into standalone queries
  • Intent classification prevents unnecessary retrieval for chitchat (saves latency)
  • Dynamic routing sends the right query type to the right agent
Q4 How do you decide between a single-agent vs. multi-agent orchestration pattern? ▼

Our project uses a multi-agent pattern (4 specialized agents + orchestrator). The decision was driven by:

  1. Separation of concerns: Each agent has a single responsibility. QueryAgent classifies. RetrievalAgent fetches. SynthesisAgent generates. Independently testable.
  2. Heterogeneous query types: 5 intents (chitchat, question, follow_up, tool, document_list). A single agent would need a monolithic prompt.
  3. Multi-collection search: The RetrievalAgent must search client + global collections, merge, deduplicate, and rerank — complex enough for its own agent.
  4. Cost control: QueryAgent uses temperature=0.1, max_tokens=256 (cheap classification), while SynthesisAgent uses temperature=0.35, max_tokens=4096 (expensive generation). Independent tuning.

When a single agent suffices: Simple, single-collection Q&A with one query type and no tool use.

Q5 Explain "tool-augmented retrieval" — how does giving an LLM access to retrieval as a callable tool change system design? ▼

In our system, retrieval is one of several callable tools the agent can choose from. The QueryAgent._detect_tool() inspects the message for tool patterns:

tool_name, tool_params = self._detect_tool(message)
if tool_name:
    intent = "tool"
    needs_retrieval = False

The orchestrator routes to ToolAgent instead of RetrievalAgent. The ToolAgent executes the tool (e.g., SafeCalculator) and passes the result to SynthesisAgent.

How this changes design:

  • The LLM doesn't compute math itself (avoids hallucinated calculations)
  • Retrieval is conditional, not mandatory
  • The AgentState carries tool_name, tool_params, tool_result alongside retrieved_chunks
  • The synthesis prompt incorporates tool results naturally
Q6 How would you architect an agentic RAG system serving both structured (SQL) and unstructured (document) sources? ▼

Our architecture is already designed for this extensibility:

  1. Add a SQL tool to ToolAgent: The ToolAgent already supports calculator/datetime. Adding sql_query follows the same pattern.
  2. Router in QueryAgent: The intent classifier already distinguishes question, tool, chitchat. Add a structured_query intent to route to a SQL agent.
  3. Shared AgentState: The tool_name, tool_params, tool_result fields are generic enough to carry SQL queries/results.
  4. Synthesis merges both: The SynthesisAgent already merges document context with tool context. SQL results flow through tool_context.
Q7 What are the trade-offs between plan-then-execute and ReAct-style agent patterns for RAG? ▼

Our system uses a plan-then-execute pattern (static DAG):

AspectPlan-then-Execute (Ours)ReAct (Reactive Loop)
LatencyLower — single planning stepHigher — multiple LLM calls per cycle
CostLower — fewer LLM invocationsHigher — each iteration costs tokens
Retrieval QualityMay miss if initial retrieval is poorCan self-correct with iterative retrieval
ComplexitySimpler DAG, easier to trace/debugComplex loop, harder to observe
PredictabilityDeterministic routing, bounded stepsMay enter infinite loops without guardrails

Our max_steps = 10 provides a safety bound. This is forward-looking for when iterative retrieval is added.

🧪 Module 1 Quiz
1. A user sends "Hello, how are you?" to the chatbot. What happens?
The system searches documents for "hello" and returns results
Query Agent classifies as "chitchat", skips retrieval, Synthesis Agent responds casually
The system returns an error because it's not a document question
✅ The Query Agent's intent classifier recognizes greetings as "chitchat" and sets needs_retrieval: false. The orchestrator routes directly to the Synthesis Agent, which uses the CHITCHAT_SYSTEM_PROMPT to respond warmly. This saves latency by skipping unnecessary document searches.
2. Why does this system use multiple specialized agents instead of one big agent?
Multiple agents are always faster than a single agent
LangGraph requires at least 4 agents to work
Each agent can be independently tuned (temp, tokens, prompts) and tested — separation of concerns
✅ Separation of concerns! The Query Agent uses temperature=0.1 for deterministic classification, while the Synthesis Agent uses temperature=0.35 for natural responses. Each agent has its own prompt, token budget, and can be tested independently. A monolithic agent would need one massive prompt covering all cases.
3. If the Redis server goes down, what happens to the chatbot?
The entire chatbot crashes and returns 500 errors
The circuit breaker activates, memory falls back to in-memory storage with LRU eviction
The chatbot works but loses all conversation history permanently
✅ The SessionBuffer has a RedisCircuitBreaker that detects failures (threshold: 5 failures). It switches to an in-memory fallback with LRU eviction and automatically tests Redis recovery every 30 seconds. This is the circuit breaker pattern — a production resilience essential you'll see in Module 7.
