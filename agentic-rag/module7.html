📊 Evaluation: Beyond "Does It Work?"

A RAG system isn't just about correct answers — it's about cost, latency, faithfulness, and coverage. Think of it like grading a restaurant: taste (answer quality), speed (latency), price (cost), and hygiene (faithfulness).

MetricWhat It MeasuresHow We Track It
Faithfulness Is the answer supported by retrieved context? Citation enforcement in prompt + LangSmith trace review
Context Precision Of retrieved chunks, what fraction is relevant? HybridSearchResult stats + reranker metadata
Context Recall Of all relevant chunks, what fraction was retrieved? Dual-source retrieval (BM25 + vector) improves recall
Answer Relevancy Does the answer address the question? Intent classification + query rewriting improve this
Cost Tokens consumed per query CostTracker (AsyncCallbackHandler) on every LLM call
Latency End-to-end response time @traceable decorators + HybridSearch timing

⚡ Caching: Every Millisecond Counts

1
Singleton Pattern — Load Expensive Models Once
@lru_cache(maxsize=1) on get_reranker(), get_embedding_function(). CrossEncoder lazy-loaded on first use. Model loads from disk only once per process lifetime.
2
HybridSearch Instance Caching
_hybrid_search_instances: Dict[str, HybridSearch] = {} — one instance per client, reused across requests. Includes the BM25 index + ChromaDB collection reference.
3
Redis Conversation Summaries
Instead of re-summarizing the entire history on every turn, compressed summaries are stored in Redis (summary_key_prefix = "summary:", TTL = 24 hours).
4
BM25 Index Persistence
Saved to disk (./data/bm25) and loaded on startup. No re-indexing unless the corpus changes.
💡
What's Not Yet Cached: Semantic Caching
A missing optimization: embed incoming queries and check similarity against recent cached queries. If a query is semantically similar (cosine > 0.95) to a recent one, return the cached response. This could eliminate ~30% of LLM calls for repeated/similar questions.

🛡️ Graceful Degradation: The Art of Failing Well

In production, things will break. The question is: does your system crash, or does it degrade gracefully? Ours has fallbacks at every critical point:

ComponentFailure ModeFallback
Redis Connection lost Circuit breaker → in-memory LRU store with token budget
Embedding Service Unavailable NoOpEmbeddingFunction returns zero vectors — avoids hard crash
Cross-Encoder Model fails to load Skip reranking, return bi-encoder order: hits[:top_k]
BM25 Index Corrupted/missing Fall back to vector-only search
LLM Call Timeout (120s) Return error message: "I encountered an error processing your request."
Any Agent Unhandled exception Per-agent error boundary catches and returns graceful fallback

🎤 Interview Questions: Evaluation & Production

Q28 What metrics do you use to evaluate an agentic RAG system beyond simple answer accuracy? ▼

Multiple dimensions:

  1. Cost tracking: CostTracker records prompt/completion tokens per call, compares against OpenAI pricing baselines (GPT-4o-mini: $0.15/$0.60 per 1M tokens).
  2. Retrieval quality: test_rag_quality.py (18.8 KB) tests precision, client isolation, hybrid vs. vector-only quality.
  3. Latency: HybridSearch.search_with_stats() tracks duration + candidate counts.
  4. Faithfulness (implicit): Citation enforcement + structured sources field.
  5. LangSmith dashboards: Per-project latency, token usage, and error trends.
Q29 How do you measure and reduce hallucination rates? ▼

Measurement: Citation enforcement (responses without [Source:] for factual claims = hallucination signal), RAG quality tests, LangSmith trace review.

Reduction: Low synthesis temperature (0.35), explicit grounding instructions, only 6 chunks × 600 chars (focused context), cross-encoder reranking (high-quality context), client context separation.

Q30 Explain the RAGAS evaluation framework. Which metrics are most useful for agentic RAG? ▼

RAGAS provides four key metrics:

  • Faithfulness: Is the answer supported by context? → Critical for our multi-agent pipeline where hallucination can creep in.
  • Answer Relevancy: Does it address the question? → Our query rewriting improves this.
  • Context Precision: Are retrieved chunks relevant? → Our reranker directly optimizes this.
  • Context Recall: Are all relevant chunks found? → Our hybrid BM25+vector search improves recall over vector-only.

Our test_rag_quality.py implements custom versions tailored to our pipeline.

Q31 How do you build a regression test suite for an agentic RAG system? ▼

Our test structure:

  • Golden tests: test_rag_quality.py — known Q&A pairs against indexed docs.
  • Client isolation: test_client_isolation.py — client A's docs never in client B's results.
  • Smoke tests: test_smoke.py — full pipeline E2E.
  • Live endpoints: test_endpoints.py (49.8 KB!) — every API endpoint exercised.
  • Config tests: test_config.py — setting changes don't break init.

CI runs make test on every commit. Embedding fingerprints catch silent model changes. LangSmith tracks metric trends.

Q32 What is "context precision" vs. "context recall"? ▼

Precision: Of the 5 final chunks, how many were actually relevant? Tracked via reranker metadata (original_score vs rerank_score).

Recall: Of all relevant chunks in the knowledge base, how many were retrieved? Our dual-source retrieval (client + global, 30 candidates each) + BM25+vector fusion maximizes recall.

The reranker metadata lets us analyze whether reranking improves precision (promoting relevant docs) or hurts recall (demoting relevant docs).

Q33 How would you A/B test two different agent strategies in production? ▼

Our architecture supports A/B testing via:

  1. Config-driven strategy: Toggle bm25_enabled, reranker_enabled, weight tuning.
  2. Multi-provider LLM: Strategy A with LM Studio, Strategy B with Groq.
  3. LangSmith project splitting: Route 50% to "agentic-rag-v1", 50% to "agentic-rag-v2", compare metrics.
  4. Cost comparison: CostTracker measures tokens per query under each approach.

Implementation: add strategy field to ChatRequest, instantiate appropriate orchestrator variant, use statistical significance tests.

Q39 How do you manage latency in an agentic RAG system with multiple retrieval rounds? ▼

Latency management strategies:

  1. Single-pass DAG: No iterative loops — Query → Retrieval → Synthesis (3 nodes, 2 LLM calls + 1 retrieval).
  2. Lightweight classification: max_tokens=256, temperature=0.1 for fast intent. Chitchat skips retrieval entirely.
  3. Semaphore-bounded reranking: asyncio.Semaphore(max_concurrent=2) prevents CPU saturation.
  4. Candidate limits: 30 candidates for reranking, not 100+. Top 5 final results.
  5. Context truncation: Max 6 chunks × 600 chars = ~3600 chars to synthesis.
Q40 What caching strategies do you use to reduce redundant retrievals and LLM calls? ▼

Multi-level caching:

  1. Redis: cache_ttl = 3600 for general caching.
  2. Singletons: @lru_cache(maxsize=1) for reranker + embedding models.
  3. Conversation summaries: Stored in Redis (session_ttl = 86400), not re-computed per turn.
  4. BM25 persistence: Saved to ./data/bm25, loaded on startup.
  5. Lazy loading: CrossEncoder model loaded on first use only.

Future: Semantic caching (embed queries, check similarity to cached queries before running retrieval).

Q41 How do you implement observability and tracing for a multi-step agentic RAG pipeline? ▼

Full-stack observability:

  1. LangSmith @traceable: Every node + sub-operation traced with latency and token counts.
  2. Structured JSON logging: Correlation ID tracking, 10MB rotation, Loki/ELK-ready.
  3. Cost tracking: CostTracker (AsyncCallbackHandler) on every LLM call, logs tokens + estimated costs.
  4. Health endpoint: /health for monitoring systems.
  5. Retrieval stats: HybridSearchResult tracks BM25/vector/fused candidate counts + duration.
Q42 How would you auto-scale an agentic RAG service to handle bursty traffic? ▼

Stateless API + shared state = horizontal scaling:

  1. Stateless API: All state in Redis + ChromaDB. Multiple replicas behind Nginx.
  2. Redis as shared state: Sessions, summaries accessible from any replica.
  3. Remote ChromaDB: url: Optional[str] supports external vector store, scaling independently.
  4. Per-instance semaphores: Each replica manages its own CPU load.
  5. IaC: Terraform (infra/main.tf) for cloud provisioning.
Q43 How do you handle rate limits and token budget management? ▼

Token budgets at every level:

  • Per-request: QueryAgent max_tokens=256, SynthesisAgent max_tokens=4096.
  • Memory: Auto-summarization at 4000 tokens, compresses to ~1000.
  • Context truncation: context[:500], summary[:300], chunks content[:600].
  • LLM timeout: timeout=120.0 prevents hung requests.
  • Reranker semaphore: Effective rate-limiting on the most CPU-intensive operation.
Q44 What is your strategy for graceful degradation? ▼

Production-grade fallbacks at every layer:

  • Redis: Circuit breaker (CLOSED → OPEN → HALF_OPEN) → in-memory LRU fallback.
  • Embeddings: NoOpEmbeddingFunction returns zero vectors.
  • Reranker: Disabled → return bi-encoder order (hits[:top_k]).
  • BM25: Missing → vector-only search.
  • Orchestrator: Catch-all error boundary returns user-friendly error.

Philosophy: Degraded service is always better than no service.

Q45 How do you deploy embedding model updates without downtime? ▼

Embedding fingerprinting system:

  1. Write: Every chunk gets embedding_fingerprint metadata: "lmstudio:nomic-embed-text:768:true:1.0"
  2. Read: verify_embedding_fingerprint() compares stored vs. current on startup.
  3. Mismatch: EmbeddingMismatchError raised — prevents silent mixed-embedding degradation.
  4. Per-client migration: Re-index one client at a time (create new collection → verify → swap → delete old).
  5. BM25 survives: BM25 uses raw text tokens, unaffected by embedding model changes.
  6. Escape hatch: verify_fingerprint=False for intentional migration scenarios.
🧪 Module 7 Quiz
1. The embedding service goes down. What happens to search?
The entire system crashes
NoOpEmbeddingFunction returns zero vectors; BM25 still works for keyword search; reranker falls back to original order

✅ The NoOpEmbeddingFunction returns zero vectors instead of crashing. BM25 continues working since it doesn't use embeddings. The system degrades gracefully — you get keyword-only results instead of nothing.
2. You swap the embedding model from nomic-embed-text to multilingual-e5. What prevents silent quality degradation?
Embedding fingerprinting detects the mismatch and raises EmbeddingMismatchError, forcing a re-index decision
ChromaDB automatically re-embeds all documents
Nothing — you need to manually check
✅ The EmbeddingFingerprint system stores a fingerprint like "lmstudio:nomic-embed-text:768:true:1.0" with each chunk. On startup, verify_embedding_fingerprint() compares it against the current model config. A mismatch raises EmbeddingMismatchError, alerting you that old embeddings are incompatible with the new model and re-indexing is needed.
