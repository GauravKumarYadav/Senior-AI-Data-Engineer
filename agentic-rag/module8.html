Adding CRAG to Our System
Because LangGraph uses a StateGraph, adding a verification node is trivial: graph.add_node("verification", self._verify_node). The AgentState already carries both response and retrieved_chunks, so the verification node has everything it needs. Trade-off: +1 LLM call (~1-3s). Worth it for legal, medical, or financial domains.

🕸️ Knowledge Graphs: Beyond Vector Search

Vector search finds semantically similar text. Knowledge graphs find structurally connected facts. Think of it like the difference between finding similar-looking books (vector) versus following the web of citations between books (graph).

🛡️ Guardrails & Safety Layers

1
Input Safety
SafeCalculator uses shunting-yard algorithm (never eval()). Rejects import, exec, __, lambda, ;. Intent validation only accepts 5 valid intents. Multi-strategy JSON parsing with regex fallbacks.
2
Data Access Safety
JWT authentication on all endpoints. Per-client collection isolation (physical separation, not filter-based). Client ID validation on document operations. Even if injection manipulates the query, retrieval is scoped to the authenticated client.
3
Output Safety
Citation enforcement grounds claims. max_tokens=4096 prevents unbounded generation. Synthesis prompt: "Be concise but complete" — no encouragement to speculate.
4
Operational Safety
Circuit breaker prevents cascade failures. LLM timeout = 120s. Per-agent error boundaries. Reranker semaphore caps CPU. Structured JSON logging with correlation IDs for audit.

🎤 Interview Questions: Advanced & System Design

Q51 How would you implement a "corrective RAG" (CRAG) pattern? ▼

Add a verification node in LangGraph after synthesis:

  1. Takes response + retrieved_chunks from AgentState.
  2. Prompts LLM: "Does this response accurately reflect the retrieved context? Flag unsupported claims."
  3. If claims are unsupported: remove them, add disclaimer, or re-retrieve with refined query.
  4. LangGraph's StateGraph makes this trivial — one add_node() + one add_conditional_edges() call.

Trade-off: +1 LLM call (~1-3s latency). Worth it for high-stakes domains.

Q52 Explain the Self-RAG paradigm. ▼

Self-RAG fine-tunes a model with special tokens ([Retrieve], [IsRel], [IsSup], [IsUse]) that the model emits during generation to signal retrieval needs and passage quality.

vs. Our system: We use prompt-based approach with off-the-shelf LLMs. More flexible (no retraining), supports multiple providers (LM Studio, Ollama, Groq), fully inspectable via LangSmith. Self-RAG has lower latency (single model, inline decisions) but needs model retraining for changes.

Q53 How do you build a graph-augmented RAG system? ▼

Our system already has a knowledge graph module:

  • graph_store.py: SQLite per-client, stores entities + relationships.
  • entity_extractor.py: LLM-based NER extracts PERSON, ORG, DATE, AMOUNT, etc.
  • graph_query.py: 2-hop traversal (kg_expansion_depth=2).
  • Integration: Chunk metadata includes detected_entities, entity_ids, semantic_type.
  • Enable: RAG__KNOWLEDGE_GRAPH_ENABLED=true

Designed for scale: SQLite now, clear path to Neo4j (entity/relationship model is graph-native).

Q54 What is Agentic Retrieval with Planning (ARP)? ▼

ARP creates an explicit retrieval plan before executing searches (what info, which sources, what order, stopping criteria).

Our system: 1-step planning (classify + route). The QueryAgent does lightweight planning, the orchestrator executes via conditional edges. Multi-step planning would be warranted for complex analytical queries requiring data from multiple documents. LangGraph edges make this extensible.

Q55 How would you integrate RLHF to improve retrieval decisions? ▼

Two areas for RLHF:

  1. Query rewriting: Collect (original, rewritten, quality_score) triples from LangSmith traces. Fine-tune to maximize retrieval quality.
  2. Intent classification: Log misclassifications, use feedback to update few-shot examples or fine-tune the classifier.

Practical path: LangSmith feedback API → collect user ratings → export as training data → fine-tune → swap model (multi-provider support makes this seamless).

Q56 Describe a collaborative multi-agent RAG system (retriever, critic, synthesizer). ▼

Our system is already collaborative multi-agent:

  • Router/Planner: QueryAgent — intent + query rewriting
  • Retriever: RetrievalAgent — hybrid search + reranking
  • Tool Executor: ToolAgent — calculator, datetime
  • Synthesizer: SynthesisAgent — response + citations
  • Coordinator: Orchestrator — LangGraph workflow

Adding a critic: ~20 lines of code — graph.add_node("critic", ...) + conditional edges. The critic evaluates retrieval quality and can route back for refined retrieval.

Q57 How do you approach guardrails and safety layers in agentic RAG? ▼

Four-layer defense:

  1. Input: SafeCalculator (shunting-yard, no eval), intent validation (5 valid intents), regex JSON parsing.
  2. Data access: JWT auth, per-client physical collection isolation, client ID validation.
  3. Output: Citation enforcement, max_tokens cap, "be concise" instruction.
  4. Operational: Circuit breaker, timeouts, error boundaries, semaphore, structured audit logging.
Q58 What is the role of embedding model fine-tuning in agentic RAG? ▼

Fine-tuning learns domain-specific semantic relationships. Our system supports it via:

  • Configurable model: RAG__EMBEDDING_MODEL=my-fine-tuned-model
  • Fingerprinting: Detects model changes, flags re-indexing needs.
  • Abstract interface: VectorStoreBase decouples embedding from storage.
  • Multiple providers: LMStudio, Ollama, Groq, dedicated microservice.

When to fine-tune: Highly specialized vocabulary, domain-specific relationships, when retrieval precision is critical. Our general-purpose nomic-embed-text works well for diverse corpora.

Q59 How would you implement speculative retrieval (pre-fetching)? ▼

After each turn, analyze conversation summary to predict likely next queries. Pre-fetch results into Redis cache (cache_ttl=300). Check prefetch cache before hitting vector store.

Our Redis infrastructure supports this. Trade-off: consumes resources on predictions that may be wrong. Best for high-traffic systems where cache hit rate justifies overhead.

Q60 How do you balance parametric knowledge with retrieved knowledge? ▼

Retrieved = priority: "Answer based on provided context when available."

Parametric = fallback: "Respond based on general knowledge but clarify it's not from documents."

Decision boundary: QueryAgent intent classification — chitchat → parametric only (CHITCHAT_SYSTEM_PROMPT), question/follow_up → retrieval-first (SYSTEM_PROMPT + context).

Q61 Design an agentic RAG system for a legal firm with 10 million case documents and strict citation requirements. ▼

Adapting our architecture:

  1. Vector store: Replace ChromaDB with Qdrant/Weaviate for 10M+ docs. Our VectorStoreBase makes this a drop-in swap. Use IVF-PQ indexing.
  2. Multi-tenant: Per-client collections already built — maps to per-firm/per-case isolation.
  3. Citations: Already implemented. Enhance with paragraph_index and exact quote extraction.
  4. Chunking: Increase to ~2000 chars for legal context. Enable respect_headings for clause alignment.
  5. Knowledge graph: Enable for entity extraction — parties, judges, case citations, dates, amounts.
  6. Hybrid search: Critical — statute numbers (BM25) + conceptual similarity (vector).
  7. Audit: LangSmith tracing + structured JSON logging for compliance.
Q62 A user reports the agent is confidently returning outdated information. Diagnosis and fix. ▼

Diagnosis:

  1. LangSmith trace: Find the query → check retrieved chunks metadata (source_filename, uploaded_at).
  2. Document currency: List docs via /documents API, check upload dates.
  3. BM25 freshness: Check if outdated docs were removed from ChromaDB but BM25 index wasn't updated.
  4. Embedding fingerprint: Stale embeddings from old model?

Fix: Delete outdated docs (both indices), upload updated versions (content-hash dedup), add timestamp-based recency filtering, rebuild BM25 if corrupted.

Q63 P95 latency spiked from 3s to 12s after deployment. Debugging process. ▼

Step 1 — Identify the slow node via LangSmith traces (per-node latency).

Step 2 — Common suspects:

  • LLM provider slowdown → check LM Studio/Groq response times
  • Reranker overload → check semaphore contention
  • ChromaDB index growth → HNSW degrades at scale
  • BM25 corruption → check file size at ./data/bm25
  • Redis latency → check circuit breaker state, memory
  • Embedding model change → EmbeddingMismatchError in logs?

Quick mitigations: Reduce initial_fetch_k (30→15), increase reranker concurrency, disable BM25 to isolate, reduce LLM timeout.

Q64 Design an agentic RAG system supporting 50+ languages. ▼

Adaptations needed:

  1. Multilingual embeddings: RAG__EMBEDDING_MODEL=multilingual-e5-large. Fingerprinting detects the change.
  2. Language-aware chunking: Token-based for CJK (no whitespace). RTL handling for Arabic/Hebrew.
  3. Multilingual BM25: Per-language stopword lists. Language-aware tokenizer for CJK.
  4. Language detection: Add to QueryAgent before rewriting. Route to language-specific prompts.
  5. Cross-lingual retrieval: Multilingual embeddings map all languages to shared space. User asks in French, docs in English → works.
  6. Per-language monitoring: Track retrieval quality per language in LangSmith.
Q65 Integrating real-time web search as a fallback without sacrificing groundedness. ▼

Design:

  1. Add web search tool: Extend QueryAgent._detect_tool() and ToolAgent.
  2. Conditional trigger: Only when retrieval returns zero results or all rerank scores below threshold.
  3. Source labeling: Web results tagged {"collection_type": "web", "source": url}.
  4. Confidence hierarchy: client docs > global docs > web results.
  5. User transparency: "Based on your documents: ..." vs "From web search (not in your knowledge base): ..."
  6. Safety: Domain whitelist, rate limiting, Redis caching, mandatory attribution.
