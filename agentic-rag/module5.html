🧠 The Memory System

Imagine you're having a long phone call. After 30 minutes, you can't remember every word — but you remember the key points. That's what auto-summarization does: compresses old messages into a summary while keeping recent ones intact.

New Message
→
Add to Redis Buffer
→
Tokens > 4000?
→
Yes: Summarize older msgs via LLM
No: Continue as-is
Python — summarizer.py
class ConversationSummarizer:
    max_context = 4000   # tokens
    target = 1000         # summary target
    window = 10           # recent msgs kept

    def needs_summarization(self, summary, msgs):
        total = self._estimate_tokens(summary)
        total += sum(self._estimate_tokens(m)
                     for m in msgs)
        return total > self.max_context

    @traceable(name="summarize")
    async def summarize(self, msgs, existing):
        prompt = SUMMARY_PROMPT.format(
            existing=existing,
            messages=format_msgs(msgs))
        result = await self.llm.ainvoke(prompt)
        return result.content
Plain English
The conversation memory manager:
→ Trigger summarization at 4000 tokens
→ Compress summary to ~1000 tokens
→ Always keep the 10 most recent messages
 
"Do we need to summarize?"
→ Count tokens in existing summary
→ Add tokens in all recent messages
→ If total > 4000 → yes, summarize!
 
The summarization itself:
→ Traced in LangSmith for observability
→ Takes the old messages + any existing summary
→ Asks the LLM to produce a concise summary
→ Preserves: key facts, decisions, current topic

✍️ Prompt Engineering: The Anti-Hallucination Prompt

Python — synthesis_agent.py
SYSTEM_PROMPT = """You are a helpful AI
assistant with access to a knowledge base.
{client_context}

## Response Guidelines:
1. Answer based on the provided context
   when available
2. Cite sources using [Source: filename]
   format
3. If the context doesn't contain enough
   information, say so clearly
4. Be concise but complete
5. If tool results are provided,
   incorporate them naturally

When no relevant context is found,
respond based on your general knowledge
but clarify that it's not from the
documents."""
Anti-Hallucination Techniques
🎯 Rule 1: Context first
→ "Answer based on provided context" — not imagination
 
📝 Rule 2: Mandatory citation
→ Forces LLM to attribute claims. Unsupported claims become visible.
 
🤷 Rule 3: Admit uncertainty
→ "Say so clearly" — better to say "I don't know" than to fabricate
 
📏 Rule 4: Concise
→ Less text = fewer chances to hallucinate
 
🏷️ Knowledge source labeling
→ Distinguishes document-based vs general knowledge
→ Users know the provenance of each claim
💡
Aha! The "Lost in the Middle" Problem
LLMs tend to focus on the beginning and end of their context window, ignoring the middle. Our system mitigates this by: (1) only passing 6 chunks (short context), (2) truncating each to 600 chars, (3) putting the most relevant chunk first (rerank-ordered), and (4) putting the user's question at the END as a recency anchor.

🎤 Interview Questions: Memory, Prompts & Safety

Q34 How do you structure the system prompt for a RAG agent to minimize hallucination? ▼

Our anti-hallucination techniques in the system prompt:

  1. Grounding first: "Answer based on provided context" — context before parametric knowledge
  2. Mandatory citation: [Source: filename] format forces attribution
  3. Explicit uncertainty: "Say so clearly" if context is insufficient
  4. Knowledge labeling: Distinguishes document-based vs general knowledge
  5. Client context injection: {client_context} prevents cross-client confusion

Natural tone preserved by: not over-constraining format, allowing Markdown, and having a separate CHITCHAT_SYSTEM_PROMPT for casual conversation.

Q35 What techniques enforce citation and attribution in agent-generated answers? ▼

Multiple layers of citation enforcement:

  1. System prompt instruction: "Cite sources using [Source: filename]"
  2. Source labels in context: _build_context() prepends [Source: name] to each chunk — same format as requested citations
  3. Structured sources API response: _build_sources() returns id, content_preview, score, source, page, section
  4. Rich metadata: ChunkMetadata includes page_number and section_heading for pinpoint citations
Q36 How do you handle the "lost in the middle" problem? ▼

LLMs focus on content at the beginning and end of context, ignoring the middle. Our mitigations:

  1. Short context: Only 6 chunks (not 20+) — no "dead zone"
  2. Truncation: Each chunk capped at 600 chars
  3. Relevance-ordered: Most relevant chunk first (rerank score order)
  4. Prompt layout: Conversation → Documents → Tools → Question at END (recency anchor)
  5. Chunk dividers: --- separators make boundaries visible to LLM
Q38 How do you guard against prompt injection attacks? ▼

Multiple defense layers:

  1. Tool safety: SafeCalculator uses shunting-yard (never eval()), rejects import, exec, __
  2. Multi-strategy JSON parsing: Intent classifier doesn't eval() LLM output — uses regex fallbacks with valid intent validation
  3. Client isolation: Even if injection manipulates the query, retrieval is scoped to authenticated client's collections
  4. JWT authentication: All endpoints require valid tokens
  5. Input truncation: context[:500], summary[:300] prevent context stuffing
  6. Separation of concerns: User message is data (HumanMessage), not system instructions
Q37 Describe your approach to dynamic few-shot example selection. ▼

Our system uses static few-shot examples in intent classification:

"Hello!" → {"intent": "chitchat", "needs_retrieval": false}
"What is the refund policy?" → {"intent": "question", "needs_retrieval": true}
"What documents do I have?" → {"intent": "document_list", "needs_retrieval": false}

Why static works: Intent classification has 5 bounded categories. Examples cover boundary cases.

For dynamic selection: Maintain a vector index of (query, correct_intent) pairs. Retrieve top-3 similar examples for each new query. The existing embedding infrastructure could serve as the store.
