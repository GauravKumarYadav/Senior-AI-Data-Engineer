🧬 What Are Embeddings?

Imagine you could reduce every sentence to a GPS coordinate. Sentences with similar meaning would be at nearby coordinates. That's what embeddings do — they convert text into lists of numbers (vectors) where proximity = similarity.

📏
768 Dimensions
Our embedding model (nomic-embed-text-v1.5) converts text into a list of 768 numbers. Each number captures a different aspect of meaning.
📐
Cosine Similarity
To find similar documents, we measure the angle between two vectors. Smaller angle = more similar. "Refund policy" and "return process" are close; "pizza recipe" is far away.
🔄
Normalized Vectors
Our config: embedding_normalize=True. This scales all vectors to length 1, so cosine similarity simplifies to a dot product — faster math!

⚠️
Collection-per-Client vs. Metadata Filtering
Many systems use one big collection with a client_id metadata filter. That's dangerous — a bug in the where clause could leak data across tenants. Our approach uses physically separate collections, making cross-client data leakage structurally impossible.

🔐 Embedding Fingerprinting: The Silent Failure Preventer

Imagine you indexed 10,000 documents with Model A, then someone changed to Model B without re-indexing. Your search results would be garbage — but silently. No errors, just wrong answers.

Python — embeddings.py
@dataclass
class EmbeddingFingerprint:
    provider: str    # "ollama"
    model: str       # "nomic-embed-text"
    dimension: int   # 768
    normalize: bool  # True
    version: str     # "1.0"

# On ingestion: save fingerprint
chunk.metadata["embedding_fingerprint"] = \
    "lmstudio:nomic-embed-text:768:true:1.0"

# On startup: verify fingerprint
def verify_embedding_fingerprint(stored):
    current = get_embedding_fingerprint()
    if current != stored:
        raise EmbeddingMismatchError()
Plain English
A "passport" for the embedding model:
→ Which provider (ollama, lmstudio)?
→ Which model name?
→ How many dimensions (768)?
→ Are vectors normalized?
→ What version?
 
When we save a document:
→ Stamp the fingerprint on every chunk's metadata
 
When the server starts:
→ Compare current model's fingerprint with stored
→ If they don't match → LOUD ERROR
→ "Hey! Your model changed! Re-index or you'll get bad results!"
Prevents the worst kind of bug: silently wrong answers

📊 Index Types: HNSW vs IVF-PQ vs Brute Force

Index TypeHow It WorksSpeedMemoryBest For
HNSW (ours) Graph-based — builds a multi-layer navigation graph ⚡ Very fast queries 📈 High (keeps full vectors in memory) <1M vectors, high recall needed
IVF-PQ Clusters + compression — groups vectors, then compresses each ⚡ Fast queries 📉 Low (~4x less than HNSW) 1M-100M vectors, memory-constrained
Brute Force Checks every single vector — exhaustive scan 🐌 Slow (O(n)) 📏 Moderate <100K vectors, perfect recall needed
🔑
Our System Uses HNSW
ChromaDB uses HNSW by default. With our 768-dimension embeddings and typical collection sizes (<1M vectors per client), HNSW handles initial_fetch_k=30 candidate retrieval in milliseconds. The reranker compensates for any ANN recall loss.

For our privacy-focused, per-client isolation use case, ChromaDB wins. For enterprise scale, we'd migrate via the VectorStoreBase abstract interface.

Q17 Explain HNSW, IVF-PQ, and brute-force ANN indexes. ▼

See the comparison table above. In our system, ChromaDB uses HNSW with 768-dim embeddings. This is appropriate for collections up to ~1M vectors. The reranker compensates for ANN recall loss — even if HNSW misses a relevant doc in initial retrieval, it won't matter if it's not in the top-30 candidates.

Q18 How do you handle incremental index updates without full re-indexing? ▼

Incremental updates at multiple levels:

  1. ChromaDB: collection.upsert() handles incremental HNSW updates. Content-hash IDs make upserts idempotent.
  2. BM25: add_documents() appends and rebuilds scoring; delete() removes by ID
  3. Hybrid Search: add_documents() updates both indices in one call
  4. Hot reload: Documents available immediately after upload — no restart needed

For modifications: Delete old chunks + add new chunks (content-hash ID change detection).

Q19 Strategies for multi-tenancy with per-tenant retrieval isolation? ▼

Our system uses collection-per-client isolation:

  1. Separate ChromaDB collections per client with sanitized names + hash suffix
  2. Separate BM25 indices via get_bm25_index(client_id)
  3. Separate HybridSearch instances cached per client
  4. API-level enforcement via JWT auth + client_id validation

This is stronger than metadata-filter-based isolation — physical separation means a bug can't leak data across tenants.

Q20 How do you version and roll back an embedding index when your model changes? ▼

Our embedding fingerprinting system:

  1. On ingestion: Each chunk gets an embedding_fingerprint metadata field
  2. On startup: verify_embedding_fingerprint() compares stored vs. current
  3. On mismatch: EmbeddingMismatchError raised — loud failure, not silent degradation

Rollback strategy: Per-client collections enable one-at-a-time re-indexing. BM25 indices are model-independent (text-only) and survive model changes. verify_fingerprint=False escape hatch for migration.

1. You change the embedding model from nomic-embed-text to a new model. What happens on the next server start?
Everything works fine — ChromaDB adapts automatically
EmbeddingMismatchError is raised — the system refuses to search with mismatched embeddings
Search works but returns random results
✅ The embedding fingerprint system catches this! On startup, verify_embedding_fingerprint() compares the stored fingerprint ("lmstudio:nomic-embed-text:768:true:1.0") with the current model's fingerprint. If they don't match, it raises EmbeddingMismatchError — a loud failure that tells you to re-index.
2. Why does this system use separate ChromaDB collections per client instead of one big collection with metadata filtering?
ChromaDB doesn't support metadata filtering
Separate collections are always faster than filtered queries
Physical separation makes cross-client data leakage structurally impossible — a filter bug can't expose other clients' data
✅ Security through architecture, not just code! With metadata filtering, a bug in the where clause (or a forgotten filter) could return another client's documents. With separate collections, there's no way to accidentally query the wrong collection. Defense in depth.
