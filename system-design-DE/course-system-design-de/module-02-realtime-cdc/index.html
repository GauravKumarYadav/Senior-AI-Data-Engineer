Designing a Real-Time Analytics Pipeline

When an interviewer says "Design a real-time analytics system," they are testing three things simultaneously: your ability to decompose an ambiguous problem into concrete requirements, your knowledge of modern streaming architectures, and your judgment about tradeoffs. This screen gives you the framework to nail all three.

Use Case Examples

Real-time analytics shows up everywhere in industry. Grounding your answer in a concrete scenario immediately signals experience:

🚗 Real-Time Dashboards (Uber)

Dispatchers and city ops teams watch live ride counts, ETAs, and surge pricing per geo-zone. Data must refresh every 1–5 seconds across millions of concurrent trips.

🛡️ Fraud Detection

A payments platform scores every transaction in under 100 ms. The system ingests 50k+ events/sec and must maintain stateful feature aggregations (e.g., "transactions from this card in the last 10 minutes").

📦 Inventory Tracking

A retailer tracks stock-level changes across 5,000 stores. Each POS scan, receiving dock event, and online order triggers an inventory delta. Buyers need sub-minute visibility.

Key Requirements to Clarify

Before drawing a single box on the whiteboard, pin down the requirements:

Architecture Overview

Almost every real-time analytics system follows this five-stage spine:

Sources → Ingestion (Kafka) → Stream Processing (Flink/Spark) → Serving (OLAP DB) → Consumption (Dashboards / APIs)

1. Sources

Application databases, clickstream, IoT sensors, mobile SDKs, third-party webhooks.

2. Ingestion

Apache Kafka acts as the durable, replayable event backbone. All events land here first.

3. Stream Processing

Flink or Spark Structured Streaming performs filtering, enrichment, windowed aggregations, and joins.

4. Serving

A specialized OLAP database (Druid, Pinot, ClickHouse) stores data optimized for sub-second queries.

5. Consumption

Dashboards (Grafana, Superset, Tableau), REST/GraphQL APIs, or downstream microservices.

💡 Interview Insight

Start by clarifying latency requirements — "real-time" means different things to different people. Sub-100 ms is operational real-time (fraud scoring, bidding). Seconds is dashboard real-time (Uber city view). Minutes is near-real-time analytics (inventory reconciliation). Naming the tier explicitly shows the interviewer you've lived through production systems where this distinction caused real arguments.

Kafka as the Ingestion Backbone

Kafka is the gravitational center of every modern streaming architecture. Understanding it deeply — not just "it's a message queue" — separates senior candidates from junior ones.

Why Kafka?

Kafka provides five properties that no traditional message broker delivers simultaneously:

💾 Durability

Messages are written to disk and replicated across brokers.

📏 Ordering

Messages within a partition are strictly ordered by offset — critical for CDC where UPDATE must follow INSERT.

⏪ Replayability

Consumers can seek to any offset and reprocess data, enabling backfills and bug fixes.

📈 Horizontal Scalability

Add partitions and brokers to scale throughput linearly.

🔌 Decoupling

Producers and consumers are fully independent.

Kafka Architecture for RT Analytics

Producers → Topics (partitioned) → Consumer Groups → Stream Processors
  • Topics are organized by domain: rides.events, payments.transactions, inventory.updates. One topic per event type keeps schemas clean.
  • Partitions determine parallelism and ordering. A topic with 64 partitions can have up to 64 consumers in parallel. The partition key determines which partition a message lands on.
  • Consumer Groups enable independent processing. The fraud-detection group and the analytics-dashboard group both consume at their own pace.
  • Retention — Hot retention of 7 days is standard. With Kafka's Tiered Storage (KIP-405, GA in Kafka 3.6+), older segments offload to S3/GCS for weeks/months of replay.

Schema Management

Without schema governance, Kafka topics become the Wild West. The Confluent Schema Registry is the industry standard:

📐 Avro / Protobuf

Schemas registered per topic (subject = <topic>-value). Schema IDs embedded in message headers for efficient deserialization.

🔄 Compatibility Modes

BACKWARD — new schema can read old data. FORWARD — old schema can read new data. FULL — both directions (safest).

📝 Evolution Strategy

Add nullable fields with defaults. Never remove or rename fields without a migration plan.

Kafka Connect

Plug-and-play framework for moving data in and out of Kafka without custom code:

📥 Source Connectors

  • JDBC Source (poll tables)
  • Debezium (CDC from WAL/binlog)
  • S3 Source (file ingestion)

📤 Sink Connectors

  • Elasticsearch Sink (search indexing)
  • S3 Sink (data lake landing)
  • JDBC Sink (write to OLTP/OLAP)

Exactly-once semantics: Combine idempotent producers (enable.idempotence=true) with transactional consumers (isolation.level=read_committed). Kafka Connect supports this natively with exactly.once.source.support=enabled.

Kafka Streams vs. External Processing

Kafka Streams — Lightweight Java library inside your app process. Perfect for simple transforms: filtering, mapping, repartitioning.

External processing (Flink/Spark) — Required for complex aggregations, multi-stream joins, windowed computations, and ML feature engineering.

Rule of thumb: if your logic fits in a KStream.filter().mapValues().to() chain, use Kafka Streams. If you need windowed joins with watermarks, reach for Flink.

💡 Interview Insight

Mention partition key design early — it determines ordering guarantees and parallelism. For a ride-sharing analytics system, partitioning by driver_id guarantees all events for a driver are ordered. But if 5% of drivers generate 50% of rides, you get hot partitions. A composite key like city_id + driver_id can spread load more evenly. Interviewers love hearing you reason through this tradeoff.

Stream Processing: Flink vs Spark

The stream processor is the brain of your pipeline. The two dominant frameworks — Apache Flink and Spark Structured Streaming — take fundamentally different approaches.

Apache Flink Deep Dive

Flink was built from the ground up as a streaming engine. Batch is a special case of streaming in Flink's worldview.

⏱️ True Event-Time

Flink natively distinguishes event time, ingestion time, and processing time. Event-time processing with watermarks is first-class.

📝 Per-Record Processing

Each event is processed individually as it arrives. No micro-batch boundary. True millisecond-level latency.

🪨 RocksDB State Backend

Stores operator state in embedded RocksDB on local SSD. Enables terabytes of state without blowing up JVM heap.

💾 Savepoints for Upgrades

Trigger a consistent snapshot before deploying a new job version. Restart from savepoint with zero data loss.

✅ Exactly-Once

Chandy-Lamport distributed snapshots checkpoint operator state and Kafka offsets atomically. No duplicates on failure recovery.

Flink Windowing

Flink's windowing is the richest in the streaming ecosystem:

🪟 Tumbling Windows

Fixed-size, non-overlapping. Example: count rides per 1-minute window.

🔄 Sliding Windows

Fixed-size, overlapping. Example: count rides in last 5 minutes, updated every 30 seconds.

🕐 Session Windows

Dynamic, gap-based. Closes after N minutes of inactivity. Example: group a user's clickstream into sessions.

⏳ Late Data Handling

allowedLateness(Duration.ofMinutes(5)) keeps windows open. Even later events go to a side output for reconciliation.

Spark Structured Streaming Deep Dive

Spark Structured Streaming builds streaming on top of the Spark SQL engine. Same DataFrame/Dataset API for both batch and streaming.

📦 Micro-Batch

Polls Kafka every 100 ms (configurable), pulls a batch, processes as DataFrame. Simple and leverages Spark's SQL optimizer — but adds at least one trigger-interval of latency.

⚡ Continuous Mode

Experimental since Spark 2.3. Processes records one-at-a-time with ~1 ms latency. Supports only map-like transforms, no aggregations. Most teams stick with micro-batch.

🌊 Watermarks

withWatermark("event_time", "10 minutes") drops events older than 10 min past the watermark. No side output like Flink — late events are silently dropped.

🔧 Stateful Processing

mapGroupsWithState and flatMapGroupsWithState enable arbitrary stateful logic per key group, with timeout support.

State SizeTerabytes (disk-backed)Gigabytes (memory-bound)

💡 Interview Insight

Don't just say "Flink is better for streaming" — explain WHY. Per-record processing eliminates the micro-batch latency floor. RocksDB handles larger state than Spark's in-memory store (critical for high-cardinality aggregations like per-user metrics across millions of users). Flink's side outputs for late data give you more control than Spark's drop-or-keep binary. But also acknowledge Spark's strengths: if the team already runs Spark for batch ETL and latency requirements are seconds (not milliseconds), Structured Streaming avoids operating a second framework.

Serving Layer: OLAP Databases for Real-Time Queries

Your stream processor produces aggregations. Now they need to be queryable — by dashboards, APIs, and analysts — with sub-second latency at thousands of concurrent queries.

Why Not Just Use Snowflake or BigQuery?

⏱️ Query Compilation Overhead

Snowflake compiles each query, spins up compute, and executes. This adds 500 ms–2 s of baseline latency even for trivial queries.

🔥 Concurrency Limits

A dashboard with 200 panels auto-refreshing every 5 seconds generates 2,400 queries/minute. Warehouses throttle or charge exorbitantly.

🐌 Ingestion Latency

Loading data via COPY or Snowpipe takes seconds to minutes. For a 1-second dashboard, this is too slow.

Apache Druid

Column-oriented, distributed OLAP database designed for sub-second queries on event data.

📊 Pre-Aggregation (Rollup)

Roll up raw events at ingest time. 1 billion clicks → manageable counts per (page, country, hour). 100x storage reduction.

🧱 Segment-Based Storage

Immutable segments partitioned by time enable efficient time-based pruning.

⚡ Real-Time + Historical

Real-time nodes ingest from Kafka with sub-second latency; historical nodes serve from S3. Queries merge both.

Apache Pinot

Built at LinkedIn for user-facing analytics at massive scale ("Who viewed your profile" across 900M+ members).

🌟 Star-Tree Index

Pre-materialized tree of aggregations across dimension combinations. Queries return in single-digit milliseconds regardless of volume.

🔄 Upsert Support

Unlike Druid (append-only), Pinot supports row-level upserts by primary key. Critical for CDC-fed tables needing latest state.

👤 User-Facing Analytics

Optimized for "1000 queries/sec, each touching a different user's data." Low tail latency is a first-class design goal.

ClickHouse

Column-oriented MPP database originally built at Yandex for web analytics (billions of events daily).

🌲 MergeTree Engine

Data stored in sorted "parts" that merge in background (like LSM trees). Fast inserts and efficient range scans.

🔄 Materialized Views

Act as insert triggers: data lands in source → transformation runs → results inserted into target. Real-time pre-aggregation without a separate processor.

📝 Strong SQL Support

Rich SQL dialect: window functions, arrays, maps, JSON. Analysts query directly without learning a custom language.

💡 Interview Insight

Pick ONE of these and know it well. Most interviewers care that you understand why you'd use a specialized OLAP DB over a general-purpose warehouse. A strong answer: "We'd use Pinot here because we need upserts from our CDC pipeline and sub-10 ms P99 for user-facing queries. Snowflake can't deliver that concurrency at that latency." That reasoning is worth more than memorizing ClickHouse MergeTree variants.

The Dual-Write Pattern & Lambda Considerations

Production real-time analytics systems almost never have a single output. They write to at least two destinations — and managing the consistency between them is one of the hardest problems in data engineering.

Dual-Write Architecture

┌──→ Real-time OLAP (Druid/Pinot) ──→ Live Dashboards Kafka → Flink/Spark ────┤ └──→ Data Lake (S3/Delta Lake) ──→ dbt → Warehouse → BI Tools

The same stream processor writes to both a real-time serving layer and a data lake. This is the dual-write pattern (sometimes called the "Kappa + batch" hybrid).

Why Dual-Write?

⚡ Real-Time OLAP

  • Serves operational dashboards with sub-second latency
  • Retains 7–30 days of data
  • Optimized for pre-defined queries (known dimensions, known metrics)

🏔️ Data Lake (S3 + Delta)

  • Serves historical analytics, data science, ad-hoc exploration
  • Retains months or years
  • Optimized for full scans, joins, and ML feature engineering

Trying to serve both patterns from a single system always results in painful compromises. The OLAP DB can't handle full table scans across 2 years. The data lake can't serve 1,000 concurrent queries at 200 ms P99.

Consistency Challenges

⏳ Eventual Consistency

Real-time path may process at T+1s while the lake path at T+5min. Queries comparing the two during that window will see different numbers.

💥 Failure Modes

Flink might write to Druid but fail on S3 (or vice versa). Without transactional writes across both sinks, you get data in one but not the other.

🔍 Reconciliation

Run periodic (hourly/daily) reconciliation comparing counts. The lake is source of truth; the OLAP DB is "fast but approximate."

Backfill Scenarios

⏪ Kafka Replay

  • If retention covers the period, reset consumer offsets and replay
  • Simple but slow for large ranges

📦 Batch Backfill

  • Read historical data from the lake, process with same logic (Flink batch or Spark batch)
  • Faster, but must ensure batch logic matches streaming logic exactly

Cost Considerations

Running two output paths costs more. Justify each by mapping to business value: real-time OLAP prevents stock-outs and catches fraud (revenue impact); the data lake enables quarterly reviews, ML models, and regulatory reporting (analytical value). If the business only needs one path, don't build both.

💡 Interview Insight

Mentioning the dual-write pattern shows architectural maturity. Explain that real-time serves operational needs (decisions in seconds) while the lake serves analytical needs (decisions in days). Then proactively address consistency: "We accept eventual consistency between the two paths and run a daily reconciliation job. For the first 5 minutes, the OLAP DB may show slightly different numbers than the lake, and we document this SLA for stakeholders." That level of pragmatism is exactly what senior interviewers look for.

Designing a CDC Pipeline — End to End

Change Data Capture is how you get data out of transactional databases and into your analytics ecosystem without destroying the source database's performance or missing intermediate state changes.

What Is CDC?

CDC captures row-level changes — every INSERT, UPDATE, and DELETE — from a source database's transaction log. Instead of periodically scanning the entire table, CDC reads the database's internal changelog and emits each change as a structured event.

Why CDC Over Batch Extraction?

AspectBatch Extraction (Query-Based)CDC (Log-Based)
LatencyMinutes to hours (cron schedule)Seconds (continuous streaming)
Source DB LoadHeavy (full table scans)Minimal (reads transaction log)
Missed ChangesYes — rapid UPDATE→UPDATE loses intermediate stateNo — every change is captured
DeletesHard to detect (row is gone)Captured as DELETE event
SchemaWhatever the current table looks likeBefore/after snapshots per change

End-to-End Architecture

OLTP DB (Postgres) → Debezium → Kafka → Stream Processor (Flink/Spark) → Delta Lake (MERGE) → dbt → Warehouse → Dashboards

Debezium Deep Dive

Debezium is the de facto open-source CDC platform. It runs as a Kafka Connect source connector:

📸 1. Initial Snapshot

On first startup, Debezium snapshots existing table data, bootstrapping the Kafka topic with current state.

🔄 2. Continuous Streaming

After snapshot, reads new changes from the transaction log (logical replication slots in Postgres, binlog in MySQL).

🌐 4. Supported DBs

Postgres (logical replication), MySQL (binlog), MongoDB (oplog), SQL Server (CT/CDC), Oracle (LogMiner/XStream).

3. Change Event Structure

{ "op": "u", "before": { "id": 42, "status": "pending", "amount": 99.99 }, "after": { "id": 42, "status": "shipped", "amount": 99.99 }, "source": { "connector": "postgresql", "db": "orders_db", "table": "orders", "lsn": 234567890, "ts_ms": 1712345678000 }, "ts_ms": 1712345678050 }
  • op — Operation: c (create), u (update), d (delete), r (read/snapshot).
  • before — Row state before the change (null for inserts).
  • after — Row state after the change (null for deletes).
  • source.lsn — Log Sequence Number in Postgres; the exact position in the WAL. Your ordering guarantee.

Kafka Topic Design for CDC

📁 One Topic Per Table

cdc.orders_db.public.orders, cdc.orders_db.public.customers. Convention: <prefix>.<db>.<schema>.<table>.

🔑 Partition by PK

Set message key to primary key. Guarantees all changes for a row are processed in order by a single consumer.

🗜️ Log Compaction

Retains only the latest message per key — a "snapshot" of every row's current state. New consumers bootstrap from the compacted topic.

MERGE into Delta Lake

MERGE INTO orders_bronze AS target USING cdc_changes AS changes ON target.id = changes.id WHEN MATCHED AND changes.op = 'd' THEN DELETE WHEN MATCHED AND changes.ts_ms > target.ts_ms THEN UPDATE SET * WHEN NOT MATCHED AND changes.op != 'd' THEN INSERT *
  • The ts_ms > target.ts_ms condition handles out-of-order events (last-writer-wins with timestamp comparison).
  • Deletes are handled explicitly via the op = 'd' condition.
  • The Bronze layer contains a near-replica of the source table with minimal transformation.
💡 Interview Insight

Walk through the FULL pipeline end-to-end. Interviewers love when you trace a single record: "A customer updates their address → Postgres writes the UPDATE to the WAL → Debezium reads the WAL entry and produces a change event with before/after snapshots → the event lands in Kafka topic cdc.users_db.public.customers partitioned by customer_id → Spark consumes the event → executes MERGE INTO Delta Lake customers_bronze → dbt transforms to customers_gold → the BI dashboard shows the updated address." This end-to-end fluency is the hallmark of a senior data engineer.

CDC Challenges: Schema Evolution, Deletes & Late Events

Building the initial CDC pipeline is the easy part. Operating it for months and years — as source schemas change, as records are deleted, as events arrive late — is where the real engineering happens.

Schema Evolution

Source databases evolve constantly. Product teams add columns, change types, rename fields. Your CDC pipeline must handle this gracefully.

Change TypeImpactNotes
Adding nullable column✅ Backward compatibleOld events deserialize fine; new field defaults to null.
Removing a column❌ Breaking changeDownstream consumers that reference the column will fail. Update consumers first.
Renaming a column❌ Effectively remove + addSchema Registry sees a new field. Need a mapping layer during transition.
Changing column type⚠️ Potentially breakingINT→BIGINT is safe (widening). STRING→INT is not (narrowing). Registry rejects under BACKWARD.

Schema Evolution Strategy (Multi-Layer Defense)

🛡️ 1. Schema Registry

Enforce BACKWARD compatibility on all CDC topics. Prevents producers (Debezium) from registering schemas that break consumers.

🔺 2. Delta Lake

Enable mergeSchema = true on writes. Automatically adds new columns when they appear in incoming data.

🔧 3. dbt

New columns arrive as nullable. dbt models reference columns explicitly, so a new Bronze column doesn't break Gold until intentionally added.

Handling Deletes

🟢 Soft Deletes (Preferred)

  • Set is_deleted = true and deleted_at = timestamp
  • Row remains for historical analysis
  • Filter WHERE is_deleted = false in Gold/serving layer

🔴 Hard Deletes

  • Use MERGE DELETE clause to physically remove
  • Simpler, but lose audit trail
  • Can't answer "what was the address before deletion?"

Tombstone events in Kafka: Debezium produces a tombstone (null value with the key) after a DELETE. With log compaction, the tombstone eventually removes the key from the compacted topic.

Recommendation: Soft deletes in Bronze/Silver layers, hard deletes only in Gold if business rules require it. This preserves lineage while keeping downstream tables clean.

Late-Arriving Events

🌊 Watermarks

Define how long the processor waits before closing a window. 10-min watermark = tolerate up to 10 min delay. Later events are dropped (Spark) or side-output (Flink).

🔄 Reprocessing

For events past the watermark, reprocess affected lake partitions. A nightly "reconciliation" job compares source row counts and patches gaps.

🔒 Idempotent MERGE

The MERGE with ts_ms > target.ts_ms is inherently idempotent. Processing the same event twice doesn't corrupt data — the second MERGE is a no-op.

Ordering Challenges

Two rapid updates (pending → shipped → delivered) may arrive out of order. Solutions:

🔑 Partition by PK

All changes for a row go to the same partition and are processed in order.

⏰ Last-Writer-Wins

The MERGE condition changes.ts_ms > target.ts_ms ensures only the latest change is applied, regardless of processing order.

🔢 Version Tracking

If the source has a monotonically increasing version column, use it instead of timestamp for stronger ordering guarantees.

💡 Interview Insight

Schema evolution is the #1 operational challenge in CDC pipelines. When discussing CDC, proactively mention: "We enforce BACKWARD compatibility in the Schema Registry to prevent breaking changes from propagating. For additive changes like new nullable columns, Delta Lake's mergeSchema handles it automatically. For breaking changes like column renames, we coordinate a migration: deploy the consumer update first, then the producer change, with a mapping layer in between." This shows you've dealt with real-world CDC operations, not just built a demo.

Data Quality & Observability in Streaming

A pipeline that produces wrong data on time is worse than one that produces right data late. Data quality is not an afterthought — it's a design requirement.

Quality Checks at Every Layer

Ingestion Layer Transformation Layer Serving Layer ├─ Schema validation ├─ Business rule checks ├─ Freshness monitoring ├─ Null / type checks ├─ Referential integrity ├─ Completeness checks ├─ Deduplication ├─ Range / distribution ├─ Query latency SLAs └─ Volume anomalies └─ Cross-source consistency └─ Anomaly detection

Shift-left philosophy: Catch bad data at ingestion before it propagates downstream. A malformed event caught at the Kafka consumer level is a log entry. The same bad data caught in a BI dashboard is a trust-destroying incident.

Key Data Quality Metrics

MetricDefinitionExample SLA
Completeness% of expected records present≥ 99.5% of source rows in lake
FreshnessTime since most recent data pointDashboard data ≤ 30 seconds old
AccuracyValidation against source of truthAggregations match within 0.1%
VolumeRow count within expected rangeDaily count within ±2σ of 30-day avg
UniquenessNo duplicate primary keys0 duplicates in Gold tables

Data Quality Tools

✅ Great Expectations

Define expectations as code (expect_column_values_to_not_be_null("order_id")), run checkpoints at pipeline stages, generate HTML data docs, integrate with Airflow.

🧪 Soda

Data contracts in SodaCL (YAML-based): row_count > 0, freshness(created_at) < 1h. Monitoring dashboard tracks results over time.

🔧 dbt Tests

Schema tests (not_null, unique, accepted_values, relationships), custom SQL tests, and freshness tests with warn/error thresholds.

Data Lineage

When something breaks, you need two answers instantly: root cause (trace upstream) and blast radius (trace downstream).

🔗 Column-Level Lineage

Track which source columns feed which targets through every transformation.

💥 Impact Analysis

Before changing a column, see every downstream dependency. Tools: DataHub, OpenLineage (Marquez standard), dbt's built-in lineage graph.

Alerting Strategy

📊 1. Threshold-Based

Row count drops below X, freshness exceeds Y. Deterministic, low false-positives. Route to Slack.

📈 2. Anomaly Detection

Statistical deviation from baseline (3σ below 30-day rolling average). Route to Slack with higher urgency.

🚨 3. SLA Breaches

Pipeline hasn't completed by deadline. Route to PagerDuty.

📢 4. Escalation

Slack (info) → Slack (warning) → PagerDuty (critical) → Incident channel.

💡 Interview Insight

Frame data quality as "shift-left testing" — the same concept from software engineering applied to data. Mention specific, quantifiable metrics: "We monitor four dimensions — freshness (SLA: < 30 seconds for real-time, < 1 hour for batch), completeness (≥ 99.5% of source rows), volume (within 2σ of the 30-day mean), and uniqueness (zero duplicate PKs in Gold). We use Great Expectations checkpoints at ingestion and dbt tests at transformation, with Slack alerts for warnings and PagerDuty for SLA breaches." Concrete numbers and tool names signal hands-on experience.

Key Takeaways

  1. Always clarify latency tiers first. "Real-time" is ambiguous. Sub-100 ms (operational), seconds (dashboards), and minutes (near-real-time) each imply radically different architectures and cost profiles. Pin this down before drawing a single box.
  2. Kafka is the universal ingestion backbone. Its durability, ordering, replayability, and decoupling properties make it the default choice. Invest in partition key design — it determines both ordering correctness and throughput scalability.
  3. Choose Flink for low-latency, stateful streaming; Spark for unified batch + stream. Flink's per-record model and RocksDB handle use cases that Spark's micro-batch cannot. But Spark's ecosystem maturity reduces operational overhead when seconds-level latency is acceptable.
  4. Specialized OLAP databases exist because warehouses can't serve sub-second, high-concurrency queries. Know one well (Druid, Pinot, ClickHouse) and articulate why it's needed.
  5. The dual-write pattern is an architectural necessity, not a hack. Writing to both a real-time OLAP DB and a data lake serves fundamentally different access patterns. Accept eventual consistency and implement reconciliation.
  6. CDC via Debezium reads the transaction log, not the table itself. Lower latency, lower source load, captures every intermediate change including deletes. Trace a single record end-to-end.
  7. Schema evolution is the #1 operational challenge in CDC. Enforce compatibility modes in the Schema Registry, use Delta Lake's mergeSchema for additive changes, and coordinate migrations for breaking changes.
  8. Handle deletes with soft deletes in Bronze/Silver and hard deletes only in Gold. Preserves audit trail and lineage while keeping downstream consumers clean. Use Kafka tombstones for topic compaction.
  9. Data quality is shift-left testing applied to data. Check at ingestion (schema, volume anomalies), transformation (business rules, referential integrity), and serving (freshness SLAs, completeness). Use concrete metrics: freshness < 30s, completeness ≥ 99.5%, volume within 2σ.
  10. End-to-end traceability wins interviews. Tracing a record from source DB change → WAL → Debezium event → Kafka message → stream processor → Delta Lake MERGE → dbt model → dashboard demonstrates the fluency that distinguishes senior data engineers.
