HARD Design a real-time fraud detection system for a payment processor handling 50K txn/sec

Requirements: 50K TPS peak (~2B/day). End-to-end latency <200ms. Real-time features (velocity, geo-distance, amount deviation) scored against an ML model. False positive rate <1%. Historical feature lookups needed (30-day user profiles).

Payment Gateway
  → Kafka (transactions topic, 128 partitions, keyed by card_id)
    → Flink Stream Processor
      ├── Feature Computation (velocity, geo, amount z-score)
      ├── Feature Store Lookup (Redis — 30-day user profile)
      ├── ML Model Scoring (embedded ONNX model in Flink)
      └── Decision: approve / decline / review
          ├── approve → response back (<200ms)
          ├── decline → response + fraud ops alert
          └── review → case management queue (Kafka → Postgres)

Data Model:

  • Transaction Event: {txn_id, card_id, merchant_id, amount, currency, geo_lat, geo_lon, timestamp, channel, device_fingerprint}
  • User Profile (Redis): card_id → {avg_30d_amount, txn_count_24h, last_5_locations[], typical_merchants[], risk_score_baseline}
  • Feature Vector: {velocity_1h, velocity_24h, geo_distance_from_last, amount_zscore, is_new_merchant, device_trust_score, time_since_last_txn}

Deep Dive — Feature Computation in Flink:

  1. Velocity features — Flink keyed state per card_id. Tumbling windows (1h, 24h). RocksDB-backed state with incremental checkpoints every 30s.
  2. Geo-distance — Haversine distance from last known location. Flag if >500km in <2h ("impossible travel").
  3. Amount z-score — (current - avg_30d) / stddev_30d from Redis. Flag if >3σ.
  4. Model scoring — ONNX Runtime embedded in Flink operator. Inference <5ms.
StageBudget
Kafka produce + consume10ms
Feature computation (Flink)20ms
Redis feature store lookup5ms
ML model inference5ms
Decision + response10ms
Total~50ms typical, <200ms p99

Tradeoffs: Embedded ONNX model eliminates network hop (<5ms) but couples deployment. Flink keyed state for short-window features avoids Redis round-trip; 30-day profiles stay in Redis. Secondary "review" path prevents hard-blocking borderline cases (~2% volume, >98% automation).

💡 Interview Insight

Always state your latency budget upfront and break it down by component. This shows you understand system design isn't about drawing boxes — it's about making each box fast enough to hit the end-to-end SLA.

SENIOR Design a data pipeline for a social media feed (Twitter/Instagram scale)

Requirements: 500M MAU, 10K new posts/sec, 300K feed reads/sec. Feed shows followed accounts' posts ranked by relevance within 2 seconds.

Architecture — Fan-Out Decision (the critical choice):

ApproachHowProsCons
Fan-out-on-writePush to every follower's cacheRead is O(1)Celebrity problem: 50M writes/post
Fan-out-on-readQuery all followed users at readWrite is O(1)Read is O(N), slow
Hybrid (Twitter)Write for <10K followers, read for celebritiesBalancedMore complex
Post Created → Kafka (posts topic)
  → Fan-out Service (checks follower count)
    ├── <10K followers: write to Redis sorted sets
    └── >10K followers: store in Celebrity Posts table
  → Analytics Pipeline (Kafka → Flink → Data Lake)

Feed Read → Timeline Service
  ├── Fetch Redis timeline cache
  ├── Fetch celebrity posts for followed celebrities
  ├── Merge + Rank (lightweight ML ranker)
  └── Return top 50 posts

Data Model: Post: {post_id, author_id, content, media_urls[], created_at}. Timeline Cache (Redis Sorted Set): key = timeline:{user_id}, score = timestamp, keep last 800 posts. Follow Graph: {follower_id, followee_id} in graph DB or sharded Postgres.

Tradeoffs: Hybrid fan-out adds complexity but is necessary at scale. Redis timeline caches: 500M × 800 × 8 bytes ≈ 3.2 TB. Ranking model must be <50ms at read time; heavy ML features pre-computed offline.

HARD Design a real-time inventory tracking system for Walmart (4,700 stores, millions of SKUs)

Requirements: 4,700 stores, ~15M unique SKUs, ~100K SKUs/store. Events: POS sales, returns, receiving, adjustments, transfers. Real-time count per store/SKU with <30s latency. Auto-replenishment triggers. Handle Black Friday (10x spikes) + network outages.

Store POS / Receiving / Adjustments
  → Store Edge Gateway (local buffer if network down)
    → Kafka (inventory-events, partitioned by store_id)
      → Flink (keyed by store_id + sku_id)
        ├── Aggregate: running inventory count
        ├── State Store (RocksDB): current_qty, pending_transfers
        ├── Replenishment Check: qty < reorder_point → emit event
        └── Sink: Cassandra (partitioned by store_id)
          → Serving API + Analytics Pipeline (Kafka → S3 → Spark)

Data Model: Event: {event_id, store_id, sku_id, event_type, quantity_delta, timestamp, source_system}. State (Cassandra): PK((store_id), sku_id) → {current_qty, reorder_point, reorder_qty, pending_transfer_qty}.

Deep Dive — Edge Buffering: Stores lose connectivity. Edge gateway buffers locally (SQLite/embedded Kafka), replays on reconnect. Events are idempotent (keyed by event_id). Flink deduplicates via keyed state TTL.

Tradeoffs: Cassandra over Postgres for write volume; eventual consistency is acceptable (shrinkage exists). Flink state for speed + Cassandra for durability. Replenishment cooldown (4h per SKU) prevents over-triggering during flash sales.

SENIOR Design an A/B testing analytics platform

Requirements: 50+ concurrent experiments. Deterministic user assignment. Compute conversion, revenue, engagement metrics. Statistical significance detection. Guardrail metrics.

User Request → Assignment Service (hash(user_id + exp_id) % 100)
  → Feature flags / variant config → client
  → User interactions → Kafka
    → Flink: enrich with experiment assignments
      → Data Lake (Bronze → Silver → Gold)
        → Metric Computation (Spark / dbt)
          → Statistical Analysis (scipy, statsmodels)
            → Results Dashboard (confidence intervals)

Data Model: Experiment Config: {experiment_id, variants[], traffic_allocation, metrics[], guardrails[], min_sample_size}. Assignment: {user_id, experiment_id, variant} (deterministic hash). Metric Result: {experiment_id, variant, metric_name, mean, stddev, ci_lower, ci_upper, p_value, is_significant}.

Statistical Engine: Welch's t-test for continuous metrics, chi-squared for proportions. 95% CIs, p < 0.05. Sequential testing for safe peeking. Guardrails (p99 latency, error rate, crash rate) auto-alert on degradation. Sample size calculator with baseline rate, MDE, and power (80%).

Tradeoffs: Batch (daily Spark) is cheaper but 24h stale; on-demand is fresher but expensive. Hybrid: batch daily + ad-hoc drill-downs. Frequentist (industry standard) vs Bayesian (more intuitive "92% chance B is better") — choose by team literacy.

COMMON Design a Customer 360 platform

Requirements: Unify data from CRM, orders, clickstream, support tickets, email, social. Build a single customer profile. Serve segmentation, churn prediction, personalization, support context.

Sources (6+)
  → CDC / API Extraction → Kafka → Bronze (raw per source)
    → Identity Resolution Engine (Silver)
      ├── Deterministic: email, phone, loyalty_id
      ├── Probabilistic: name+address fuzzy (Jaro-Winkler >0.85)
      └── Output: unified_customer_id mapping
    → Profile Builder (Silver → Gold)
      ├── Merge on unified_customer_id
      ├── Conflict resolution: most-recent-wins / source-priority
      └── Derived: lifetime_value, churn_risk, preferred_channel
        → Gold: customer_360 table
          ├── Warehouse (BI / segmentation)
          ├── Redis (real-time support lookup)
          └── Reverse ETL → Salesforce, Marketo

Identity Resolution (the hard part): Step 1: Deterministic keys (~70% matches). Step 2: Probabilistic — Jaro-Winkler on name+address, threshold >0.85, use blocking by zip to avoid O(n²). Step 3: Graph-based — connected components for transitive matches.

Tradeoffs: Aggressive vs conservative matching (false merge vs split profile — start conservative). Batch vs real-time resolution (batch is simpler for fuzzy matching; real-time appends events to the latest resolved profile).

HARD Design a data quality platform for a large organization

Requirements: 500+ tables, 30+ pipelines. Define/execute/monitor quality rules. Quality scores per table. Lineage-aware alerting. Remediation workflows.

Pipeline Execution (Airflow / dbt / Spark)
  → OpenLineage events → Metadata Collector
  → Quality Rule Engine
    ├── Scheduled checks (Great Expectations / Soda)
    ├── Inline checks (dbt tests, Spark validation)
    └── ML anomaly detection (volume, distribution drift)
  → Quality Score Calculator
    ├── Per-table: weighted (freshness 30%, completeness 30%, validity 20%, uniqueness 20%)
    └── Per-pipeline: min(table scores)
  → Lineage-Aware Alert Router
    ├── Root cause table → alert owner only
    ├── Downstream → suppress, show "upstream failure"
    └── Channels: PagerDuty / Slack / email digest
  → Incident Management (Jira/ServiceNow)
  → Quality Dashboard (scorecard, trends, drill-down)

Lineage-Aware Alerting: Naive = alert every failing table (15 alerts for 1 root cause). Smart = use lineage graph to identify furthest-upstream failure, alert only that owner. Downstream shows "degraded — upstream dependency failure."

Tradeoffs: Centralized platform vs embedded in pipelines — hybrid is best (embed lightweight checks, aggregate to central via API). ML anomaly detection catches unknown-unknowns but has false positives; layer with threshold rules for known invariants.

SENIOR Design a cost optimization system for cloud data infrastructure

Requirements: Track/optimize costs across Spark/Databricks, S3/GCS, Snowflake/BQ, Kafka, Airflow. Attribute to teams/projects/pipelines. Anomaly detection and optimization recommendations.

Cloud Billing APIs (AWS Cost Explorer, GCP Billing, Databricks API)
  → Cost Collector (daily batch)
    → Cost Attribution Engine
      ├── Join billing with resource tags (team, project)
      ├── Untagged → heuristic attribution (cluster → DAG → team)
      └── Shared → proportional (Kafka by topic bytes, S3 by prefix)
    → Anomaly Detector
      ├── Z-score on daily cost (>3σ alert)
      ├── Week-over-week (>50% increase alert)
      └── New resource detection
    → Recommendation Engine
      ├── Storage: lifecycle policies (IA 30d, Glacier 90d)
      ├── Compute: idle clusters, right-sizing
      ├── Queries: full-table scans → suggest partitioning
      ├── Spot/preemptible for fault-tolerant workloads
      └── Zombie resources: unused tables (no reads 90d)
    → Dashboard + Savings Tracker

Tradeoffs: Tag-based vs heuristic attribution (mandate tags in IaC, heuristics as fallback). Automated vs recommended optimization — start with recommendations + approval; graduate to automation for well-understood patterns (e.g., auto-hibernate dev clusters at 7pm).

⚖️ Trade-off Discussions

Every senior interview includes trade-offs. They don't want the "right" answer — they want you to hold two opposing ideas and articulate when each wins.

CORE Batch vs Streaming Processing

BatchStreaming
DefinitionProcess in discrete chunks on scheduleProcess continuously as data arrives
ProsSimple to build/debug; cheaper (spot instances); easier exactly-once; mature toolingLow latency (seconds); real-time dashboards; event-driven fit
ConsStale data until next run; large catch-up on failureComplex state management; harder debugging; higher ops cost

Choose Batch: consumers tolerate hourly/daily freshness, large historical transforms, team lacks streaming expertise, cost priority.

Choose Streaming: latency <5 min, event-driven use cases (fraud, alerts, personalization), data naturally streams (clickstream, IoT).

💡 Most production systems are hybrid — streaming for time-sensitive paths, batch for historical aggregations and backfills. Say this to show pragmatism.

SENIOR Schema-on-Read vs Schema-on-Write

💥 Failure Scenarios

Interviewers love these because they reveal operational maturity. Anyone designs the happy path. Senior engineers design for failure. For each: blast radius → triage → fix → prevention.

HARD Kafka broker goes down mid-stream — what happens?

What Breaks: Partitions led by that broker become temporarily unavailable. Producers get NotLeaderForPartitionException and retry. Consumers rebalance (~10–30s). If replication factor is 1 — data is lost.

Triage: Check broker health. Verify ISR count — if ISR < min.insync.replicas, producers are blocked. Monitor consumer lag (Burrow).

Fix: Hardware failure → replace broker (Kafka auto-reassigns). Software crash → restart, it rejoins and catches up from replicas. Verify consumers resumed and lag is decreasing.

Prevention: Replication factor ≥ 3 with min.insync.replicas=2. Rack-aware replication across AZs. Consumer idempotency (MERGE/upsert on sink). Alert on ISR shrink and consumer lag.

HARD Spark job fails halfway through a 4-hour batch run — how do you recover?

What Breaks: 4 hours of compute wasted. Partial data on S3 (some partitions complete, others corrupt). Downstream DAGs time out or run on stale data.

Triage: Check Spark UI/driver logs — OOM? Data skew? Schema change? Determine which partitions/stages completed.

Fix — Design for Restartability:

  1. Partition-level checkpoints: Track completed partitions in a checkpoint table. On restart, skip completed ones.
  2. Delta/Iceberg transactions: Failed writes are auto-rolled back. Simply re-run.
  3. Idempotent writes: Use MERGE or INSERT OVERWRITE PARTITION — safe to re-run.

Prevention: Break 4-hour jobs into smaller increments. Dynamic resource allocation (spark.dynamicAllocation.enabled=true). Handle data skew (salted keys, AQE). Airflow retries with idempotent tasks.

SENIOR Source database schema changes without warning (column renamed)

What Breaks: Ingestion fails with column-not-found (strict) or silently produces NULLs (permissive). Downstream Silver/Gold tables fail or produce wrong results. Dashboards break. Trust lost.

Triage: Alert fires from schema change detector or dbt test. Identify which column changed. Check blast radius via lineage.

Fix: Update ingestion to alias new → old column name. Re-run for affected window. Validate downstream with quality checks.

Prevention: Schema Registry (Avro/Protobuf) on Kafka — producers can't change without backward-compatibility checks. Data Contracts — PR-like review with deprecation period. Nightly schema diff detection. Defensive ingestion: Bronze stores raw, schema enforcement in Silver (easier to fix and reprocess).

SENIOR Pipeline produces duplicate records — how do you detect and fix?

What Breaks: Inflated aggregations (revenue doubled). Join fan-outs. Business decisions on wrong numbers.

Detection: dbt unique test on PK. SQL: GROUP BY pk HAVING COUNT(*) > 1. Volume anomaly (row count doubles). Business user reports (worst way).

Fix: Deduplicate in Silver: ROW_NUMBER() OVER (PARTITION BY pk ORDER BY updated_at DESC) = 1. Root cause: Kafka rebalance reprocessing? → idempotent consumer. Pipeline retry? → use MERGE. Source sent duplicates? → dedup in Bronze→Silver. Multiple pipeline instances? → max_active_runs=1.

Prevention: Idempotent writes (MERGE on PK). Exactly-once (Kafka transactions + Flink checkpoints). dbt unique tests on every table. Use dbt tests or Great Expectations as PK enforcement layer.

HARD Corrupt data in Silver propagated to Gold and served to dashboards

Blast Radius: Gold tables wrong. Dashboards showing bad metrics. Business decisions may already be made on bad data.

Incident Response:

  1. Detect & Alert: Quality checks should have caught this. If not, first gap = quality coverage.
  2. Triage (15 min): Identify affected time range/tables. Lineage trace: Gold → Silver → Bronze → source. Is it bad source, transform bug, or infra failure?
  3. Contain: Time travel rollback: RESTORE TABLE gold_daily_revenue TO VERSION AS OF 42. Flag dashboards with banner: "Data under review."
  4. Fix: Fix Silver transform or source issue. Reprocess Silver from Bronze, then Gold from corrected Silver. Validate before re-serving.
  5. Post-Incident: Blameless post-mortem. Add quality checks that would have caught it. Update runbooks.

Prevention: Quality gates between layers (halt on failure). Time travel retention ≥7 days. ML-based anomaly detection for distribution shifts and volume anomalies.

SENIOR Kafka consumer lag is growing — consumers can't keep up

What Breaks: Processing falls behind. Real-time dashboards stale. If lag exceeds retention → data loss.

Triage: Check lag per partition. Is it uniform (consumer is slow) or concentrated (data skew / hot partition)? Are consumers alive or stuck?

Fix (by speed):

  1. Scale consumers (up to partition count max).
  2. Fix hot partitions — re-key or salt partition key.
  3. Optimize processing — batch writes, reduce per-record time, async I/O.
  4. Increase Kafka retention (buy time).
  5. Backpressure (Flink handles automatically; custom consumers need rate limiting).

Prevention: K8s HPA based on lag metric. Alert at 5min (warning) / 30min (critical). Load test at 2x peak. Partition count ≥ expected peak consumer count.

COMMON Airflow DAG fails at 3am — walk me through alerting and recovery

Alerting Chain: Task fails → retry (retries=3, delay=5min) → all exhausted → on_failure_callback → PagerDuty (pages on-call) + Slack #data-alerts. Alert includes: DAG, task, execution date, error log, Airflow UI link.

On-Call Response:

  1. Acknowledge within 15min SLA.
  2. Read logs — OOM? Source unavailable? Permission denied?
  3. Assess blast radius via dependencies / lineage.
  4. Fix: transient error → clear & re-trigger. Data issue → fix source, re-trigger. Code bug → fix, deploy, re-trigger.
  5. Verify downstream completed. Check data quality.
  6. Document in on-call handoff.

Prevention: SLA monitoring (DAG deadline alerts). Idempotent tasks. ExternalTaskSensor for dependencies. Per-DAG runbooks.

HARD Critical dimension table hasn't updated for 3 days — nobody noticed

Why Undetected: Pipeline "succeeded" but processed zero records. No freshness monitoring. Airflow task was "success" (ran without errors) but data is stale. Downstream JOINs used stale dimension values silently.

Fix: Investigate source stoppage (expired creds? dropped table? dead CDC connector?). Backfill 3 days. Reprocess Gold tables that used stale dimension.

Prevention — Multi-Layer Freshness:

  1. Table-level freshness SLI: SELECT MAX(updated_at) — alert if >24h stale.
  2. Row count checks: Alert if pipeline succeeds with 0 rows (silent failure).
  3. Freshness SLO: "dim_product updated within 24h, 99.5% of the time." Track compliance.
  4. Quality dashboard: Red/yellow/green for freshness, completeness, volume.
  5. Anomaly detection: dim_product updates daily ~2 AM. No update by 6 AM → auto-alert.

💡 Silent failures are more dangerous than loud ones. Monitor the data, not just the pipeline status. Key phrase: "I monitor data freshness and volume, not just Airflow task status."

⚡ Rapid Fire Q&A

No preamble. Short questions. Tight answers. This is the speed round.

1. What is backpressure in streaming?
When downstream can't keep up with upstream. Flink handles it automatically by slowing source reads. Without it: OOM or data loss.
2. CAP theorem in 15 seconds?
Distributed systems: pick 2 of 3 — Consistency, Availability, Partition tolerance. Since partitions are inevitable, you choose CP (HBase) or AP (Cassandra).
3. ACID vs BASE?
ACID = strict transactions (relational DBs). BASE = Basically Available, Soft state, Eventually consistent — relaxes consistency for scale (NoSQL).
4. SLA vs SLO vs SLI?
SLI = metric measured (freshness: 2h). SLO = target (freshness <2h, 99%). SLA = contractual commitment with consequences if SLO breached.
5. What is a dead letter queue?
Holding area for failed messages — malformed records, schema violations. Bad records go to DLQ instead of blocking the pipeline.
6. Event sourcing in one sentence?
Store every state change as an immutable event — current state is derived by replaying all events in order.
7. What is partition pruning?
Query engine skips partitions that don't match the filter. WHERE event_date = '2026-04-06' → only that partition is scanned.
8. Delta Lake vs Iceberg — one sentence each?
Delta: Databricks-native, JSON transaction log, best with Spark/Databricks. Iceberg: vendor-neutral, manifest-tree metadata, best for multi-engine (Spark+Trino+Flink).
9. What is data skew and how to fix it?
One partition has disproportionately more data → one task runs 100x longer. Fix: salted keys, Spark AQE skew join, pre-filter hot keys.
10. Explain watermarks in stream processing.
A timestamp declaring "all events before this point have arrived." Tells the processor when to close a window. Late events are dropped or sent to side output.
11. What is Z-ordering?
Multi-dimensional data layout co-locating related data in files. Z-order by (region, product_id) → queries on either column scan fewer files.
12. Star schema vs snowflake schema?
Star: central fact + denormalized dimensions (1 join). Snowflake: normalized sub-dimensions (more joins, less redundancy). Star preferred for BI performance.
13. What is a slowly changing dimension?
Dimension whose attributes change over time. Type 1: overwrite (lose history). Type 2: new row with valid_from/to (preserve history). Type 3: add previous-value column.
14. Explain idempotency in pipelines.
Same result whether you run once or ten times. Use MERGE/upsert, partition overwrite, or dedup on natural key. Critical for safe retries.
15. What is predicate pushdown?
Push filters to the storage layer so only matching data is read. Parquet supports this via min/max stats in row group metadata.
16. Hot partition — what and how to fix?
One partition gets disproportionate reads/writes → bottleneck. Fix: salt/random suffix on partition key, composite keys, or pre-split partitions.
17. What is a compacted topic in Kafka?
Retains only the latest value per key, deleting older records. Used for changelogs/state topics. Key-based retention, not time-based.
18. Event-time vs processing-time?
Event-time: when it actually happened (embedded timestamp). Processing-time: when system processes it. Event-time is correct but needs watermarks. Processing-time is simpler but inaccurate for late data.
19. What is a feature store?
Centralized ML feature management. Offline store (data lake, batch training) + online store (Redis, low-latency inference). Prevents train-serve skew.
20. What is data lineage and why does it matter?
Tracks how data flows source→destination through transforms. Matters for: debugging (root cause), impact analysis, compliance (PII tracking), and trust (provenance).

🎯 The 6-Step Design Framework

Memorize this. Internalize it. Use it for every question. The steps should feel as natural as breathing.

Step 1: Clarify Requirements (3–5 min)

Never start drawing. Start asking. Functional: sources, consumers, queries. Non-Functional: latency, volume, SLA, compliance. Scale: events/sec, growth rate. Signal: "Before I design anything, let me clarify scale and latency requirements."

Step 2: High-Level Architecture (5–7 min)

Draw: Sources → Ingestion → Storage → Processing → Serving → Consumers. Plus Orchestration and Governance. Name technologies and explain why: "Kafka for ingestion because we need 50K TPS with ordering and durability."

Step 3: Data Model (3–5 min)

Show key schemas. Bronze/Silver/Gold layers. Primary entity, key attributes, partition key, file format choice (Parquet for analytics, Avro for streaming, JSON for flexibility).

Step 4: Deep Dive (10–15 min)

Pick 2–3 critical components. Go deep on internals: CDC vs batch ingestion, Spark optimizations, Flink stateful processing, data quality rules, query performance, caching. This is where you differentiate.

Step 5: Failure Handling (3–5 min)

Proactively address: "Here's what happens when things go wrong." Source down? (Kafka retention, retries). Job fails? (Idempotent writes, checkpoints). Bad data? (Quality gates, DLQ, time travel rollback). Monitoring? (SLIs, SLOs, alerting).

Step 6: Cost & Scale (2–3 min)

How does this scale to 10x? What are cost drivers? Where to optimize first? (Spot instances, storage tiering, query optimization). Growth plan: start managed, graduate to self-hosted at scale.

🏆 10 Golden Rules

1

Ask before you draw. 3–5 min on requirements = #1 differentiator between mid and senior.

2

Name technologies and justify them. "Kafka because we need ordered, durable, replayable streaming at 50K TPS" — not just "a message queue."

3

State your latency budget. Whether 200ms for fraud or 6 AM for a dashboard — make the constraint explicit.

4

Show the data model. Schemas, partition keys, format choices. A design without a data model lacks substance.

5

Talk about failure before they ask. "Let me address what happens when this component fails." Signals operational maturity.

6

Quantify everything. "50K events/sec × 1KB = 50MB/sec → Kafka with 32 partitions handles this easily." Numbers build credibility.

7

Discuss tradeoffs, not just choices. "Flink over Spark Structured Streaming for sub-second latency. Tradeoff: more operational complexity for state and checkpoints."

8

Layer in governance. Data quality checks, access control, lineage, monitoring. Separates senior from mid-level.

9

Don't over-engineer. Daily batch with 10GB/day ≠ Flink with exactly-once. Match complexity to requirements.

10

Practice the whiteboard. Explain your design out loud while sketching. The framework should take 35–40 min, leaving 5–10 for Q&A.

🔥 The Final Word

You've survived the Gauntlet. No one expects every answer memorized. Interviewers want your framework — how you break down ambiguity, reason about tradeoffs, and design for the real world where things fail, data is messy, and budgets are finite. Requirements → Architecture → Data Model → Deep Dive → Failure Handling → Cost & Scale. Make it second nature. Now close this, open a blank whiteboard, and design a system from scratch. You've got this.
