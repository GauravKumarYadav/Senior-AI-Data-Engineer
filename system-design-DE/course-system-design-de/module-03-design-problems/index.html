🚕 Uber Surge Pricing Pipeline

Design a real-time data pipeline that powers Uber's surge pricing. The system must continuously calculate price multipliers for every geographic zone based on real-time supply and demand, blended with historical demand patterns.

Demand & Supply Signals

📈 Demand Signals

  • 🔸 Ride requests per area per minute
  • 🔸 Destination hotspots (concerts, airports)
  • 🔸 App opens without booking (latent demand)
  • 🔸 Scheduled ride requests for upcoming hours

🚗 Supply Signals

  • 🔹 Active drivers per area (GPS every 4s)
  • 🔹 Driver ETA to zone (adjacent zones)
  • 🔹 Driver shift patterns (predicted drop-off)
  • 🔹 Ride completions about to free a driver

Geo-Spatial Partitioning with H3 Hexagons

Uber uses H3, a hierarchical hexagonal grid system. Why hexagons? — 6 equidistant neighbors, hierarchical resolutions, and a single integer index for aggregation.

🔷 Uniform Tiling

Hexagons tile the earth's surface without distortion near the poles, unlike square grids.

🔗 6 Equidistant Neighbors

Squares have 4 edge + 4 corner neighbors at different distances. Hexagons have 6 equidistant neighbors — no bias.

📐 Hierarchical

Res 7 hexagons (~5.16 km²) nest cleanly inside res 6 (~36.13 km²). Zoom in or out as needed.

For surge pricing, resolution 7 (~1.2 km edge) is the sweet spot: granular enough to detect a stadium emptying, broad enough for meaningful supply counts.

Flink Streaming Job

Tumbling windows of 60 seconds, keyed by h3_index. Computes: request_count, active_drivers, supply_demand_ratio. Emits per-hex features to the pricing service.

Batch Pipeline

Daily Spark job trains an ML model predicting baseline demand per hex per hour. Incorporates day-of-week seasonality, holidays, weather, local events. Predictions written to a feature store (Redis) for sub-ms lookup.

⚠️ Train-Serve Skew & Serving Flow

The Train-Serve Skew Problem

The most insidious bug in ML systems: if features used during training differ from those used during serving, the model performs poorly in production despite great offline metrics.

🐛 Different Code Paths

Spark SQL computes a feature one way; Python serving code computes it differently. Subtle divergence = silent degradation.

⏰ Different Freshness

Training uses T-1 daily data. Serving uses real-time data. The model never saw real-time feature distributions.

✅ The Fix

Compute features ONCE in a shared pipeline. Both training and serving read from the SAME store, same code, same definitions.

End-to-End Serving Flow

User opens app → API Gateway → Recommendation Service → Fetch user features from online store (Redis) ~5ms → Fetch candidate item features from online store ~5ms → Run model inference (TensorFlow Serving) ~20ms → Apply business rules (filter out-of-stock, etc.) ~2ms → Return top 20 recommendations ← Response to client Total: ~35ms

Feature Registry (Metadata Layer)

📋 Schema Definitions

Name, type, description, owner for every feature.

🔄 Versioning

Feature v1 vs v2 with backward compatibility.

🔗 Lineage

Which raw sources feed which features.

📊 Monitoring

Data quality checks, drift detection, freshness SLAs.

💡 Interview Insight

The key insight is the DUAL feature store — offline for training, online for serving. Show you understand the train-serve skew problem: "If training and serving compute features differently, model accuracy degrades silently in production. The feature store ensures both paths use identical feature definitions." This separates senior candidates from juniors.

💳 Event-Driven Fintech Architecture

Design a transaction processing platform for a fintech company (Stripe/Square). Financial-grade correctness, real-time fraud detection, and an immutable audit trail.

Event Sourcing — The Foundation

In fintech, you never update or delete a record. Every state change is an immutable event:

Event 1: TransactionCreated { txn_id: T-100, amount: $50.00, merchant: "Coffee Shop", ts: 14:00:01 } Event 2: TransactionAuthorized { txn_id: T-100, auth_code: "A-999", ts: 14:00:02 } Event 3: TransactionSettled { txn_id: T-100, settled_amount: $50.00, ts: 14:01:30 } --- Event 4: TransactionCreated { txn_id: T-101, amount: $500.00, merchant: "Electronics", ts: 14:02:00 } Event 5: TransactionDeclined { txn_id: T-101, reason: "FRAUD_SUSPECT", ts: 14:02:00 }

The current account balance is a derived view — replay all events from the beginning and you reconstruct it perfectly.

CQRS — Separating Reads from Writes

WRITE SIDE READ SIDE API Request ──► Validate ──► Append to ──► Projection ──► Materialized (command) business Event Store (Flink/ Views: rules (Kafka + consumer) • Account Balance append-only • Transaction History DB) • Monthly Statement • Analytics Cube

Exactly-Once Processing

Financial systems cannot double-charge or lose a transaction. End-to-end exactly-once requires:

🔑 Idempotency Keys

Every API request includes a client-generated UUID. Retries return the original result.

⚛️ Kafka Transactions

Producer writes to multiple topics atomically. Consumer commits offsets with the DB write.

🔄 DB Upserts

INSERT ... ON CONFLICT (txn_id) DO NOTHING — safe to replay.

💡 Interview Insight

Event sourcing + CQRS is the gold standard for fintech. Say: "I'd use event sourcing because financial regulations require an immutable audit trail — every state change is an append-only event. CQRS separates the write path from the read path (materialized views)." Then emphasize exactly-once processing with idempotency keys.

🕸️ Data Mesh & SCD at Scale

Data Mesh is an organizational paradigm, not a technology. It addresses the failure mode of centralized data teams becoming bottlenecks.

Four Principles

1️⃣ Domain-Oriented Ownership

Each business domain team (Payments, Inventory, Customer) owns their data end-to-end. The Payments team builds the payments data product because they understand payment logic best.

2️⃣ Data as a Product

Discoverable, addressable, self-describing, interoperable, trustworthy. Standard URIs, schema registry, published SLAs for freshness and accuracy.

3️⃣ Self-Serve Platform

Platform team provides Spark, Flink, Airflow, dbt, S3 as managed services. Domain teams build without managing infrastructure.

4️⃣ Federated Governance

Global: PII rules, naming, retention, access control. Local: teams choose scheduling, transformation logic, internal models.

Challenges (discuss in interviews — shows maturity)

🔄 Org Change

Organizational change management is harder than any technology migration.

📋 Data Duplication

Payments and Customer both store customer data. Cross-domain joins require contract negotiation.

⚖️ Platform Sizing

Too small → teams blocked. Too large → becomes the new centralized team.

SCD Handler at Scale — Spark + Delta Lake MERGE

Requirements: Track 500M dimension records with daily changes affecting ~2% (10M changes/day).

MERGE INTO dim_product_scd2 AS target USING staged_changes AS source ON target.product_id = source.product_id AND target.is_current = true -- Close the old record WHEN MATCHED AND (target.price != source.price OR target.category != source.category) THEN UPDATE SET valid_to = source.change_date, is_current = false -- Insert the new version WHEN NOT MATCHED THEN INSERT (product_key, product_id, name, price, category, valid_from, valid_to, is_current) VALUES (generate_surrogate_key(), source.product_id, source.name, source.price, source.category, source.change_date, '9999-12-31', true);

Optimizations

#️⃣ Hash Partition

Partition by hash of product_id (mod 256) — even distribution, avoids hot partitions from sequential IDs.

📥 Incremental CDC

Only changed records delivered. MERGE only touches affected partitions.

📊 Z-Ordering

Co-locates data for the same product across files, making lookups fast.

Time-Travel Queries

-- What was the product's price on Black Friday 2024? SELECT * FROM dim_product_scd2 WHERE product_id = 'SKU-12345' AND valid_from <= '2024-11-29' AND valid_to > '2024-11-29'; -- Or with Delta Lake time travel: SELECT * FROM dim_product_scd2 VERSION AS OF '2024-11-29' WHERE product_id = 'SKU-12345' AND is_current = true;

🧩 Cross-Cutting Concepts: Building Blocks

These concepts appear in every system design interview. Master them once, apply them everywhere.

Exactly-Once Processing

True end-to-end exactly-once requires all three:

1️⃣ Idempotent Producers

Unique ID per message; the broker deduplicates on retry.

2️⃣ Transactional Consumers

Commit consumer offset AND downstream write in one atomic transaction.

3️⃣ Idempotent Sinks

UPSERT/MERGE at the destination so replayed messages don't create duplicates.

Reality check: Most systems achieve effectively once through idempotency — it's safe to process a message multiple times because the result is the same.

Hot Partition Fix with Salting

# Problem: account "uber_eats" generates 50% of all events # Solution: salt the partition key partition_key = f"{account_id}_{random.randint(0, 9)}" # Now "uber_eats" spreads across 10 partitions # Read query must fan out to all 10 and merge results

Backfill Strategies

🔄 Full Reprocess

Re-run entire pipeline from raw data. Simple but expensive. For schema changes or logic rewrites.

📦 Incremental Backfill

Reprocess only affected partitions. Use airflow dags backfill -s ... -e ....

⏪ Kafka Replay

Reset consumer group offset to a past timestamp. Reprocess without re-ingesting.

🟢 Blue-Green Tables

Write backfilled data to new table, validate, then swap. Zero downtime, easy rollback.

🔧 Idempotency, Late Data & Schema Evolution

Idempotent Pipelines

Same input → same output, no matter how many times you run it.

🔄 MERGE/UPSERT

INSERT ON CONFLICT UPDATE — safe to replay any number of times.

📦 Partition Overwrite

INSERT OVERWRITE PARTITION — replaces the entire partition atomically.

🔍 Deduplication

SELECT DISTINCT on (event_id, timestamp) before writing.

⚙️ Deterministic

Avoid now(), random(), or non-deterministic UDFs in transforms.

Late-Arriving Data

🌊 Watermarks

Define how long to wait (e.g., 5 min after window end). Events within the watermark are included normally.

⏰ Allowed Lateness

Events after the watermark but within "allowed lateness" are processed and flagged as late.

🔁 Reprocessing

Data arriving days late triggers a re-run of the affected partition. Idempotent pipeline overwrites the old result.

Schema Evolution at Scale

Best practices: Schema registry enforces compatibility on every write. New columns always have defaults. Never rename or delete — add new ones, deprecate old ones.

🎯 Key Takeaways

  1. Streaming + batch merge is the standard pattern for real-time systems. The streaming path provides low-latency signals; the batch path provides historical context and ML predictions. Design both paths and show how they merge at the serving layer.
  2. H3 hexagonal grids are the industry standard for geo-spatial partitioning. 6 equidistant neighbors, hierarchical resolutions, a single integer index for GROUP BY. Resolution 7 (~5 km²) is the common starting point.
  3. Define the grain first, always. Before drawing any schema, state: "The grain of this fact table is one row per ___." This single sentence drives every downstream decision about dimensions, measures, and aggregations.
  4. SCD Type 2 preserves history through valid_from, valid_to, and is_current. At scale (500M+ records), use Delta Lake MERGE with incremental CDC — never full-scan when only 2% of records change.
  5. The dual feature store solves train-serve skew. Offline store (S3/Delta) for training, online store (Redis/DynamoDB) for serving, both fed by the same feature pipelines. This is the #1 architecture pattern for production ML systems.
  6. Tiered storage (hot/warm/cold) optimizes cost vs. query speed. 7 days hot in Elasticsearch, 90 days warm in Parquet+Athena, years cold in Glacier. Always present this hierarchy in log analytics designs.
  7. Event sourcing + CQRS is the gold standard for fintech. Immutable events provide the audit trail regulators require. CQRS separates append-only writes from query-optimized reads. Exactly-once via idempotency keys + transactional consumers + UPSERT sinks.
  8. Data Mesh is an organizational pattern, not a technology. Domain ownership, data as a product, self-serve platform, federated governance. The hardest challenge is people and culture, not tooling.
  9. Idempotent pipelines are non-negotiable. Same input → same output regardless of how many times it runs. MERGE/UPSERT, partition overwrite, dedup on event_id, deterministic transformations.
  10. Use the 6-step design framework in every interview: Clarify requirements → High-level architecture → Data model → Deep-dive components → Failure handling → Cost and scale.
