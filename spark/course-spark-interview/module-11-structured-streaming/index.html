The Structured Streaming Model

Forget everything you know about traditional stream processing. Structured Streaming flips the model: every streaming source is just a table that keeps getting new rows appended to it. Your query runs against this ever-growing unbounded table, and Spark figures out how to execute it incrementally.

The brilliance of this design is API unification. The same DataFrame/Dataset/SQL code you write for a batch job works for streaming — no new abstractions. A groupBy().count() just works, and Spark handles incremental execution automatically via the Catalyst optimizer and Tungsten engine.

This is a fundamentally different execution model from DStreams. DStreams were RDD-based micro-batches — a loop of tiny batch jobs with no native event-time support. Structured Streaming gets predicate pushdown, whole-stage code generation, logical plan optimization, and native event-time windowing for free.

# End-to-end: Kafka → Windowed Agg → Console from pyspark.sql import SparkSession from pyspark.sql.functions import window, col spark = SparkSession.builder.appName("Demo").getOrCreate() # Step 1: Read from Kafka df = (spark.readStream .format("kafka") .option("kafka.bootstrap.servers", "broker:9092") .option("subscribe", "clickstream") .option("startingOffsets", "latest") .load() ) # Step 2: Parse — same DataFrame API as batch! parsed = (df .selectExpr( "CAST(value AS STRING) AS raw_event", "timestamp AS kafka_ts" ) .where("raw_event IS NOT NULL") ) # Step 3: Windowed aggregation windowed = (parsed .groupBy(window("kafka_ts", "5 minutes")) .count() ) # Step 4: Write to console query = (windowed.writeStream .outputMode("update") .format("console") .start() ) query.awaitTermination()

Under the hood, Spark doesn't re-run the entire query on each trigger. The planner converts your logical plan into an incremental execution plan. For a groupBy().count(): read only new data, update an internal state store (key → running count), and emit results based on the output mode.

💡 Interview Insight

"Structured Streaming is NOT micro-batching DStreams. It's a completely different execution model built on Catalyst and Tungsten, using DataFrames — which means you get all the optimizations (predicate pushdown, whole-stage code generation) for free. The key abstraction is the unbounded table: your query is logically applied to the entire input history, but Spark executes it incrementally using a state store."

Output Modes — Append, Complete, Update

When Spark finishes processing a micro-batch, it has an updated Result Table. The output mode controls which part of that result table gets written to the sink.

📤 Append Mode

Only newly added rows are written. Once emitted, rows are final — never changed or retracted. Like a printer: once printed, it's done. Requires watermark for aggregations.

📋 Complete Mode

The entire result table is written every trigger. Previous output is overwritten. Like a whiteboard — erase and redraw every time. Only works with aggregations.

🔄 Update Mode

Only changed rows are written. Most flexible — works with and without aggregations. Like a notification system: you only hear about changes.

Common mistakes: (1) Append mode + aggregations without watermark → AnalysisException. (2) Complete mode with huge cardinality → sink overwhelmed. (3) Update mode with file sink → AnalysisException (files are immutable).

💡 Interview Insight

"Start with update mode as your default mental model — it's the most flexible and efficient. Switch to append for write-once sinks like raw Parquet files or Kafka topics. Use complete only when the downstream system needs the full picture every time, and only when the result table has bounded cardinality."

Triggers — When Does Processing Happen?

A trigger defines when Spark kicks off the next micro-batch. It doesn't control what data is processed — that's determined by source offsets. It controls the cadence of processing.

⚡ Default (No Trigger)

Next batch starts immediately after the previous one finishes. Lowest latency (~100ms–few seconds). Standard "streaming" mode for always-on clusters.

⏱️ Fixed Interval

.trigger(processingTime="30s") — waits at least 30s between batch starts. If a batch takes longer, starts immediately. A polite neighbor on shared clusters.

🔂 Trigger Once

.trigger(once=True) — processes all available data in one batch, then stops. The underrated superpower: streaming reliability + batch cost efficiency.

📦 Available Now (3.3+)

.trigger(availableNow=True) — like Once, but splits into multiple batches respecting rate limits. Prevents OOM on large backlogs.

# Trigger Once — scheduled batch superpower # Run hourly in Airflow, keep exactly-once query = (result.writeStream .trigger(once=True) .format("delta") .option("checkpointLocation", "/chk/my_q") .start("/output/events") ) # Available Now — backlog-safe version query = (result.writeStream .trigger(availableNow=True) .option("maxOffsetsPerTrigger", 100000) .format("delta") .start("/output/events") ) # Continuous (experimental) — true ~1ms latency query = (result.writeStream .trigger(continuous="1 second") .format("kafka") .start() )

Watermarks & Late Data

In reality, events arrive late — mobile apps buffer on subways, sensors lose connectivity, Kafka consumers lag. If you're doing event-time windowed aggregations, late data means a window you thought was complete can get new events at any time.

Without any mechanism to handle this, Spark keeps every window's state forever → unbounded state growth → OutOfMemoryError.

A watermark tells Spark: "Data can arrive late, but I don't care about data more than X time units late. Discard it." The watermark is defined as max(event_time) - threshold. It only moves forward, never backward.

# Full example: Kafka → watermarked agg → Parquet from pyspark.sql.functions import window, col, from_json from pyspark.sql.types import StructType, StringType, TimestampType schema = (StructType() .add("user_id", StringType()) .add("event_time", TimestampType()) ) events = (spark.readStream .format("kafka") .option("subscribe", "user_events") .load() .select(from_json(col("value").cast("string"), schema).alias("data")) .select("data.user_id", "data.event_time") ) # Watermark MUST come before groupBy windowed = (events .withWatermark("event_time", "10 minutes") .groupBy( window("event_time", "5 minutes"), "user_id" ) .count() ) query = (windowed.writeStream .outputMode("append") .format("parquet") .option("checkpointLocation", "/chk/uc") .start("/output/user_counts") )

Choosing the right duration: Too short (1 min) → lose legitimate late data. Too long (24 hrs) → huge state in memory. Analyze your lateness distribution — if 99.9% arrive within 15 minutes, set 15–20 min. Many production systems use 2–3× expected maximum lateness.

💡 Interview Insight

"Watermarks solve two problems simultaneously: they prevent unbounded state growth (which would cause OOM), and they define when results are 'final' enough to emit in append mode. The formula is simple: watermark = max(event_time) - threshold. Any event older than the watermark is dropped. In production, always set a watermark for windowed aggregations — the question is not if, but how long."

Streaming Joins

Streaming joins come in three flavors, each with increasing complexity. Understanding the differences separates candidates who've built streaming pipelines from those who've only read the docs.

Stream-Static Joins: Join a streaming DataFrame with a regular (batch) DataFrame. No state store needed — the static side is just a lookup table, cached and reused across micro-batches.

# Stream-Static: enrich events with user data user_profiles = spark.read.parquet("/data/users") click_events = (spark.readStream.format("kafka") .option("subscribe", "clicks").load() .selectExpr("CAST(value AS STRING) AS user_id") ) enriched = click_events.join(user_profiles, "user_id") # Stream-Stream: match impressions with clicks impressions = (spark.readStream.format("kafka") .option("subscribe", "impressions").load() .selectExpr("CAST(value AS STRING) AS ad_id", "timestamp AS imp_time") .withWatermark("imp_time", "2 hours") ) clicks = (spark.readStream.format("kafka") .option("subscribe", "clicks").load() .selectExpr("CAST(value AS STRING) AS ad_id", "timestamp AS click_time") .withWatermark("click_time", "3 hours") ) matched = impressions.join(clicks, expr(""" impressions.ad_id = clicks.ad_id AND click_time >= imp_time AND click_time <= imp_time + INTERVAL 1 HOUR """))

💡 Interview Insight

"Stream-stream joins are the hardest Structured Streaming topic. The key insight is about state management: inner joins work without watermarks but state grows forever. Outer joins REQUIRE watermarks because Spark needs to know when to emit NULL for unmatched rows — it's the deadline for 'giving up' on finding a match. Always pair watermarks with a time-range condition in the join predicate to bound state size."

Exactly-Once & Fault Tolerance

End-to-end exactly-once processing requires three components working together. If any one is missing, you degrade to at-least-once (or worse).

🔄 Replayable Source

The source must support reading from a specific offset. Kafka does this natively. File sources use file lists as offsets. A random TCP socket does NOT.

💾 Checkpointing

Spark records processing progress in a checkpoint directory. Before each batch, offsets are logged. On failure, Spark resumes from the last committed batch.

✅ Idempotent Sink

Since Spark replays on failure, the sink may receive duplicates. Must handle gracefully — file overwrite, Delta Lake MERGE, or application-level dedup.

Recovery flow: (1) Read commits/ to find last completed batch. (2) Read offsets/ to find planned-but-not-committed batch. (3) Replay that batch from source. (4) Idempotent sink handles any duplicate writes from the failed attempt.

Checkpoint compatibility: Safe changes include adding output columns, relaxing filters, changing trigger intervals. Breaking changes (require new checkpoint): changing grouping keys, window duration, watermark threshold, or adding/removing stateful operations. Pro tip: version your checkpoint directories (/checkpoints/my_query/v2/) for rollback safety.

💡 Interview Insight

"Exactly-once in Structured Streaming requires three things working together: a replayable source (like Kafka with offset-based replay), an idempotent sink (like Delta Lake with MERGE or file overwrite semantics), and checkpointing (which stores offset ranges and state). If any one is missing, you degrade to at-least-once. The formula is: replayable source + idempotent sink + checkpointing = exactly-once."

Quiz — Test Your Streaming Knowledge

Q1: What's the difference between append, complete, and update output modes?

A) They all write the same data, just to different sink types
B) Append = new rows only; Complete = entire result table; Update = only changed rows
C) Append = aggregated results; Complete = raw events; Update = deleted rows
D) They're aliases for the same behavior
✅ Append writes only newly added rows (once, final). Complete rewrites the entire result table every trigger (only works with aggregations). Update writes only rows that changed since the last trigger — most flexible. Key gotcha: append mode with aggregations requires a watermark because Spark needs to know when a window's result is final.

Q2: Your streaming aggregation runs out of memory after a few hours. Most likely cause and fix?

A) The Kafka topic has too many partitions — reduce them
B) The output mode is wrong — switch to append
C) A watermark is missing — without it, Spark keeps all windowed state forever
D) The executor memory is too low — just increase spark.executor.memory
✅ Without a watermark, Spark keeps state for every window ever seen — state grows linearly with time → OOM. Adding .withWatermark("event_time", "10 minutes") tells Spark to discard state for windows older than the watermark. Increasing memory (D) only delays the inevitable.

Q3: You want to run a streaming pipeline hourly in Airflow with exactly-once semantics between runs. What trigger?

A) .trigger(processingTime="1 hour") — it pauses for an hour between batches
B) .trigger(once=True) or .trigger(availableNow=True) — process all available data then stop
C) .trigger(continuous="1 hour") — runs continuously with 1-hour checkpoints
D) No trigger needed — just kill the job and restart it hourly
✅ Trigger.Once() processes all available data in a single batch and stops, saving checkpoint state. Trigger.AvailableNow() (Spark 3.3+) does the same but splits into multiple batches for better memory management. Both maintain checkpoint state between runs. Option A keeps the app running for the full hour, wasting resources. Option D loses checkpoint state.

Q4: Can you perform an outer join between two streams without a watermark?

A) Yes — Spark buffers everything and eventually matches
B) No — outer joins require watermarks so Spark knows when to emit NULL for unmatched rows
C) Yes, but only for full outer joins, not left or right
D) No — stream-stream joins of any type require watermarks
✅ For outer joins, Spark needs to decide when an unmatched row will never find a partner — that's when it emits (row, NULL). Without a watermark, it can never make this determination. Inner joins work without watermarks (D is wrong) but should still have watermarks to prevent unbounded state growth.

Q5: What three components are needed for end-to-end exactly-once processing?

A) Kafka, Spark, and HDFS
B) Watermarks, triggers, and output modes
C) Replayable source, idempotent sink, and checkpointing
D) State store, RocksDB, and Delta Lake
✅ The three pillars: (1) Replayable source — can re-read from a specific offset (Kafka, files). (2) Idempotent sink — handles duplicate writes gracefully (Delta Lake, file overwrite). (3) Checkpointing — records offsets and state for recovery. Kafka/HDFS (A) are specific technologies, not abstract requirements. RocksDB/Delta (D) are implementation choices.

Q6: A streaming query fails mid-batch and restarts. How does Spark avoid duplicates?

A) Spark contacts Kafka to decommit the consumed offsets
B) The checkpoint stores committed offsets; on restart, Spark replays from the last committed offset, and the idempotent sink handles any duplicate writes
C) Spark uses distributed locks to prevent any re-processing
D) Duplicate data is unavoidable — Spark provides at-most-once guarantees
✅ Recovery: (1) Read commits/ to find last completed batch. (2) Read offsets/ to find the planned-but-not-committed batch. (3) Replay that batch from source. (4) Idempotent sink handles partial writes from the failed attempt. Spark does NOT interact with Kafka's consumer group offsets — it manages offsets independently in its own checkpoint.
