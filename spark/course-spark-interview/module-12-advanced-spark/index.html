Partitioning Strategy Deep Dive

Every shuffle operation — joins, aggregations, groupBy, distinct — produces output partitions. The default spark.sql.shuffle.partitions = 200 is almost never right for production. It's a compromise that breaks down at TB-scale: 200 partitions on 1 TB = 5 GB per partition = guaranteed OOM.

The industry-standard rule: ideal_partitions = total_shuffle_data_size ÷ 128 MB. Why 128 MB? It amortizes task scheduling overhead (~50–100 ms), fits in executor memory, and aligns with HDFS/cloud block sizes. Check "Shuffle Write" in the Spark UI to estimate shuffle data size.

Bucketing — The Hidden Join Optimization

Bucketing is a write-time optimization that pre-organizes data on disk by a key column. Spark hashes each row's bucket column, assigns it to one of N buckets (files), and optionally sorts within each bucket. The result: same-key data always lives in the same file.

Analogy: Imagine a mail room. Without bucketing, every morning you dump both filing cabinets on the floor and re-sort to match employee records with payroll (that's the shuffle). With bucketing, both cabinets are permanently organized by employee_id — joining is just opening matching drawers.

# STEP 1: Bucket both tables identically orders_df.write \ .bucketBy(256, "user_id") \ .sortBy("user_id") \ .saveAsTable("orders_bucketed") users_df.write \ .bucketBy(256, "user_id") \ .sortBy("user_id") \ .saveAsTable("users_bucketed") # STEP 2: Join — zero shuffle! result = spark.table("orders_bucketed") \ .join(spark.table("users_bucketed"), "user_id") # Verify: no Exchange node in plan result.explain()

File Formats — Parquet vs ORC vs Avro

Columnar (Parquet, ORC) stores values column-by-column: [Alice, Bob, Carol] [28, 35, 42] [NYC, LA, NYC]. Reading SELECT avg(age) on a 100-column table skips ~97% of I/O. Same-type values compress dramatically — a column of ages compresses far better than mixed-type rows.

Row-based (Avro, CSV, JSON) stores complete rows: [Alice, 28, NYC] [Bob, 35, LA]. Simple appends, low-latency writes — ideal for streaming and CDC. But reading one column means reading every column.

Parquet's internal structure: Row Groups (~128 MB) contain Column Chunks with min/max stats. If you filter WHERE age > 60 and a chunk's max is 45, Spark skips it entirely — that's data skipping, not just predicate pushdown.

📊 Analytics / OLAP

Parquet — columnar, Spark-native, great compression, ecosystem standard for data lakes.

📨 Kafka / Streaming

Avro — row-based, schema registry integration, compact binary, schema evolution champion.

🐝 Hive-Centric

ORC — Hive-optimized, bloom filters for high-cardinality lookups, built-in ACID.

🔄 CDC Pipelines

Avro → Parquet — Avro for transport (change events), Parquet + Delta/Iceberg for storage.

💡 Interview Insight

"Never say 'Parquet is always best.' The senior answer: Parquet for analytics reads, Avro for streaming transport, ORC for Hive-heavy environments. The key distinction is columnar vs row-based — analytics benefits from columnar (read only needed columns), streaming benefits from row-based (append a full record quickly). Modern data lakes often use Avro for ingestion and Parquet for the serving layer."

The Small Files Problem & Compaction

The small files problem is the most common production Spark issue. Imagine a directory with 5,000 files averaging 34 KB each. Each file requires separate metadata requests (HDFS NameNode call or S3 HEAD+GET). The driver lists all files and parses footers — millions of small files can OOM the driver before executors start. Poor compression, task overhead, and rate limits compound the damage.

📉 Coalesce Before Write

Calculate target files: output_size ÷ 256 MB. Cheap operation (no shuffle) when reducing partitions. Best for batch with known output size.

📏 maxRecordsPerFile

Spark splits output into files of ~N records each. Prevents large files but won't combine small ones. Best when output size is unknown.

🔧 Compaction Job

Read all small files → coalesce → write as fewer large files. Schedule as periodic maintenance. Best for fixing existing messes.

⚡ Delta OPTIMIZE

For Delta Lake: OPTIMIZE table auto-compacts. Pair with Z-ORDER for co-locating related data. Best for streaming sinks.

# Coalesce before write output_size_mb = 5000 # 5 GB output target_files = max(output_size_mb // 256, 1) df.coalesce(target_files).write.parquet(path) # maxRecordsPerFile — size-based splitting df.write \ .option("maxRecordsPerFile", 1_000_000) \ .parquet(path) # Smart partitionBy: pair with repartition df.repartition(100, "date") \ .write.partitionBy("date") \ .parquet(path) # DANGER: high-cardinality partitioning df.write.partitionBy("user_id").parquet(path) # 10M users = 10M directories! 💀

Dynamic Partition Pruning (DPP)

DPP solves a classic star-schema problem. Consider: SELECT sum(amount) FROM sales JOIN dim_date ON date_key WHERE year = 2024. Without DPP, Spark reads ALL 10 TB of sales — the filter is on the dimension table, not the fact table. With DPP (Spark 3.0+), Spark first resolves the filter to get 365 date_key values, then injects them as a runtime filter on the sales scan. 10 TB → ~300 GB. Potentially 30× faster.

-- Star-schema query with DPP SELECT sum(s.amount) FROM sales s JOIN dim_date d ON s.date_key = d.key WHERE d.year = 2024 -- DPP conceptually transforms to: SELECT sum(s.amount) FROM sales s WHERE s.date_key IN ( SELECT d.key FROM dim_date d WHERE d.year = 2024 ) JOIN dim_date d ON s.date_key = d.key WHERE d.year = 2024

"DPP is the runtime cousin of predicate pushdown. Static pushdown handles explicit filters like WHERE date = '2024-01-15'. Pandas UDFs & PySpark Performance

PySpark has an architectural bottleneck: Spark's engine runs on the JVM, but Python code runs in a separate process. Regular Python UDFs serialize data row-by-row via pickle between JVM and Python — for 100M rows, that's 100M serialize/deserialize round trips. Easily 10–100× slower than Scala.

Pandas UDFs fix this using Arrow — a columnar format both JVM and Python understand. Data transfers in batches of thousands of rows (not one at a time), and Python operates on vectorized pandas Series with NumPy operations. Result: 10–100× faster than regular UDFs.

# ❌ SLOW: Regular Python UDF (row-by-row pickle) from pyspark.sql.functions import udf from pyspark.sql.types import DoubleType @udf(returnType=DoubleType()) def slow_normalize(value, mean, std): return (value - mean) / std # ✅ FAST: Pandas UDF (Arrow-based vectorized) import pandas as pd from pyspark.sql.functions import pandas_udf @pandas_udf("double") def fast_normalize(s: pd.Series) -> pd.Series: return (s - s.mean()) / s.std() # ✅ Iterator pattern — load model once @pandas_udf("double") def predict(iter: Iterator[pd.Series]) -> Iterator[pd.Series]: model = load_heavy_model() # once! for batch in iter: yield pd.Series(model.predict(batch))

JVM-only, zero serialization. 300+ built-in functions: upper(), split(), when(). Always try this first.

🥈 Pandas UDFs

Arrow-based batches, vectorized pandas/NumPy. 10–100× faster than regular UDFs. Use when built-ins aren't enough.

💡 Interview Insight

"Three-level PySpark performance answer: (1) Use native Spark SQL functions — JVM-only, no serialization, always fastest. There are 300+ built-ins. (2) If you need Python logic, use Pandas UDFs — vectorized via Arrow, batches of rows, 10–100× faster than regular UDFs. (3) Never use plain Python UDFs or RDD .map() with lambdas in production — the row-by-row JVM↔Python pickle serialization kills performance. For ML inference, Iterator Pandas UDFs load the model once and score in batches."

Quiz — Test Your Mastery

Q1: Your job shuffles 50 GB. How many shuffle partitions should you target?

A) Keep the default 200 — it's fine for most jobs
B) Set to 50 — one partition per GB
C) Set to ~400 — using the 128 MB rule: 50,000 MB ÷ 128 MB ≈ 390, round to 400
D) Set to 2,000 — more partitions is always better
✅ The 128 MB rule gives 50,000 ÷ 128 ≈ 390 → round to 400. Default 200 = 250 MB partitions (survivable but large). 2,000 = 25 MB each — too small, scheduler overhead dominates. Option C balances parallelism and per-task efficiency.

Q2: Two large tables (100 GB + 80 GB) are joined daily on user_id. What eliminates the shuffle entirely?

A) Broadcast the smaller table
B) Cache both tables in memory
C) Bucket both tables by user_id with the same bucket count using saveAsTable()
D) Repartition both tables by user_id before each join
✅ At 80 GB, broadcasting is impossible (default limit 10 MB). Caching doesn't eliminate shuffle. Repartitioning still shuffles every run. Bucketing pre-partitions on disk — same column, same count, read via spark.table() → zero shuffle. It's a write-time investment for repeated read-time wins.

Q3: When would you choose Avro over Parquet?

A) For analytics dashboards with columnar aggregations
B) For Kafka message serialization with schema registry integration
C) When you need the best compression ratio for cold storage
D) For Spark SQL tables that are read frequently
✅ Avro is row-based — perfect for serializing individual records in streaming. It integrates with Schema Registry (4-byte schema ID per message), supports schema evolution, and has compact binary encoding. Parquet is superior for analytics (columnar reads), but Avro excels at write-heavy, record-level streaming workloads.

Q4: After one week, your streaming job has 500,000 files averaging 45 KB each. Problem and fix?

A) Normal for streaming — just add more executors
B) Switch from Parquet to CSV for smaller files
C) Increase spark.sql.shuffle.partitions to create more partitions
D) Small files problem. Fix: coalesce before write, maxRecordsPerFile, or Delta OPTIMIZE. Target 128 MB–1 GB per file.
✅ 500K files at 45 KB = severe small files problem. Each file open has metadata overhead, the driver may OOM on listing, and compression is poor. Coalesce in each micro-batch, use maxRecordsPerFile, or use Delta OPTIMIZE for automated compaction. Never increase shuffle partitions — that creates MORE files.

Q5: What is Dynamic Partition Pruning (DPP), and how does it differ from static predicate pushdown?

A) DPP pushes WHERE clause filters to storage — same as predicate pushdown
B) DPP dynamically adjusts the number of partitions during a shuffle
C) DPP derives filter values from a join's broadcast side at runtime to prune fact table partitions — pushdown handles explicit filters at planning time
D) DPP is another name for AQE's coalesce shuffle partitions
✅ Static pushdown handles explicit filters (WHERE date = '2024-01-15') known at planning time. DPP handles implicit filters derived from joins: filtering dim_date.year = 2024 resolves matching date_keys at runtime and prunes sales partitions. Requirements: fact partitioned by join key, dimension broadcastable, Spark 3.0+. They're complementary.

Q6: A PySpark job with regular Python UDFs is 50× slower than the Scala version. Why, and how do you fix it?

A) Python is inherently 50× slower than Scala — nothing you can do
B) Use cache() on the input DataFrame
C) Increase executor memory to compensate for Python's overhead
D) Row-by-row pickle serialization between JVM and Python. Replace with Pandas UDFs (Arrow-based, 10–100× faster) or native Spark SQL functions (zero serialization).
✅ The bottleneck isn't Python's speed — it's serialization overhead. Each row is pickled, sent via socket, unpickled, processed, repickled, sent back, and unpickled again. For 100M rows = 100M round trips. Pandas UDFs use Arrow columnar batches (thousands of rows, zero-copy). Native Spark SQL functions are best — entirely JVM, zero Python.
