What Is a Lakehouse?

For over a decade, data teams lived in a painful split reality. Data Lakes (Hadoop/S3) gave you cheap storage and any format — but zero governance, no ACID, and silent data corruption. Data Warehouses (Snowflake, Redshift, BigQuery) gave you structure and ACID — but at massive cost, vendor lock-in, and no support for ML workloads.

Most enterprises ran both — a lake for raw/ML data and a warehouse for BI — with fragile ETL pipelines copying data between them. Double storage cost, double governance headaches, stale data.

A Lakehouse puts warehouse-like capabilities directly on top of data lake storage. You keep the cheap, open, scalable storage but add ACID transactions, schema enforcement, time travel, indexing, and unified batch + streaming — all in open file formats.

🔍 Query Engine Layer

Spark · Trino · Presto · Flink · Dremio — the engines that run your queries. They read metadata from the table format layer to know what to scan.

📋 Table Format Layer

Delta Lake · Apache Iceberg · Apache Hudi — the magic layer. A metadata specification that manages transactions, schema, and statistics on top of open files.

💾 Storage Layer

S3 · ADLS · GCS · HDFS · MinIO — cheap, durable, scalable object storage. The actual Parquet/ORC files live here.

💡 Interview Insight

When asked "What is a Lakehouse?", say: "A lakehouse gives you warehouse-like reliability — ACID transactions, schema enforcement, time travel — on top of data lake economics. The key enabler is the table format layer (Delta, Iceberg, or Hudi) which adds transactional metadata on top of open file formats like Parquet stored on cheap object storage like S3."

Delta Lake Deep Dive

Delta Lake is an open-source table format created by Databricks. At its core, a Delta table is just Parquet files + a transaction log. That transaction log transforms a messy pile of Parquet files into a proper ACID-compliant table.

Every Delta table has a _delta_log/ directory — the single source of truth. Each JSON commit file records which Parquet files were added or removed. Every 10 commits, a checkpoint consolidates the state so readers start from the latest checkpoint.

# MERGE — the most interview-asked op from delta.tables import DeltaTable target = DeltaTable.forPath(spark, "/data/customers") source = spark.read.parquet("/data/customer_updates") target.alias("t").merge( source.alias("s"), "t.customer_id = s.customer_id" ).whenMatchedUpdate( condition="s.updated_at > t.updated_at", set={"name": "s.name", "email": "s.email"} ).whenNotMatchedInsert( values={"customer_id": "s.customer_id", "name": "s.name", "email": "s.email"} ).execute()

Under the hood, MERGE joins source against target, rewrites affected Parquet files entirely (Delta doesn't do in-place edits), and records which old files were removed and which new files were added in the commit log. The whole operation is atomic.

💡 Interview Insight

"How does Delta Lake handle concurrent writes?" — Say: "Delta uses optimistic concurrency control. Multiple writers proceed independently without locking. At commit time, Delta checks the transaction log for conflicting changes. If two writers modified the same partition, one gets a ConcurrentModificationException and can retry. This is more performant than pessimistic locking because conflicts are rare."

Time Travel & Schema Evolution

Because Delta's transaction log records every change as a versioned commit, you can read any historical version of your table. This is called time travel.

Schema enforcement rejects writes with mismatched schemas — no more silent downstream failures. Schema evolution lets you add columns with mergeSchema (additive, safe) or replace the schema entirely with overwriteSchema (breaking changes).

💡 Interview Insight

"How does time travel work?" — Say: "Each commit creates a new version by recording which Parquet files were added or removed. When you query version N, Delta reads the log up to commit N and determines which files composed the table at that point. No data is duplicated — unchanged files are shared across versions. The only constraint is that VACUUM can delete old files, so retention periods must align with time travel needs."

Optimization Operations

Streaming jobs and frequent appends create thousands of tiny Parquet files. This destroys read performance — each file requires a separate I/O operation, object store LIST operations are slow, and Spark creates one task per file with massive scheduling overhead.

📦 OPTIMIZE

Reads small files and rewrites them into larger, optimally-sized files (~1GB). Solves the small files problem. Old files remain for time travel until VACUUM.

🗂️ Z-ORDER BY

Co-locates related data within files by chosen columns. Enables data skipping — Spark reads file-level min/max stats and skips irrelevant files. Choose 2-4 high-cardinality columns you filter on frequently.

🧹 VACUUM

Deletes old Parquet files no longer referenced within the retention period (default: 7 days / 168 hours). After VACUUM, time travel beyond retention is impossible — those files are gone.

# Production maintenance playbook from delta.tables import DeltaTable table = DeltaTable.forPath(spark, "/data/events") # Step 1: Compact + cluster by key columns table.optimize().executeZOrderBy( "user_id", "event_date" ) # Step 2: Clean up old files table.vacuum(retentionHours=168) # Step 3: Update statistics spark.sql("ANALYZE TABLE events COMPUTE STATISTICS FOR ALL COLUMNS")

Apache Iceberg

Apache Iceberg is an open table format created at Netflix and donated to Apache. Like Delta, it brings ACID transactions to data lakes — but with a different architecture. Iceberg is a table format specification, not a query engine. Spark, Trino, Presto, Flink, Dremio, and Snowflake all support it natively.

Iceberg's killer feature is hidden partitioning. In traditional Hive-style partitioning, users MUST know and filter on partition columns explicitly or suffer full table scans. With Iceberg, partition transforms are defined in metadata — users write natural queries and the engine prunes automatically.

# Iceberg hidden partitioning spark.sql(""" CREATE TABLE my_catalog.db.events ( event_id BIGINT, user_id BIGINT, event_ts TIMESTAMP, payload STRING ) USING iceberg PARTITIONED BY ( days(event_ts), bucket(16, user_id) ) """) # Users query naturally — no partition knowledge needed! SELECT * FROM my_catalog.db.events WHERE event_ts BETWEEN '2024-01-15' AND '2024-01-16' AND user_id = 42; # Engine auto-prunes day-partitions AND user-buckets

Partition evolution lets you change partitioning strategy without rewriting data. Old data stays in the old layout, new data uses the new layout, and queries on both work seamlessly. Iceberg also tracks columns by unique IDs (not name/position), so renames and reorders are metadata-only — zero data rewriting.

💡 Interview Insight

"What makes Iceberg different from Delta?" — Say: "Hidden partitioning — users write natural queries without knowing partition structure, and the engine prunes automatically. Iceberg also supports partition evolution (change partitioning without rewriting data) and full schema evolution by tracking columns with unique IDs. Architecturally, Iceberg is engine-agnostic — used natively by Spark, Trino, Flink, Snowflake — whereas Delta has historically been Databricks-centric."

Delta vs Iceberg vs Hudi

All three solve the same core problem (ACID on data lakes), but each originated from different pain points. Hudi was built at Uber for high-frequency upserts. Its unique contribution is two storage types: Copy-on-Write (fast reads, slow writes) and Merge-on-Read (fast writes, reads merge delta logs on the fly).

🏢 Choose Delta Lake

Databricks-native shop. Tightest integration, best tooling, largest community, managed by Databricks.

🌐 Choose Iceberg

Multi-engine environment (Spark + Trino + Flink). Engine-agnostic design, no vendor lock-in. Native Snowflake support.

⚡ Choose Hudi

Streaming CDC with millions of upserts/sec. Merge-on-Read storage type optimized for high write throughput.

The formats are converging: Delta UniForm emits Iceberg-compatible metadata, Apache XTable translates between all three, and Snowflake/BigQuery/Redshift are adopting Iceberg natively. Understanding the differences is what separates senior from junior in interviews.

💡 Interview Insight

"Which table format would you choose?" — Never say one is universally better. Say: "It depends on the stack. Delta for Databricks shops — tightest integration. Iceberg for multi-engine environments — engine-agnostic with hidden partitioning. Hudi for streaming-heavy CDC — Merge-on-Read is hard to beat. The formats are converging via Delta UniForm and Apache XTable." This shows trade-off thinking.

Quiz — Test Your Lakehouse Knowledge

Q1: What problem does the Lakehouse architecture solve?

A) It replaces both data lakes and data warehouses with a new storage system
B) It combines data lake economics (cheap open storage) with warehouse capabilities (ACID, schema enforcement, governance)
C) It eliminates the need for ETL pipelines
D) It makes all queries run in real-time
✅ The Lakehouse doesn't replace storage systems — it adds a metadata/transaction layer (Delta, Iceberg, Hudi) on top of existing cheap object storage to provide ACID, schema enforcement, and time travel that previously required an expensive warehouse.

Q2: How does Delta Lake achieve ACID transactions?

A) By using a distributed lock manager that prevents concurrent writes
B) Through a transaction log (_delta_log/) that records every change as an ordered, atomic JSON commit
C) By writing data to a relational database first and syncing to Parquet
D) By using write-ahead logs in the Spark driver
✅ The _delta_log/ directory contains sequentially numbered JSON files recording which Parquet files were added or removed in each commit. Atomicity comes from writing the commit file only after all data files succeed. Checkpoints every 10 commits speed up log replay.

Q3: A bad ETL job corrupted your Delta table 3 hours ago. The table has a 7-day VACUUM retention. How do you fix it?

A) Delete the Parquet files manually and re-run the ETL
B) Use RESTORE TABLE to roll back to the pre-corruption state
C) Re-create the table from scratch using a backup
D) Use VACUUM to clean up the corrupted files
✅ Since the corruption is within the 7-day retention, all historical Parquet files still exist. RESTORE TABLE creates a new commit that reverts the file list to the pre-corruption state. No data is copied — it just changes which files the table references.

Q4: What is Iceberg's "hidden partitioning" and why does it matter?

A) Partitions are encrypted so users can't see the data
B) Partition transforms are defined in metadata — users query naturally without knowing the partition structure, and the engine prunes automatically
C) Partitions are hidden from the query optimizer to prevent biased plans
D) Data is stored without partitions but Iceberg simulates partitioning at read time
✅ In Hive-style partitioning, users must filter on partition columns explicitly or get full table scans. With Iceberg, transforms like days(event_ts) are defined at table creation. Users query WHERE event_ts > '2024-01-15' naturally, and Iceberg auto-prunes. This decouples physical layout from logical queries.

Q5: Your Delta table has 50,000 small files from streaming micro-batches. What's your maintenance plan?

A) Run VACUUM immediately to delete old files
B) Run OPTIMIZE (with optional ZORDER BY) to compact files, then VACUUM to clean up old files after the retention period
C) Increase the number of Spark partitions to spread the load
D) Switch to Iceberg to solve the problem
✅ OPTIMIZE compacts the 50,000 small files into fewer, optimally-sized files. ZORDER BY co-locates data for data skipping. VACUUM (after retention period) deletes old files. Going forward, enable autoOptimize.optimizeWrite and autoCompact to prevent recurrence.

Q6: When would you choose Hudi over Delta Lake or Iceberg?

A) When you need the best schema evolution support
B) When your primary workload is streaming CDC/upserts — Hudi's Merge-on-Read storage type is optimized for high write throughput
C) When you need to query from multiple engines like Trino and Flink
D) When you want the largest community support
✅ Hudi was built at Uber for high-frequency upsert workloads. Merge-on-Read writes updates as small delta log files (fast appends) instead of rewriting Parquet files. For schema evolution, Iceberg is strongest (A). For multi-engine (C), Iceberg wins. For community (D), Delta leads.
